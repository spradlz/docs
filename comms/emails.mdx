**Emails** is the email channel: a teammate, the AI agent (Myli), or a bulk campaign sends mail to a lead, and
the reply threads back on the lead. The governing rule is simple тАФ **the *from* identity decides everything**:
the sending alias, the rail (Gmail for humans, Resend for AI/bulk), the signature, and where replies come back.

## How an email flows

1. A send resolves the **from identity** тАФ a teammate (Gmail OAuth), Myli (Resend), or system/bulk (Resend).
2. That identity sets the **alias, signature, and reply-to** тАФ sent from one place, replies can come back to another.
3. Each recipient send is logged (`hl_email_send_log`); campaigns fan out one row per member.
4. Inbound replies thread back on the lead and fire a notification.

For the end-to-end architecture (inbound capture, one-writer logging, and how email fits alongside texts and
calls), see the comms subsystem map: **[CRM Comms тАФ Texts, Calls, iMessage](/guides/crm-comms)**.

| What you send | Goes out AS | Rail | Working today |
|---|---|---|---|

| **Home-Value Report** | `notifications@houseloop.ai` | Resend | тЬЕ |
| **Campaign / branded / home-search** (the "warmup pool") | `recommendations@` / `my-listing@` / `my-saved-home@` / `instant-updates@` (`@mail.houseloop.ai`) | Gmail pool | тЭМ **pool not OAuth-connected тАФ dormant by design** |

**The facts, confirmed live:**
- **The system / notifications sender is `notifications@houseloop.ai`** (SMS system line `913-800-4470`). Resolved
  from `hl_settings.notification_sender.email_from`.
- **Three team mailboxes are connected + signed** тАФ `ana@`, `myli@`, `zach@houseloop.ai` (`hl_oauth_connections`).
  A workflow set to any of them sends as that identity; a 1:1 composer send uses that person's real Gmail.
- **The identity resolver never returns a null sender** тАФ `resolveAgentSendIdentity` (`lib/agent-runtime/resolve-send-identity.ts`)
  resolves the assigned agent, else falls back to the system sender.
- **The warmup pool (`@mail.houseloop.ai` / `@email.houseloop.ai`) is deliberately NOT connected** (all 8 inboxes
  `oauth_connected=false`). Branded / marketing / home-search email does **not** send yet тАФ this is the intended
  "warming up, no mass sends yet" state, **not a bug.** Connecting the pool (or the DWD alternative) is a future
  decision, gated on being ready to send mass email.
- **Reality check:** in the trailing 30 days the only sends were 2 manual composer tests. Automated volume is
  effectively zero today.

## тнР Email Engine тАФ the one-sender redesign (Caleb, in progress)

<Info>
**Living status тАФ last updated 2026-07-15.** Texts have one writer (`logMessage`) тАФ one place the rules live.
Email did not: a three-agent code audit found the send path shattered. This section is the single record of the
redesign that makes email as clean as texts, and exactly what has shipped vs. what is still coming.
</Info>

### Why (the audit)

A file-line-grounded audit of every email path found the subsystem fragmented and, in places, broken:

| Finding | Detail |
|---|---|
| **2 transports, 4 senders** | Resend + Gmail, realized as four separate sender implementations. |
| **3 duplicate Gmail stacks** | The RFC-2822 builder + OAuth token refresh + `messages/send` fetch were copy-pasted in three routes. |
| **7 hand-rolled log writes** | Each feature wrote its own `INSERT INTO hl_email_send_log` with a different column set; one logged elsewhere, one logged nothing. |
| **3 merge engines** | `{{token}}` substitution done three different ways (one is inline regex). |
| **Live bugs** | Open/click tracking silently dead for Gmail-composed + template email; the Gmail marketing path skipped unsubscribe/footer; scheduled campaigns never send (no runner); the Home-Value Report was never logged (threw on a NOT NULL, swallowed). |

The strong parts are kept, not replaced: the send-identity resolver (`resolve-send-identity.ts`), the canonical
signature system, the branded shell/footer, and the lead-scoped composer from-picker.

### The model тАФ one of everything

Every email flows through one function тАФ the email analogue of `logMessage`. That function owns transport,
identity, merge, compliance, tracking, and logging, so a rule lives in exactly one place.

**Transport decision (locked, Hybrid B):** system + marketing send via **Resend** (full open/click tracking);
agent 1-to-1 sends via the agent's **real Gmail** mailbox, with our own open-pixel + click-wrapper so tracking
is transport-independent. This kills the "tracking only works on Resend" bug at the root.

### Where the code lives

- **The one writer:** `apps/houseloop-crm/lib/email/log-email-send.ts` (`logEmailSend`).
- **The one Gmail transport:** `apps/houseloop-crm/lib/email/gmail-send.ts` (`sendViaGmail`, `gmailRawSend`, `buildRfc2822`, `refreshHlOAuthToken`).
- **The Resend transport:** `apps/houseloop-crm/lib/email/resend-send.ts` (`sendViaResend` тАФ the only path with the undeliverable-address guard today; 1c extends the guard to Gmail too).
- **Identity (kept):** `apps/houseloop-crm/lib/agent-runtime/resolve-send-identity.ts`.

<Warning>
**1c is load-bearing** тАФ a bug in `sendEmail()` breaks *all* email at once, and it decides the transport-source
question deferred in 1b. It is built + verified as its own focused pass, with a real test send fired through each
migrated caller тАФ never rushed. Until 1c lands, the 8 callers still send directly but all log through the one
`logEmailSend`.
</Warning>
