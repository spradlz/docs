## 2 · How a call routes (`app/api/phone/webhooks/voice/route.ts`)

Webhook decision order, top to bottom:
1. **Caller-based forwarding rules** (`hl_phone_forward_rules`, `/settings/phone`) — an explicit human rule
   wins for ANY caller.
2. **Owned line** (platform=twilio): resolve the line → its inbox trigger seat → ring the routing targets /
   `forwards_to` chain. Owned lines NEVER hand off to the FUB front-router.
3. **Spam gate**: a repeat caller with a 180-day spam history and no lead relationship gets `<Reject>` —
   but the trigger event still fires (`routed_by='spam-gate'` — a gate is routing, not a signal eraser).
4. **FUB front-router fallback** (non-owned lines, non-HouseLoop callers) → `<Redirect>` to FUB's webhook.
5. **Per-dialer default** (`hl_dialer_assignments.call_forwarding`).

**The loop-guard (CONSTITUTIONAL — the 2026-07-14 call-forward loop autopsy), three belts in code:**
1. **Pool guard** — never `<Dial>` a number WE own (pool→pool re-entry is structurally impossible).
2. **Echo guard** — never dial the caller back.
3. **Hop breaker** — ≥4 webhook entries for the same caller→line pair in 60s = a loop through an external
   hop → dialing stops, voicemail hard-stop.

**Answered vs missed (dial-status, 2026-07-16):** `<Dial action>` asks the webhook what's next when a dial
ends — answered → clean `<Hangup/>` (a caller can NEVER fall through into the voicemail recorder after a
finished conversation); missed → the no-answer forward chain → voicemail. Handler:
`app/api/phone/webhooks/dial-status/route.ts`; shared TwiML fragments: `lib/phone/twiml.ts`.

**Mint-at-ring + caller context (2026-07-16, live-proven on an organic Zillow caller):**
- Every inbound call/text on an owned line **fires a trigger event** — lead-less included, stamped with the
  line's source key (the board's per-inbox counts read exactly this). Catalog + fire-pipeline law:
  [Trigger Events](/guides/trigger-events).
- Unknown caller + the line's **Creates-Lead** toggle ON → the lead is **minted DURING the ring**
  (`lib/leads/mint-phone-lead.ts` — ONE writer, shared with Myli's wrap-up; source owner/stage defaults,
  mig 0416 — a phone-minted lead is never ownerless). **Two cases:** a **truly-unknown** caller mints a bare
  "Unknown Caller" lead (voicemail triage, §4, later updates the name/callback or deletes it if spam); a caller
  the **FUB mirror knows** is a candidate for the **FUB pull-in** — the `creates_lead` behaviour that imports
  their FUB context + browsing (never their comms). See [Trigger Events → the FUB pull-in](/guides/trigger-events).
- **Caller NAME — the HouseLoop lead WINS (2026-07-17, the "Zach Test" autopsy).** When the call matches an
  existing HouseLoop lead, **that lead's own name is used**. The FUB people mirror (`callerContext`,
  `lib/phone/caller-context.ts`) is consulted **only as a fallback** — to name/pull a caller we don't have yet.
  A stale FUB contact sitting on one of OUR own numbers (there was a "Zach Test" contact on Zach's cell) must
  never override the real lead. **Twilio `ForwardedFrom`** still tells us WHICH past agent's old number was
  dialed. All of it rides the call row, the card, and the minted lead ("Sachind Jamesh — Alisha Hanks' old
  line", never "Unknown Caller").
- **Line NAME comes from the NAMED source, never a code map (2026-07-17).** The line a call reached resolves via
  `lineLabelForNumber` (called number → `hl_phone_numbers` → `lead_source_id` → `hl_lead_sources.name`, else the
  number's `label`) — rename a source and the card/notification follow it. The old hardcoded `CALL_LINE_LABEL`
  map is a fallback only for an unregistered number.
- **One call = one trigger event** — the lane that ANSWERS the number fires it. FUB's `callsCreated` echo
  of a call that landed on our Twilio is skipped; FUB-native lines (Zach's seat) fire via the FUB router
  into the inbox lane.

## 3 · Recording & AI analysis

**The settings surface** — CRM sidebar → Phone & Voice → **Voice & Recording** (`/settings/voice-recording`):

| Setting | Scope | Who can change | Persisted to | Default |
|---|---|---|---|---|
| My Call Recording Preference (`recordCalls`) | Per-user opt-out | Any signed-in user | `hl_users.data.recordCalls` | on |
| Enable Call Recording (`recordingEnabled`) | Team | Admin / owner only | `hl_settings` key `call_recording` | off |
| Conversational Intelligence AI (`aiAnalysisEnabled`) | Team | Admin / owner only | `hl_settings` key `call_recording` | off |

A call records when the team toggle is ON **and** the user has not opted out (`shouldRecordForUser`,
`lib/voice-recording.ts`). API: `app/api/settings/voice-recording/route.ts` (GET / PUT admin-only / PATCH
personal). Opt-out model mirrors SureSend.

**The capture + pipeline (LIVE — shipped and proven 2026-07-16):**
- **Voicemail** records via `<Record>`; **answered forwards** record via `record-from-answer-dual` on the
  `<Dial>` legs (dual-channel — caller on one channel, answerer on the other).
- The recording callback (`app/api/phone/webhooks/recording/route.ts`) disambiguates via Twilio's
  **`RecordingSource`**: `RecordVerb` = our voicemail (flips outcome, fires the voicemail pings/triage);
  `DialVerb` = an answered forward (outcome `answered`, no voicemail framing, analysis still runs).
- **Analysis** (`lib/calls/analyze.ts`, Inngest `call-analyze`): Deepgram transcription — **multichannel on
  dual recordings → `Caller:` / `Agent:` speaker labels** (mono voicemails use acoustic diarization) →
  Claude Haiku summary bullets + sentiment + suggested tasks → written to the `hl_calls` row.
- **The call card** (lead detail, All + Calls tabs — ONE `ActivityHeroCard`, never a parallel copy):
  headline + line context + "New lead created" pill + FUB link + player + summary + suggested tasks +
  collapsible transcript.
- **SMS delivery truth** (E1, 2026-07-16): outbound texts carry a `StatusCallback` →
  `app/api/phone/webhooks/sms-status/route.ts` stamps `hl_messages.status` forward-only
  (queued→sent→**delivered/undelivered/failed** + error code). The escalation ladder reads REAL delivery,
  never Twilio's initial "accepted" (the Renita lesson).

## 10 · Number porting — the port-out playbook

**The God-tier endgame:** every business line lives in **our own Twilio**, not FUB or Google Voice. Then inbound
SMS/voice hits a **Twilio webhook → HouseLoop native → the Myli workflow**, and outbound sends **AS the exact
line** the person dialed. No middleman, no forwarding shell game, no lost source. We already own **two Twilio
accounts** (HouseLoop + SRG, `support@spradlingrealtygroup.com`, vault / Master Sheet row 78).

### Port a Google Voice (Workspace) number → Twilio (the 913-954-7123 recipe, battle-tested 2026-07-09)

A **Google Workspace** Voice number (managed under our domain) ports differently from a consumer GV number — there's
no `voice.google.com/about/port` unlock screen. The credentials + the address all live in the **Admin Console**.

1. **Get the port-out credentials from Google Admin.** Admin Console → Apps → Google Workspace → **Google Voice →
   Number porting** → the **"Port-out info"** button (top-right of the Ports list). It shows the **US/Canada
   port-out PIN** (10 digits; the reveal eye/copy are trusted-click only — read it off-screen if automating).
   - **Account number** = the **Google Billing ID** from the invoice (e.g. `9620-6665-8983` → `962066658983`) — **NOT**
     the 10-digit phone number, despite what Google's generic help says.
   - **PIN** = that port-out PIN.  **BTN** = `+1<number>`.
2. **Submit the port-in via the Twilio Porting API — not the beta UI.** The "Port-in US Local (Beta)" wizard's
   eligibility step downloads a CSV via a blob link that **silently kills the request when driven by automation** (it
   never persists). Use the API (creds `TWILIO_ZACH_ACCOUNT_SID` / `TWILIO_ZACH_AUTH_TOKEN`):
   - Upload the billing statement: `POST https://numbers-upload.twilio.com/v1/Documents` (multipart `File`,
     `document_type=utility_bill`) → returns `RD…` doc SID.
   - Create the port-in: `POST https://numbers.twilio.com/v1/Porting/PortIn` — **must** include top-level
     `account_sid` + `target_port_in_date` + `notification_emails[]` + `losing_carrier_information{…}` +
     `phone_numbers[{phone_number, pin}]` + `documents[RD…]`.
   - Twilio auto-generates the **LOA** and emails it via **HelloSign** to `authorized_representative_email` for
     signature. On signature → `port_submitted`. Full runbook: memory `reference_google_voice_port_out_playbook`.
3. **Land it.** Point the number's **Messaging + Voice webhooks at HouseLoop** → native inbound; send AS it.

<Warning>
**⚠️ THE #1 REJECTION CAUSE — `INVALID_ADDRESS` (code 22154) + the 2-BUSINESS-DAY PROPAGATION TRAP.**
The port validates the service address against the **losing carrier's (Bandwidth) record**, which = the **Google
Voice "Service address"** (voice.google.com/settings → Service address, per-number), **NOT** the billing/invoice
address. Two attempts for Ana's number bounced `INVALID_ADDRESS`: first the stale Kansas address, then the *correct*
Florida one — because Google's own screen says **"Service addresses register within 2 business days."** When you
**change** the service address it renders instantly in the UI but takes **up to 2 business days to propagate to
Bandwidth**, and the port validates against the not-yet-updated carrier record. **So: fix the Google Voice service
address → WAIT ~2 business days → THEN (re)submit the port with that exact string.** `customer_name` + address must
match letter-for-letter. Twilio opens a Zendesk ticket per attempt but only echoes the `22154` code; reply-to-ticket
reaches Porting Ops if you need the exact CSR address.
</Warning>

<Warning>
**Secondary GV gotcha (VoIP direct-reject):** if a GV number rejects for reasons *other* than address, the
battle-tested fallback = a **two-step port**: GV → a cheap **prepaid mobile SIM** (Tello / US Mobile, ~$10) first,
let it settle a few days, **then** port that SIM → Twilio. Slower, near-100%. Try direct first.
</Warning>

<Warning>
**Two things this creates:**
1. **Ported numbers leave FUB.** Once 320-0906 / 354-8600 / 954-7123 are on Twilio, FUB no longer logs them —
   so each call is **mirrored back into FUB via the API** (`POST /v1/calls`, the `mirror_to_fub` step) to keep

   read as "someone called the legacy line" — so they resolve to ONE subtype (`phone_call_past_agent`). FUB
   still holds the exact number if per-number detail is ever needed. **Caller-ID pass-through** must be set on
   every forward, or you see the FUB inbox number instead of who actually called.
</Warning>

Twilio port-in. Confirm each line in **FUB Admin → Phone Numbers** first; each port is a real **cutover**, not a
