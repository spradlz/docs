## The root cause (one bug, three channels)

A text / call / email is glued to a lead by **phone- or email-string match**, then every thread is
scoped by **`lead_id` alone** and "cleaned" with subtractive band-aid filters.

- Every outbound text runs `sendSms` → `logMessage`. `logMessage` decides which lead a message belongs
  to by matching the **recipient's phone** against every lead (`resolveLeadByContact`,
  `lib/messages/log-message.ts:181`).
- Zach's cell (407-403-8088) **is** the phone on the Zach Spradling test lead. So **anything** texted to
  that number — a digest, a "saved a home" alert, a system report — phone-matches that lead and is
  written into its Texts thread. It reads "as Ana" because the send physically left Ana's 648-9363 line.
- The band-aids only *relabel the sender* (`actor='system'`) or *hide the row from the "All" feed*
  (`isAgentAlert`). They do **not** stop the row from being filed into the lead's Texts thread — and the
  digest carries **no tag at all**, so not one filter even looks at it.
- **Emails and calls have the identical flaw** — same string-match, same `lead_id`-only scoping. Texts is
  just the loudest channel, so it's where it shows first.

## PART 3 — One conversation-identity model · 🟡 THE REAL FIX (3.1 Texts SHIPPED)

<Info>
**3.1 — Texts: SHIPPED 2026-07-15 (commit `041c345816`).** `hl_messages` now carries `our_identity_user_id`
+ `conversation_key` (migration 0410). A real two-party text is keyed `text:{our_identity_user_id}:{lead_id}`;
a notification / digest / report / agent-alert / broadcast is keyed **NULL** and the three Texts readers now
require `conversation_key IS NOT NULL` — so a broadcast structurally cannot appear in a thread. The
website-activity digest + Myli/partner handoff briefs pass `kind:'notification'`; dispatcher alerts already
carry `recipient_role='agent'`. Verified on the Zach Spradling test lead: the leaked digest is gone, the real
conversation still renders. **Calls (3.2) + Emails (3.3) get the same treatment next.**
</Info>


Stop deriving threads with subtractive filters. Give every comms row a **persisted conversation key**, so
a message's placement is decided by *what it is*, never by whose phone it reached.

- **Write-side:** stamp `our_identity_user_id` (already computed via `resolveActingIdentity` /
  `resolveAgentSendIdentity` / `resolve-inbound-owner`) + a `conversation_key` of
  `(channel_family, our_identity, lead_id)` on `hl_messages` (`log-message.ts`), `hl_calls` (voice
  webhooks), and both email tables. A migration adds the columns + backfills from existing sender/recipient.
- **Read-side:** every thread reader scopes by the key, per lane — **delete** the band-aids
  (`recipient_role` not-equal `agent`, `source` not-equal `fub`, the Twilio-only handshake CTE). One
  reader per channel, one scoping rule — fixing the current 4-readers-4-rules divergence across
  `app/(crm)/leads/[id]/page.tsx`, `api/bb/thread`, `api/email/thread`, `api/calls/thread`.
- **Dual-lane by identity, not name-regex.** Split lanes on `our_identity_user_id` (agent line vs Myli
  line) — kill the `(AI)` name heuristics in `CenterLane.tsx` + `TextThreadStream.tsx`. Cover iMessage
  (`hl_imessage_assignments.bb_address`, not just dialer digits — the gap in the 2026-07-14 handshake).
- **Notifications never borrow a lead thread.** A notification / report / agent-alert sent to an agent's
  number gets its OWN key (or none) — so it *structurally cannot* reappear in a lead thread by
  recipient-phone match, with no hardcoded line exclusion. **This is the digest / "Lucy Hartman" fix.**
- **Finish Emails + Calls to the same model.** Calls use `lib/phone/resolve-subtype.ts` so a
  `company` / `sign_call` / `past_agent` call isn't slotted into the assigned-agent lane.
- **Lock the model** in `guides/crm-comms.mdx` so it can't drift back to `lead_id`-scoping.

**Verify (test lead):** overlay Texts / Calls / Emails — only the assigned-line↔lead handshake shows; the
digest and the 800-4470 notification are absent because they have **no key on this lead**, not because
they're filtered out.

## PART 5 — AI SEND GOVERNANCE + attribution truth · ✅ SHIPPED + LIVE-VERIFIED (Adam, 2026-07-22)

<Info>
**This is the AI-actor half of comms truth — it sits ON TOP of Caleb's conversation-identity model (Part 3).**
Part 3 answers *"which lead does a message belong to."* Part 5 answers *"is this AI actor even allowed to send,
and who really sent it."* Three production incidents (a clone texting a lead when clones were "off"; Myli
nudging a `mic=none` lead; a phantom "Myli texted" that never sent) traced to ONE disease: AI sends fired from
many paths that never checked assignment/mic, got mis-attributed, and were read/written through divergent
tables. Shipped commit `961979fd32`; verified live end-to-end (a clone text + a Myli text fired to the Zach
test lead's phone, correctly attributed). Full receipts: memory `project_houseloop_clone_unauthorized_send_faduma`.
</Info>

- **THE ONE GATE — `canAiActOnLead(actor, leadId)`** (`lib/ai/can-act.ts`). Every AI send/write path calls it
  FIRST. It enforces **THE ASSIGNMENT LAW** (an AI actor may write/send/be-attributed on a lead ONLY if it is
  the actor ASSIGNED to that lead), **THE MIC** (`ai_responder='none'` = silence, everywhere), and **THE
  MASTER**. It delegates to the ONE resolver `resolveActorStamp` → `resolveAgentSendIdentity` so "who may act"
  and "who a send goes out as" can never drift. Adopted at the fire-time choke point (`inngest-fn.ts`
  reviewBeforeFire — covers the esign nudge / floor / daily-home / clone dispatchers at once), the workflow
  `ai_message` node, `schedule-esign-nudge`, and the floor-backstop query.
- **NO JUNK WRITES — the phantom, killed at the source (Zach's law: never a filter to HIDE a row).** The
  BlueBubbles webhook (`app/api/bb/inbound/route.ts`) no longer writes an **empty-body outbound echo** to
  `hl_messages` at all (it used to land there and only the run-log MIRROR filtered it → the leads table read
  it as a phantom "Myli texted" on a lead Myli wasn't assigned to). Don't write junk; don't filter it after.
- **⚠ THE 4th TEXTS READER Part 3.1 MISSED — `lib/leads-query.ts`.** Part 3.1 scoped the three thread readers
  by `conversation_key IS NOT NULL`, but the **leads-table** reader (`leads-query.ts`) is a fourth reader it
  never listed — it read `hl_messages` raw, so a broadcast/echo still surfaced there. Now every leads-query
  read (display columns, "Agent Action", the texted/not-texted filter predicates, AND the sort keys) applies
  the identical `conversation_key IS NOT NULL` scope — one canonical scope, no per-surface divergence.
- **CORRECT ATTRIBUTION.** A clone/Myli text is **never** logged as `human`: the workflow `ai_message` node now
  passes `actorRole='ai_agent'` through `sendTextAs` (the Twilio branch used to default to `human`, hiding that
  it was AI + skipping the owner ping). The AI-composed **email** now writes the canonical `logActivity('email_sent')`
  on the LIVE path (was backfill-only). Facts + the dossier refresh stamp the real actor via `resolveActorStamp`
  (see [AI Cockpit](/guides/ai-cockpit) actor-attribution).
- **LANE PURITY.** `hl_workflow_runs` carries `actor_type` (mig 0440), stamped at run open, so a clone-run
  workflow can never surface on Myli's sacred RIGHT lane (the `ai-activity` API filters by lane).

**Verify (done 2026-07-22, live):** master ON via the cockpit → fire the FUB tour workflow on the test lead with
`mic=clone` → the clone text delivers to the phone, `actor_role=ai_agent`, run `actor_type=clone`. Flip
`mic=myli` → the Myli text sends from Myli's line, `actor=Myli`, run `actor_type=myli`. With the master OFF,
neither fires. (Found in the same pass: the FUB tour first-touch **compose** puts the wrong name in the body —
"This is Ana" from Myli — a vibe-coded CONTENT bug, separate from this governance layer, queued for the workflow redo.)
