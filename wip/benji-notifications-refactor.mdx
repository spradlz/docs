#### GT — HOUSELOOP NOTIFICATIONS REFACTOR (PRD)
Owner: **Benji** (build) · **Lee** (spec + review) · Zach LOCKED 2026-07-01 · Status: PENDING → building
Indexed in: `GT_HOUSELOOP_MASTER.md` §6. Design source (Zach-signed): `.claude/state/notification-classification-audit.md`
+ `.claude/state/benji-notifications-handoff.md`. Build plan: `.claude/plans/encapsulated-munching-truffle.md`.

##### THE LAW
`/settings/notifications` is **catalog-driven** and organized into **7 stable sections**. The engine is THE
INVERSION: every trigger event in `hl_trigger_events` carries a **`notify_class`** — many events → ONE class.
The notification list is the DISTINCT set of classes, so **the catalog can grow (new sources, new events) and
the notification list stays STABLE**; a new source just folds its events into existing classes. The fire-time
message injects the source: **"&#123;class&#125; via &#123;source&#125;"** ("Tour Requested via Follow Up Boss").

##### WHY
The old page hardcoded a `SECTIONS` list (`page.tsx:36`) that drifts every time we add a lead source or event,
and there was NO HouseLoop-native notification control for the trigger catalog (the RealLoop→HouseLoop cutover
had nowhere to live). This makes notifications a first-class, self-maintaining surface + the home of the
RealLoop→HouseLoop switch, and adds two surfaces Zach wants: "notify me when my AI does X" and a site-wide
traffic digest.

##### THE 7 SECTIONS (Zach-signed)
1. **Lead Activity** (catalog-derived `notify_class`): New Lead · Tour · Contact · Like · Save · Share · Saved
   Search · Property Viewed (default OFF) · Seller Request · Home Value Report.
2. **Communications**: New inbound message · **Email received (new)** · **Phone call received (new)**.
3. **Lead Lifecycle**: Assigned · Stage Changed · Collaborator · Handoff/Ack · Clone stepped in · Reassigned ·
   Appointment scheduled · **Appointment completed (new)**.
4. **🤖 AI Follow-Up (new)**: 6 groups / ~15 toggles — Calls · Messages · Facts · Homes · Stages · Appointments,
   from the AI slice of the activity log.
5. **🌐 Website Activity (new)**: a site-wide DIGEST (active users · registrations · homes viewed · new leads ·
   searches · high-intent), daily + optional live threshold. Separate engine.
6. **Team**: Invited to join a team.  7. **System**: health alerts.

##### ARCHITECTURE (reuse — do not rebuild)
- **Prefs** = `hl_users.data.notification_prefs` JSONB `{[key]:{email,sms}}` via `/api/notification-preferences`.
  Classes + AI toggles are new KEYS; **no new prefs table.**
- **Dispatch** (unchanged, keyed by the same string): `lib/notifications/dispatch.ts` `dispatchNotification(key)`
  → prefs → `template-registry.ts` → `hl_templates` → `sendViaResend()`/`sendSms()` → `hl_notification` +
  `hl_run_log`. Fired by `createNotification()` (`lib/notifications/create.ts`) from `lead-events.ts` /
  `website-activity.ts` / `fub-lead-import.ts`.
- **RealLoop→HouseLoop** = `hl_fub_webhook_settings.notify_owner` (mig 0270).
- **AI Follow-Up taxonomy** already exists: `components/ai-follow-up/constants.ts` (kinds proven live via
  `/api/ai-follow-up/stats` over `hl_lead_activity WHERE actor_role='ai_agent'`).
- **Digest** clones the saved-search pattern (web cron → `houseloop/*.digest-due` → CRM Inngest fn → workflow →
  Resend — INNGEST_OR_IT_DOESNT_EXIST). Metrics cheap: `hl_pixel_event` (bot-filtered) + `hl_leads` + GA4.

##### BUILD PHASES
- **P0 (DONE):** migration `0277` `notify_class` (24 events → 12 classes) · `getNotificationClasses()` ·
  `/api/notifications/classes`.
- **P1:** `page.tsx` → 7-section shell; Section 1 renders from `/api/notifications/classes`; other sections
  re-grouped from existing keys. Prefs unchanged.
- **P2:** new per-event classes + `NOTIFICATION_TEMPLATES` + `PREF_DEFAULTS` + seed `hl_templates` migration +
  fire from triggers.
- **P3:** "&#123;class&#125; via &#123;source&#125;" — inject source into the `{{notify.*}}` merge scope in `dispatch.ts`.
- **P4:** AI Follow-Up section — `ai_*` toggles + an activity-log notifier gated by the toggle.
- **P5:** Website Activity digest engine (web cron → CRM fn → workflow → email), per-metric toggles + threshold.
