#### GT — TRIGGER EVENTS: THE ONE CANONICAL CATALOG (PRD)
Owner: Benji · Zach-signed 2026-07-01 (terminology + DB-table decision locked) · Status: PENDING → build

##### THE LAW
There is **ONE canonical table of Trigger Events: `hl_trigger_events`**. Every trigger in the system is a
row here. **Everything derives from it — no second list, no patchwork.** A "Trigger Event" = a trigger
(it starts a workflow) that fires on an event (a lead did something). One word across the whole app.

##### WHY (the drift this kills)
Today the same trigger info is duplicated across **4 code lists + 3 DB tables**, hand-synced — which is
why adding a trigger under Lead Sources / System Events did NOT flow into the workflow builder:
- `WF_TRIGGERS` (`lib/workflows/registry.ts`) — the builder palette (hardcoded, drifts)
- `CANONICAL_LEAD_EVENTS` (`components/settings/lead-sources/lead-source-event-constants.ts`) — event defs
- `EVENT_TITLES` (`components/settings/lead-flow/lead-flow-shared.tsx`) — duplicate labels
- `EVENT_TRIGGER` / `DOOR_EVENT_KEYS` — event→trigger map + Inngest door registration
- DB `hl_lead_source_events` (per-source wiring, incl. `workflow_id`), `hl_lead_sources`, `hl_fub_webhook_settings`

##### THE MODEL (Zach-locked)
Every Trigger Event has a **CATEGORY** and (for registration) a **LEAD SOURCE**:
- **Registration** → grouped by Lead Source (HouseLoop.ai · Follow Up Boss · Manual · future: Google PPC →
  its own landing-page trigger events, e.g. "$400K homes in Overland Park" = a trigger event under that source)
- **System** → the "System Events" page (existing-lead actions)
- **Follow-up** → future
- **Other** → future / TBD

UI keeps friendly labels ("Registration Events", "System Events") — the DATA unit is always a Trigger Event.

##### SCHEMA — `hl_trigger_events`
```sql
CREATE TABLE hl_trigger_events (
  id            uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  type          text UNIQUE NOT NULL,        -- the trigger_type (workflows.trigger_type joins on this)
  category      text NOT NULL,               -- 'registration' | 'system' | 'follow_up' | 'other'
  lead_source_key text,                       -- hl_lead_sources.key (registration only); NULL otherwise
  label         text NOT NULL,               -- display label
  description   text,                         -- "Start when …"
  icon          text,                         -- lucide icon name
  platforms     text[] NOT NULL DEFAULT '{houseloop}',
  fire_endpoint text,                          -- dev-facing "triggered when" web route
  sort_order    int NOT NULL DEFAULT 100,
  is_active     boolean NOT NULL DEFAULT true,
  is_system     boolean NOT NULL DEFAULT true, -- system-seeded vs UI-created (landing-page triggers)
  created_at    timestamptz NOT NULL DEFAULT now(),
  updated_at    timestamptz NOT NULL DEFAULT now()
);
```
Seed from the union of today's registration + system triggers (WF_TRIGGERS registration+system ∪
CANONICAL_LEAD_EVENTS ∪ EVENT_TRIGGER) — dedupe by `type`, fix the 2 naming bugs (below).

##### WHAT DERIVES FROM IT (all reads, zero duplication)
- **Lead Sources page** = `WHERE category='registration'` grouped by `lead_source_key`.
- **System Events page** = `WHERE category='system'`.
- **Workflow builder palette** = ALL rows grouped by `category` + platform; **plus** the code-only
  INTERNAL controls (stage_changed / tag_added / appointment_completed / manual_enrollment — NOT lead
  trigger events, they stay in code and are appended).
- **Workflows LIST page** (P3) = every Trigger Event LEFT JOIN `hl_workflows` ON `type=trigger_type`;
  built → the workflow; unbuilt → **"Coming Soon"**. Grouped by category / lead source.
- **event→trigger + titles** = derived (kill `EVENT_TRIGGER` static map + `EVENT_TITLES`).
- **Inngest door registration** = `DOOR_EVENT_KEYS` derives from `SELECT type FROM hl_trigger_events WHERE is_active`.
  (Firing note: Inngest fns are code-registered at boot, so a genuinely-new `type` still needs a deploy to
  get its door; a GENERIC door for UI-created trigger events is a fast-follow so landing-page triggers fire
  with no deploy.)

##### KNOWN BUGS TO FIX IN THE SEED (readers found these)
- registry `seller_request_registration` ≠ event-map `home_value_report_registration` — reconcile to ONE.
- registry `seller_started_registration` ≠ event-map `seller_get_started_registration` — reconcile to ONE.
  **Before renaming any `type`, verify NO live `hl_workflows.trigger_type` uses the old string** (else re-map it).
- The 3 saved-search trigger events (`saved_search` client · `saved_search_agent` · `saved_search_system`)
  live under System, each its own trigger, each attachable to its own workflow.

##### BUILD PHASES

- **P1** repoint the two settings pages + the builder palette to READ the table.
- **P2** derive `WF_TRIGGERS` / `EVENT_TITLES` / `EVENT_TRIGGER` / `DOOR_EVENT_KEYS` from the table (delete the
  hardcoded lists; keep INTERNAL controls in code, appended).
- **P3** Workflows LIST page = every Trigger Event as a row, "Coming Soon" when unbuilt, grouped by source.
- **P4** builder Lead-Source ↔ System-Event toggle (from `category`).
- **P5** generic Inngest door so UI-created trigger events fire without a deploy (fast-follow).
