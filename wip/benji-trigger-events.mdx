**North star (Zach-signed 2026-07-01):** ONE canonical DB table `hl_trigger_events` is the single source
of truth for every trigger in the system. Everything derives from it — the Lead Sources page, the System
Events page, the workflow builder palette, the Workflows list — no second list, no patchwork. A "Trigger
Event" = a trigger (starts a workflow) that fires on an event (a lead did something); one word everywhere.

**Why:** today the same trigger info is duplicated across 4 code lists (`WF_TRIGGERS`, `CANONICAL_LEAD_EVENTS`,
`EVENT_TITLES`, `EVENT_TRIGGER`) + 3 DB tables, hand-synced — which is why a trigger added under Lead Sources
/ System Events never flowed into the workflow builder. Researched with 3 sub-agents; full map in
`.claude/checkpoints/workflow-trigger-catalog.md`.

**Model:** each trigger event = `type · category (registration/system/follow_up/other) · lead_source_key ·
label · description · icon`. Registration triggers group by Lead Source (HouseLoop.ai · FUB · Manual · future
Google PPC → its own landing-page trigger events). Every trigger event gets a matching workflow ("Coming Soon"
if unbuilt).
