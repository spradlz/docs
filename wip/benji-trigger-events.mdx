### Background & model (rolled in — the original reconciliation)

**North star (Zach-signed 2026-07-01):** ONE canonical DB table `hl_trigger_events` is the single source of truth for
every trigger. Everything derives from it (Lead Sources page, Trigger Events page, workflow builder palette, Workflows
list) — no second list. A "Trigger Event" = a trigger (starts a workflow) that fires on an event a lead did.

**Why:** the same trigger info was duplicated across 4 code lists (`WF_TRIGGERS`, `CANONICAL_LEAD_EVENTS`,
`EVENT_TITLES`, `EVENT_TRIGGER`) + 3 DB tables, hand-synced — so a trigger added under Lead Sources never flowed into
the builder. Full map: `.claude/checkpoints/workflow-trigger-catalog.md`.

**Attribution (PRD §3):** every lead = **Platform** (the system it arrived through — Follow Up Boss / houseloop.ai / any)
› **Lead Source** (the finer origin within the platform — Zillow Preferred / houseloop.ai Organic) › **Tags** (page /
campaign). Shown on lead detail; grouped under platform on the scoreboard. `platform=any` is a first-class scope.

**Config (PRD §9):** two axes — Platform = the tab (`houseloop`/`fub`/`any`); Category = the group
(registration/system/follow-up). Per-platform config = two toggles (Creates-Lead, Workflow-Active) + FUB's Process.
Workflows bind to **(trigger × platform-scope)**; `any` = agnostic. Notification = read-only `notify_class` rolling up
to the Notifications section. **The event always fires — toggles gate the lead + the workflow, never the firing.**

##### 1. Context / problem
Under **Settings → Pipeline & Workflow** the trigger/event config is spread across **4 redundant routes**
(`/settings/lead-sources`, `/settings/lead-platform`, `/settings/lead-platform/configure/[id]` + `/settings/fub`,
`/settings/trigger-events`). The same event rows appear in 2+ places with *different* columns, so you can't tell in
one view "does this create a lead / is its workflow on / what notification does it fire." Zach's model: **Platform is
the spine.** Each platform owns its trigger events + workflows. A trigger event **always fires** once it exists — the
only real knobs are *does it create a lead* and *is its workflow active*. Notifications are **classified, not toggled**
here (they live in the separate Notifications section). Goal: **two screens** — a Lead Sources KPI scoreboard, and a
platform-tabbed Trigger Events config — with one enriched row shape, plus a platform-scoped workflow builder.

##### 2. The model (already ~80% built — REUSE, don't reinvent)
Two layers already exist:
1. **Catalog = `hl_trigger_events`** (mig 0276) — source-agnostic master list of event *types*: `type`, `category`
   (registration / system / follow-up), `platforms` (array), `label`, `description`, `icon`, `fire_endpoint`,
   **`notify_class`** (mig 0277, many-events→one-notification), `is_active`, `is_system`.
2. **Per-(platform × event) config** — native doors in **`hl_lead_source_events`** (`is_active`, `workflow_id`);
   FUB in **`hl_fub_webhook_settings`** (`enabled`, `creates_lead`, `notify`, `notify_owner` realloop|houseloop,
   `workflow_id`). These carry the toggles.

**The one rule:** an event's **`category`** decides *grouping* + whether Creates-Lead is offered. **Platform** decides
which *tab* it appears under. The UI reads the catalog as the spine and joins the per-platform config — nothing dupes.

##### 3. Attribution model — Platform &gt; Lead Source &gt; Tags (RESTORE on lead detail)
Every lead carries **three levels of attribution** (Benji had this on the lead-detail page and REMOVED it — restore):
1. **Platform** — the system it arrived through: `Follow Up Boss` | `houseloop.ai` (| `any`). Cols exist:
   `hl_leads.platform`, `hl_lead_activity.platform` + `__platform_houseloop__` / `__platform_fub__` pseudo-sources
   (mig 0247). *Later platforms = Sierra Interactive, other CRMs — each a new platform.*
2. **Lead Source** — the finer source *within* the platform: FUB → `Zillow Preferred`; houseloop.ai →
   `houseloop.ai Organic`; later → `houseloop.ai/ppc`. Lead source lives UNDER a platform.
3. **Tags** — the third rich layer: which page/campaign it came from (existing tag system).

These are **lead attributes** (shown on lead detail; grouped under platform on the scoreboard). They do NOT change
which trigger events exist. ~~`platform = any` is a first-class scope~~ — **superseded by the wireframe lock: there
is no `any` platform.** Platform is exactly `houseloop` or `follow_up_boss`. Manual entry uses platform `houseloop`
with a real source + the §5c fire-gate. Behavioral/aggregate triggers (§6) remain a **future note only**, not a tab
or a scope you can pick today.

##### 6. Workflow builder — Trigger Event × Platform scope
A workflow's **Trigger node = two selectors**: (1) **Trigger Event** (the canonical concept — platform-agnostic in
`hl_trigger_events`); (2) **Platform scope** — `houseloop` | `fub` | **`any`**. Bind to **(trigger, platform-scope)**,
never to a platform-welded trigger. **Fire rule:** an event carries its `platform` (from the lead); the router runs
every workflow where `trigger matches` **AND** (`workflow.platform == event.platform` **OR** `workflow.platform ==
'any'`). `any` = wildcard; specific + `any` coexist (both fire). **Notifications are agnostic by default** (roll up via
`notify_class` on the canonical trigger) — pick `platform=any` on a workflow only for an agnostic *action/nurture*,
never a notification. **Build now:** add `platform_scope` to the workflow trigger binding (default = lead's platform;
`any` selectable). Don't clone triggers per platform.

###### Behavioral / aggregate triggers (FUTURE engine — NOTE ONLY, do not build)
Zach's scaling case: **"viewed a home 10× in 3 days"** as `platform=any`. A distinct trigger *kind* — a
**count-over-window on the NORMALIZED activity stream**. Works only if raw per-platform events (Zillow view via FUB, a
HouseLoop web view) are **canonicalized to ONE activity type** (`property_view`) BEFORE counting, so the count **sums
across platforms** at scope=`any` (and filters to one at scope=`fub`/`houseloop`). Shape: `(canonical activity, count
≥ N, window = Xd, platform-scope)`. Needs an aggregate/behavioral engine over `hl_lead_activity` that doesn't exist
yet. **Note in catalog + builder so Trigger×Platform-scope stays forward-compatible; build no aggregate trigger now.**
