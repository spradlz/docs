### Background & model (Flow 1 — the original reconciliation, rolled in)
**North star (Zach 2026-07-01):** ONE canonical DB table `hl_trigger_events` is the SSOT for every trigger — Lead Sources
page, Trigger Events page, workflow builder palette, Workflows list all derive from it. A "Trigger Event" = a trigger
(starts a workflow) that fires on an event a lead did. **Why:** the same trigger info was duplicated across 4 code lists
+ 3 DB tables, hand-synced, so a trigger added under Lead Sources never flowed into the builder.
**Attribution (PRD §3):** every lead = **Platform** › **Lead Source** › **Tags**.
**Config (PRD §9):** Platform = the tab; Category = the group (registration/system/follow-up). Per-platform config = two
toggles (Creates-Lead, Workflow-Active) + FUB's Process. Workflows bind to (trigger × platform-scope). **The event always
fires — toggles gate the lead + the workflow, never the firing.**

##### 6. Workflow builder — Trigger Event × Platform scope
A workflow's **Trigger node = two selectors**: (1) **Trigger Event** (the canonical concept — platform-agnostic in
`hl_trigger_events`); (2) **Platform scope** — `houseloop` | `fub` | **`any`**. Bind to **(trigger, platform-scope)**,
never to a platform-welded trigger. **Fire rule:** an event carries its `platform` (from the lead); the router runs
every workflow where `trigger matches` **AND** (`workflow.platform == event.platform` **OR** `workflow.platform ==
'any'`). `any` = wildcard; specific + `any` coexist (both fire). **Notifications are agnostic by default** (roll up via
`notify_class` on the canonical trigger) — pick `platform=any` on a workflow only for an agnostic *action/nurture*,
never a notification. **Build now:** add `platform_scope` to the workflow trigger binding (default = lead's platform;
`any` selectable). Don't clone triggers per platform.
