##### 1. THE PROBLEM
HouseLoop is the website AND the CRM in one product — and Follow Up Boss is a legacy CRM piping in leads with
history. "Lead source" alone can't express that. We spent a day circling because two things were tangled:
**where a lead came from** vs **what it does on our site**. This PRD untangles it.

##### 2. WHAT THE RESEARCH SAID (5 CRMs — HubSpot, Salesforce, UTM/GA4, FUB/kvCORE/Sierra, Pipedrive/GHL/AC)
Unanimous findings (full briefs in session history):
- **Two fields, never one:** a TRUE ORIGIN (Zillow) kept separate from the CONNECTOR (Follow Up Boss). Origin
  is first-touch + immutable. (HubSpot Original vs Record Source; Salesforce LeadSource vs Source System.)
- **Events ≠ source.** Activity is a separate growing stream; source is a stamp set once. (We already do this:
  `hl_lead_activity`.)
- **Trigger on the EVENT, filter on SOURCE/PLATFORM.** Never one workflow per source. Merge fields fill the
  words. (Every CRM. Sierra/GHL/AC explicit.)
- **No "type" field** for origin — "type" = the event (already our catalog) or the pipeline role (already our
  Buyer/Seller pipelines). Don't add it.
- **Don't prefix `fub-zillow.com`** — keep source = the real origin (Zillow), connector separate (live FUB data
  with 48k people confirmed this). Platform = the connector/world.

##### 3. THE MODEL (LOCKED)

###### 3.1 The keystone — TWO buckets, TWO routing keys
| Bucket | Keyed on | Meaning |
|---|---|---|
| **New Lead Registration** | the lead's **ORIGIN** (source + platform) | "how they came in." realtor.com/Zillow via Follow Up Boss → that source's registration (the reactivation). Origin is **set once, permanent** — a merge field forever, but stops STEERING after registration. |
| **(HouseLoop) System Events** | the **SURFACE** (where the action fired) | "what they do on HouseLoop." ANY lead — regardless of origin — who likes a home / requests a tour on HouseLoop.ai fires the **HouseLoop** workflow. |
| **Follow-Up** | schedule | the nurtures |

**Proof (Zach's, universal):** lead acquired platform=Follow Up Boss, source=Zillow.com → registration runs
**Reactivation** ("you worked with Alicia… homes on HouseLoop.ai"). Origin stays Zillow forever. They then like
a home **on HouseLoop.ai** → SURFACE = HouseLoop → runs the **HouseLoop liked-home workflow**, not the FUB one.

###### 3.2 The cascade (first match wins)
```
A lead's event fires →
  1. SPECIFIC workflow for (this source + this event)?    → run it
  2. else a workflow for (this platform)?                 → run it   (e.g. FUB → Reactivation)
  3. else GENERAL REGISTRATION (the catch-all floor)      → always runs ({{lead_source}} fills the words)
```
Already engine-true: `dispatchDoor` routes existing-lead events by event SURFACE and keeps registration
ORIGIN-scoped (built P3). New sources work instantly off the floor; specials are opt-in.

###### 3.3 Scale = MERGE FIELDS
One event → one workflow → `{{first_name}}` `{{home_address}}` `{{lead_source}}` `{{platform}}` fill the rest.
Never a workflow per source. Adding a source = a new ROW, not a rebuild.
