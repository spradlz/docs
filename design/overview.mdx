## The one sentence

Every CRM surface is assembled from a **closed set of components**. Each block is a **brand-blue header (title + description)** welded to **one body** — rows, a table, a table with a menu, KPIs, or nav-cards. **Nothing floats** on the bare page, **nothing is a hand-rolled card**, there are **no raw hex or pixel values**, and **green means the SMS channel (or a real success state) only**. The linter — not a person remembering — keeps it that way.

<Note>
This is the human-readable companion to the constitutional rule `.claude/rules/UI_DESIGN_DOCTRINE.md`. The rule is the law; this tab is the reference. They stay in sync.

## The five body types

A block is exactly **one** of these. If what you need is not one of the five, **stop and ask Zach** — do not invent a sixth.

| # | Body type | Built with | Use it for |
|---|---|---|---|
| 1 | **Rows** | `SettingSection` + `SettingRow` / `SettingGroup` | Key → control settings (toggles, values, buttons) |
| 2 | **Table** | `SettingsTable` | A list with a blue header + derived count |
| 3 | **Table + menu** | `MenuTable` | A table driven by tabs / filters / a date range |
| 4 | **KPIs** | `StatStrip` under a blue header | A row of metric cards |
| 5 | **Nav-cards** | `NavCardGrid` | A launcher page (home, marketplace) — **nav only, never config** |

Every one of the five carries the **same brand-blue header**. The only thing that changes is the body.

## The closed component set

Build with these. If you need something else, grep for it; if it still does not exist, **stop and ask Zach** — never hand-roll a lookalike (`REUSE_NOT_RECREATE`).

| Primitive | Component | It is |
|---|---|---|
| Page frame | `PageShell` / `PageHeader` / `PageContent` | The page contract. The page top is the **only** navy strip. |
| Section | `SettingSection` (+ `SettingGroup`, `SettingRow`) | Blue header welded to a body of rows. Never a nested card. |
| Table | `SettingsTable` | The same blue header welded to a full-width `DataTable`. Count = row count. |
| Menu-table | `MenuTable` | Blue header holding title + count + tabs + filters, welded to a table. The canonical menu. |
| Data table | `DataTable` / `HlDataTable` | The grid. Group rows are a light **blue-tint** row (never navy). |
| Header bar | `SectionBand` | The shared brand-blue header both Section and Table render. |
| Stat strip | `StatStrip` | KPI cards, one normalized color, inside a blue-header section. Never a rainbow, never floating. |
| Nav card | `NavCard` / `NavCardGrid` | The one sanctioned floating tile — **launchers only**. |
| Filter pills | `HlFilterPills` | The single-select filter bar. Do not re-implement it inline. |
| Buttons | `.hl-btn` (6 variants) | No raw `<button style>`. |
| Pills / toggles | `.hl-pill*`, `.hl-status-pill*`, `.hl-toggle*` | No hand-rolled pills or toggles. |

## The gate — answer before the first line

1. **Which of the five body types is this block?** Name it.
2. **What existing component renders it?** Name the file. If none, grep. If still none, stop and ask.
3. **Am I about to write a raw card, a floating header, a nested card, a raw button, or a hand-rolled pill?** If yes, stop — use the component.
4. **Is there a count?** It comes from the row count in the header. Never hand-typed.
5. **Any raw hex / pixel / color?** Replace it with a token.

## Why this exists

A full 9-slice, 112-page audit (2026-07-11) found the CRM had drifted: ~54 pages with hand-rolled card sections, floating tab strips (only Run Log used the real menu component), rainbow KPI cards (no shared `StatStrip` existed), 87 files with raw hex, 134 raw `<button style>`, and green misused everywhere. The root cause was that there was no single answer to "what is a UI block," and the two linter gates that would catch pills and buttons did not exist. This doctrine makes the block the first decision, the closed set the only way to build one, a new type a Zach-approved event, and the linter the thing that stops the drift.
