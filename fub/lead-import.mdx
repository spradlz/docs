# 1. Overview & the one-source map 🔒

There is **one source** (FUB, fed by Zillow). Its signals arrive over **one real-time trigger** (the native
webhook), with the **scraper as a recovery/history backstop**, land in **one store** (`fub.*` on the Railway
mirror), get minted **forward-only** into **our own events** (`hl_lead_activity`), and are read by HouseLoop
surfaces — with **HouseLoop as the sole notifier**. The whole integration is governed by **ONE master switch**.

```
   Zillow (Preferred · Rentals · RealScout)
                  │  inquiries · tours · views · saves · ML intent tags · texts/calls/emails
                  ▼
            FOLLOW UP BOSS              ← the hub. FUB owns the signal first.
                  │
   NATIVE WEBHOOK (sole real-time trigger)          SCRAPER (Chrome, Camp David) — BACKSTOP only
   crm.houseloop.ai/api/fub/webhook → fubWebhookRouter    · Lane-2 masked-body recovery (fub.body_recovery_queue)
   3 lanes: visible → notify + store ·                    · history/photo/in-app backfill on import
            masked → recovery queue ·                     (NOT a coequal real-time pipe; the realloop.co
            record-edit → store only                       dashboard receiver + catch-up pollers are RETIRED)
        └───────────────────────┬───────────────────────┘
                  ▼   storeRecord: texts/calls/emails/events stored UNCONDITIONALLY in the mirror
        fub.* on Railway MIRROR              ← THE ONE STORE (relay: Camp David → Railway)
                  │  classifyFubEvent / convert-fub-signal → logActivity()   ·   FEED IS FORWARD-ONLY
                  ▼
        hl_lead_activity              ← OUR EVENT STORE (the truth; one run-log, no fork)
                  │
   ┌──────────────┼───────────────────────────────┐
   ▼              ▼                                ▼
 HouseLoop     HouseLoop CRM                   FUB's own UI
 notifications (feed w/ real timestamps,       (followupboss.com)
 (sole notifier) import WIZARD, request cards)
```

**The principle (LOCKED):** every FUB signal becomes **one of OUR OWN events** — minted via `logActivity()` into
the one run-log `hl_lead_activity` (no new store, no fork; the `ALL_FEED_IS_EVERYTHING` law). `fub.*` is the raw
landing zone; our event store is the truth. **The feed is FORWARD-ONLY** (2026-07-10): a pulled-in lead starts
clean — pre-import comms stay context-only under the FUB tab; new activity flows into the feed going forward.
**FUB is lead source #1** — the pattern (acquire → import → live-sync → events) is source-agnostic.

# 2. Data acquisition — webhooks + scraper 🔒 *(the photo "race" fix = 🚧, see §7/I1)*

Two acquisition paths, one destination. *"It's all webhooks"* and *"we're still scraping"* are **both true** —
they cover different gaps.

```
                ┌─ 19 LIVE WEBHOOKS ─ signals + inquiry property (real-time)┐
FUB  ──────────►│                                                          ├─► Camp David fub.*  ─► [RELAY] ─► Railway fub.*  ─► HouseLoop FUB Activity panel
                └─ SCRAPER (Chrome) ─ recovers ONLY what the API hides ─────┘                       (fub-relay.mjs)               (lead detail → "Follow Up Boss Activity")
```

- **Webhooks** = FUB pings us instantly with clean structured data for most things.
- **Scraper** = logs into FUB as a human to grab the **two things the API hides**: masked message/email/call
  **bodies**, and the **deep historical Zillow browse list** (incl. photos).
- **Relay** (`tools/fub-sync/fub-relay.mjs`, pm2 `fub-relay`) mirrors Camp David `fub.*` → Railway `fub.*`.
- **HouseLoop panel** = `apps/houseloop-crm/components/leads/detail/LeadContextPanel.tsx` — reads Railway `fub.*`
  by `hl_leads.fub_person_id`.

### What comes from where (the canonical table)
| Data | Webhook / API delivers it? | Needs the scraper? |
|---|---|---|
| **Property inquiries** | ✅ `eventsCreated` — full clean property (street, MLS, price, beds/baths) | ❌ |
| **Viewed homes** | ❌ **FUB no longer emits a view webhook** (Zach LOCKED 2026-07-01) | ✅ **scraper reads the ENTIRE view history + backfills it at lead-import** |
| **Saved homes** | ❌ **no save webhook** (Zach LOCKED 2026-07-01) | ✅ **scraper reads the full saved list + backfills at lead-import** |
| **Lead notes** | ✅ `notesCreated` (body included) | ❌ |
| **Lead / contact / stage / tags / deals** | ✅ `peopleCreated/Updated/StageUpdated`, `peopleTagsCreated`, `dealsCreated/Updated` | ❌ |
| **Text — that one happened** | ✅ `textMessagesCreated` (to/from/time) | — |
| **Text/email *body*** | ❌ masked ("hidden for privacy") | ✅ scraper recovers it |
| **Call AI summary / transcript** | ❌ not in API | ✅ scraper |
| **Deep historical browse list** | ❌ not in the API at all | ✅ scrape-only |

### Webhooks (the real-time signal feed)
**19 active subscriptions**, system `BossLoop` (`X-System` + `FUB_SYSTEM_KEY`), all pointed at
`apps/dashboard/server/fub-webhooks.ts` (`POST /api/fub/webhook`).

<Warning>
`GET /v1/webhooks` returns **0 unless you send `X-System: BossLoop`** — webhooks are system-scoped. They are
live; a bare GET just can't see them.
</Warning>

- **Lean payload** `{ eventId, event, resourceIds[], uri }` — the handler **fetches back** the full resource from
  `uri` (Basic auth + `X-System`/`X-System-Key`). A **`Property Inquiry`** fetch-back returns the
  full `property` object (street/city/zip/MLS/price/beds/baths — **no lat/lng**, geocode ourselves). ⚠️ There is
  **no `Viewed Property` webhook** — that public event type **died May 21** (see the 🔑 note in §2). Views/saves do
  NOT come from a webhook fetch-back; they come from the scraper intercept below.
- **The handler** `handleEvent()` → `fub.events`; `parseZillowViewNote()` parses "Property Viewed" automation notes
  → `fub.zillow_home_activity` **as a re-scrape TRIGGER only** — the note lags & misses, so it just says "go
  re-scrape"; the authoritative, complete view list is the **`GET /api/v1/events?hasProperty=true` intercept**.
  The handler also handles texts/emails/calls/notes/deals/tasks/appts; queues masked-body items for the scraper;
  fires Twilio/Telegram alerts; has **catch-up pollers** (2–5 min) as a webhook-drop backstop.
- **Signature** `FUB-Signature` = `HMAC-SHA256( base64(rawBody), X-System-Key )` (hex) — verify on the RAW body.
- **Limits** events GET 20/10s; return 2xx within 10s; FUB retries up to ~8h.

### Scraper (recovers what the API hides)
FUB masks bodies in its public API. The scraper logs in via a real CDP Chrome and recovers the bodies a human
sees (`content_status` `hidden` → `visible`). Repo SSOT `tools/fub-scraper/`; deploy `tools/fub-scraper/deploy.sh`
— **never hand-edit files on the box.**

| **fub-webhook-scraper** | `fub-webhook-scraper.mjs` | Real-time — polls `fub.rehydration_queue` (`FOR UPDATE SKIP LOCKED`) |
| **fub-watchdog** | `fub-backfill-watchdog.mjs` | Self-healing — every 60s relaunches a dead/wedged Chrome (A **and** B), auto-logs-in via Google SSO, restarts a stalled worker |

### ⭐ The scrape-on-entry doctrine — the real-time per-lead intercept (THE entry mechanism) 🔒
**This is how every new lead's full history reaches us, in ~half a second.** The moment a new Zillow inquiry fires
the FUB `eventsCreated` webhook, the handler **spins up a per-lead scrape on Camp David** that intercepts FUB's own
authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire** timeline — grabbing
**every home the lead ever touched: the Zillow URL, zpid, MLS, price, and the photo** — in one shot (242 → 534 homes
on the test lead James Ross). It writes each home keyed by FUB's stable `fub_event_id` into `fub.zillow_home_activity`
via `saveZillowHomeActivity` (`tools/fub-scraper/fub-scraper-lib.mjs`). Trigger paths: the webhook auto-enqueues it
(`fub.rehydration_queue` → `fub-webhook-scraper`); on demand, **`mcp__fub-scraper__fub_scrape_lead` /
`mcp__camp-david__fub_scrape_person`** re-run the full intercept for one person. **It is deployed + LIVE.** This
scraper is the front door for ALL new lead data — when a lead's cards look wrong, re-running it is the first move.

### 🗄️ TWO databases — and `zillow_home_activity` is OFF the relay (read this before any photo/home-card debugging) 🔒
There are **two** FUB stores, and they are NOT kept in sync the same way for every table:
- **Camp David `fub.*`** (`DATABASE_URL` on the box) — where the scraper WRITES. Ground truth for scraped data.
- **Railway mirror `fub.*`** (`FUB_RAILWAY_DATABASE_URL`) — **what the HouseLoop CRM READS** (`fubQuery`,
  `LeadContextPanel`, the import, the card minter). The CRM never reads Camp David directly.

Most tables reach the mirror via the **relay** (`fub-relay`, upsert-by-id). **`fub.zillow_home_activity` is the
exception — it is OFF the relay (Zach 2026-06-27).** The **scraper is its SINGLE WRITER to the mirror**: right after
`saveZillowHomeActivity` writes Camp David, it does a **per-person FULL REPLACE to the mirror**
(`syncPersonHomeActivityToMirror` — `DELETE person rows + INSERT the fresh set`). So a home's photo reaches the CRM
**only when the scraper runs for that person.** If the scraper hasn't re-run since the photos were captured, **Camp
David has the photos but the mirror does not** — and the CRM, reading the mirror, shows blank.

### ✅ The blank-photo bug — ROOT CAUSE FOUND + FIXED (2026-06-28)
Every blank photo + stale lead traced to **ONE bug**: `mirrorUrl()` in `fub-scraper-lib.mjs` resolved its `~/.env`
fallback with **`require('node:fs')` — but that file is an ES module, so `require` is undefined and threw**, silently
returning `null`. The per-person mirror sync then no-op'd on **every on-demand scrape** (which doesn't set
`RAILWAY_DATABASE_URL`). pm2 scrapers "worked" only because their env sets the var. So a scrape wrote Camp David
perfectly but **never updated the Railway mirror the CRM reads** → blank. *Proven:* Alberto Camp David **199/199**
photos, mirror **829 rows / 0 photos** — until the fix flipped it to **207/207**.

**Fixed (shipped):**
- `mirrorUrl()` uses the ESM-imported `fs` (no `require`).
- `syncPersonHomeActivityToMirror` now **sources the FULL set from Camp David** (not the flaky in-memory scrape
  result, which the intercept returns non-deterministically — 0/4/199 on the same lead), **always runs**, **never
  shrinks** the mirror below Camp David, and **fails LOUD** (`✅ mirror replaced: N` / `❌ mirror sync FAILED` — never
  silent). The loud log is what surfaced the `require` bug on the first run.
- **Card heal:** `resyncHomePhotos` + `stampActivityPhoto` (the sanctioned append-only exception) stamp photos onto
  already-minted cards (all `evt`/`zpid`/slug source_id variants); for leads minted from the old broken mirror,
  **re-import** (`fub/lead.import {backfill}`) mints clean `evt`-keyed cards, then a Rule#0-approved delete removes the
  stale `fub_home:` non-evt duplicates. **Junk guard** in `convertHomeActivity`: an address that is a URL / agent-page
  (`realscout.com/users/...`) / doesn't start with a street number **never mints a card**.

> **One-line debug rule:** blank home card → check the **Railway mirror** (`FUB_RAILWAY_DATABASE_URL`), not Camp
> David. 0 photos there → re-run `fub_scrape_lead` (now mirrors loud); cards stale → re-import the lead.

### 🔑 Where property VIEWS actually come from — the internal feed, NOT the notes (Zach 2026-06-28)
FUB has **no property-view webhook** (the public "Viewed Property" event died May 21). Your **"Property Viewed" note
automation** fires `notesCreated` — but it **lags and misses** (proven: James had 6 views, Alberto 5, newer than their
newest note; rentals/heavy browsers fall behind). The **complete, ordered, current** view list lives ONLY in FUB's
**internal `GET /api/v1/events?hasProperty=true` feed**, which the scraper intercepts. So: **source views from the
intercept, never the notes.** The note webhook is fine as a *trigger* ("go re-scrape"), but the data is the feed.

# 3. The import workflow 🔒

A lead's card is born from a **webhook, not a poll**. The chain, grounded on real leads (Juan Martín Munguía
42729, James Ross 50308):

1. **Source** — a Zillow in-app message / inquiry lands in FUB (e.g. "interested in touring 15111 E 44th St S on
   Jun 26 at 12pm", source **Zillow Preferred**, MLS# 2626844). FUB fires an `eventsCreated` webhook.
2. **Pull-in decision** — the **native webhook router** (`fub-webhook-router.ts`) classifies the signal
   (`classifyFubEvent`) and fires Inngest **`fub/lead.import`** ([Inngest or it doesn't exist](/guides/inngest)).
   (The old dashboard `dispatchFubImport` / `fub-webhooks.ts EVENT_IMPORT_TRIGGER` path is RETIRED — capture is the
   native webhook now; see [FUB Trigger Events](/fub/trigger-events).)
3. **The import mints (the key steps):**
   - **2.6 — the canonical request card** — `kind` from `classifyFubEvent(message, source)`,
     `source_table='fub_request'`, `source_id='fubreq|<fub.events.id>'`, ONE per inquiry, home resolved by MLS# →
     photo + `/home` link. **The card we keep.** (`fub.events` is now stored **unconditionally** by the router's
     `storeRecord`, so inquiries/tours always land for this step — 2026-07-10.)
   - **2.7 — behavior + comms backfill** — `backfillFubBehavior` mints the lead's FUB behavior history
     (views/saves/searches/insights); `backfillFubComms` bridges texts/calls/emails. Both are **watermarked
     FORWARD-ONLY** (floored at the lead's `created_at`) — a pulled-in lead starts CLEAN, and only new comms flow
     into the feed; old comms stay context-only under the FUB tab (2026-07-10). Idempotent.
   - **2.8 — Myli context is NOT built on import.** `buildLeadContext` was removed from the import path (2026-06-29):
     facts refresh **before Myli communicates** (the conversation path / a pre-outreach workflow step / on-demand
     when the lead is opened), never on a data pull-in. See §6.
   - **3 — registration door log** — `kind='registration'`, `source_table='fub_lead_import'`, source "VIA HouseLoop".
     **This currently DUPLICATES the request card** (tracked issue I2, §7).
- **Link + source + owner:** `hl_leads.fub_person_id`; lead source = `Zillow.com`, platform = `follow_up_boss`; a
  lead is **never ownerless** — `ensureLeadAiOwner` falls back to the platform default AI agent (Myli) when no human
  owner and no AI agent is resolved (2026-07-10).

<Warning>
**Lead comms go through workflows ONLY (LOCKED 2026-06-26).** The legacy canned lead-facing ack in
`request-alerts.ts` is removed — a FUB lead's tour/contact acknowledgement is the job of the **Lead Flow workflow**
bound to that door, never a hidden canned send. Agent-facing alerts remain.
</Warning>

### The bulk import wizard — select → assign → tag (Settings → Integrations → Follow Up Boss → Import) 🔒
The webhook path above imports ONE lead on a live signal. The **wizard** (`FubImportWizard.tsx`) is the operator
tool to pull people in **in bulk**, all decided on-screen (never hardcoded). It reads live from the synced mirror
and dispatches the same `fub/lead.import` per person via `fub/import-all.run` (the SILENT path — no doors, no
workflows, no notifications). Four steps:

1. **WHO — filter who comes in.** Pick FUB **stages** (with live people-counts + a per-stage → HouseLoop-stage
   mapping) and, optionally, narrow by **FUB tags**: a searchable, scrollable modal over **all live FUB tags**
   (~1,276, most→least by count, from `fub.people.tags`). Facet logic: OR within stages, OR within tags, **AND**
   across the two — a live "N match" comes from `/api/integrations/fub/import/count`.
2. **HOW — depth + tags to apply.** Light-seed (default) vs full-history; silent claimable accounts; and the
   **HouseLoop tags to apply to every imported lead** — pick from existing `hl_tags` OR type a **new** name to
   create it (`ensureTag` find-or-creates). These apply on top of the origin tag + the person's own imported FUB tags.
3. **OWNERSHIP — assign to a person.** A dropdown of human `hl_users` (or **Myli-AI-only**, never ownerless);
   the pick threads to `assigned_user_id` on every minted lead.
4. **REVIEW — the Rule #0 gate.** Dry runs + test batches (≤100) are open; the **full run stays disabled** until
   Zach types the Rule #0 approval (`approval:'ZACH-TYPED-OK'`) — a bulk write to `hl_leads` never fires from the browser.

Stage mapping is applied one-shot at import (`opts.stageId ?? source default`); the flow is **one-way** — FUB → HouseLoop, no stage sync-back. Files: `components/settings/fub/FubImportWizard.tsx`, `app/api/integrations/fub/import/{preview,run,count}/route.ts`, `lib/inngest/functions/fub-import-all.ts`, `lib/fub-import/import-fub-lead.ts`.

### Layer 1 — real-time sync (event-driven, NO polling) 🔒
```
Camp David fub.*  →  relay  →  Railway fub.*  ──NOTIFY fub_lead_changed(fub_person_id)──►
   HouseLoop fubListener  →  lead-detail SSE (/api/leads/[id]/stream)  →  card updates instantly
```
The relay fires a Postgres `NOTIFY` on each write; `fubListener` + the lead-detail SSE push it to any open card —
**no polling** (HouseLoop law). The FUB Activity panel reads **real counts** from `fub.*` (e.g. Website **3,923**),
never FUB's display number (capped at 100).

### The two pipes 🔒 — *(scope evolved: behaviors are now first-class events too)*
A FUB lead's activity flows through **two pipes that do not feed each other**:
- **Pipe A — the raw FUB history** (the **Follow Up Boss Activity** panel, `LeadContextPanel.tsx`): the lead's FULL
  FUB history — texts, calls, notes, every inquiry, Zillow views — read **live from `fub.*` by link**. This is
  Myli's reading context (§6).
- **Pipe B — our event store** (`hl_lead_activity`, the **All** + **Website Activity** tabs): the lead's activity
  minted as **our own first-class events**. **Originally only tour/contact requests crossed; now behavior events
  cross too** — backfilled + forward-minted, **scoped to leads already in HouseLoop** (the Landing, §4). FUB
  texts/calls/emails now ALSO cross into Pipe B, but **FORWARD-ONLY** (CleanSlate, 2026-07-10): only comms after the

# 4. The event model & Data & Signals Matrix 🔒

### The coarse-type problem
FUB stores **all** lead-initiated property contact under one `type = "Property Inquiry"`. The real intent lives
**only in the message + `source`**. One classifier — `classifyFubEvent({type, message, source})`
(`apps/houseloop-crm/lib/fub/classify-fub-event.ts`) — resolves it; every surface uses that one verdict. The ONE
import gate is `SERVICED_FUB_EVENT_TYPES` in that file (replaces the old hardcoded `type IN (...)` SQL — adding a
type is now ONE place).

### The three request types (derived from the message, never the FUB type)
| Request type | Message signal | Our event |
|---|---|---|
| **Contact request** | *reach out · call me · like to talk* | `contact_request` |
| **Property inquiry** | *interested in &lt;addr&gt;* (no tour/callback) | `property_inquiry` |
| **Property tour request** | *tour · showing · seeing &lt;addr&gt; on &lt;date&gt;* | `tour_request` |

<Warning>
**FUB's `General Inquiry` (~4,088) is a junk bucket** — 86% blank, rest mostly phone-call notices + Zillow system
messages. Parsed: real questions → `contact_request`, phone notices → calls feed, noise → dropped (`null`).
</Warning>

|---|---|---|
| Property Inquiry (tour language) | `eventsCreated` | `tour_request` |

### One-to-one parity — Zillow ↔ HouseLoop (LOCKED 2026-06-26)
Same event catalog on both platforms; only the **source badge** differs (`Zillow.com` vs `HouseLoop.ai`). A home
viewed on Zillow and on houseloop.ai are the same `kind` (`property_view`). The one asymmetry: **liked home is
HouseLoop-only** (Zillow has no "like"). Pre-qualified is the same kind, branded **ZHL** (Zillow) vs **Lending
Loop** (HouseLoop). Add an event to the catalog (`activity-badges.ts` / `BEHAVIOR_BADGES`) → it gets a card on
every surface automatically (no per-type hardcoding — the `ALL_FEED_IS_EVERYTHING` law).

The Zillow side is minted from: views/saves → `notesCreated` → `fub.zillow_home_activity`; shared → scraper;
repeat-views/area → the raw note; tour/inquiry/contact → `fub.events` → `classifyFubEvent`; saved search → scraper;
high-intent/reengaged/specific-home/pre-qual → the Zillow tags. **`Zillow Smart List` = the 13-day "active now"
window** (add → `behavior_active`, remove → `behavior_window_closed`) — not itself a behavior.

### 🛬 The Landing — ship it without boiling the ocean (LOCKED)
1. **Only a request creates a NEW lead.** A lead is pulled in **only** by a property inquiry / tour request.
   **Behavior events NEVER create a lead** (that would flood HouseLoop with events for leads we have no workflows for).
2. **Behavior events are first-class — but only for leads already in HouseLoop.** Once a request pulls a lead in:
   **(a) backfill** its whole FUB behavior history into `hl_lead_activity`; **(b) forward**-mint every new behavior
   in real-time; **(c) scope** — a behavior for a FUB lead NOT in HouseLoop is ignored. No flood, no orphans.
3. **Pre-seed every trigger token → a "Coming Soon — Not Built Yet" stub** in the Workflows list. The trigger is
   visible, the event flows first-class, **nothing fires** until a real workflow replaces the stub.

Each conversion runs through ONE converter writing via `logActivity()` (events → `classifyFubEvent`; notes →
view/save + the insight parsers; home activity → view/save/share deduped per home **with the photo**; saved
searches → `saved_search`; tags → the `behavior_*` events + Smart-List window). Each fires at webhook-time +
a per-lead backfill (gated to leads already in HouseLoop).

# 6. Myli context 🔒

One lead, three layers:
```
FUB lead (fub.* on Railway)
   ├─ Layer 1  REAL-TIME SYNC ........ live counts + activity on the lead card (§3)
   ├─ Layer 2  FACTS (deterministic) . the 12-domain fact store — structured, no LLM
   └─ Layer 3  DOSSIER (on-demand LLM) . one Lead Context = [FUB section] + [HouseLoop section]
```
- **UI** — the lead-detail `[Lead Context] / [Follow Up Boss Activity]` toggle + the FUB Activity panel
  (`LeadContextPanel.tsx`), reading the full FUB history live by `fub_person_id` (**Pipe A**, §3).
- **Layer 2 — Facts** — on first pull, `buildLeadContext` structures every FUB fact into the **12-domain fact
  store** (name, source, timeframe, price range, views/saves/searches, Zillow Insights). No LLM, fully structured.
  Built on every pull-in as import step 2.8.
- **Layer 3 — Dossier** — ONE Lead Context = a **Follow Up Boss** section (imported FUB history via `loadFubBrief`,
  Sonnet, on-demand `POST /api/leads/[id]/context`) + a **HouseLoop** section (live). Not two dossiers. Sonnet once
  on import · Haiku delta on update · never auto-per-fact.

### ⭐ The Watermark Law — read-once, merge-forward (universal) 🔒
Governs **every** LLM rollup (dossier, facts, Hot Buyers, any summary): **never re-process data already consumed.**
The dossier stores its text + a `source_cursor` (high-water mark). Each regen: read only the delta
(`WHERE synced_at > source_cursor`) → merge into the existing dossier (don't re-summarize) → advance
`source_cursor = max(synced_at)`. First pull = full history (Sonnet, cursor = now); every pull after = the delta,
merged with Haiku. **Tokens paid once per row, ever.**

<Warning>
**The gotcha:** watermark on **`synced_at` (when WE ingested the row), NOT the event's own date.** A 2-year-old
home view the scraper *just* backfilled is **new to us** — `synced_at > cursor` catches it. Watermark on the event
date and you'd silently skip every freshly-backfilled historical row.
</Warning>
