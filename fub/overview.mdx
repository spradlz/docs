## The one-source map (memorize this)

There is **one source** (FUB, fed by Zillow), pulled by **two pipes** (webhooks + scraper), into **one store**
(`fub.*` on Railway), read by **many surfaces**.

```
   Zillow (Preferred · Rentals · RealScout)
                  │  inquiries · tours · views · saves · ML intent tags
                  ▼
            FOLLOW UP BOSS              ← the hub. FUB owns the signal first.
                  │
        ┌─────────┴─────────┐
   19 WEBHOOKS          SCRAPER (Chrome)
   real-time signals    only what the API hides:
   + clean property     · message/email/call bodies
   (inquiries, views,   · saved searches
    notes, tags…)       · the deep browse list
        └─────────┬─────────┘
                  ▼
        fub.* on Railway              ← THE ONE STORE (relay: Camp David → Railway)
                  │
   ┌──────────────┼───────────────────────────┐
   ▼              ▼                            ▼
 Telegram     HouseLoop CRM               FUB's own UI
 + SMS alert  (FUB Activity panel,        (followupboss.com)
              lead import, request cards)
```

## The principle (LOCKED)

**Every FUB signal becomes one of OUR OWN events.** We don't leave data trapped in FUB's shape and read it
live in a panel — each signal (inquiry, view, save, saved search, insight note, ML intent tag) is minted as a
canonical HouseLoop event (via `logActivity()` → the one run-log). Once it's our event it's our data, it
renders on our cards, and it can trigger our workflows. `fub.*` stays the raw landing zone; our event store is
the truth. See [Data & Signals Matrix](/fub/data-matrix).
