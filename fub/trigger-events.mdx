## The principle (LOCKED 2026-07-09 by Zach)

**Notifications are the source of truth. The scraper is a dumb, on-demand grabber of HIDDEN content only.**

The native HouseLoop webhook is what decides *what happened*; when the thing that happened has content FUB hides
(a masked message body), the notification layer *tells the scraper* to go recover exactly that one body — nothing
else. The scraper never decides what's new, never crawls, never re-scrapes on a record edit. This replaces the
old "push for speed, pull for truth" dual-path / differential-reconciler design, which is **dead** (it read the
lagging mirror and flooded Zach with 2022-era activity — see the post-mortem below).

## Vocabulary (LOCKED 2026-07-07)

- **Integration** = the *pipe* a lead flows through — **HouseLoop.ai** (our website pixel) or **Follow Up Boss**
  (the Zillow-connected CRM). Managed at **Settings → Lead Integrations** (`/settings/lead-platform`).
- **Lead Source** = the *brand* on the lead — Zillow, Referral, etc. A source lives *under* an integration.
- **Trigger Event** = a *door* — a thing that can happen (a tour request, an inbound text, a property save) that
  can start a lead, run a workflow, and/or fire a notification. One row in **`hl_fub_webhook_settings`**,
  configured at `crm.houseloop.ai/settings/trigger-events`.

## The four independent toggles (this is the model)

Each FUB trigger event carries **four independent switches** — none implies another:

| Toggle | Column | What it controls |
|---|---|---|
| **Sync** | `enabled` | Do we ingest/process this FUB signal at all. Off = we don't even mirror it. |
| **Creates Lead** | `creates_lead` | Does firing this door mint (or match) a HouseLoop lead. |
| **Workflow** | `workflow_id` + `workflow_active` | Which workflow the door runs, and whether it runs. |
| **Notify** | `notify` | Fire an owner-only HouseLoop notification when this door fires — **independent of Creates Lead.** |

<Note>
**Notify is decoupled from Creates Lead (Zach 2026-07-08).** You can turn ON a notification for a FUB trigger
event **even if you do not create a lead** — a "heads-up, this happened" ping with no lead minted. The card then
renders "**HouseLoop — no lead created**" (no HouseLoop link) alongside the always-present FUB person link. The
gate is two-tier: the per-event row toggles over a `_global` master row (`webhook_type='_global'`).

**The `_global` master now has a switch on the wall (Lee, 2026-07-09 night).** It was the ONE breaker for the
whole FUB pipeline — every per-row Sync/Creates-Lead/Notify toggle is gated behind `_global.enabled` on the
server (`fub-body-recovered.ts`, `fub-lead-import.ts`) — but its only UI lived in the never-mounted
`FubChannelBoard`, so with `_global` OFF, flipping the 23 per-row toggles did nothing and there was no control
to fix it. Fixed: `FubChannelRows` now renders a **master-breaker banner** above the FUB rows on the Trigger
Events board (red "OFF — nothing below runs" / blue "ON — the pipeline is live"), writing `_global.enabled`.
This is the go-live switch. **Flipped ON 2026-07-09 night → the pipeline is LIVE** (see
[Where the build stands](#where-the-build-stands)).
</Note>

## Behavioral signals MINT when Creates Lead is ON

**The `Creates Lead` toggle means the same thing on every row — behavioral events included (Zach 2026-07-11).**
A `property_saved` / `property_viewed` / `in_app_message` from a person who is **not yet** a HouseLoop lead now
**mints a new lead when that event's `creates_lead` is ON** (and `_global` on), instead of the old notify-only
short-circuit. Turned OFF, it stays a lead-less "heads-up" notify. Each behavioral event is independent — Zach
turns Create-Lead on only for the ones he wants to become leads.

- **What fires on a mint:** a SAVE fires the real **`saved_home_registration`** door (its seat row already lives
  in `hl_lead_source_events`); a view/in-app has no registration door of its own so it mints via the generic
  first-touch **`direct_registration`**. Either way `importFubLead` mints/links the `hl_leads` row (source folded
  to its six-bucket canonical), `backfillFubBehavior` logs the save as a `save_home` feed card, `dispatchDoor`
  writes the `hl_trigger_fire` ledger row + runs any assigned workflow, and the owner gets a **New-lead** card
  carrying the saved home.
- **An EXISTING lead's behavioral signal is unchanged** — activity card + the `FUB_NOTIFY[webhookType]` ping, no
  registration door (the `NON_DOOR_WEBHOOKS` gate still holds for `created = false`).
- **Fail-closed preserved:** minting is still gated on `_global` + the row's `creates_lead`; nothing mints
  without an explicit toggle. Code: `lib/inngest/functions/fub-lead-import.ts` (behavioral fast-path →
  fall-through to the mint path; `saved_home` trigger/door; `doorEligible` opens only when `created`).

## The 3-lane architecture

Every FUB event enters the native webhook and `fubWebhookRouter` sorts it into exactly one lane:

### Lane 1 — Visible activity (~95%): router notifies directly, scraper never touches
Home saved/viewed (`notesCreated`), inquiries (`eventsCreated`), visible SMS, call metadata, appointments,
tasks, deals. (In-app messages are NOT here — they are the inbox-app sweep lane below.) The API hands us clean
content → the router classifies it and fires
the owner-only notification immediately.

### Lane 2 — Masked comms: the ONLY thing that wakes the scraper
The three masked cases — masked SMS bodies, email bodies, and call notes. The flow is a **surgical body-fetch**,
never a re-scrape:

```
masked textMessagesCreated / emailsCreated  →  fubWebhookRouter writes metadata + enqueues ONE job
      ↓ fub.body_recovery_queue (on Camp David — the relay is CD→Railway one-way, so the job lives CD-side)
      { person_id, resource_type: 'text' | 'email', resource_id, requested_at }
      ↓ the scraper drains the queue — reads THAT ONE body from the FUB UI
      ↓ writes the recovered body → flips content_status  hidden → visible
      ↓ trg_fub_new_message fires  (now AFTER INSERT OR UPDATE, transition-guarded, text + email)
      ↓ pg_notify('fub_new_message') → the native notify path
   →  HouseLoop notification fires WITH the body
```

**No metadata-only notify for a masked body** — a card with no message is useless, so we hold the notify until
the body is recovered. Born-visible content (in-app, visible SMS) is Lane 1 and notifies on the INSERT.

### Lane 3 — Record edits: update the person, never scrape
`peopleUpdated` / `peopleStageUpdated` / `peopleTagsCreated` / `peopleDeleted`. These update the person record
only. **They never enqueue a scrape.** (A stage change may fire `stage_changed` if the Notify toggle is on; that
is a notification, not a scrape.)

## ⭐ How an in-app (Zillow Messages) message becomes a trigger event

Zillow in-app chat is the ONE signal FUB does **not** expose through its API or webhooks (proven 2026-07-09 on
James Ross: his in-app messages carry synthetic ids and never appear in `GET /v1/textMessages`; no
`textMessagesCreated` fires). They live in FUB's **"Zillow Messages" inbox app** (`inboxApp id 1`, created by
the original account owner's seat — `createdById 1`, which is why a departed owner's Zillow connection still
fronts these leads). So capture is: find the people via the REST API, scrape their pages for the bodies — all
lawful, gated, and Inngest-visible:

1. **Detect — the REST API, not the inbox SPA (proven 2026-07-09):** message BODIES aren't in the API, but WHO
   has in-app activity IS: **`GET /v1/people?sort=-lastInboxAppMessage&fields=id,lastInboxAppMessage`** returns
   every person ordered by their last in-app message, with the timestamp. Reliable, no browser, cursor-pageable.
   (The FUB inbox web endpoint `/api/v1/inbox/…/conversations` **hangs** on any direct/replay fetch and its
   virtualized list won't page on scroll — only the app's own live XHR ever fires; the people-sort is the robust
   surface.) Incremental takes only people newer than our mirror's newest `zillow_inapp`; backfill takes all.
   The registered Inngest cron **`fub-inapp-sweep-tick`** (Run Log `comms` lane; event-only until sign-off, then
   cron `*/3`) enqueues one sentinel `inbox_sweep` job into the Railway FUB mirror queue —
   `UNIQUE(resource_type,resource_id)` = at most one sweep pending (no pile-up).
2. **Extract (reuse, never reinvent):** the Camp David `fub-body-recovery` consumer claims the job and runs
   **`tools/fub-scraper/fub-inapp-sweep.mjs`**, which — for each discovered person — reuses the proven extractor
   in `fub-scraper-lib.mjs` (`scrapePersonFull` + `matchAndUpdateMessages`: `scrapeTextContainers` classifies a
   timeline row `zillow_inapp` by its **`zillow.png`** circle-icon; the synthetic id
   **`900000000 + personId*1000 + seq`** dedupes against the existing rows, `ON CONFLICT (id) DO NOTHING`). The
   relay carries the new born-visible rows Camp David → Railway.
3. **Notify (native, gated — one notifier):** for each NEW inbound in-app message the sweep POSTs
   `/api/fub/body-recovered`; **`fub-body-recovered`** maps `message_type='zillow_inapp'` → `in_app_message` →
   a card, through the gate stack: **existing-lead gate** (no HouseLoop lead ⇒ store-only, NO card, NO mint) +
   `_global` on + `in_app_message` Notify on + recency. The legacy realloop `notify-scraper` Twilio path is
   retired (behind `REALLOOP_SCRAPER_NOTIFY`, off) so there is exactly ONE notification per message.
4. **Backfill is silent:** `fub-inapp-sweep.mjs --backfill` pulls every in-app message into the mirror + All
   feed with **zero** notifications (no 1,198-card flood). `--measure` prints the FUB-vs-mirror gap.

<Note>
**Live proof (2026-07-09, dark run).** `--measure`: FUB **1,303** in-app people vs our mirror **2,464** people /
**10,588** messages (coverage gap 0). Incremental sweep: discovered 1 person newer than the cursor → reused
extractor scraped → **2 new in-app messages stored** → POSTed body-recovered → **notified: 0** (gate held:
`_global` + `in_app_message` Notify both off) → `hl_notification` new_message = 0. Discover → scrape → store →
gate, proven end-to-end, zero cards. In-app messages are **buyer-initiated** (an agent can't originate one), so
the only remaining live event is a real buyer message once the toggles flip.
</Note>

**Why not a webhook:** in-app *inquiries* do fire an `eventsCreated` (Property Inquiry) our native router
already receives — but pure-conversation messages ("What's up with this place?") fire nothing. So the sweep is
the mandatory floor that catches every message. **Never** set `REALLOOP_FUB_AUTONOMY=1` — that
revives the poller + FUB-write-autoreply flood stack retired in the one-writer lockdown.

## Native capture — the webhook that severs realloop.co

**The native CRM webhook `crm.houseloop.ai/api/fub/webhook` is the SOLE real-time trigger.** FUB posts
`{ event, resourceIds[] }`; the receiver responds 2xx immediately (FUB's 10s SLA), then fans out one
`fub/webhook.received` Inngest event per resource → `fubWebhookRouter` fetches the record (native `lib/fub/fub-api.ts`,
Basic-auth `FUB_API_KEY`), classifies it into a lane, reads `hl_fub_webhook_settings` directly, and dispatches the
existing `fub/lead.import` → `fubLeadImport` → `createNotification` chain. Reuse targets (no re-invention):
`fubWebhookRouter`, `classifyFubEvent`, `SERVICED_FUB_EVENT_TYPES`, `hl_fub_webhook_settings`, `createNotification`.

**Zero realloop.co in the path.** The old realloop.co dashboard receiver + `dispatchFubImport` forwarder is the
contamination this severs — realloop.co could be deleted tomorrow and notifications would not flinch. There is no
mirror-reading poller and no differential reconciler; those are dead.

## The recency gate + the flood post-mortem

**Recency gate:** a `fub_created` gate on the notify path — **old activity never notifies**, even during a
one-off catch-up. A signal whose FUB activity date is old is captured/stored but does not fire a card.

**Why (2026-07-09 flood):** the retired realloop.co handler re-queued a person for a **full re-scrape on every
`peopleUpdated`** — and FUB fires that on any tag/stage/field edit. It reset already-captured people to "pending,"
so the scraper re-walked the entire 49K book, re-inserting **old** activity with a **fresh** `synced_at` + new
ids. The mirror-reading pollers then treated those old rows as new and flooded Zach with 2022–2023 activity as if
it were live (e.g. Lucy Hartman "viewed" from 2026-07-03 written to the mirror on 2026-07-09; Jessica Marroquin
in-app from 2023-12-09). **Root cause: record edits re-scraping (Lane 3 violation) + the mirror deciding what's
"new."** Fix: retire the re-scrape loop, correct the **25,626** people the bug had wrongly reset to `pending`
(flipped back to done), rip the mirror-reading pollers, and make the webhook the sole trigger + the recency gate
the backstop. The backfill is done — **49,059** people carry `fub.rehydration_log.status = 'rehydrated'` (the
terminal "done" state; there is no separate `fully_scraped` flag), 0 pending, forward-only.

## How a FUB trigger event fires end-to-end

```
FUB signal → POST crm.houseloop.ai/api/fub/webhook (native receiver, 2xx immediately)
          → fub/webhook.received (Inngest, per resourceId)
          → fubWebhookRouter: fetch record → classify → sort into a lane
              Lane 1 (visible):  read hl_fub_webhook_settings → if notify: createNotification (owner-only, deduped, recency-gated)
                                                              → if creates_lead: fub/lead.import → mint/match HL lead
              Lane 2 (masked):   write metadata + enqueue fub.body_recovery_queue → scraper recovers body
                                                              → content_status flip → trg_fub_new_message → notify WITH body
              Lane 3 (record):   update the person record only (no scrape)
```
