## The big picture — two acquisition paths, one destination

There are **two ways** FUB data reaches us, and they cover **different gaps**. This is the thing that
confuses everyone: *"it's all webhooks"* and *"we're still scraping"* are **both true** — they do
different jobs.

```
                ┌─ 19 LIVE WEBHOOKS ─ signals + CLEAN property (real-time) ─┐
FUB  ──────────►│                                                          ├─► Camp David fub.*  ─► [RELAY] ─► Railway fub.*  ─► HouseLoop FUB Activity panel
                └─ SCRAPER (Chrome) ─ recovers ONLY what the API hides ─────┘                       (CD→RW, fub-relay.mjs)        (lead detail → "Follow Up Boss Activity")
```

- **Webhooks** = FUB pings us instantly when anything happens, with clean structured data for most things.
- **Scraper** = logs into FUB as a human to grab the **two things the API hides**: masked message/email/call **bodies**, and the **deep historical Zillow browse list**.
- **Relay** (`tools/fub-sync/fub-relay.mjs`, pm2 `fub-relay`) mirrors Camp David `fub.*` → Railway `fub.*`. Rebuilt + backfilled 2026-06-22 (all tables, all columns, ~1.6M rows).
- **HouseLoop panel** = `apps/houseloop-crm/components/leads/detail/LeadContextPanel.tsx` — the "Follow Up Boss Activity" toggle on the lead detail page, reads Railway `fub.*` by `hl_leads.fub_person_id`.

## What can we get, and from where? (the canonical table)

| Data | Webhook / API delivers it? | Needs the scraper? |
|---|---|---|
| **Property inquiries** (lead inquired on a home) | ✅ `eventsCreated` — **full clean property** (street, MLS, price, beds/baths) | ❌ no |
| **Viewed homes** (individual event) | ✅ `eventsCreated` "Viewed Property" — clean property | ⚠️ only the full *historical* 1,000+ browse list |
| **Saved homes** | ⚠️ "Saved Property" event type exists, but FUB marks some events **app-only** (may not fire via API) | ⚠️ for the full saved list |
| **Lead notes** | ✅ `notesCreated` (body included) | ❌ no |
| **Lead / contact / stage / tags / deals** | ✅ `peopleCreated/Updated/StageUpdated`, `peopleTagsCreated`, `dealsCreated/Updated` | ❌ no |
| **Text messages — that one happened** | ✅ `textMessagesCreated` (to/from/time) | — |
| **Text message *content / body*** | ❌ **masked** ("Body is hidden for privacy reasons") | ✅ **scraper recovers the body** |
| **Email *body*** | ⚠️ often masked (esp. UI-hidden emails) | ✅ scraper |
| **Call AI summary / transcript** | ❌ not in API | ✅ scraper |
| **The deep historical browse list** | ❌ not in the API at all | ✅ **scrape-only** |

**One sentence:** the webhooks get every *signal* + the *clean property* on inquiries/views in real-time;
the scraper exists ONLY to (1) un-mask text/email/call **bodies** and (2) pull the **historical browse list**.

## 1. Webhooks (the real-time signal feed)

**19 active subscriptions**, system `BossLoop` (`X-System` + `FUB_SYSTEM_KEY`), all pointed at the handler
`apps/dashboard/server/fub-webhooks.ts` (endpoint `POST /api/fub/webhook`, via the camp-zach Tailscale URL).

<Warning>
`GET /v1/webhooks` returns **0 unless you send the `X-System: BossLoop` header** — webhooks are
**system-scoped**. They are live; a bare GET just can't see them.
</Warning>

- **Payload is lean:** `{ eventId, event, resourceIds[], uri }` — the handler **fetches back** the full
  resource from `uri` (Basic auth + `X-System`/`X-System-Key`). A `Viewed Property` / `Property Inquiry`
  fetch-back returns the **full `property` object** (street/city/zip/MLS/price/beds/baths — **no lat/lng**, geocode ourselves).
- **The handler** (`fub-webhooks.ts`): `handleEvent()` → `fub.events` (with property); `parseZillowViewNote()`
  parses the agent's "Property Viewed" **automation notes** (zillow.com/homedetails URLs) → `fub.zillow_home_activity`;
  handles texts/emails/calls/notes/deals/tasks/appts; queues masked-body items for the scraper; fires Twilio/Telegram
  inquiry alerts; has **catch-up pollers** (events/people/calls every 2–5 min) as a webhook-drop backstop.
- **Signature:** `FUB-Signature` header = `HMAC-SHA256( base64(rawBody), X-System-Key )` (hex). Verify on the RAW body.
- **Limits/retries:** events GET 20/10s; must return 2xx within 10s; FUB retries up to ~8h.
- **Register / list / delete:** `POST|GET|DELETE /v1/webhooks` (owner-only, max 2 per event per system, `X-System` required).

## 2. Scraper (recovers what the API hides)

Follow Up Boss masks message/email/call bodies in its public API (`[CONTENT HIDDEN]` / "Body is hidden for
privacy reasons"). The **FUB Scraper** logs into FUB via a real CDP-controlled Chrome session and recovers the
bodies a human can see in the FUB UI, writing them to Camp David `fub.*` (`content_status` `hidden` → `visible`).

### Processes (pm2 on Camp David)

| pm2 name | script | role |
|---|---|---|
| **fub-backfill-a / -b** | `fub-parallel-scraper.mjs` | Master scraper — pending-first from `fub.rehydration_log` |
| **fub-webhook-scraper** | `fub-webhook-scraper.mjs` | Real-time — polls `fub.rehydration_queue` (`FOR UPDATE SKIP LOCKED`), scrapes webhook-triggered leads |
| **fub-watchdog** | `fub-backfill-watchdog.mjs` | Restarts a stalled backfill worker |
| **fub-relay** | `tools/fub-sync/fub-relay.mjs` | Camp David `fub.*` → Railway `fub.*` mirror (all tables/columns) |

Scraping logic lives in **`fub-scraper-lib.mjs`** (`scrapePersonFull`, `scrapeEmailCards`, `matchAndUpdateEmails`,
`applyApiTexts`, `matchAndUpdateMessages`, `refreshLeadCounts`). Repo SSOT: `tools/fub-scraper/` — deploy with
`tools/fub-scraper/deploy.sh`; never hand-edit files on the box. **It is tricky — do not poke the Chrome on port
18800 (shared with the live scrapers) without knowing what you're doing.**

<Warning>
`fub-webhook-sink` is a **dead, redundant duplicate** that OOM-crash-looped and was **deliberately stopped
2026-06-02**. It is *supposed* to be off — it is NOT the live scraper. The live ones are `fub-backfill-a/b`
+ `fub-webhook-scraper`. Reading its "down/crashed" status as a live failure is a known trap.
</Warning>

### How body recovery works

1. **DOM extraction** — open the lead's email/text tabs in the authenticated Chrome session; `scrapeEmailCards` parses each card.
2. **Timeline-API interception** — the newest message often never renders; the scraper intercepts the FUB SPA's
   own authenticated `GET /api/v1/timeline?...&types=ThreadNew` whose `lastEmail.bodyHtmlVisibleClean` carries the
   full unmasked body. (A direct `fetch` returns `401` — you must intercept the UI's own request.)
3. **Write + flag** — recovered rows get `content_status='visible'`.

### ⚠️ UI-hidden emails are unrecoverable
If a FUB user sets emails private/hidden in the FUB UI, the body shows `[CONTENT HIDDEN]` in the UI itself — there
is nothing to scrape (`content_status='hidden'`, permanent). This is a FUB privacy feature, not a bug. A large
hidden count is often expected.

## Data model

| Table | Holds |
|---|---|
| `fub.people` | Leads — `phones` (jsonb), `tags` (text[]), `content_status`, materialized `*_count` |
| `fub.text_messages` · `fub.emails` · `fub.calls` · `fub.notes` | Activity, each with `content_status` (`hidden` → `visible`) |
| `fub.events` | Lead events — Property Inquiry / Viewed Property / Seller Inquiry, with the `property` jsonb (webhook-delivered, clean) |
| `fub.zillow_home_activity` | Homes the lead viewed/saved (note-parsed + scraped) — what the HouseLoop panel's homes tab reads |
| `fub.rehydration_log` | Backfill queue (`pending`/`done`) — drives `fub-backfill-a/b` |
| `fub.rehydration_queue` | Real-time webhook queue (`FOR UPDATE SKIP LOCKED`) — drives `fub-webhook-scraper` |

```bash
ssh lee@192.168.1.211 "PGPASSWORD=*** psql -h localhost -U bossloop -d bossloop -c \"

```

## Open item (2026-06-22)

The webhook handler writes Viewed/Inquiry events to `fub.events`, but the HouseLoop panel's homes tab reads
`fub.zillow_home_activity` — so webhook-delivered home **views/saves** flow to HouseLoop **without scraping** only
once the handler also writes those events into `fub.zillow_home_activity` (or the panel reads `fub.events`). Small
wire-up; until then the panel's homes tab shows note-parsed + scraped homes. The deep historical browse list always
stays scrape-backfilled.
