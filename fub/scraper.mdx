## One-line truth

Follow Up Boss's **public API masks** message/email/call bodies and never exposes the full historical browse
list. The scraper **logs into FUB as Zach in a real Chrome (CDP)**, reads what a human sees, and writes it to
`fub.*` — the store the CRM reads. **If the FUB UI itself hides something, the scraper cannot recover it. That is
not a bug** (see [Hidden emails](#what-it-can-t-recover)).

## Why it exists — what the API hides

| Data | API/webhook gives it? | Needs the scraper? |
|---|---|---|
| Property inquiries, viewed/saved (single events), notes, stage, tags, deals | ✅ clean, real-time | ❌ |
| Text / email **that one happened** | ✅ (to/from/time) | — |
| Text / email **body** (masked ~7% SMS · most email) + masked **call note** (5,690) | ❌ masked "hidden for privacy" | ✅ recovers it (Lane 2) |
| Call **transcript** | ❌ doesn't exist anywhere (0 rows; FUB has none) | ❌ would need our own transcription of `recording_url` |
| **Deep historical Zillow browse list** (every home ever touched + photo) | ❌ not in the API at all | ✅ scrape-only |

The webhooks (covered in [Lead Import §2](/fub/lead-import)) carry the real-time signals; the scraper fills the
two gaps the API can't: **masked bodies** and the **full browse history**.

## Role in the notification architecture (LOCKED 2026-07-09 — the scraper is a dumb Lane-2 grabber)

Under the 3-lane model (full detail: [Follow Up Boss Trigger Events](/fub/trigger-events)), **notifications are
the source of truth and the scraper is a dumb, on-demand grabber of HIDDEN content only.** It does not decide
what's new, does not crawl, and **never re-scrapes on a record edit.**

- **The only thing that wakes it is Lane 2** — a masked body. The native webhook writes a targeted job to
  **`fub.body_recovery_queue` (Camp David)** `{ person_id, resource_type:'text'|'email', resource_id }`; the
  scraper reads **that one body**, writes it back, and flips `content_status` hidden→visible (which fires
  `trg_fub_new_message` → the notification, now with the body). A surgical fetch, not a re-walk.
- **Masked = exactly three things:** SMS bodies (~7%), most email bodies, and 5,690 call notes. In-app messages,
  saves/views, inquiries, and call metadata are **exposed** and never touch the scraper (Lane 1).
- **The lock (precise mechanism):** the one-time 49K backfill is done — **49,059 people carry
  `fub.rehydration_log.status = 'rehydrated'`** (the terminal "done" state; there is **no** separate
  `fully_scraped` flag/status), **0 pending**, forward-only. The 2026-07-09 fix corrected **25,626** people that
  a `peopleUpdated` bug had wrongly reset to `pending` (flipped them back to `rehydrated`). A record edit
  (`peopleUpdated`/stage/tag) **never** re-queues a scrape.

## Architecture

```
FUB (logged in as Zach, real Chrome via CDP)
   │  DOM extraction + intercept of FUB's OWN authenticated SPA calls
   │  (GET /api/v1/timeline?…ThreadNew  ·  GET /api/v1/events?hasProperty=true)
   ▼
Camp David Postgres  fub.*   ── MASTER (host 192.168.1.211, db bossloop)
   │
   ├─ most tables ─ relay (fub-relay, upsert-by-id) ─────────────┐
   └─ fub.zillow_home_activity ─ OFF the relay; the SCRAPER is    │
      its single writer to the mirror (per-person FULL REPLACE)   ▼
                                              Railway  fub.*  ── MIRROR (what the CRM READS)
                                                 │  fubQuery / LeadContextPanel / lead import
                                                 ▼
                                              HouseLoop CRM
```

**Two databases (memorize this before any photo/blank-card debugging):**
- **Camp David `fub.*`** — where the scraper **writes**. Ground truth for scraped data.
- **Railway `fub.*`** (`FUB_RAILWAY_DATABASE_URL`) — what the **CRM reads**. The CRM never reads Camp David directly.
- Most tables sync via the **relay** (`fub-relay`). **`fub.zillow_home_activity` is OFF the relay** — the scraper
  is its **single writer to the mirror** (a per-person full-replace right after it scrapes). So a home's photo
  reaches the CRM **only when the scraper runs for that person**. Blank photo in the CRM → check the **mirror**,
  then re-run the scraper for that lead.

**ONE Chrome (Zach LOCKED 2026-07-08 — "go down to one chrome, harden it up").** A single persistent Chrome —
**Chrome A** on CDP port **18800** (`~/.chrome-fub-profile`), holding a live Google session — runs the scrape.
`fub-parallel-scraper.mjs` fans out `FUB_WORKERS` (default 3) worker *pages* inside that one Chrome. The old
second Chrome (B, `:18802`) + its `fub-backfill-b` worker are **retired**: two Chromes plus a watchdog meant
multiple processes reaching into shared browsers over CDP and closing each other's pages — the exact contention
that crash-looped the scraper. One browser, one boss. (The watchdog runs with `FUB_SINGLE_CHROME=1`, which skips
all Chrome-B logic.)

## The login (this is what "stuck / can't log in" means)

FUB's real login is **"Log in with Google"** as `zach@spradlingrealtygroup.com`. The persistent Chrome profile
holds a live Google session, so navigating to the app **auto-completes SSO** — no password prompt in the normal
case. If the Google session itself has lapsed, `fub-login-bot.mjs` drives the SSO by hand: **email → password →
account-chooser → consent**, in that order, using `FUB_WEB_GOOGLE_PASSWORD` (→ `FUB_WEB_PASSWORD` fallback). Only a
**genuine 2FA challenge** — a real code box or a phone push, with nothing else the bot can fill — alerts a human via
`notify.sh`. **Option text alone ("Try another way", "use a passkey", "verify it's you") is NOT a 2FA wall** — those
appear next to the normal password field and the bot must ignore them (see the 2026-07-08 incident below).

When the session expires, the scraper's **startup session-check** navigates fresh to `/2/people`; if it lands on
the login page it **delegates to `fub-login-bot.mjs` (Google SSO)** and re-verifies. (Before 2026-07-08 it clicked
a `input[type=submit].u-bigBlueButton` — the DEAD legacy FUB email/password form button. FUB is Google SSO now, so
that click always failed → an instant 10-minute cooldown → the "scraper down / won't log back in" symptom. The
delegation to the real login bot fixed it.)

**The `fub-watchdog` self-heals a real logout AUTONOMOUSLY — a human should almost never touch it.** Every 60s it
checks Chrome A (18800). If a login tab is up, it runs the login-bot; on a persistent real logout it does a HARD
relaunch (kill + fresh-profile launch + re-login), same as `fub_chrome_restart`. It only alerts a human via
`notify.sh` if even that can't re-auth (a genuine 2FA/password wall).

<Warning>
**The watchdog must WATCH, never HARM (Caleb 2026-07-08 — the "watchdog was the killer" fix).** Its health probe
`isCdpConnectable()` used to `connectOverCDP()` then `browser.close()` — and on this Chrome build that close
**tore down the browser's live pages.** So every 60s the watchdog killed the actively-scraping worker, mis-read
the wreckage as "WEDGED," and hard-relaunched Chrome — **568 relaunches**, an endless self-inflicted crash loop
that looked exactly like "the scraper won't stay logged in." Two guards now prevent it, permanently:
1. **Passive probe** — a raw DevTools websocket open/close (Node global `WebSocket`) that never calls
   `browser.close()`, so probing a healthy Chrome is harmless.
2. **Fresh-log guard** — if the `fub-backfill-a` log is fresh (the worker wrote scrape output seconds ago), the
   watchdog **never probes or relaunches**: an actively-producing worker is definitive proof CDP is healthy.

Rule for anyone touching the watchdog: a monitor may **look** but must never **close/relaunch a browser that has a
live worker.** Code: `fub-backfill-watchdog.mjs` — `isCdpConnectable` + the `workerFresh` guard in `check()`.
</Warning>

**Manual override (rarely needed):**
```
mcp__camp-david__fub_chrome_restart
```
Kills + relaunches the FUB Chrome (port 18800), re-logs-in, returns `logged_in: true` + the landing URL. Use it if
you want an instant fix instead of waiting for the ≤2-min watchdog cycle, or if a 2FA alert fired. **Before
2026-07-03 the watchdog only ran the in-session login-bot and, on a real logout, gave up + alerted a human — so it
could sit spinning until someone ran this tool by hand. That gap is now closed.**

## The scrape-on-entry doctrine

The moment a new Zillow inquiry fires the FUB `eventsCreated` webhook, the handler spins up a **per-lead scrape**
that intercepts FUB's own authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire
timeline** — every home they ever touched (Zillow URL, zpid, MLS, price, **photo**) — in one shot, keyed by FUB's
stable `fub_event_id` into `fub.zillow_home_activity`, then **full-replaces** that person's rows in the Railway
mirror. On demand, re-run it with **`mcp__fub-scraper__fub_scrape_lead`** or **`mcp__camp-david__fub_scrape_person`**.
This is the front door for all new lead data — when a lead's cards look wrong, **re-running the scrape is the first move.**

## What it can't recover

**If a FUB user sets their emails private/hidden in the FUB UI, the DOM scraper cannot recover them** — they read
`[CONTENT HIDDEN]` in the actual UI, so there is nothing to scrape. This is a FUB privacy feature, not a scraper
failure (concrete case: Aaron Foster's UI-hidden emails store `[CONTENT HIDDEN]` permanently). So a large `hidden`
count is **largely expected** and must not be "fixed." What it **can** recover: anything visible in Zach's FUB UI,
including the newest message body via the intercepted timeline API (`lastEmail.bodyHtmlVisibleClean`; a direct
public-API fetch returns `401`).

## Health & runbook

**Fast path (MCP tools):**
- `mcp__camp-david__fub_scraper_status` — running/stuck/down, Chrome alive on 18800, heartbeat, current person.
- `mcp__camp-david__fub_backfill_status` — per-process `health` verdict, heartbeat, `content_status` distribution.
  Read `health` and compare `last_crash_at` vs `last_deploy_at` — a crash older than the last deploy means the fix is holding.
- `mcp__camp-david__fub_chrome_restart` — the session-expired fix (relaunch + re-login).

**Healthy =** `fub-backfill-a` + `fub-webhook-scraper` + `fub-relay` + `fub-watchdog` all `online`; the webhook log
shows `email:N` recoveries on leads that have visible emails; the pending backlog trends down. A big `hidden` count
is **not** automatically bad (UI-hidden emails). A pending backlog in the tens of thousands is **throughput, not a bug**.

**The two known "down but fine" processes:** `fub-webhook-sink` and `fub-sync` are deliberately stopped — don't restart them.

  destroying its own scraper. Root cause chain, each fixed:
  1. **The health probe was destructive.** `isCdpConnectable()` did `connectOverCDP()` + `browser.close()`; on this
     Chrome build that close tears down the browser's live pages. So every 60s the watchdog killed the
     actively-scraping worker mid-lead, mis-read the aftermath as "WEDGED," and hard-relaunched Chrome —

     `browser.close()`) + a **fresh-log guard** (a worker that wrote output seconds ago proves CDP is healthy — the

     FUB email/password form button. FUB is Google SSO now, so it always failed → instant 10-min cooldown. → **Fix:**

  Result: `restarts:0`, throughput **65/hr → ~236/hr**, stable through a hands-off window. Commits `62b28ef234`,
