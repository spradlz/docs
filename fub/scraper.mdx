## One-line truth

Follow Up Boss's **public API masks** message/email/call bodies and never exposes the full historical browse
list. The scraper **logs into FUB as Zach in a real Chrome (CDP)**, reads what a human sees, and writes it to
`fub.*` ‚Äî the store the CRM reads. **If the FUB UI itself hides something, the scraper cannot recover it. That is
not a bug** (see [Hidden emails](#what-it-can-t-recover)).

## Why it exists ‚Äî what the API hides

| Data | API/webhook gives it? | Needs the scraper? |
|---|---|---|
| Property inquiries, viewed/saved (single events), notes, stage, tags, deals | ‚úÖ clean, real-time | ‚ùå |
| Text / email **that one happened** | ‚úÖ (to/from/time) | ‚Äî |
| Text / email / call **body** | ‚ùå masked "hidden for privacy" | ‚úÖ recovers it |
| Call AI summary / transcript | ‚ùå not in API | ‚úÖ |
| **Deep historical Zillow browse list** (every home ever touched + photo) | ‚ùå not in the API at all | ‚úÖ scrape-only |

The webhooks (covered in [Lead Import ¬ß2](/fub/lead-import)) carry the real-time signals; the scraper fills the
two gaps the API can't: **masked bodies** and the **full browse history**.

**Two databases (memorize this before any photo/blank-card debugging):**
- **Camp David `fub.*`** ‚Äî where the scraper **writes**. Ground truth for scraped data.
- **Railway `fub.*`** (`FUB_RAILWAY_DATABASE_URL`) ‚Äî what the **CRM reads**. The CRM never reads Camp David directly.
- Most tables sync via the **relay** (`fub-relay`). **`fub.zillow_home_activity` is OFF the relay** ‚Äî the scraper
  is its **single writer to the mirror** (a per-person full-replace right after it scrapes). So a home's photo
  reaches the CRM **only when the scraper runs for that person**. Blank photo in the CRM ‚Üí check the **mirror**,
  then re-run the scraper for that lead.

**Two Chromes (parallel workers):** Chrome **A** on CDP port **18800** (`~/.chrome-fub-profile`) + Chrome **B** on
**18802** (`~/.chrome-fub-profile-2`). Each is a persistent profile holding a live Google session.

## The login (this is what "stuck / can't log in" means)

FUB's real login is **"Log in with Google"** as `zach@spradlingrealtygroup.com`. The persistent Chrome profile
holds a live Google session, so navigating to the app **auto-completes SSO** ‚Äî no password prompt in the normal
case. Fallbacks if the session lapsed: `FUB_WEB_GOOGLE_PASSWORD` ‚Üí `FUB_WEB_PASSWORD`. A **2FA challenge alerts via
`notify.sh`** (a human must clear it).

When the session expires and auto-login can't re-establish it, the scraper sits **stuck on the FUB login page** and
the workers crash-loop (heartbeat error: *"FUB session expired ‚Äî auto-login failed"*). **The fix is one tool:**

```
mcp__camp-david__fub_chrome_restart
```
It kills + relaunches the FUB Chrome (port 18800), re-logs-in, and returns `logged_in: true` + the landing URL.
The `fub-watchdog` does this automatically every 60s; run it by hand when the status shows the session expired.

## The scrape-on-entry doctrine

The moment a new Zillow inquiry fires the FUB `eventsCreated` webhook, the handler spins up a **per-lead scrape**
that intercepts FUB's own authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire
timeline** ‚Äî every home they ever touched (Zillow URL, zpid, MLS, price, **photo**) ‚Äî in one shot, keyed by FUB's
stable `fub_event_id` into `fub.zillow_home_activity`, then **full-replaces** that person's rows in the Railway
mirror. On demand, re-run it with **`mcp__fub-scraper__fub_scrape_lead`** or **`mcp__camp-david__fub_scrape_person`**.
This is the front door for all new lead data ‚Äî when a lead's cards look wrong, **re-running the scrape is the first move.**

## What it can't recover

**If a FUB user sets their emails private/hidden in the FUB UI, the DOM scraper cannot recover them** ‚Äî they read
`[CONTENT HIDDEN]` in the actual UI, so there is nothing to scrape. This is a FUB privacy feature, not a scraper
failure (concrete case: Aaron Foster's UI-hidden emails store `[CONTENT HIDDEN]` permanently). So a large `hidden`
count is **largely expected** and must not be "fixed." What it **can** recover: anything visible in Zach's FUB UI,
including the newest message body via the intercepted timeline API (`lastEmail.bodyHtmlVisibleClean`; a direct
public-API fetch returns `401`).

## Health & runbook

**Fast path (MCP tools):**
- `mcp__camp-david__fub_scraper_status` ‚Äî running/stuck/down, Chrome alive on 18800, heartbeat, current person.
- `mcp__camp-david__fub_backfill_status` ‚Äî per-process `health` verdict, heartbeat, `content_status` distribution.
  Read `health` and compare `last_crash_at` vs `last_deploy_at` ‚Äî a crash older than the last deploy means the fix is holding.
- `mcp__camp-david__fub_chrome_restart` ‚Äî the session-expired fix (relaunch + re-login).

**Healthy =** `fub-backfill-a/b` + `fub-webhook-scraper` + `fub-relay` + `fub-watchdog` all `online`; the webhook log
shows `email:N` recoveries on leads that have visible emails; the pending backlog trends down. A big `hidden` count
is **not** automatically bad (UI-hidden emails). A pending backlog in the tens of thousands is **throughput, not a bug**.

**The two known "down but fine" processes:** `fub-webhook-sink` and `fub-sync` are deliberately stopped ‚Äî don't restart them.
