## Verdict in one paragraph

The **core pipeline is healthy**: one hardened Chrome, the watchdog fix is holding (zero crash-loops since
2026-07-08), the backfill drained the book (49,059 rehydrated / **0 pending**), and the Camp David → Railway relay
is **real-time to the millisecond** with the notify triggers enabled on the mirror. What's broken is the
**edges**: the "real-time" rehydrator has been silently dead for ~6 weeks over a 49k-row graveyard queue, the
Lane-2 body-recovery queue has zero writers and zero readers, the mirror carries 46k ghost rows the master
deleted, monitoring lies twice (a startup-snapshot heartbeat + a pm2-online-but-wedged process), and the Lane-2
queue as designed was **unreachable from the CRM** (Camp David is LAN-only) — relocated to the mirror as part of
the joint plan.

## Findings — ranked

<AccordionGroup>
  <Accordion title="F1 🔴 The Lane-2 queue was unreachable from the CRM — relocated to the mirror (DECIDED)">
    `fub.body_recovery_queue` was created on Camp David, but the enqueuing router runs on **Railway**, which has
    no route to LAN postgres (the tailscale funnel proxies HTTPS only). **Decision (Caleb+Lee 2026-07-09): the
    queue lives on the RAILWAY mirror.** Lee enqueues there natively; Caleb's consumer polls it from Camp David
    (CD→RW connections already exist — the relay does exactly this). The relay can never clobber it because the
    relay is **allowlist-based** (16 named tables) and the queue is simply never added. The empty CD copy is dead
    — dropped only with Zach's typed approval (Rule #0). DDL updated at creation: `resource_type` CHECK extended
    to `('text','email','call_note')` so masked call-notes can queue too.
  </Accordion>
  <Accordion title="F2 🔴 The 'real-time' rehydrator has been dead 6 weeks while pm2 said online">
    `fub-webhook-scraper`: pm2 online, heartbeat last beat **2026-05-30**, queue last drained ~May 9, 270 silent
    restarts, 49,231 pending rows piled up. Nobody noticed because pm2 "online" was trusted over the heartbeat.
    **Fix:** the joint plan retires this path entirely (Lane 2 replaces it); the systems-health cockpit (Phase D)
    must alert on **heartbeat recency**, never process status.
  </Accordion>
  <Accordion title="F3 🟠 The mirror carries 46,046 ghost rows the master deleted">
    `fub.text_messages`: Camp David 400,920 rows vs Railway **446,966** — same id range. The relay never deletes
    (by design, Rule #0), so Camp David's group_sms dedup never propagated. Ghost rows can surface as duplicate
    texts in any CRM thread that reads the mirror. **Fix:** a one-time reconciled cleanup on the mirror (delete
    RW rows whose id no longer exists on CD, table named, count stated) — **needs Zach's typed approval** — plus
    a documented rule that master-side deletes require a mirror-side reconcile pass.
  </Accordion>
  <Accordion title="F4 🟠 Monitoring lies twice">
    (a) The scraper heartbeat computes `total` **once at startup** (`fub-parallel-scraper.mjs:370-373`) and never
    refreshes — today it still displays the pre-lock flood backlog (26,284/25,625 pending) while the real queue is
    **0 pending**. (b) pm2 "online" ≠ working (see F2). **Fix:** heartbeat re-counts the queue each beat (or
    stamps `queue_total_at_start` honestly as a label), and health checks read heartbeats + queue depth, not
    process state. Feeds Phase D probes: queue depth stale &gt;N = RED · webhook recency &gt;30min = RED ·
    consumer heartbeat.
  </Accordion>
  <Accordion title="F5 🟠 ecosystem.config.cjs drift — the pm2 config can't reproduce the live box">
    The ecosystem file still defines the dead `fub-webhook-sink` and does NOT define the live-named
    `fub-webhook-scraper` that `deploy.sh` reloads (`deploy.sh:18-22` vs `ecosystem.config.cjs:56-70`). A fresh
    `pm2 start ecosystem.config.cjs` would start the wrong world. Also: DB creds + FUB creds are hardcoded in the
    file, and `fub-parallel-scraper.mjs:25` imports the lib from a hardcoded `/Users/lee/` path. **Fix:** rewrite
    the ecosystem to exactly the four intended apps (backfill-a, watchdog, relay, and the Lane-2 consumer when it
    ships), env from `.env`, retire dead blocks.
  </Accordion>
  <Accordion title="F6 🟡 Empty-queue exit loop burns pm2 restarts">
    On an empty queue the parallel scraper exits instantly (`:376-380`) under pm2's `min_uptime` (30s), so the
    restart counter climbs forever (280k) and every relaunch pays full startup. Harmless today but it masks real
    crashes inside the same counter. **Fix:** idle-sleep (e.g. 5 min) before the empty-queue exit, mirroring the
    existing `cooldownExit()` error path (`:87-92`).
  </Accordion>
  <Accordion title="F7 🟡 The 216k cleanup (zillow_home_activity dedup) — tool ready, run pending">
    `fub.zillow_home_activity` accumulated ~77% duplicate rows (volatile upsert key + NULL list_position; 1.78M →
    ~411k target) plus URL-as-address rows from the note-parser path. The reversible cleanup tool exists
    (`tools/fub-scraper/dedup-home-activity.mjs`, Zach-approved design 2026-06-27): collapse to one row per
    (person, zpid/address, activity_type), fix the unique indexes, run CD first then mirror. `--execute` is
    Rule #0 — **needs Zach's typed approval in-session** before running.
  </Accordion>
  <Accordion title="F8 ✅ What's confirmed healthy (don't touch)">
    The watchdog fix (passive probe + fresh-log guard) — zero kill-loops since 2026-07-08. The login bot
    (Google SSO, actionable-inputs-first). The relay (ms-level parity, allowlist, never-delete). The
    `peopleCreated` enqueue gate — flood fix held (0 pending). The mirror notify triggers (enabled, transition
    guard verified). Throughput after the one-Chrome consolidation: 65/hr → 236/hr.
  </Accordion>
</AccordionGroup>
