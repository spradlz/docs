## Parse vs scrape — where each piece comes from

| Card data | How it arrives | Scrape needed? |
| --- | --- | --- |
| **Property inquiry / tour request** — address, city, zip, **MLS**, price, beds, message, source | `eventsCreated` webhook → `fub.events.property` (structured by FUB) | ❌ No |
| **Property views / saves** | `notesCreated` webhook ("Property Viewed/Saved: \<url\>") → URL parsed | ⚠️ deep history = scraper |
| **Saved searches** | **No webhook** (Zillow killed it Apr 2026) → an inquiry fires a timeline-walk → **scraper** | ✅ scraper only |
| **Text/email/call bodies** | scraper unmasks | ✅ for the body |

Every webhook FUB fires is logged in `fub.webhook_event_log` (queryable per lead).

## The coarse-type problem

FUB stores **all** lead-initiated property contact under one `type = "Property Inquiry"`. The real intent lives
**only in the message + the `source`**. One classifier (`classifyFubEvent`) resolves it; every surface uses
that one verdict.

## The three request types

Derived from the **message**, never from a FUB event type:

| # | Request type | What the lead is saying | Message signal | Our event |
| --- | --- | --- | --- | --- |
| 1 | **Contact request** | "general question / reach out to me" | *reach out · call me · like to talk* | `contact_request` |
| 2 | **Property inquiry** | "specific question about THIS property" | *interested in \<addr\>* — no tour/callback | `property_inquiry` |
| 3 | **Property tour request** | "I want to tour / see THIS property" | *tour · showing · seeing \<addr\> on \<date\>* | `tour_request` |

<Warning>
**FUB's `General Inquiry` (4,088) is a junk bucket** — 86% blank, the rest mostly phone-call notifications +
Zillow system messages. Parse it: real questions → `contact_request`, phone notices → calls feed, noise → dropped.
</Warning>

## Behavior-Based Follow-Up

- **Time-Based Follow-Up** = fixed cadence ("new lead → text daily ×10 → Nurture Warm → Cold"). The *timeline* drives it.
- **Behavior-Based Follow-Up** = triggered by what the lead **DID**. The *behavior* drives it. Not "hot" — a **behavioral trigger**.

<Info>
**We document, we do not replicate.** Zillow runs its own ML + playbooks inside FUB. We document **what each
signal means** so the team + Myli understand the behavior and run **our** follow-up off it.
</Info>

### The Behavior-Based Follow-Up events

| Behavior | FUB signal | What it means | Our event |
| --- | --- | --- | --- |
| **High-intent buyer** | tag `Zillow High Intent Buyer` | 5× likelier to transact in 3 months | `behavior_high_intent_buyer` |
| **High-intent seller** | tag `Zillow High-Intent Seller` | likely to sell in 3 months | `behavior_high_intent_seller` |
| **Re-engaged buyer** | tag `Zillow Reengaged Buyer` | searching again after ≥14 days inactive | `behavior_reengaged_buyer` |
| **Specific-home interest** | tag `Zillow specific home interest` | repeatedly viewed specific homes | `behavior_specific_home_interest` |
| **ZHL pre-qualified** | the 3 `ZHL connection: …` tags | referred + pre-qual'd by a ZHL LO | `behavior_zhl_prequalified` |
| **Repeat property views** | note "viewed N× in 7 days" | hammering one home | `behavior_repeat_views` |
| **Area interest** | note "now looking in \<ZIP\>" | shifted search area | `behavior_area_interest` |
| **Property tour** | Property Inquiry (tour) | asked to tour a home | `tour_request` |
| **Contact request** | Property/General Inquiry (reach out) | asked an agent to reach out | `contact_request` |
| **Saved search** | scraper → `fub.saved_searches` | saved search criteria | `saved_search` |

**`Zillow Smart List` = the "active now" 13-day window, NOT a behavior.** Add → `behavior_active`, remove →
`behavior_window_closed`.

## 🛬 The Landing — how we ship it without boiling the ocean

The matrix is the map; this is the scope lock. **Two rules:**

<Steps>
  <Step title="Rule 1 — only a request creates a NEW lead">
    A lead is pulled into HouseLoop **only** by a **property inquiry / tour request**. **Behavior events NEVER
    create a lead** — that would flood HouseLoop with events for leads we have no workflows for. The
    lead-creation door stays inquiry/tour only.
  </Step>
  <Step title="Rule 2 — behavior events are first-class, but only for leads already in HouseLoop">
    Once a request has pulled a lead in: **(a) backfill** its entire FUB behavior history into first-class
    `hl_lead_activity` events (same status as a tour request); **(b) forward** — every new behavior event for
    THAT lead is minted in real-time; **(c) scope** — a behavior event for a FUB lead NOT in HouseLoop is
    ignored. No flood, no orphan events.
  </Step>
  <Step title="The UI — the 'Follow Up Boss' backfill (lead detail, tablet view)">
    A **"Follow Up Boss"** control on the lead-detail FUB Activity panel backfills all the lead's FUB history
    into first-class events; everything forward shows under the behavior-based event stream for that lead.
  </Step>
</Steps>

Workflows that *act* on behavior events are built **later, one at a time** — safe, because the events are
already first-class and scoped.

## How each conversion runs

One converter per signal class, all writing through `logActivity()`:

<Steps>
  <Step title="Events">`fub.events` → `classifyFubEvent` → tour / inquiry / contact / rental / seller.</Step>
  <Step title="Notes">`notesCreated` → `property_viewed` / `property_saved` + the insight parsers (`behavior_repeat_views`, `behavior_area_interest`).</Step>
  <Step title="Saved searches">`fub.saved_searches` → `saved_search` (minted, not read-live).</Step>
  <Step title="Tags">`peopleUpdated` / `peopleTagsCreated` → the `behavior_*` events + Smart List window via `fub.tag_history`.</Step>
</Steps>

Each fires at webhook-time (real-time) + a per-lead backfill (gated to leads already in HouseLoop — see The Landing).
