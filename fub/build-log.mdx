## 2026-06-28 — ROOT CAUSE FOUND: the photo "race" (why Alberto is blank, James is full)

**This supersedes the earlier "no fallback / Listing-only" diagnosis.** Grounded in live data + code:

- **James Ross:** 442 `property_view` cards, **432 carry the Zillow photo.** Alberto Lopez: 113 cards, **0 carry a photo** — even though his `fub.zillow_home_activity` has **199 rows, all 199 with a photo** (FUB-hosted `s.followupboss.com/properties/…` URLs the scraper captured). The photos exist; they're stranded.
- **The minter is INSERT-ONLY.** Cards are written by `backfillFubBehavior` → `convertHomeActivity` (carries `photo: h.photo`) → `logActivity`, which is **idempotent on `(source_table, source_id)`**. A card copies the photo **only if it already exists in `zillow_home_activity` at mint-time**, and **never updates it later.**
- **The import does NOT wait for the scrape.** The webhook dispatches the CRM import via `setImmediate(() => dispatchFubImport(...))` (`apps/dashboard/server/fub-webhooks.ts`) — it races the CDP complete-home-activity scrape. The import reads whatever's in `zillow_home_activity` *at that instant* and mints.
- **Net:** James was fully pre-scraped (test lead) → minted with photos. Alberto's live import **beat his scrape** → minted blank → scrape later loaded all 199 photos → insert-only minter never backfilled them → **blank forever.**
- **Documented intent (the doctrine Zach named):** "one-time complete scrape of every house on entry" — `Scraper — complete home activity via API intercept` (FUB `/api/v1/events?hasProperty=true`, 242→534 on James). It IS deployed; it just isn't *gated before the mint*.

**The fix (two parts):**
1. **Re-sync (heals existing blank cards now):** patch `photo` onto already-minted cards from `zillow_home_activity` (match by `fub_event_id`, else address). The one thing insert-only can't do. Fixes Alberto's 113 immediately; makes the race harmless going forward.
2. **Gate the mint on the scrape (prevents recurrence):** don't mint behavior cards until the complete-home-activity scrape has finished — the same "scrape-first, wait up to 120s" pattern the inbound-text alert already uses. Every new lead becomes the James path: scrape first, mint second.

## What we're building (per the PRD)

Every behavioral event — FUB/Zillow today, native HouseLoop tomorrow — is a **first-class `hl_lead_activity`
event** rendered through **ONE catalog-driven card**, identical on **All · Website Activity · Follow Up Boss
Activity**, each carrying **house (address + photo + link) · what it is · source (`Zillow.com` | `HouseLoop.ai`)**.
One-to-one Zillow↔HouseLoop parity. FUB is the test data; this is the HouseLoop event model.

## The card system — TWO render tiers (no new components, catalog-driven)

`FeedRow` (exported from `PixelFeedStream.tsx`, reused by `CenterLane.tsx`) draws every card from the one
`activity-badges.ts` catalog. Exactly two shapes:

- **HERO** — `ActivityHeroCard.tsx`. Big card: accent border · actor pill · title · "via" source badge · time ·
  photo/icon tile · address + facts ($price · bd · ba · MLS#) · `View on HouseLoop │ View on Zillow` ·
  quote/message · calendar chip. Fired by `request:true`, `registration`, and `hero:true`.
- **COMPACT** — the row card. Thumbnail (80×80) · actor pill · label+icon · "via" badge · time · address ·
  facts · View links. Fired by everything else.

**HERO is ONLY for property tours, inquiries, and registrations (Zach LOCKED 2026-06-28).** Everything else —
all behavior triggers / signals / ML flags, saved searches, comms — is a regular COMPACT card. (The earlier
"all behaviors are hero" call was reversed.) **Every HERO request card looks IDENTICAL to the canonical Tour
Request card; the ONLY difference is a Tour Request shows the date as a calendar chip.**

## Image pull-in — why photos are blank + the fix (Issue B, CORRECTED 2026-06-28)

**The photo source is the Zillow photo the scraper already captured** into `fub.zillow_home_activity`
(`s.followupboss.com/properties/…`). **NO Google Street View. NO Google Maps. The Zillow listing IS the
picture** (Zach LOCKED 2026-06-28). The real problem is the **race** (see "ROOT CAUSE FOUND" above), not a
missing fallback — the photos exist, they're just stranded because the insert-only minter ran before the scrape.

The mint chain for a home card's photo: HouseLoop `Listing` photo (in-market) → the home's **Zillow photo from
`fub.zillow_home_activity`** (match by `fub_event_id`/zpid, else address) → if still none, the complete-home-
activity **scrape hasn't run yet** → that's the gate to fix, not a third image source.

**The fix:**
1. **Re-sync** — patch `photo` onto existing minted cards from `fub.zillow_home_activity` (the insert-only minter
   can't update; this can). Heals Alberto's 113 blank cards immediately.
2. **Gate the mint on the scrape** — mint behavior cards only after the complete-home-activity scrape finishes
   (the "scrape-first, wait up to 120s" pattern the inbound-text alert already uses).
3. **Junk filter** — drop an "address" that's a URL / doesn't start with a house number (`realscout.com/agent/...`)
   at the mint layer + the backfill SQL, so URL-junk never becomes a card.

Net: **every home card shows its real Zillow picture**; no Street View; URL-junk never renders.

## How a card is created — the trigger path (real, traced on Juan Martín Munguía 42729)

A Zillow lead's card is born from a webhook, not a poll. The chain, grounded in real data:

1. **Source** — a Zillow in-app message / inquiry lands in FUB (e.g. Juan → Ben Froeschl: "interested in touring
   15111 E 44th St S on Jun 26 at 12pm", source **Zillow Preferred**, MLS# 2626844). FUB fires an `eventsCreated` webhook.
2. **Pull-in decision** — `apps/dashboard/server/fub-webhooks.ts` `EVENT_IMPORT_TRIGGER` maps the event type
   (`Property Inquiry → 'inquiry'`, `Virtually Toured Property → 'tour'`, `Seller Inquiry → 'seller'`,
   `Registration → 'new_lead'`) → `dispatchFubImport(personId, trigger)` → CRM `/api/fub/import` → Inngest **`fub/lead.import`**.
3. **The import mints two kinds of row:**
   - **The canonical request card** — step 2.6, `kind` from `classifyFubEvent(message)`, `source_table='fub_request'`,
     `source_id='fubreq|<fub.events.id>'`, ONE per inquiry, home resolved by MLS# → photo + `/home` link. **The card we keep.**
   - **A registration door log** — step 3, `kind='registration'`, `source_table='fub_lead_import'`, source "VIA HouseLoop".
     **This DUPLICATES the request card** (tracked issue I2).

**Proven on Juan:** 2 `Property Inquiry` events (Zillow Preferred, MLS# 2626844) → 2 `tour_request` cards
(`fubreq|1786386`, `fubreq|1786387`) + 2 `registration` cards. His home (MLS# 2626844) IS on HouseLoop → photo +
blue link resolve — exactly why Juan's cards look right and Alberto's (off-market / relisted MLS#) don't.
