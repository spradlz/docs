## What we're building (per the PRD)

Every behavioral event — FUB/Zillow today, native HouseLoop tomorrow — is a **first-class `hl_lead_activity`
event** rendered through **ONE catalog-driven `ActivityHeroCard`**, identical on **All · Website Activity ·
Follow Up Boss Activity**, each carrying **house (address + photo + link) · what it is · source
(`Zillow.com` | `HouseLoop.ai`)**. One-to-one Zillow↔HouseLoop parity (lone asymmetry: `liked_home` is
HouseLoop-only; pre-qual = ZHL ↔ Lending Loop). This is the HouseLoop 2 event model; FUB is the test data.

## Where it stands on James (lead 25246ca6 / fub 50308)

The **data is there** — 397 events minted (274 `property_view`, 41 `save_home`, repeat/area/saved-search/tags).
What's wrong is the **render**: "Viewed Home" rows show no address/photo/source; only the Tour Request renders as
the correct hero card. The build closes that gap (PRD §6 P1–P7).

## The two-writer split (still holds — why there are no double tours)

The import owns the **request spine** (keyed `fub_request` / `fubreq|<id>`); the **backfill owns behavior only**
(views/saves/notes/searches/tags). `logActivity` dedupes on `(source_table, source_id)`, so re-runs add zero rows
and the two never double-log.
