## THE ONE LAW (this is the whole point)

**When a lead is pulled into HouseLoop, their activity feed must keep flowing — the HouseLoop lead stays LOCK-STEP
in sync with Follow Up Boss, forever. A save, a view, a saved search, an in-app message, a text — anything the lead
does in FUB after pull-in appears in HouseLoop. There is no reason not to.**

If a pulled-in lead's HouseLoop activity is behind what FUB shows, that is a bug — every time.

## 🎯 THE TRIGGER-EVENT DOCTRINE (Zach, 2026-07-16 — the target architecture)

This is the north star for the whole FUB pipeline. The current code does **not** fully match it yet (see the
honest status + the audit checklist at the bottom) — this section is the TARGET we are hardening toward.

1. **A trigger event fires EVERY time something happens.** View a home 50 times in a day = **50 trigger events.**
   Not gated, not deduped, not sampled. Every single thing FUB gives us is one trigger event.
2. **The trigger event is the ONE spine — everything is downhill from it:**
   `TRIGGER EVENT → notification · workflows · the re-scrape · the activity/website-activity feed`.
   Today these are three decoupled systems (the `hl_trigger_fire` ledger, the `hl_lead_activity` feed, the
   `hl_notification` card). The target is **one thing**: the trigger event is what fires, and the notification,
   the feed, the workflow, and the scraper all hang off it.
3. **Workflows attach to trigger events; rules attach to workflows.** That is the dependency chain.
4. **A trigger event is NOT the same as "Creates Lead."** Creates-Lead is a **separate** toggle, currently OFF on
   purpose (we do not want FUB auto-creating leads). **Trigger events still fire when Creates-Lead is off.**
   Creates-Lead decides only whether a *lead is born* — never whether the *trigger fires*. When Creates-Lead IS on
   for an event (e.g. "viewed a home"), the FIRST house they view creates the lead and HouseLoop backfills the
   person's ENTIRE FUB history.
5. **The comms policy (intentional, locked):** website activity (views/saves/shares/searches/inquiries/in-app/
   insights) is backfilled in full AND kept lock-step forever. **Texts, calls, and emails are deliberately NOT
   backfilled into the lead-detail activity feed, and new ones are not leaked into it either** — HouseLoop's
   lead-detail stays clean of raw comms. **The FUB Context section on the lead shows ALL comms** (backfilled +
   forward). One line: *activity feed = website activity, lock-step; FUB Context section = all comms.*
6. **~23–24 trigger events, one per FUB signal**, each driving the notification + the scraper + the feed + any
   attached workflow from the ONE trigger event.

## The pipeline in one picture

```
Zillow (Preferred · Rentals · RealScout)
        │  inquiries · tours · views · saves · saved searches · in-app · texts/calls/emails · ML intent
        ▼
   FOLLOW UP BOSS  ── the hub. FUB owns the signal first.
        │
   ┌────┴─────────────────────────────────────────────┐
   │ NATIVE WEBHOOK (real-time trigger)                 │  SCRAPER (Chrome on Camp David) — the DUMB helper
   │ crm.houseloop.ai/api/fub/webhook → fubWebhookRouter│  recovers ONLY what the API hides:
   │ 20 subscriptions, all Active                       │   · masked text/email/call BODIES (Lane 2)
   │ stores every signal in the mirror + notifies       │   · the deep Zillow BROWSE list (views/saves + photos)
   │                                                     │   · saved searches (timeline-only)
   │                                                     │   · Zillow in-app messages (inbox-app sweep)
   └────┬───────────────────────────────────────────────┘
        ▼   relay (Camp David → Railway, ms lag)
   fub.*  on the RAILWAY MIRROR  ── the ONE store the CRM reads
        │  backfillFubBehavior / convert-fub-signal → logActivity()
        ▼
   hl_lead_activity  ── OUR event store (the All + Website Activity feed) — forward-only
```

**The split, in one line:** the **webhook** is the real-time source of truth for *what happened*; the **scraper**
is a **dumb helper** our system calls to fetch the few things FUB's API hides. The scraper never decides what's new.

## What FUB actually gives us, per activity (verified live 2026-07-15)

The core question for lock-step sync: for each thing a lead does, **what signal do we get, and where does the data
live?**

| Activity | Real-time signal | Where the DATA lives | Notes |
|---|---|---|---|
| **Property inquiry / tour / contact** | ✅ `eventsCreated` webhook (full clean property) | `fub.events` | The one signal that carries its own complete data — no scrape needed. |
| **Saved a home** | ✅ `notesCreated` webhook — "Property Saved: `<zpid-url>`" | authoritative list = scraper's `fub.zillow_home_activity` | The **note is lossy** — a bare zpid URL, **no address/price/photo**. The note is a *"go re-scrape"* trigger; the real row comes from the scraper's `GET /api/v1/events?hasProperty=true` intercept. |
| **Viewed a home** | ✅ `notesCreated` webhook — "Property Viewed: `<zpid-url>`" | same as above | Same lossy-note rule. |
| **Saved search** | ❌ **no dedicated webhook** (its `eventsCreated` died Apr 23) | scraper-only → `fub.saved_searches` (read off the FUB activity timeline) | Detect that it happened via the **person activity heartbeat** below. |
| **Zillow in-app message** | ❌ not in the API/webhooks at all | inbox-app sweep → `fub.text_messages` (`zillow_inapp`) | Captured by `fub-inapp-sweep` → `/api/fub/body-recovered`. Full flow in [Trigger Events](/fub/trigger-events). |
| **Text / email / call — that it happened** | ✅ `textMessagesCreated` / `emailsCreated` / `callsCreated` | `fub.text_messages/emails/calls` | Bodies masked (~7% SMS, most email, call notes) → Lane-2 recovery. |
| **ANY activity, per person** | ✅ **`GET /v1/people?sort=-updated`** — the activity heartbeat | — | Every person ordered by most-recent activity, timestamps to the second, cursor-pageable. This is the *"something happened to person X"* signal for anything without its own webhook (e.g. saved search). Proven: 48,962 people, newest-first. Mirrors the in-app sweep's `sort=-lastInboxAppMessage`. |

**The rule that falls out of this table:** we **never blind-rescrape**. A real signal (a `notesCreated` webhook, or
a bump on the `sort=-updated` heartbeat) tells us *person X did something* → we surgically re-scrape **that one
person** → the dumb scraper fetches the details → they mint into the feed.

## 🔴 The gap found 2026-07-15 (why leads go stale after pull-in)

Traced end-to-end on **Shamir Pearson (FUB 50442)**, who went dark in HouseLoop after his tour request even though
FUB had a day of newer activity:

1. He saved `2435752` + viewed `2442158` on Jul 14 → FUB notes fired → `notesCreated` webhook stored them in
   `fub.notes` ✅ (the owner alert fired).
2. Those notes are **bare zpid URLs with no address** → the note→`zillow_home_activity` parser can't build a real
   row → nothing landed in the mirror.
3. The enrichment needs a **re-scrape** of his browse feed — but **an existing HouseLoop lead never gets
   re-scraped on activity.** The per-person priority scrape runs **once, at initial mint only** (the `created`
   gate in `fub-lead-import.ts`); the comment says it outright: *"Only NEW behavioral FUB mints wait; existing-lead
   + backfill skip."* The old real-time re-scrape worker (`fub-webhook-scraper`, drained `fub.rehydration_queue`)
   was **retired ~May 30**; its "high-priority lane so an active lead scrapes in seconds" replacement was **spec'd
   but never built** (see the lead-import changelog).
4. So his mirror froze at his mint-time scrape, and his 5 Jul-14 saved searches (scraper-only) never arrived.

**One-line root cause:** *after pull-in, a lead's ongoing Zillow behavior **notifies** but never triggers the
per-person re-scrape the feed depends on.* The alert path works; the **data-into-the-feed path is severed**.

The **manual "sync" the UI used to expose** (`POST /api/fub/rebackfill` → re-runs `backfillFubBehavior`, minting a
lead's fresh FUB activity into `hl_lead_activity`) is the **back half** of the loop and still exists — but with no
automated trigger in front of it, keeping a lead in sync became a button nobody presses. Pulling that button out
removed the only way the sync ran by hand. The fix is to make it automatic, so there's no button to forget.

## ✅ The fix — the activity-sweep (lock-step sync, automated)

Build the active-lead re-scrape the lead-import doc listed as NEXT. It reuses the exact rail the in-app sweep
already runs on — no new infrastructure:

1. **Detect** — a registered Inngest tick polls `GET /v1/people?sort=-updated` for people with `updated > cursor`
   (everyone active since the last tick), **filtered to HouseLoop leads** (`fub_person_id ∈ hl_leads`). PLUS the
   instant lane: a `notesCreated` (save/view) webhook for a HouseLoop lead enqueues immediately, without waiting
   for the poll.
2. **Enqueue** — a targeted per-person re-scrape job into `fub.body_recovery_queue` (the **same queue the live
   `fub-body-recovery` consumer already drains** for the in-app sweep sentinel).
3. **Scrape** — the dumb scraper re-walks that ONE person → refreshes `fub.zillow_home_activity` (views/saves) +
   `fub.saved_searches`.
4. **Mint** — the scraper POSTs `POST /api/fub/rebackfill` → `backfillFubBehavior` mints the new activity into
   `hl_lead_activity` → it appears in **All + Website Activity**, in step with FUB.

Debounced + HouseLoop-leads-only → surgical (a few per minute for active leads), never a 49k re-walk. This one
bridge keeps every pulled-in lead lock-step with FUB and rides the proven in-app rail.

### ⚠️ The re-scrape is necessary but NOT sufficient — mint note-only saves from the note (2026-07-15)

Live-testing on Shamir surfaced a deeper truth: **his 07-14 saves/views were NEVER in FUB's browse feed** (the
`GET /api/v1/events?hasProperty=true` list the scraper reads) — they existed ONLY as `notesCreated` "Property
Saved/Viewed: `<bare-zpid>`" notes. So **no re-scrape can ever recover them.** The fix: `convertNote`
(`lib/fub/convert-fub-signal.ts`) now emits a `save_home`/`property_view` straight from the note, and
`backfillFubBehavior` resolves the bare zpid → the full home (address/price/photo) from the person's OWN captured
rows for that zpid. **The scraper stays primary** for any view it has (richer, real photo); **notes fill only the
gaps** the browse feed misses — so note-only saves land AND a view never renders twice. Proven: Shamir's 07-14
save (4224 E 104th Ter) + view (10516 Richmond Ave) now mint, zero doubles.

### The double-card dedup + the in-app surfacing
- **Double view card:** the same view was written to `fub.zillow_home_activity` by two writers (the note-parse id
  `146xxx` and the scraper browse-feed id `18xxxxx`) — a fub_event_id key double-minted it. Fixed by deduping the
  mint query on the LOGICAL view identity (home + type + minute), keeping the richer row.
- **Zillow in-app → All + Messages** (not just the FUB tab): still pending on this pass.

## Sync control surfaces (reference)

- **The master switch:** `hl_fub_webhook_settings._global.enabled` (`POST /api/fub/sync-master`) — the ONE breaker
  for the whole FUB pipeline. Verified **ON** 2026-07-15. The UI banner lives on the Trigger Events board
  (`FubChannelRows`). One flag the UI shows and the runtime obeys — it must never orphan (that killed all FUB
  notifications for hours on 2026-07-10).
- **Per-lead re-mint:** `POST /api/fub/rebackfill` (shared-secret gated) — re-runs `backfillFubBehavior` for a lead;
  idempotent, de-duped, never mints. This is the callback the scraper fires when a re-scrape finishes.
- **The per-event toggles:** `hl_fub_webhook_settings` (Sync · Creates Lead · Workflow · Notify) — deep law in
  [Trigger Events](/fub/trigger-events).
