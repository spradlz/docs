### тнР The scrape-on-entry doctrine тАФ the real-time per-lead intercept (THE entry mechanism) ЁЯФТ
**This is how every new lead's full history reaches us, in ~half a second.** The moment a new Zillow inquiry fires
the FUB `eventsCreated` webhook, the handler **spins up a per-lead scrape on Camp David** that intercepts FUB's own
authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire** timeline тАФ grabbing
**every home the lead ever touched: the Zillow URL, zpid, MLS, price, and the photo** тАФ in one shot (242 тЖТ 534 homes
on the test lead James Ross). It writes each home keyed by FUB's stable `fub_event_id` into `fub.zillow_home_activity`
via `saveZillowHomeActivity` (`tools/fub-scraper/fub-scraper-lib.mjs`). Trigger paths: the webhook auto-enqueues it
(`fub.rehydration_queue` тЖТ `fub-webhook-scraper`); on demand, **`mcp__fub-scraper__fub_scrape_lead` /
`mcp__camp-david__fub_scrape_person`** re-run the full intercept for one person. **It is deployed + LIVE.** This
scraper is the front door for ALL new lead data тАФ when a lead's cards look wrong, re-running it is the first move.

### ЁЯЧДя╕П TWO databases тАФ and `zillow_home_activity` is OFF the relay (read this before any photo/home-card debugging) ЁЯФТ
There are **two** FUB stores, and they are NOT kept in sync the same way for every table:
- **Camp David `fub.*`** (`DATABASE_URL` on the box) тАФ where the scraper WRITES. Ground truth for scraped data.
- **Railway mirror `fub.*`** (`FUB_RAILWAY_DATABASE_URL`) тАФ **what the HouseLoop CRM READS** (`fubQuery`,
  `LeadContextPanel`, the import, the card minter). The CRM never reads Camp David directly.

Most tables reach the mirror via the **relay** (`fub-relay`, upsert-by-id). **`fub.zillow_home_activity` is the
exception тАФ it is OFF the relay (Zach 2026-06-27).** The **scraper is its SINGLE WRITER to the mirror**: right after
`saveZillowHomeActivity` writes Camp David, it does a **per-person FULL REPLACE to the mirror**
(`syncPersonHomeActivityToMirror` тАФ `DELETE person rows + INSERT the fresh set`). So a home's photo reaches the CRM
**only when the scraper runs for that person.** If the scraper hasn't re-run since the photos were captured, **Camp
David has the photos but the mirror does not** тАФ and the CRM, reading the mirror, shows blank.

### тЬЕ The blank-photo bug тАФ ROOT CAUSE FOUND + FIXED (2026-06-28)
Every blank photo + stale lead traced to **ONE bug**: `mirrorUrl()` in `fub-scraper-lib.mjs` resolved its `~/.env`
fallback with **`require('node:fs')` тАФ but that file is an ES module, so `require` is undefined and threw**, silently
returning `null`. The per-person mirror sync then no-op'd on **every on-demand scrape** (which doesn't set
`RAILWAY_DATABASE_URL`). pm2 scrapers "worked" only because their env sets the var. So a scrape wrote Camp David
perfectly but **never updated the Railway mirror the CRM reads** тЖТ blank. *Proven:* Alberto Camp David **199/199**
photos, mirror **829 rows / 0 photos** тАФ until the fix flipped it to **207/207**.

**Fixed (shipped):**
- `mirrorUrl()` uses the ESM-imported `fs` (no `require`).
- `syncPersonHomeActivityToMirror` now **sources the FULL set from Camp David** (not the flaky in-memory scrape
  result, which the intercept returns non-deterministically тАФ 0/4/199 on the same lead), **always runs**, **never
  shrinks** the mirror below Camp David, and **fails LOUD** (`тЬЕ mirror replaced: N` / `тЭМ mirror sync FAILED` тАФ never
  silent). The loud log is what surfaced the `require` bug on the first run.
- **Card heal:** `resyncHomePhotos` + `stampActivityPhoto` (the sanctioned append-only exception) stamp photos onto
  already-minted cards (all `evt`/`zpid`/slug source_id variants); for leads minted from the old broken mirror,
  **re-import** (`fub/lead.import {backfill}`) mints clean `evt`-keyed cards, then a Rule#0-approved delete removes the
  stale `fub_home:` non-evt duplicates. **Junk guard** in `convertHomeActivity`: an address that is a URL / agent-page
  (`realscout.com/users/...`) / doesn't start with a street number **never mints a card**.

> **One-line debug rule:** blank home card тЖТ check the **Railway mirror** (`FUB_RAILWAY_DATABASE_URL`), not Camp
> David. 0 photos there тЖТ re-run `fub_scrape_lead` (now mirrors loud); cards stale тЖТ re-import the lead.

### ЁЯФС Where property VIEWS actually come from тАФ the internal feed, NOT the notes (Zach 2026-06-28)
FUB has **no property-view webhook** (the public "Viewed Property" event died May 21). Your **"Property Viewed" note
automation** fires `notesCreated` тАФ but it **lags and misses** (proven: James had 6 views, Alberto 5, newer than their
newest note; rentals/heavy browsers fall behind). The **complete, ordered, current** view list lives ONLY in FUB's

intercept, never the notes.** The note webhook is fine as a *trigger* ("go re-scrape"), but the data is the feed.

## 3. The import workflow ЁЯФТ

1. **Source** тАФ a Zillow in-app message / inquiry lands in FUB (e.g. "interested in touring 15111 E 44th St S on

   native webhook now; see [FUB Trigger Events](/guides/trigger-events).)
3. **The import mints (the key steps):**
   - **2.6 тАФ the canonical request card** тАФ `kind` from `classifyFubEvent(message, source)`,
     `source_table='fub_request'`, `source_id='fubreq|<fub.events.id>'`, ONE per inquiry, home resolved by MLS# тЖТ
     photo + `/home` link. **The card we keep.** (`fub.events` is now stored **unconditionally** by the router's
     `storeRecord`, so inquiries/tours always land for this step тАФ 2026-07-10.)
   - **2.7 тАФ behavior + comms backfill** тАФ `backfillFubBehavior` mints the lead's FUB behavior history

     FORWARD-ONLY** (floored at the lead's `created_at`) тАФ a pulled-in lead starts CLEAN, and only new comms flow

   - **2.8 тАФ Myli context is NOT built on import.** `buildLeadContext` was removed from the import path (2026-06-29):
     facts refresh **before Myli communicates** (the conversation path / a pre-outreach workflow step / on-demand
     when the lead is opened), never on a data pull-in. See ┬з6.
   - **3 тАФ registration door log** тАФ `kind='registration'`, `source_table='fub_lead_import'`, source "VIA HouseLoop".

- **Link + source + owner:** `hl_leads.fub_person_id`; lead source = `Zillow.com`, platform = `follow_up_boss`; a
  lead is **never ownerless** тАФ `ensureLeadAiOwner` falls back to the platform default AI agent (Myli) when no human

### тнР The Watermark Law тАФ read-once, merge-forward (universal) ЁЯФТ
Governs **every** LLM rollup (dossier, facts, Hot Buyers, any summary): **never re-process data already consumed.**
The dossier stores its text + a `source_cursor` (high-water mark). Each regen: read only the delta
(`WHERE synced_at > source_cursor`) тЖТ merge into the existing dossier (don't re-summarize) тЖТ advance
`source_cursor = max(synced_at)`. First pull = full history (Sonnet, cursor = now); every pull after = the delta,
merged with Haiku. **Tokens paid once per row, ever.**

<Warning>
**The gotcha:** watermark on **`synced_at` (when WE ingested the row), NOT the event's own date.** A 2-year-old
home view the scraper *just* backfilled is **new to us** тАФ `synced_at > cursor` catches it. Watermark on the event
date and you'd silently skip every freshly-backfilled historical row.

## The principle (LOCKED 2026-07-09 by Zach)

**Notifications are the source of truth. The scraper is a dumb, on-demand grabber of HIDDEN content only.**

The native HouseLoop webhook is what decides *what happened*; when the thing that happened has content FUB hides
(a masked message body), the notification layer *tells the scraper* to go recover exactly that one body тАФ nothing
else. The scraper never decides what's new, never crawls, never re-scrapes on a record edit. This replaces the
old "push for speed, pull for truth" dual-path / differential-reconciler design, which is **dead** (it read the
lagging mirror and flooded Zach with 2022-era activity тАФ see the post-mortem below).

## Vocabulary (LOCKED 2026-07-07)

- **Integration** = the *pipe* a lead flows through тАФ **HouseLoop.ai** (our website pixel) or **Follow Up Boss**
  (the Zillow-connected CRM). Managed at **Settings тЖТ Lead Integrations** (`/settings/lead-platform`).
- **Lead Source** = the *brand* on the lead тАФ Zillow, Referral, etc. A source lives *under* an integration.
- **Trigger Event** = a *door* тАФ a thing that can happen (a tour request, an inbound text, a property save) that
  can start a lead, run a workflow, and/or fire a notification. One row in **`hl_fub_webhook_settings`**,
  configured at `crm.houseloop.ai/settings/trigger-events`.

## The four independent toggles (this is the model)

Each FUB trigger event carries **four independent switches** тАФ none implies another:

| Toggle | Column | What it controls |
|---|---|---|
| **Sync** | `enabled` | Do we ingest/process this FUB signal at all. Off = we don't even mirror it. |
| **Creates Lead** | `creates_lead` | Does firing this door mint (or match) a HouseLoop lead. |

## The 3-lane architecture

Every FUB event enters the native webhook and `fubWebhookRouter` sorts it into exactly one lane:

### Lane 1 тАФ Visible activity (~95 percent): router notifies directly, scraper never touches
Home saved/viewed (`notesCreated`), inquiries (`eventsCreated`), visible SMS, call metadata, appointments,
tasks, deals. (In-app messages are NOT here тАФ they are the inbox-app sweep lane below.) The API hands us clean
content тЖТ the router classifies it and fires
the owner-only notification immediately.

### Lane 2 тАФ Masked comms: the ONLY thing that wakes the scraper
The three masked cases тАФ masked SMS bodies, email bodies, and call notes. The flow is a **surgical body-fetch**,
never a re-scrape:

```
masked textMessagesCreated / emailsCreated  тЖТ  fubWebhookRouter writes metadata + enqueues ONE job
      тЖУ fub.body_recovery_queue (on Camp David тАФ the relay is CDтЖТRailway one-way, so the job lives CD-side)
      { person_id, resource_type: 'text' | 'email', resource_id, requested_at }
      тЖУ the scraper drains the queue тАФ reads THAT ONE body from the FUB UI
      тЖУ writes the recovered body тЖТ flips content_status  hidden тЖТ visible
      тЖУ trg_fub_new_message fires  (now AFTER INSERT OR UPDATE, transition-guarded, text + email)
      тЖУ pg_notify('fub_new_message') тЖТ the native notify path
   тЖТ  HouseLoop notification fires WITH the body
```

**No metadata-only notify for a masked body** тАФ a card with no message is useless, so we hold the notify until
the body is recovered. Born-visible content (in-app, visible SMS) is Lane 1 and notifies on the INSERT.

### Lane 3 тАФ Record edits: update the person, never scrape
`peopleUpdated` / `peopleStageUpdated` / `peopleTagsCreated` / `peopleDeleted`. These update the person record
only. **They never enqueue a scrape.** (A stage change may fire `stage_changed` if the Notify toggle is on; that
is a notification, not a scrape.)

## тнР How an in-app (Zillow Messages) message becomes a trigger event

Zillow in-app chat is the ONE signal FUB does **not** expose through its API or webhooks (proven 2026-07-09 on
James Ross: his in-app messages carry synthetic ids and never appear in `GET /v1/textMessages`; no
`textMessagesCreated` fires). They live in FUB's **"Zillow Messages" inbox app** (`inboxApp id 1`, created by
the original account owner's seat тАФ `createdById 1`, which is why a departed owner's Zillow connection still
fronts these leads). So capture is: find the people via the REST API, scrape their pages for the bodies тАФ all
lawful, gated, and Inngest-visible:

1. **Detect тАФ the REST API, not the inbox SPA (proven 2026-07-09):** message BODIES aren't in the API, but WHO
   has in-app activity IS: **`GET /v1/people?sort=-lastInboxAppMessage&fields=id,lastInboxAppMessage`** returns
   every person ordered by their last in-app message, with the timestamp. Reliable, no browser, cursor-pageable.
   (The FUB inbox web endpoint `/api/v1/inbox/тАж/conversations` **hangs** on any direct/replay fetch and its
   virtualized list won't page on scroll тАФ only the app's own live XHR ever fires; the people-sort is the robust
   surface.) Incremental takes only people newer than our mirror's newest `zillow_inapp`; backfill takes all.
   The registered Inngest cron **`fub-inapp-sweep-tick`** (Run Log `comms` lane; cron `*/3` ARMED since
   2026-07-09 sign-off) enqueues one sentinel `inbox_sweep` job into the Railway FUB mirror queue тАФ
   `UNIQUE(resource_type,resource_id)` = at most one sweep pending (no pile-up).
2. **Extract (reuse, never reinvent):** the Camp David `fub-body-recovery` consumer claims the job and runs
   **`tools/fub-scraper/fub-inapp-sweep.mjs`**, which тАФ for each discovered person тАФ reuses the proven extractor
   in `fub-scraper-lib.mjs` (`scrapePersonFull` + `matchAndUpdateMessages`: `scrapeTextContainers` classifies a
   timeline row `zillow_inapp` by its **`zillow.png`** circle-icon; the synthetic id
   **`900000000 + personId*1000 + seq`** dedupes against the existing rows, `ON CONFLICT (id) DO NOTHING`). The
   relay carries the new born-visible rows Camp David тЖТ Railway.
3. **Notify (native, gated тАФ one notifier):** for each NEW inbound in-app message the sweep POSTs
   `/api/fub/body-recovered`; **`fub-body-recovered`** maps `message_type='zillow_inapp'` тЖТ `in_app_message` тЖТ
   a card, through the gate stack: **existing-lead gate** (no HouseLoop lead тЗТ store-only, NO card, NO mint) +
   `_global` on + `in_app_message` Notify on + recency. The legacy realloop `notify-scraper` Twilio path is
   retired (behind `REALLOOP_SCRAPER_NOTIFY`, off) so there is exactly ONE notification per message.
4. **Backfill is silent:** `fub-inapp-sweep.mjs --backfill` pulls every in-app message into the mirror + All
   feed with **zero** notifications (no 1,198-card flood). `--measure` prints the FUB-vs-mirror gap.

<Note>
**Live proof (2026-07-09, dark run).** `--measure`: FUB **1,303** in-app people vs our mirror **2,464** people /
**10,588** messages (coverage gap 0). Incremental sweep: discovered 1 person newer than the cursor тЖТ reused
extractor scraped тЖТ **2 new in-app messages stored** тЖТ POSTed body-recovered тЖТ **notified: 0** (gate held:

</Note>

**Why not a webhook:** in-app *inquiries* do fire an `eventsCreated` (Property Inquiry) our native router
already receives тАФ but pure-conversation messages ("What's up with this place?") fire nothing. So the sweep is
the mandatory floor that catches every message. **Never** set `REALLOOP_FUB_AUTONOMY=1` тАФ that
revives the poller + FUB-write-autoreply flood stack retired in the one-writer lockdown.

## The recency gate + the flood post-mortem

**Recency gate:** a `fub_created` gate on the notify path тАФ **old activity never notifies**, even during a
one-off catch-up. A signal whose FUB activity date is old is captured/stored but does not fire a card.

**Why (2026-07-09 flood):** the retired realloop.co handler re-queued a person for a **full re-scrape on every
`peopleUpdated`** тАФ and FUB fires that on any tag/stage/field edit. It reset already-captured people to "pending,"
so the scraper re-walked the entire 49K book, re-inserting **old** activity with a **fresh** `synced_at` + new
ids. The mirror-reading pollers then treated those old rows as new and flooded Zach with 2022тАУ2023 activity as if
it were live (e.g. Lucy Hartman "viewed" from 2026-07-03 written to the mirror on 2026-07-09; Jessica Marroquin
in-app from 2023-12-09). **Root cause: record edits re-scraping (Lane 3 violation) + the mirror deciding what's
"new."** Fix: retire the re-scrape loop, correct the **25,626** people the bug had wrongly reset to `pending`
(flipped back to done), rip the mirror-reading pollers, and make the webhook the sole trigger + the recency gate
the backstop. The backfill is done тАФ **49,059** people carry `fub.rehydration_log.status = 'rehydrated'` (the
terminal "done" state; there is no separate `fully_scraped` flag), 0 pending, forward-only.

## The login (this is what "stuck / can't log in" means)

FUB's real login is **"Log in with Google"** as `zach@spradlingrealtygroup.com`. The persistent Chrome profile
holds a live Google session, so navigating to the app **auto-completes SSO** тАФ no password prompt in the normal

account-chooser тЖТ consent**, in that order, using `FUB_WEB_GOOGLE_PASSWORD` (тЖТ `FUB_WEB_PASSWORD` fallback). Only a
**genuine 2FA challenge** тАФ a real code box or a phone push, with nothing else the bot can fill тАФ alerts a human via
`notify.sh`. **Option text alone ("Try another way", "use a passkey", "verify it's you") is NOT a 2FA wall** тАФ those
appear next to the normal password field and the bot must ignore them (see the 2026-07-08 incident below).

When the session expires, the scraper's **startup session-check** navigates fresh to `/2/people`; if it lands on
the login page it **delegates to `fub-login-bot.mjs` (Google SSO)** and re-verifies. (Before 2026-07-08 it clicked
a `input[type=submit].u-bigBlueButton` тАФ the DEAD legacy FUB email/password form button. FUB is Google SSO now, so
that click always failed тЖТ an instant 10-minute cooldown тЖТ the "scraper down / won't log back in" symptom. The
delegation to the real login bot fixed it.)

**The `fub-watchdog` self-heals a real logout AUTONOMOUSLY тАФ a human should almost never touch it.** Every 60s it
checks Chrome A (18800). If a login tab is up, it runs the login-bot; on a persistent real logout it does a HARD
relaunch (kill + fresh-profile launch + re-login), same as `fub_chrome_restart`. It only alerts a human via
`notify.sh` if even that can't re-auth (a genuine 2FA/password wall).

<Warning>
**The watchdog must WATCH, never HARM (Caleb 2026-07-08 тАФ the "watchdog was the killer" fix).** Its health probe
`isCdpConnectable()` used to `connectOverCDP()` then `browser.close()` тАФ and on this Chrome build that close
**tore down the browser's live pages.** So every 60s the watchdog killed the actively-scraping worker, mis-read
the wreckage as "WEDGED," and hard-relaunched Chrome тАФ **568 relaunches**, an endless self-inflicted crash loop
that looked exactly like "the scraper won't stay logged in." Two guards now prevent it, permanently:
1. **Passive probe** тАФ a raw DevTools websocket open/close (Node global `WebSocket`) that never calls
   `browser.close()`, so probing a healthy Chrome is harmless.
2. **Fresh-log guard** тАФ if the `fub-backfill-a` log is fresh (the worker wrote scrape output seconds ago), the
   watchdog **never probes or relaunches**: an actively-producing worker is definitive proof CDP is healthy.

Rule for anyone touching the watchdog: a monitor may **look** but must never **close/relaunch a browser that has a
live worker.** Code: `fub-backfill-watchdog.mjs` тАФ `isCdpConnectable` + the `workerFresh` guard in `check()`.
**Known gap (audit F9, 2026-07-09):** a Chrome whose DevTools websocket has wedged (HTTP alive, ws hangs forever)
can hide behind the fresh-log guard, because the empty-queue exit loop keeps writing "Nothing to scrape" тАФ a log
write, not real scrape output. The freshness signal must key on *scrape output*, and CDP connects need an
external timeout, not an in-process race.
</Warning>

## The scrape-on-entry doctrine

The moment a new Zillow inquiry fires the FUB `eventsCreated` webhook, the handler spins up a **per-lead scrape**
that intercepts FUB's own authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire
timeline** тАФ every home they ever touched (Zillow URL, zpid, MLS, price, **photo**) тАФ in one shot, keyed by FUB's
stable `fub_event_id` into `fub.zillow_home_activity`, then **full-replaces** that person's rows in the Railway
mirror. On demand, re-run it with **`mcp__fub-scraper__fub_scrape_lead`** or **`mcp__camp-david__fub_scrape_person`**.
This is the front door for all new lead data тАФ when a lead's cards look wrong, **re-running the scrape is the first move.**

### The real activity time тАФ `activity_at` (2026-07-12)

Each home row carries **`activity_at timestamptz`** тАФ the REAL moment the lead viewed/saved the home, lifted
from FUB's browse-list `occurred` field. It exists because `activity_date` is **date-only and UTC-sliced**, so
an evening view (e.g. Jul 11 9:24pm ET) rolled its date forward a day and the CRM feed fell back to
**scrape-time** for the timestamp (Austin Banh's 6303 N Bellefontaine showed "Jul 12 5:46am" тАФ the scrape тАФ vs
FUB's Jul 11). The CRM converter (`homeOccurredAt`) now uses `activity_at` first, so the feed + the FUB Context
view render FUB's real time 1:1.

Two gotchas: **(1)** after scp'ing a new `fub-scraper-lib.mjs`, **pm2-restart the worker** тАФ a running worker
holds the old lib in memory, so `activity_date` populates while `activity_at` stays NULL (the tell). **(2)** a
lead minted BEFORE the fix keeps scrape-time `occurred_at` тАФ `logActivity` is `DO NOTHING` on conflict and the
heal path never touches `occurred_at` (write-once identity), so the per-lead fix is a **re-scrape + purge/re-mint**
(`scripts/austin-remint-loop.ts`); a fleet-wide `occurred_at` heal needs Zach's sign-off.

## What it can't recover

**If a FUB user sets their emails private/hidden in the FUB UI, the DOM scraper cannot recover them** тАФ they read
`[CONTENT HIDDEN]` in the actual UI, so there is nothing to scrape. This is a FUB privacy feature, not a scraper
failure (concrete case: Aaron Foster's UI-hidden emails store `[CONTENT HIDDEN]` permanently). So a large `hidden`
count is **largely expected** and must not be "fixed." What it **can** recover: anything visible in Zach's FUB UI,
including the newest message body via the intercepted timeline API (`lastEmail.bodyHtmlVisibleClean`; a direct
public-API fetch returns `401`).

# тХР THE FUB FOLLOW-UP WIDGET (absorbed from fub/follow-up-widget, 2026-07-17) тХР

## What it is

A Follow Up Boss **embedded app** ("Houseloop") that renders in the right column of every FUB
person view. It loads the lead's **recovered** history (the bodies the [FUB scraper](/fub/master-scraper-ssot)
unmasked), acts as the agent's AI clone, and lets them **text or call the lead** without leaving
FUB. Served from `https://realloop.co/fub-widget`.

- **Auth:** FUB passes a `context` (base64 JSON) plus an HMAC `signature`; the widget verifies
  `HMAC-SHA256(context, FUB_EMBEDDED_APP_API_KEY)` before rendering. No `X-Frame-Options` (it's an iframe).
- **Freshness:** every panel re-reads the database on open (`Cache-Control: no-store`) so it always
  reflects the latest activity.

## The three tabs

| Tab | Shows |
|---|---|
| **Actions** | Claude "what should I do?" тАФ summary, intent, single best next step, a channel-aware drafted reply, **Approve & Send**, **Call lead**, Regenerate. |
| **Activity** | Texts / Calls / Emails / Homes тАФ the real recovered content, homes are clickable Zillow links. |
| **Context** | The living brain тАФ who the lead is + buying intent, regenerated from live data. |

The AI reads a **single unified, timestamped timeline** across *every* channel тАФ texts, emails,
calls, team notes, and home-activity events тАФ with the most-recent line flagged, so suggestions
reflect what just happened rather than a stale early message.

## First-touch outreach mode

When a lead has a **team-exit / prior-agent tag** (e.g. `Team Exit - April Atkinson`), no recent
outbound from us, and a real in-market property to anchor on, the widget switches to a cold
first-touch script:

1. **Who** тАФ "Zach Spradling, team lead at The Spradling Group in Overland Park тАФ the local Zillow
   team that runs showings in your area."
2. **Bridge** тАФ names the prior agent: "April's moved on from our team, so I'm personally looking
   after your search now."
3. **The property** тАФ the most-recently-viewed home **in the lead's primary market** (out-of-state
   browsing noise filtered out), pulled from the notes.
4. **The ask + open door** тАФ "Saw you checking out `[address]` тАФ want to go see it? Happy to set up
   that one or any others you've got your eye on."
5. The exact Zillow URL is appended deterministically (the model never edits the link).

## Sending тАФ text and call

There is **no FUB API that sends a message** (`POST /v1/textMessages` only *logs*; it never
transmits). So the widget sends through a number we control and logs into FUB for the timeline.

- **Number:** `+1 (913) 286-1761` "Zach AI Outbound" on our Twilio account тАФ A2P brand **approved**,
  campaign **verified**, in a Messaging Service.
- **Text** тАФ `POST /fub-widget/send` sends via the Twilio Messages API (`From` = the AI-outbound
  number **and** `MessagingServiceSid` together, for A2P compliance), then logs to the FUB timeline
  via the registered-system API (gated on `FUB_SYSTEM_KEY`).
- **Call (click-to-call bridge)** тАФ `POST /fub-widget/call` rings the agent's cell first, then
  `/fub-widget/voice/outbound` returns TwiML that dials the lead with the AI-outbound number as
  caller ID and bridges the two. The agent talks on their normal phone; the lead sees the
  AI-outbound number.

<Note>
Logging sent/received texts onto the **FUB timeline** requires registering the system at
`apps.followupboss.com/system-registration` and setting `FUB_SYSTEM` + `FUB_SYSTEM_KEY`. Sending,
calling, and inbound routing all work without it тАФ only the FUB timeline row waits on that key.
</Note>

## Inbound routing тАФ one number, two systems

A lead lives in **exactly one** system тАФ FUB (this widget) or HouseLoop CRM тАФ never both, and keeps
the same number when they graduate FUB to HouseLoop. Inbound to the AI-outbound number therefore
routes by **ownership = does an `hl_leads` row exist for that phone** (HouseLoop-first, FUB-fallback):

```
Lead texts/calls the AI-outbound number
   -> CRM webhook (front door) checks hl_leads by phone tail-10
        - HouseLoop lead          -> handled in the CRM (unchanged)
        - not a HouseLoop lead     -> forwarded to the FUB widget:
             SMS   -> POST realloop.co/fub-widget/sms/inbound  (X-HL-Forward-Secret + MessageSid dedupe)
             Voice -> TwiML Redirect to realloop.co/fub-widget/voice/inbound (whisper "call from <lead>")
```

Graduation is automatic: the moment an `hl_leads` row exists for that number, the next inbound
resolves to HouseLoop and the widget stops receiving it тАФ **no double-send, no flag to flip.** The
shared `FUB_FORWARD_SECRET` guards the SMS forward; `MessageSid` / `CallSid` dedupe Twilio retries.

<Note>
True lead caller-ID on a forwarded inbound call is **not possible on raw Twilio** (anti-spoofing /
STIR-SHAKEN). The agent's phone shows the AI-outbound number; a spoken **whisper** announces who's
calling instead.
</Note>
