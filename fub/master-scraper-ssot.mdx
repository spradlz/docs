### ⭐ The scrape-on-entry doctrine — the real-time per-lead intercept (THE entry mechanism) 🔒
**This is how every new lead's full history reaches us, in ~half a second.** The moment a new Zillow inquiry fires
the FUB `eventsCreated` webhook, the handler **spins up a per-lead scrape on Camp David** that intercepts FUB's own
authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire** timeline — grabbing
**every home the lead ever touched: the Zillow URL, zpid, MLS, price, and the photo** — in one shot (242 → 534 homes

(`fub.rehydration_queue` → `fub-webhook-scraper`); on demand, **`mcp__fub-scraper__fub_scrape_lead` /
`mcp__camp-david__fub_scrape_person`** re-run the full intercept for one person. **It is deployed + LIVE.** This
scraper is the front door for ALL new lead data — when a lead's cards look wrong, re-running it is the first move.

### 🗄️ TWO databases — and `zillow_home_activity` is OFF the relay (read this before any photo/home-card debugging) 🔒
There are **two** FUB stores, and they are NOT kept in sync the same way for every table:
- **Camp David `fub.*`** (`DATABASE_URL` on the box) — where the scraper WRITES. Ground truth for scraped data.
- **Railway mirror `fub.*`** (`FUB_RAILWAY_DATABASE_URL`) — **what the HouseLoop CRM READS** (`fubQuery`,
  `LeadContextPanel`, the import, the card minter). The CRM never reads Camp David directly.

Most tables reach the mirror via the **relay** (`fub-relay`, upsert-by-id). **`fub.zillow_home_activity` is the
exception — it is OFF the relay (Zach 2026-06-27).** The **scraper is its SINGLE WRITER to the mirror**: right after
`saveZillowHomeActivity` writes Camp David, it does a **per-person FULL REPLACE to the mirror**
(`syncPersonHomeActivityToMirror` — `DELETE person rows + INSERT the fresh set`). So a home's photo reaches the CRM
**only when the scraper runs for that person.** If the scraper hasn't re-run since the photos were captured, **Camp
David has the photos but the mirror does not** — and the CRM, reading the mirror, shows blank.

### ✅ The blank-photo bug — ROOT CAUSE FOUND + FIXED (2026-06-28)
Every blank photo + stale lead traced to **ONE bug**: `mirrorUrl()` in `fub-scraper-lib.mjs` resolved its `~/.env`
fallback with **`require('node:fs')` — but that file is an ES module, so `require` is undefined and threw**, silently
returning `null`. The per-person mirror sync then no-op'd on **every on-demand scrape** (which doesn't set
`RAILWAY_DATABASE_URL`). pm2 scrapers "worked" only because their env sets the var. So a scrape wrote Camp David
perfectly but **never updated the Railway mirror the CRM reads** → blank. *Proven:* Alberto Camp David **199/199**
photos, mirror **829 rows / 0 photos** — until the fix flipped it to **207/207**.

**Fixed (shipped):**
- `mirrorUrl()` uses the ESM-imported `fs` (no `require`).
- `syncPersonHomeActivityToMirror` now **sources the FULL set from Camp David** (not the flaky in-memory scrape
  result, which the intercept returns non-deterministically — 0/4/199 on the same lead), **always runs**, **never
  shrinks** the mirror below Camp David, and **fails LOUD** (`✅ mirror replaced: N` / `❌ mirror sync FAILED` — never
  silent). The loud log is what surfaced the `require` bug on the first run.
- **Card heal:** `resyncHomePhotos` + `stampActivityPhoto` (the sanctioned append-only exception) stamp photos onto
  already-minted cards (all `evt`/`zpid`/slug source_id variants); for leads minted from the old broken mirror,
  **re-import** (`fub/lead.import {backfill}`) mints clean `evt`-keyed cards, then a Rule#0-approved delete removes the
  stale `fub_home:` non-evt duplicates. **Junk guard** in `convertHomeActivity`: an address that is a URL / agent-page
  (`realscout.com/users/...`) / doesn't start with a street number **never mints a card**.

> **One-line debug rule:** blank home card → check the **Railway mirror** (`FUB_RAILWAY_DATABASE_URL`), not Camp
> David. 0 photos there → re-run `fub_scrape_lead` (now mirrors loud); cards stale → re-import the lead.

### 🔑 Where property VIEWS actually come from — the internal feed, NOT the notes (Zach 2026-06-28)

### Pulling in a FRESH person — the mirror lag + the scrape front door (Lee, 2026-07-22) 🔒

the FUB API**. A person created minutes ago (proven: **Kenneth Eugene Hunt, FUB 50544**, while the mirror's newest

### ⭐ The Watermark Law — read-once, merge-forward (universal) 🔒
Governs **every** LLM rollup (dossier, facts, Hot Buyers, any summary): **never re-process data already consumed.**
The dossier stores its text + a `source_cursor` (high-water mark). Each regen: read only the delta
(`WHERE synced_at > source_cursor`) → merge into the existing dossier (don't re-summarize) → advance
`source_cursor = max(synced_at)`. First pull = full history (Sonnet, cursor = now); every pull after = the delta,
merged with Haiku. **Tokens paid once per row, ever.**

<Warning>
**The gotcha:** watermark on **`synced_at` (when WE ingested the row), NOT the event's own date.** A 2-year-old
home view the scraper *just* backfilled is **new to us** — `synced_at > cursor` catches it. Watermark on the event
date and you'd silently skip every freshly-backfilled historical row.
</Warning>

# ═ SIGNAL CAPTURE + THE TOGGLES (absorbed from fub/trigger-events, 2026-07-17 — the EVENT CATALOG moved to /guides/trigger-events) ═

## The principle (LOCKED 2026-07-09 by Zach)

**Notifications are the source of truth. The scraper is a dumb, on-demand grabber of HIDDEN content only.**

The native HouseLoop webhook is what decides *what happened*; when the thing that happened has content FUB hides
(a masked message body), the notification layer *tells the scraper* to go recover exactly that one body — nothing
else. The scraper never decides what's new, never crawls, never re-scrapes on a record edit. This replaces the
old "push for speed, pull for truth" dual-path / differential-reconciler design, which is **dead** (it read the
lagging mirror and flooded Zach with 2022-era activity — see the post-mortem below).

## Vocabulary (LOCKED 2026-07-07)

- **Integration** = the *pipe* a lead flows through — **HouseLoop.ai** (our website pixel) or **Follow Up Boss**
  (the Zillow-connected CRM). Managed at **Settings → Lead Integrations** (`/settings/lead-platform`).
- **Lead Source** = the *brand* on the lead — Zillow, Referral, etc. A source lives *under* an integration.
- **Trigger Event** = a *door* — a thing that can happen (a tour request, an inbound text, a property save) that
  can start a lead, run a workflow, and/or fire a notification. One row in **`hl_fub_webhook_settings`**,
  configured at `crm.houseloop.ai/settings/trigger-events`.

## The four independent toggles (this is the model)

Each FUB trigger event carries **four independent switches** — none implies another:

| Toggle | Column | What it controls |
|---|---|---|
| **Sync** | `enabled` | Do we ingest/process this FUB signal at all. Off = we don't even mirror it. |
| **Creates Lead** | `creates_lead` | Does firing this door mint (or match) a HouseLoop lead. |
| **Workflow** | `workflow_id` + `workflow_active` | Which workflow the door runs, and whether it runs. |
| **Notify** | `notify` | Fire an owner-only HouseLoop notification when this door fires — **independent of Creates Lead.** |

<Note>
**Notify is decoupled from Creates Lead (Zach 2026-07-08).** You can turn ON a notification for a FUB trigger
event **even if you do not create a lead** — a "heads-up, this happened" ping with no lead minted. The card then
renders "**HouseLoop — no lead created**" (no HouseLoop link) alongside the always-present FUB person link. The
gate is two-tier: the per-event row toggles over a `_global` master row (`webhook_type='_global'`).

**The `_global` master now has a switch on the wall (Lee, 2026-07-09 night).** It was the ONE breaker for the
whole FUB pipeline — every per-row Sync/Creates-Lead/Notify toggle is gated behind `_global.enabled` on the
server (`fub-body-recovered.ts`, `fub-lead-import.ts`) — but its only UI lived in the never-mounted
`FubChannelBoard`, so with `_global` OFF, flipping the 23 per-row toggles did nothing and there was no control
to fix it. Fixed: `FubChannelRows` now renders a **master-breaker banner** above the FUB rows on the Trigger
Events board (red "OFF — nothing below runs" / blue "ON — the pipeline is live"), writing `_global.enabled`.
This is the go-live switch. **Flipped ON 2026-07-09 night → the pipeline is LIVE** (see
[Where the build stands](#where-the-build-stands)).
</Note>

   scraper reads it (see "How an in-app message becomes a trigger event" below). Once a sweep writes it to the
   mirror it is born-visible (never masked) and notifies via the native card, gated to existing leads. Only
   **SMS** ever masks a body.
</Note>

## The 3-lane architecture

Every FUB event enters the native webhook and `fubWebhookRouter` sorts it into exactly one lane:

### Lane 1 — Visible activity (~95 percent): router notifies directly, scraper never touches
Home saved/viewed (`notesCreated`), inquiries (`eventsCreated`), visible SMS, call metadata, appointments,
tasks, deals. (In-app messages are NOT here — they are the inbox-app sweep lane below.) The API hands us clean
content → the router classifies it and fires
the owner-only notification immediately.

### Lane 2 — Masked comms: the ONLY thing that wakes the scraper
The three masked cases — masked SMS bodies, email bodies, and call notes. The flow is a **surgical body-fetch**,
never a re-scrape:

```
masked textMessagesCreated / emailsCreated  →  fubWebhookRouter writes metadata + enqueues ONE job
      ↓ fub.body_recovery_queue (on Camp David — the relay is CD→Railway one-way, so the job lives CD-side)
      { person_id, resource_type: 'text' | 'email', resource_id, requested_at }
      ↓ the scraper drains the queue — reads THAT ONE body from the FUB UI
      ↓ writes the recovered body → flips content_status  hidden → visible
      ↓ trg_fub_new_message fires  (now AFTER INSERT OR UPDATE, transition-guarded, text + email)
      ↓ pg_notify('fub_new_message') → the native notify path
   →  HouseLoop notification fires WITH the body
```

**No metadata-only notify for a masked body** — a card with no message is useless, so we hold the notify until
the body is recovered. Born-visible content (in-app, visible SMS) is Lane 1 and notifies on the INSERT.

### Lane 3 — Record edits: update the person, never scrape
`peopleUpdated` / `peopleStageUpdated` / `peopleTagsCreated` / `peopleDeleted`. These update the person record
only. **They never enqueue a scrape.** (A stage change may fire `stage_changed` if the Notify toggle is on; that
is a notification, not a scrape.)

## ⭐ How an in-app (Zillow Messages) message becomes a trigger event

Zillow in-app chat is the ONE signal FUB does **not** expose through its API or webhooks (proven 2026-07-09 on
James Ross: his in-app messages carry synthetic ids and never appear in `GET /v1/textMessages`; no
`textMessagesCreated` fires). They live in FUB's **"Zillow Messages" inbox app** (`inboxApp id 1`, created by
the original account owner's seat — `createdById 1`, which is why a departed owner's Zillow connection still
fronts these leads). So capture is: find the people via the REST API, scrape their pages for the bodies — all
lawful, gated, and Inngest-visible:

1. **Detect — the REST API, not the inbox SPA (proven 2026-07-09):** message BODIES aren't in the API, but WHO
   has in-app activity IS: **`GET /v1/people?sort=-lastInboxAppMessage&fields=id,lastInboxAppMessage`** returns
   every person ordered by their last in-app message, with the timestamp. Reliable, no browser, cursor-pageable.
   (The FUB inbox web endpoint `/api/v1/inbox/…/conversations` **hangs** on any direct/replay fetch and its
   virtualized list won't page on scroll — only the app's own live XHR ever fires; the people-sort is the robust
   surface.) Incremental takes only people newer than our mirror's newest `zillow_inapp`; backfill takes all.
   The registered Inngest cron **`fub-inapp-sweep-tick`** (Run Log `comms` lane; cron `*/3` ARMED since
   2026-07-09 sign-off) enqueues one sentinel `inbox_sweep` job into the Railway FUB mirror queue —
   `UNIQUE(resource_type,resource_id)` = at most one sweep pending (no pile-up).
2. **Extract (reuse, never reinvent):** the Camp David `fub-body-recovery` consumer claims the job and runs
   **`tools/fub-scraper/fub-inapp-sweep.mjs`**, which — for each discovered person — reuses the proven extractor
   in `fub-scraper-lib.mjs` (`scrapePersonFull` + `matchAndUpdateMessages`: `scrapeTextContainers` classifies a
   timeline row `zillow_inapp` by its **`zillow.png`** circle-icon; the synthetic id
   **`900000000 + personId*1000 + seq`** dedupes against the existing rows, `ON CONFLICT (id) DO NOTHING`). The
   relay carries the new born-visible rows Camp David → Railway.
3. **Notify (native, gated — one notifier):** for each NEW inbound in-app message the sweep POSTs
   `/api/fub/body-recovered`; **`fub-body-recovered`** maps `message_type='zillow_inapp'` → `in_app_message` →
   a card, through the gate stack: **existing-lead gate** (no HouseLoop lead ⇒ store-only, NO card, NO mint) +
   `_global` on + `in_app_message` Notify on + recency. The legacy realloop `notify-scraper` Twilio path is
   retired (behind `REALLOOP_SCRAPER_NOTIFY`, off) so there is exactly ONE notification per message.
4. **Backfill is silent:** `fub-inapp-sweep.mjs --backfill` pulls every in-app message into the mirror + All
   feed with **zero** notifications (no 1,198-card flood). `--measure` prints the FUB-vs-mirror gap.

## Architecture

**Two databases (memorize this before any photo/blank-card debugging):**
- **Camp David `fub.*`** — where the scraper **writes**. Ground truth for scraped data.

## The login (this is what "stuck / can't log in" means)

FUB's real login is **"Log in with Google"** as `zach@spradlingrealtygroup.com`. The persistent Chrome profile
holds a live Google session, so navigating to the app **auto-completes SSO** — no password prompt in the normal
case. If the Google session itself has lapsed, `fub-login-bot.mjs` drives the SSO by hand: **email → password →
account-chooser → consent**, in that order, using `FUB_WEB_GOOGLE_PASSWORD` (→ `FUB_WEB_PASSWORD` fallback). Only a
**genuine 2FA challenge** — a real code box or a phone push, with nothing else the bot can fill — alerts a human via
`notify.sh`. **Option text alone ("Try another way", "use a passkey", "verify it's you") is NOT a 2FA wall** — those
appear next to the normal password field and the bot must ignore them (see the 2026-07-08 incident below).

When the session expires, the scraper's **startup session-check** navigates fresh to `/2/people`; if it lands on
the login page it **delegates to `fub-login-bot.mjs` (Google SSO)** and re-verifies. (Before 2026-07-08 it clicked
a `input[type=submit].u-bigBlueButton` — the DEAD legacy FUB email/password form button. FUB is Google SSO now, so
that click always failed → an instant 10-minute cooldown → the "scraper down / won't log back in" symptom. The
delegation to the real login bot fixed it.)

**The `fub-watchdog` self-heals a real logout AUTONOMOUSLY — a human should almost never touch it.** Every 60s it
checks Chrome A (18800). If a login tab is up, it runs the login-bot; on a persistent real logout it does a HARD
relaunch (kill + fresh-profile launch + re-login), same as `fub_chrome_restart`. It only alerts a human via
`notify.sh` if even that can't re-auth (a genuine 2FA/password wall).

<Warning>
**The watchdog must WATCH, never HARM (Caleb 2026-07-08 — the "watchdog was the killer" fix).** Its health probe
`isCdpConnectable()` used to `connectOverCDP()` then `browser.close()` — and on this Chrome build that close
**tore down the browser's live pages.** So every 60s the watchdog killed the actively-scraping worker, mis-read
the wreckage as "WEDGED," and hard-relaunched Chrome — **568 relaunches**, an endless self-inflicted crash loop
that looked exactly like "the scraper won't stay logged in." Two guards now prevent it, permanently:
1. **Passive probe** — a raw DevTools websocket open/close (Node global `WebSocket`) that never calls
   `browser.close()`, so probing a healthy Chrome is harmless.
2. **Fresh-log guard** — if the `fub-backfill-a` log is fresh (the worker wrote scrape output seconds ago), the
   watchdog **never probes or relaunches**: an actively-producing worker is definitive proof CDP is healthy.

Rule for anyone touching the watchdog: a monitor may **look** but must never **close/relaunch a browser that has a
live worker.** Code: `fub-backfill-watchdog.mjs` — `isCdpConnectable` + the `workerFresh` guard in `check()`.
**Known gap (audit F9, 2026-07-09):** a Chrome whose DevTools websocket has wedged (HTTP alive, ws hangs forever)
can hide behind the fresh-log guard, because the empty-queue exit loop keeps writing "Nothing to scrape" — a log
write, not real scrape output. The freshness signal must key on *scrape output*, and CDP connects need an
external timeout, not an in-process race.
</Warning>

**Manual override (rarely needed):**
```

```
Kills + relaunches the FUB Chrome (port 18800), re-logs-in, returns `logged_in: true` + the landing URL. Use it if
you want an instant fix instead of waiting for the ≤2-min watchdog cycle, or if a 2FA alert fired. **Before

could sit spinning until someone ran this tool by hand. That gap is now closed.**

## The scrape-on-entry doctrine

The moment a new Zillow inquiry fires the FUB `eventsCreated` webhook, the handler spins up a **per-lead scrape**
that intercepts FUB's own authenticated SPA call `GET /api/v1/events?hasProperty=true` and walks the lead's **entire
timeline** — every home they ever touched (Zillow URL, zpid, MLS, price, **photo**) — in one shot, keyed by FUB's
stable `fub_event_id` into `fub.zillow_home_activity`, then **full-replaces** that person's rows in the Railway
mirror. On demand, re-run it with **`mcp__fub-scraper__fub_scrape_lead`** or **`mcp__camp-david__fub_scrape_person`**.
This is the front door for all new lead data — when a lead's cards look wrong, **re-running the scrape is the first move.**

### The real activity time — `activity_at` (2026-07-12)

Each home row carries **`activity_at timestamptz`** — the REAL moment the lead viewed/saved the home, lifted
from FUB's browse-list `occurred` field. It exists because `activity_date` is **date-only and UTC-sliced**, so
an evening view (e.g. Jul 11 9:24pm ET) rolled its date forward a day and the CRM feed fell back to
**scrape-time** for the timestamp (Austin Banh's 6303 N Bellefontaine showed "Jul 12 5:46am" — the scrape — vs
FUB's Jul 11). The CRM converter (`homeOccurredAt`) now uses `activity_at` first, so the feed + the FUB Context
view render FUB's real time 1:1.

Two gotchas: **(1)** after scp'ing a new `fub-scraper-lib.mjs`, **pm2-restart the worker** — a running worker
holds the old lib in memory, so `activity_date` populates while `activity_at` stays NULL (the tell). **(2)** a
lead minted BEFORE the fix keeps scrape-time `occurred_at` — `logActivity` is `DO NOTHING` on conflict and the
heal path never touches `occurred_at` (write-once identity), so the per-lead fix is a **re-scrape + purge/re-mint**
(`scripts/austin-remint-loop.ts`); a fleet-wide `occurred_at` heal needs Zach's sign-off.

## What it can't recover

**If a FUB user sets their emails private/hidden in the FUB UI, the DOM scraper cannot recover them** — they read
`[CONTENT HIDDEN]` in the actual UI, so there is nothing to scrape. This is a FUB privacy feature, not a scraper
failure (concrete case: Aaron Foster's UI-hidden emails store `[CONTENT HIDDEN]` permanently). So a large `hidden`
count is **largely expected** and must not be "fixed." What it **can** recover: anything visible in Zach's FUB UI,
including the newest message body via the intercepted timeline API (`lastEmail.bodyHtmlVisibleClean`; a direct
public-API fetch returns `401`).

    without `process.exit()`, so the `connectOverCDP` websocket kept the event loop alive and it hung idle
    "online", never re-scanning (the root of F9's "empty-queue exit loop"). Now the workers loop forever (claim →
    on empty `nap POLL_IDLE_MS` → re-poll), **re-verify the FUB session every ~75 scrapes** so a multi-day sweep
    self-heals its login, and handle SIGTERM cleanly. A lead enqueued later is picked up within the poll interval
    with **no manual restart**. `claimLead` also orders **HouseLoop leads first**.
  - **Per-person relay completeness (the F11 root fix, generalized).** On every successful scrape the worker
    re-stamps `synced_at = NOW()` on that person's `text_messages` / `calls` / `emails` / `notes` / `events` +
    `people`. The CD→Railway relay forwards by a `synced_at` cursor, so a rehydrated person's historical rows
    (stamped months ago, behind the cursor) would otherwise **never reach the mirror** — "scraper runs for days,
    mirror stays behind." Now every rehydrated person's COMPLETE current set propagates. Proven end-to-end: 5 test
    people, mirror `text_messages` count == Camp David exactly. **Zillow in-app** (`message_type='zillow_inapp'`)
    rides the same path and lands.
  - **realloop.co fully removed from the scraper.** All 6 legacy POSTs (3 `sync-message`, 2 `sync-email`, 1
    `notify-scraper`) deleted from `fub-scraper-lib.mjs`. The relay carries CD→Railway; in-app notify is the native
    `fub-inapp-sweep → /api/fub/body-recovered → fub-body-recovered` path (SOLE in-app notifier, gated). **Zero
    realloop references remain** — this scraper no longer touches realloop.co, matching the trigger-events doctrine.
  - **Two-Chrome parallel backfill (throughput ×2).** `fub-backfill-b` revived on **Chrome B (:18802)** — launched
    via `launch-chrome2.mjs` with FUB cookies **injected from Chrome A** (shared session, no second SSO). 6 workers
    total. **Reset-guard:** each scraper's startup now only resets `in_progress` rows older than **15 min** (real
    crashes), never the sibling's live claims — without it a second scraper's startup clobbered the first's
    in-flight work and both re-scraped the same person. `fub-chrome-guard` caps BOTH ports (`MAX_CHROME=3`,
    `MAX_TABS=12`) and never touches the MLS/billing Chromes.
  - **Safe by construction — no notification flood.** The mirror triggers are transition-guarded: `notify_new_message`
    fires only on INSERT or a `hidden→visible` transition (a `visible→visible` re-sync does **not** fire), and
    `auto_queue_lead` only queues `content_status IS NULL` people. Re-stamping `synced_at` on already-visible rows
    therefore fires **zero** notifications — verified against `hl_notification` on prod (the 2026-07-09 flood cannot recur).
  - **Op note — Chrome wedge on rapid restart.** Restarting a scraper twice in quick succession can leave its Chrome
    with orphaned `about:blank` tabs → workers hang on blank pages (0 sub-step logs, CPU idle). Fix:
    `mcp fub_chrome_restart` (fresh Chrome A), then restart the scraper. If worker pages show `about:blank` instead
    of `…/people/view/<id>`, that's the wedge.
</Update>

# ═ THE FUB FOLLOW-UP WIDGET (absorbed from fub/follow-up-widget, 2026-07-17) ═


<Note>
**Quarantined FUB section.** Part of the **FUB system** on RealLoop infrastructure
(realloop.co, future ConvertLoop) — **not** HouseLoop web/CRM. Documented here only because FUB
leads migrate into HouseLoop eventually. The one shared touch-point with the CRM (inbound call/text
routing) is described under [Inbound routing](#inbound-routing-one-number-two-systems) and is
deliberately the only crossover.
</Note>

## What it is

A Follow Up Boss **embedded app** ("Houseloop") that renders in the right column of every FUB
person view. It loads the lead's **recovered** history (the bodies the [FUB scraper](/fub/master-scraper-ssot)
unmasked), acts as the agent's AI clone, and lets them **text or call the lead** without leaving
FUB. Served from `https://realloop.co/fub-widget`.

- **Auth:** FUB passes a `context` (base64 JSON) plus an HMAC `signature`; the widget verifies
  `HMAC-SHA256(context, FUB_EMBEDDED_APP_API_KEY)` before rendering. No `X-Frame-Options` (it's an iframe).
- **Freshness:** every panel re-reads the database on open (`Cache-Control: no-store`) so it always
  reflects the latest activity.

## The three tabs

| Tab | Shows |
|---|---|
| **Actions** | Claude "what should I do?" — summary, intent, single best next step, a channel-aware drafted reply, **Approve & Send**, **Call lead**, Regenerate. |
| **Activity** | Texts / Calls / Emails / Homes — the real recovered content, homes are clickable Zillow links. |
| **Context** | The living brain — who the lead is + buying intent, regenerated from live data. |

The AI reads a **single unified, timestamped timeline** across *every* channel — texts, emails,
calls, team notes, and home-activity events — with the most-recent line flagged, so suggestions
reflect what just happened rather than a stale early message.

## First-touch outreach mode

When a lead has a **team-exit / prior-agent tag** (e.g. `Team Exit - April Atkinson`), no recent
outbound from us, and a real in-market property to anchor on, the widget switches to a cold
first-touch script:

1. **Who** — "Zach Spradling, team lead at The Spradling Group in Overland Park — the local Zillow
   team that runs showings in your area."
2. **Bridge** — names the prior agent: "April's moved on from our team, so I'm personally looking
   after your search now."
3. **The property** — the most-recently-viewed home **in the lead's primary market** (out-of-state
   browsing noise filtered out), pulled from the notes.
4. **The ask + open door** — "Saw you checking out `[address]` — want to go see it? Happy to set up
   that one or any others you've got your eye on."
5. The exact Zillow URL is appended deterministically (the model never edits the link).

## Sending — text and call

There is **no FUB API that sends a message** (`POST /v1/textMessages` only *logs*; it never
transmits). So the widget sends through a number we control and logs into FUB for the timeline.

- **Number:** `+1 (913) 286-1761` "Zach AI Outbound" on our Twilio account — A2P brand **approved**,
  campaign **verified**, in a Messaging Service.
- **Text** — `POST /fub-widget/send` sends via the Twilio Messages API (`From` = the AI-outbound
  number **and** `MessagingServiceSid` together, for A2P compliance), then logs to the FUB timeline
  via the registered-system API (gated on `FUB_SYSTEM_KEY`).
- **Call (click-to-call bridge)** — `POST /fub-widget/call` rings the agent's cell first, then
  `/fub-widget/voice/outbound` returns TwiML that dials the lead with the AI-outbound number as
  caller ID and bridges the two. The agent talks on their normal phone; the lead sees the
  AI-outbound number.

<Note>
Logging sent/received texts onto the **FUB timeline** requires registering the system at
`apps.followupboss.com/system-registration` and setting `FUB_SYSTEM` + `FUB_SYSTEM_KEY`. Sending,
calling, and inbound routing all work without it — only the FUB timeline row waits on that key.
</Note>

## Inbound routing — one number, two systems

A lead lives in **exactly one** system — FUB (this widget) or HouseLoop CRM — never both, and keeps
the same number when they graduate FUB to HouseLoop. Inbound to the AI-outbound number therefore
routes by **ownership = does an `hl_leads` row exist for that phone** (HouseLoop-first, FUB-fallback):

```
Lead texts/calls the AI-outbound number
   -> CRM webhook (front door) checks hl_leads by phone tail-10
        - HouseLoop lead          -> handled in the CRM (unchanged)
        - not a HouseLoop lead     -> forwarded to the FUB widget:
             SMS   -> POST realloop.co/fub-widget/sms/inbound  (X-HL-Forward-Secret + MessageSid dedupe)
             Voice -> TwiML Redirect to realloop.co/fub-widget/voice/inbound (whisper "call from <lead>")
```

Graduation is automatic: the moment an `hl_leads` row exists for that number, the next inbound
resolves to HouseLoop and the widget stops receiving it — **no double-send, no flag to flip.** The
shared `FUB_FORWARD_SECRET` guards the SMS forward; `MessageSid` / `CallSid` dedupe Twilio retries.

<Note>
True lead caller-ID on a forwarded inbound call is **not possible on raw Twilio** (anti-spoofing /
STIR-SHAKEN). The agent's phone shows the AI-outbound number; a spoken **whisper** announces who's
calling instead.
</Note>

## Endpoints

| Route | Purpose |
|---|---|
| `GET /fub-widget` | The iframe (Actions / Activity / Context), HMAC-gated |
| `GET /fub-widget/ai` | Claude read + channel-aware draft (no-store) |
| `POST /fub-widget/send` | Send text via Twilio + log to FUB |
| `POST /fub-widget/call` | Click-to-call bridge |
| `POST /fub-widget/sms/inbound` | Receive a FUB-owned inbound text forwarded by the CRM front-router |
| `POST /fub-widget/voice/inbound`, `/voice/whisper`, `/voice/outbound` | Inbound forward + whisper + outbound bridge TwiML |
