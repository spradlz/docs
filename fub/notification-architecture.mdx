## The core principle: FUB is an unreliable source — reconcile, never trust one path

Every notification failure this cycle came from relying on a **single delivery path**. The fix is not "make one
path reliable" — it's a **self-healing system** where each path backstops the others and a reconciler guarantees
eventual completeness.

## The three capture paths

| Path | Catches | Fails when |
|---|---|---|
| **Webhooks** — `POST /api/fub/webhook` | ~95% of events (new leads, texts, calls, property views/saves, inquiries) — instant | Rate-limit storms drop events; **never fires for free-form Zillow in-app messages** |
| **API pollers** — `GET /v1/events`, `/v1/calls` (cursor) | The ~5% of inquiries webhooks drop | Masks message bodies; **no in-app messages** in the API at all |
| **Scraper** — reads FUB's UI → `fub.zillow_inapp_ghost_quarantine` / `fub.text_messages` | **Zillow in-app messages** (the only source — API-masked) + hidden text/call bodies | Crash-loops / lags → misses the notify window |

None is complete alone. Proof from the recon: a lead's property *view* webhooks fine while their in-app *message*
never arrives; ~5% of inquiries fire no webhook; the in-app scraper died June 30 while webhooks kept flowing.

## The differential reconciler — the god-tier core

Instead of trusting deliveries, **ask FUB the count and heal the gap.** On a cadence, per active lead:

```
for each active lead:
  fubTotal   = GET /v1/{notes|textMessages|events}?personId=X  → _metadata.total   (cheap count)
  ourTotal   = SELECT count(*) FROM fub.{notes|text_messages|events} WHERE person_id=X
  if fubTotal > ourTotal:
      fetch the missing rows → store → notify (first-capture)
      if the missing row is an API-masked in-app message:
          enqueue a TARGETED scrape for THIS lead (not a blind crawl)
```

This **self-heals every failure mode** — dropped webhooks, poller misses, scraper lag — because it compares
against FUB's own truth and cannot silently under-report. And it's the reliable trigger for in-app messages: FUB's
API **counts** a masked message even though it hides the body, so the differential detects "a message exists we
don't have" and commands a targeted scrape to fetch just that body. That turns the fragile scraper from a blind
49k-deep crawler into an on-demand worker the reconciler points at a known gap.

<Note>
**This is NOT how the old poller worked.** RealLoop's `fubEventsPoll` was a dumb global cursor tail
(`/v1/events?sort=-created&limit=50`, process anything newer than a cursor) — it missed under volume and only
covered `/v1/events`. The differential is per-lead, count-based, and covers every activity type. The tail poller
stays as a fast safety net; the differential is the source of truth.
</Note>

## The five layers

| Layer | Job | Trust |
|---|---|---|
| **1. Webhooks** | Latency — react now | Best-effort hint |
| **2. Cursor tail poller** | Catch webhook drops (`/v1/events`, `/v1/calls`) | Safety net |
| **3. Differential reconciler** ⭐ | Count-compare per lead → fetch the delta | Source of truth |
| **4. Scraper (on-demand)** | Fetch API-masked bodies **when Layer 3 flags a gap** | Targeted worker |
| **5. Notifier** | One owner-only card per signal, deduped by `(lead, signal-id)` | Deterministic |

## The notification contract

- **Owner-only, one-card.** `notifyLeadEvent(...)` → `createNotification` fans ONLY to the lead owner (never
  collaborators, never AI agents) → `dispatchNotification` renders the single `notify_card` and appends provenance
  **"from {source} via Follow Up Boss."**
- **Two-tier gate.** Per-trigger **Notify** toggle (`hl_fub_webhook_settings.notify`) over a `_global` master —
  ported from RealLoop's `fubGate` `{enabled, creates_lead, notify}`. Notify is **decoupled** from Creates-Lead.
- **Dedup by signal-id**, not by time. Any layer can catch a signal; you're notified exactly once. This structurally
  kills the old "double message" fan-out.
- **Notify-on-first-capture.** Notify the first time a signal is captured regardless of age — replaces the brittle
  `< 2h old` gate (`fub-scraper-lib.mjs`) that silently dropped alerts whenever the scraper ran behind.

## RealLoop is the proven base — what we port

RealLoop's webhook receiver + inline handlers already implement the reliability tricks; the port to HouseLoop
native carries them (`apps/dashboard/server/fub-webhooks.ts`):

- Respond **2xx before processing** (FUB's 10s SLA).
- **peopleCreated 404-race retry** (5s for calls/texts, 30s for people).
- **Serial FUB API queue** so a webhook burst can't trigger a 429 storm (FUB limit 125 req/10s).
- **Scrape-first-then-notify** wait loop for masked text bodies (`notifyZachInboundTextWhenReady`).
- `X-System-Key` only on **writes**; Basic-auth on reads.

The pollers (`fubEventsPoll`, `fubCallsNotifyPoll`) exist but are **dead** (only the new-lead `catchupPoll` runs);
re-enabling them as HouseLoop-native **Inngest** functions (INNGEST_OR_IT_DOESNT_EXIST) is Phase 3.
