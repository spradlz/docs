<Note>
**One card, kind-driven, every surface.** There is a single `RequestCard` component rendered identically on
**FUB Activity · Website Activity · All Activity** (and mirrored in the Telegram alert formatting). It is
**display-only** — no accept/decline buttons. Acting on a request is a [workflow](/guides/workflows), not a
button on the card. Public mirror of `GT_HOUSELOOP_FUB_EVENTS_AND_REQUEST_CARD_SSOT.md` §7.
</Note>

## The three card kinds

<CardGroup cols={3}>
  <Card title="Property Inquiry" icon="house">
    Photo · full address · MLS · price/beds · message. No calendar chip.
  </Card>
  <Card title="Tour Request" icon="calendar-check">
    Same + a calendar chip (showing date/time) + "View all tour requests".
  </Card>
  <Card title="Saved Search" icon="bookmark">
    Criteria line — property types · beds · baths · price · for-rent · location.
  </Card>
</CardGroup>

Inquiry sub-variants ride on the Property Inquiry card by classifier: **contact request** (message + call-back
framing) and **rental inquiry** (`source = Zillow Rentals`, rental price). See the
[three request types](/fub/data-matrix#the-three-request-types).

## The card anatomy (target = Zillow's card)

```
┌─────────────────────────────────────────────────────────┐
│ ① Tour Request   VIA Zillow Preferred          Jun 25    │  type (classified) + source pill + timestamp
│ ───────────────────────────────────────────────────────  │
│ ② [PHOTO]   ③ 📅 Jun 26, 6:30 pm                          │  photo (far left) + calendar chip (date+time)
│             ④ 2324 SW 34th St, Topeka, KS 66611          │  FULL address (city+state+zip), blue link
│             ⑤ MLS #165929 · $260,000 · 3bd/3ba           │  MLS + price + beds/baths
│ ───────────────────────────────────────────────────────  │
│ ⑥ "I'm interested in 2324 SW 34th St… seeing Jun 26…"     │  full message line, under the photo (Zillow-style)
│ ───────────────────────────────────────────────────────  │
│ ⑧ View all tour requests                                  │  per-person link (tour cards only)
└─────────────────────────────────────────────────────────┘
```

| Field | Source |
| --- | --- |
| ① type | `classifyFubEvent(message, source)` — tour / inquiry / contact |
| ② photo | our `Listing` (in-market) → fallback Zillow `zpid` from `property.url` (out-of-market) |
| ③ calendar chip | parsed showing date/time from the message (tour cards only) |
| ④ full address | `property` jsonb — street + city + state + zip |
| ⑤ MLS · price · beds/baths | `property` jsonb |
| ⑥ message | `event.message` |
| ⑧ view all | filter the person's tour requests |

## Why the data is all there

Every `eventsCreated` fetch-back stores the full clean object in `fub.events.property` — street, **city, state,
zip**, MLS, price, beds/baths, lat/lng, and the Zillow URL (with `zpid`). So a missing city/zip or MLS on a card
is a **render bug, not a data gap**. The only field not in the row is the **photo**: in-market homes resolve one
from our `Listing` DB; out-of-market homes (e.g. Topeka) fall back to the Zillow listing photo by `zpid`.

See the [Data & Signals Matrix](/fub/data-matrix) for how each request becomes one of our own events.
