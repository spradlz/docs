## One-line truth

**Webhook *delivery* works.** Every categorical theory (wrong key, pond scope, user scope, "automation notes
never webhook") was tested and **disproven live**. The real story is that Zillow signals arrive through **three
different capture paths**, and **two of the three were dead** while the third only covers part of the traffic.

## The three capture paths (this is the mental model)

| Path | What it catches | Status (2026-07-08) |
|---|---|---|
| **Webhooks** → `POST /api/fub/webhook` | Most events: new leads, texts, calls, property views/saves (as notes), ~95% of inquiries | ✅ **Alive** — this is why Zach still got "tons" from RealLoop |
| **Events poller** → `fubEventsPoll` polling `GET /v1/events` every 2 min | The **~5% of inquiries that fire NO webhook** (verified: 38/40 events webhooked, 2/40 poller-only) | ❌ **Disabled May 31** (`fub-webhooks.ts:1754`) |
| **Scraper** → reads FUB's UI, writes `fub.zillow_inapp_ghost_quarantine` | **Zillow in-app inbox messages** ("send me pics", "is it available this weekend") — these are **masked in FUB's public API**, so the scraper is the ONLY path | ❌ **Stopped June 30** (last write Jun 29, last scrape Jun 30 00:07) |

## The four breaks (verified)

### 1. In-app message scraper died June 30
Zillow free-form inbox messages (buyer typing to the agent) are **not exposed by FUB's public API** — verified by
checking all five endpoints for the affected leads:

| Endpoint | Result |
|---|---|
| `/v1/events` | in-app *inbox* messages absent (only structured "Property Inquiry" events) |
| `/v1/notes` | only "Property Viewed/Saved" |
| `/v1/textMessages` | old + `* Body is hidden for privacy reasons *` (FUB masks bodies) |
| `/v1/em` (emails) | none |
| `/v1/threads` | `You do not have access to this API endpoint` |

The only capture is the scraper writing to **`fub.zillow_inapp_ghost_quarantine`** (9,884 rows historically).
That table's **last message is June 29, last scrape June 30 00:07 — nothing since.** So every Zillow inbox
message since June 30 has been dropped. Ubah Mindingall's and Evelyn Morales's recent messages are gone for this
reason; the one from **June 27 exists** because it predates the break. (This is a *distinct* failure from the
scraper crash-loop of 2026-07-08 — the in-app component flatlined over a week earlier.)

### 2. Events poller disabled May 31
`fubEventsPoll` polled `GET /v1/events` every 2 min and fired `notifyZachInquiry` for new **Property/Seller/General
Inquiries and Zillow in-app messages**. It was turned off on May 31:

```js
// fub-webhooks.ts:1754
// Notification pollers disabled — FUB webhooks handle all notifications directly
console.log('[FUB Pollers] Disabled — using FUB webhook sync only');
```

The cursor `fub.poll_state.last_event_id` is **frozen at 1754483 since 2026-05-31 13:53**, while FUB is now past
event 1795454 (~41k events behind). The assumption "webhooks handle everything" is **wrong**: a live test showed
**2 of 40 recent events fire no webhook at all** — those only exist in `/v1/events`, and the poller was their only
catch. (The other 38 webhook fine — which is exactly why the loss was invisible.)

### 3. `sales@spradlingrealtygroup.com` mailbox deleted → Zillow emails black-holed
Zillow Premier Agent forwards notification emails to `sales@spradlingrealtygroup.com`. That mailbox was **deleted
(Aaron)**, so the emails are accepted by the domain and dropped. Note: `spradlingrealtygroup.com` is a **separate
Google Workspace** from `houseloop.ai` — our admin service account (`bossloop-delegation@…`, customer `C01dtfk5x`,
domains houseloop.ai / mail / email) returns **403** on it, so the mailbox must be fixed from the SRG admin
console (super-admin login = 2FA), not our API. Fix = recreate `sales@` (or re-point the dead Gmail **routing**
rule) then parse the Zillow emails.

### 4. Lead contamination — new leads merging onto old records
A brand-new Zillow confirm from **`cheyennmay@gmail.com`** did not create a new lead — its email was attached to
**James Ross (person 50308)** as a *second* `isPrimary` email:

```
50308 | James Ross | [james.randall.ross@gmail.com (primary), cheyennmay@gmail.com (primary)]
```

Two people fused onto one FUB record → no new-lead event → no notification. This also inflates 50308's activity
(it showed 661 webhooks — likely several people's activity merged), which is why "James Ross" looked hyperactive
during the recon.

## What was ruled OUT (don't re-chase these)

- **"We have the wrong API key."** No — our key works (`/v1/me` = Zach Spradling, user 93, admin). Webhooks are
  registered under a separate system app; that does not scope *which* people fire.
- **"Webhook delivery is blocked for some leads."** No — a live API-created note on a "dark" lead (Ahmed, 49133)
  **webhooked to our receiver in ~1 second.** Delivery is fine.
- **"It's pond / user assignment."** No — 152 user-1 leads and 147 in-a-pond leads got webhooks in 24h.
- **"Automation notes never webhook."** Partly a red herring — the deeper cause is capture-path death, not
  author suppression.

## The repair plan (prioritized)

1. **Scraper health** (Caleb) — restores in-app message capture (#1) and is the backstop for contamination (#4).
   Verify `fub.zillow_inapp_ghost_quarantine` starts writing again (fresh `quarantined_at`).
2. **Re-enable `fubEventsPoll` + `fubCallsNotifyPoll`** (#2) — run every 2 min, wired to HouseLoop's owner-only
   notifier (NOT the retired RealLoop SMS). **Bootstrap the cursor forward** (do not replay 41k old events).
3. **Recreate `sales@` + Zillow-email parser** (#3) — SRG-console recreate (Zach), then a HouseLoop inbound parser
   → notification. This path survives scraper + API-mask failures.
4. **Lead-merge / contamination audit** (#4) — find records with multiple `isPrimary` emails; split, and fix the
   intake match so a new Zillow confirm mints a new lead instead of merging.

## Why this stayed invisible

No single break read as "notifications are down." Webhooks kept delivering ~95% of traffic (Zach kept getting
alerts), so the missing ~5% of inquiries (poller), the in-app messages (scraper), the emails (dead mailbox), and
the occasional merged lead each looked like a one-off — until they were mapped together here. **The lesson: these
are three capture paths, not one, and each needs its own health check.**
