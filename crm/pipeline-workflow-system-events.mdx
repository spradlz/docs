## What it is

A **trigger event** is a thing that can happen (a lead did something) that can **start a workflow**. There is exactly
**ONE canonical catalog** — the DB table **`hl_trigger_events`** — and everything derives from it (this page, the Lead
Sources config, the Workflow Builder palette). No second list.

The page is **tabbed by platform** and, inside each tab, **grouped by category**:

- **Platform tabs:** `HouseLoop.ai` · `Follow Up Boss` · `Manual` · `Any`. Each platform emits its own events; the
  **Any** tab holds cross-platform behavioral triggers (see *Platform scope* below).
- **Categories (groups within a tab):**
  - **Registration** — the event **creates a lead** (New Lead — Liked Home, Shared Home, Tour Request, …).
  - **System** — an **existing** lead's on-site behavior (Property Viewed / Liked / Saved / Shared, Saved Search, …).
  - **Follow-up** — internal **nurture** triggers (timers / no-reply), fired by the system, not by a source.

## The two layers (why there isn't a duplicate)

- **`hl_trigger_events`** = the event **definitions** — the one catalog, the ground truth for *what can happen*.
- **`hl_lead_source_events`** (native) + **`hl_fub_webhook_settings`** (FUB) = the per-source **config / routing** —
  *for this source, is it on, does it create a lead, which workflow fires*. This is **not** a copy of the catalog; it's
  how every door actually routes (read by the workflow dispatch in `lib/workflows/runtime.ts` and the website
  lead-intake). Both layers are required.

## The row

Each row is one trigger event. **The event always fires once it exists** — the toggles gate the lead + the workflow,
never the firing itself.

| Column | What it does |
|---|---|
| **Trigger Event** | Icon + label + description (from `hl_trigger_events`). |
| **Triggered** | How many times it fired (`hl_run_log` door counts + the FUB mirror). **Click it → the leads that fired it, newest first.** |
| **Creates Lead** | Toggle — does this event spin up a NEW lead? **Category-gated:** System behaviors default off (a property view never mints a lead); a few (Virtually Toured) are toggleable; Follow-up is N/A. |
| **Workflow Active** | Toggle — do you want the attached workflow to fire? (The event still fires + notifies regardless.) |
| **Notification** | **Read-only** — which notification class it rolls up to (`notify_class`) + a **Realloop / HouseLoop** indicator. The real on/off + channels + templates live under [Notifications](/guides/notifications). |
| **Process** *(FUB only)* | Whether we ingest that FUB webhook type into the lead's history at all. Native events always process. |
| **Actions (⋮)** | Open in Builder · Clone & assign · (create workflow if none bound). |

## Platform scope on workflows

A workflow's Trigger node = **Trigger Event × Platform scope** (`houseloop` / `fub` / `any`). Firing = *trigger
matches* **AND** (*workflow platform == the event's platform* **OR** *workflow platform == any*). `any` is a wildcard —
one workflow that fires for a tour from any platform. **Notifications are platform-agnostic by default** (they ride the
`notify_class` on the canonical trigger), so you only pick `any` on a workflow for an agnostic *action/nurture*.

## Data source

Everything is read/written live from the CRM Postgres — no separate config store. Catalog: `hl_trigger_events`
(mig 0276) + `notify_class` (0277). Per-source config: `hl_lead_source_events` / `hl_fub_webhook_settings`.
Toggle/attach: `PATCH /api/lead-flow`. Create/clone workflow: `POST /api/workflows` / `.../duplicate`. Fire-count
drill-through: `/api/trigger-events/fires`.

## How it fits

Trigger Events is the middle of the pipeline: **[Lead Sources](/guides/lead-sources) → Trigger Events →
[Workflows](/guides/workflows)**. Lead Sources is where each platform's events are *configured* (Configure deep-links
into this page's tab); Trigger Events is the *catalog*; the Workflow Builder is where the bound workflow is *authored*.
