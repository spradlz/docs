## The roster тАФ business cards in department bands

ONE blue header drives the page: **Members | Roles** tabs. The Members tab renders every non-clone member
as a **fixed business card** (the same `FixedTeamCard` the AI roster uses тАФ 300px, 64px headshot + status
dot, four fixed slots: **Leads ┬╖ Connections (the channel icons тЬЙЁЯУЕЁЯУЮЁЯТм, bright=connected) ┬╖ AI Clone (a LIVE toggle on the card тАФ same wire as the tab) ┬╖ Access**, one button).
Cards group into **department bands** derived from the roles people wear (Real Estate Agents ┬╖ Showing
Partners ┬╖ AI Agents ┬╖ тАж); the "Group by" control flips to a flat Everyone view. Human cards open the
member page; AI cards open the agent's own `/ai/agents/[id]` page. The Add Member ghost opens the one
create modal (`InviteMemberModal` тАФ human or AI agent). Global AI policy (`AiFailoverPolicyCard`,
`AiModelsTable`) stays stacked below тАФ policy, not roster.

## The member page тАФ one flat template (the nesting kill)

`/settings/team/agents/[id]` тАФ humans only (an AI agent's id redirects to its `/ai/agents/[id]` page:
one surface per actor). ONE flat row of tabs, deep-linkable with `?tab=`:

0. **Overview** тАФ the PulseDeck (`/api/team/pulse?scope=member:<id>`): the member's own ten/eight-card KPI
   deck with the Today/7d/MTD/YTD picker. The page is ONE infinite scroll тАФ the pill row at the top is pure
   JUMP navigation (nothing hides); every section below is always expanded.
1. **Profile** тАФ name, email, phone, time zone (debounced auto-save). `/settings/profile` lands here.
2. **Permissions** тАФ the Access select (owner/admin/member) with the can't-edit-your-own-access guard.
3. **Connections** тАФ the star: **Email ┬╖ Google Calendar ┬╖ Phone ┬╖ Text**, four flat sections
   (`MemberConnections`; Phone = the Twilio line + AI-voice line policy; Text = default channel +
   iMessage handle). "Configure email, text, phone" answered on one screen тАФ for AI members too.
4. **Roles** тАФ the member's human **job hats** (chips + one add-role picker). AI roles never appear on
   a human here.
5. **AI Clone** тАФ see below.

The header carries the Active toggle and the **Actions** kebab (Emulate User ┬╖ Delete Member тАФ the
dependency-aware delete with the reassign flow, now shared via `MemberDeleteFlow`).

## The AI clone тАФ built in, not a role

**Every human member has an AI clone built in** тАФ off by default, starting from the shared default
persona. You never "add" a clone and it never appears in a role list. The member page's AI Clone tab
(`CloneSection`) is the entire surface: **one master toggle** (first enable mints the clone row and seats
the plumbing silently, all idempotent; OFF = disarm only тАФ nothing is ever deleted), the **takeover
minutes**, and **"Configure the clone тЖТ"** which opens the clone's own agent page.

<Note>
**One truth under the hood:** a clone's persona/capabilities live on its HUMAN's `hl_member_role` rows тАФ
the exact rows the runtime reads (`loadCloneContext`). The clone's `/ai/agents/[id]` page reads and
writes *through* the human (`clone_of_user_id`), and the roster card API resolves the same way, so what
you see is always what runs.
</Note>

## Roles тАФ job hats, one library

The **Roles tab** is the one role library (AI and human kinds in one table). Roles are JOB HATS тАФ one
member can wear several (Myli: AI ISA today, Receptionist and Content Manager as those services ship).
"AI Clone" in the library is the built-in default template every member's clone starts from тАФ edit it to
change the default; it is not assignable. Role shells (persona ┬╖ capability gates ┬╖ docs ┬╖ model) edit at
`/settings/team/roles/[id]`; per-member overrides edit on the member/agent pages.
