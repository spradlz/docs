**In the app:** CRM sidebar → **Phone & Voice** → **Phone Provider** — [`/settings/phone-provider`](https://crm.houseloop.ai/settings/phone-provider)

## What it does

Phone Provider is where an admin connects the CRM's telephony backend — **Twilio** — and manages the phone numbers the team dials and texts from.

On load the page fetches live status straight from the Twilio REST API and shows it inline:

- **Connection state** — "Twilio Connected" (credentials verified, account `active`) vs "Twilio Saved — Not Verified".
- **Capability pills** — **Voice Ready / Inactive** and **SMS Ready / Inactive**, derived from the capabilities of the numbers on the account.
- **A2P 10DLC campaign** — the messaging compliance campaign status (Approved / Pending TCR / Failed), use case, and submission date, pulled from the account's first messaging service.
- **Phone Numbers table** — every number on the account, each number's Voice/SMS/MMS capabilities, who it's assigned to, call-forwarding target, and voice/text default flags. Rows can be configured, assigned to a member, or a new number can be provisioned.
- **Twilio Account Details** — friendly name, masked Account SID, optional masked Subaccount SID, live account status, and the messaging service name.

The page is **role-aware**: admins/owners see the full Twilio account chrome + all numbers; non-admins see only a "My Phone" view listing the numbers assigned to them (defensive — today everyone is an admin).

The numbers table is the shared `DialerNumbersTable` component, the same one embedded on a member's profile page, so the two surfaces can never drift.

## What you configure

Credentials are entered in the **Add / Edit Phone Provider** form. Only Twilio is supported (the provider type is fixed to Twilio).

| Field | Required | Notes |
|---|---|---|
| Friendly Name | No | Display label; defaults to "Main Twilio Account". |
| Account SID | Yes | Twilio `AC…` account SID. |
| Auth Token | Yes | Twilio auth token; stored server-side, never returned to the browser (shown masked as a prefix/suffix). |
| Subaccount SID | No | Optional; leave blank to use the main account. |

On save the CRM calls the Twilio API to verify the credentials; the record is marked `verified` only when Twilio returns account status `active`. **Disconnect** deletes the stored credentials. Per-number settings (nickname, call forwarding, voice/text default, member assignment) are edited from the numbers table's row actions, and new numbers are provisioned via **Get a New Phone Number**.

## Data source

- **`hl_settings`** — the Twilio credentials + verification flag are stored as a single JSON row under `key = 'phone_provider'` (account SID, auth token, optional subaccount SID, optional API key SID/secret, friendly name, `verified`).
- **`hl_dialer_assignments`** — one row per user mapping a `user_id` to a `phone_number` (plus nickname, `messaging_service_sid`, `call_forwarding`, `is_default`, `is_voice_default`, `is_text_default`, `webhooks_configured`). One number, one living owner is enforced on assign.
- **`hl_users`** — team members joined to their assignment (`is_active` members only).

APIs:

- **`GET/POST/DELETE /api/phone/provider`** — read connection + live Twilio status, save & verify credentials, disconnect.
- **`GET/POST /api/phone/dialers`** — list members + their assignment; assign a number to a member (409 if already held by another active member).
- **`GET /api/me`** — resolves the current user's access level for the admin/non-admin branch.

Live account, number, messaging-service, and A2P data is fetched at request time from the **Twilio REST API** (`api.twilio.com` + `messaging.twilio.com`) using the stored credentials — it is not cached in the database.

## How it fits

Phone Provider is the account-level root of the CRM's Phone & Voice stack. The credentials it stores are what power outbound calling/texting and inbound handling elsewhere in the CRM, and the `hl_dialer_assignments` rows it manages are what bind each teammate to the number they dial and text from. Because it mounts the same `DialerNumbersTable` used on member profiles, number configuration stays consistent whether an admin edits it here or a teammate's number is managed from their profile.
