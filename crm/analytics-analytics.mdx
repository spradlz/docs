**In the app:** CRM sidebar → **Analytics** → **Analytics** — [`/analytics`](https://crm.houseloop.ai/analytics)

## What it does

The top-level `/analytics` route is a **placeholder surface, not a built feature yet.** The page (`app/(crm)/analytics/page.tsx`) renders a single `ComingSoon` component — it has no tabs, charts, or data fetching of its own. It exists so the CRM nav matches the intended product surface 1:1 while the UI is deferred to a later build phase.

<Info>Do not confuse this with the **Website Analytics** surface at `/settings/website/analytics` (GA4 + Google Business Profile), which is a separate, built page with its own components and API routes. This top-level `/analytics` route is unrelated to those and is still stubbed.</Info>

## The tabs / sections

None. The page renders the shared `ComingSoon` stub inside the standard page shell (`PageShell` → `PageHeader` + `PageContent`), showing the title, a one-line blurb, and a "build later" phase note. There are no analytics views wired up.

The blurb in `page.tsx` records the **planned** scope for this surface (not currently implemented):

- Workflow success rate
- Recovery %
- Response times
- Disposition changes
- State-transition frequencies
- Interruption rates
- LLM cost

## Data source

No data source is wired to this page today. `page.tsx` imports only `ComingSoon` (`components/shell/ComingSoon.tsx`), which renders through `app/components/PageShell/PageShell.tsx`. The `ComingSoon` component text notes generically that stub surfaces are "backed by their `hl_*` tables" once built, but this page issues **no queries and calls no API routes**. The planned metrics above (workflow success, LLM cost, etc.) would be sourced when the surface is actually built.

## How it fits

This is the reserved home for cross-workflow / AI operational analytics (success, recovery, response time, LLM cost). Until it's built, the live analytics that do exist live on the separate Website Analytics page.
