## Overview

When a lead comes into HouseLoop from **Follow Up Boss**, this is the layer that turns it into a
fully-contextualized HouseLoop lead — its facts, its live activity, and Myli's dossier on it.

<Note>
**This is the bridge, not the plumbing.** *How FUB data lands in our `fub.*` DB* is the
[FUB Data System](/fub/scraper) (webhooks + scraper). **This page** is the layer above it. FUB is
**lead source #1** — the whole pattern (import → facts → live sync → dossier) is source-agnostic and
future sources (Zillow-direct, etc.) live under this same "Lead Sources" group.
</Note>

The model: one lead, three independent layers.

```
FUB lead (fub.* on Railway)
   │
   ├─ Layer 1  REAL-TIME SYNC ........ live counts + activity on the lead card (event-driven, no polling)
   ├─ Layer 2  FACTS (deterministic) . the 12-domain fact store — structured, no LLM
   └─ Layer 3  DOSSIER (on-demand LLM) . one Lead Context = [FUB section] + [HouseLoop section]
```

## Tour & Contact requests → the HouseLoop feed (the two-pipe rule)

There are **two separate pipes** for a FUB lead's activity, and they do not feed each other:

- **Pipe A — Myli's context** (the **Follow Up Boss Activity** panel): the lead's FULL FUB history —
  texts, calls, notes, every property inquiry, Zillow views — read live from `fub.*` by link. **All FUB
  history lives here.**
- **Pipe B — the HouseLoop activity feed** (the **All** + **Website Activity** tabs, `hl_lead_activity`):
  only **tour/contact REQUESTS** cross into here. FUB texts/calls/old activity never cross.

The rules (LOCKED 2026-06-26):

| Rule | Behavior |
| --- | --- |
| **One event per inquiry** | Each FUB tour/contact request becomes its **own** HouseLoop activity event (keyed per property), so a lead with 4 tour requests shows **4 request cards** — not one. |
| **First touch vs request** | The lead's **first** touch = a `registration` (New Lead Registration). Every request **after** = a `tour_request` / `contact_request` event → shows under **Website Activity → Requests** + **All**. |
| **Request SOURCE = the portal** | Each request carries the portal it came FROM — `fub.events.source` (e.g. **Zillow Preferred**, Homes.com, Opcity, realtor.com) for FUB, **HouseLoop** for native — shown as a badge on the row. This is **separate** from the LEAD source (which stays `SRG–Follow Up Boss`). |
| **No backfill (watermark)** | Per the Watermark Law, only inquiries newer than a per-lead `synced_at` cursor (starting **now**) create HouseLoop events. Past FUB texts/tours/website activity are **not** mirrored into the HouseLoop feed — they stay in Pipe A. |

## The architecture in detail

### Layer 1 — Real-time sync (event-driven, NO polling)
```
Camp David fub.*  →  relay  →  Railway fub.*  ──NOTIFY fub_lead_changed(fub_person_id)──►
   HouseLoop fubListener  →  lead-detail SSE (/api/leads/[id]/stream)  →  card updates instantly
```
The relay fires a Postgres **`NOTIFY`** on each write. HouseLoop's `fubListener` + the lead-detail SSE
stream push the change to any open card — **no polling** (HouseLoop law). The panel reads the **real
counts** from `fub.zillow_home_activity` / `fub.notes` / etc., never FUB's display number.

### Layer 2 — The Facts (deterministic extractor)
On first pull, the extractor structures every FUB fact into HouseLoop's **12-domain fact store** — name,
source, timeframe, price range, home views/saves/searches, Zillow Insights. **No LLM, fully structured.**

### Layer 3 — One Lead Context + the dossier (on-demand LLM)
| Section | Source | When it generates |
|---|---|---|
| **Follow Up Boss** | imported FUB history (notes, texts, calls, views, saved searches, insights) via `loadFubBrief` | **on-demand** — `POST /api/leads/[id]/context` (Sonnet, rich) |
| **HouseLoop** | everything happening in HouseLoop going forward | live |

Myli reads the **union** (FUB KB + FUB dossier + live HouseLoop) for any action. Regen is **on-demand**,
never auto-per-fact.

## ⭐ The Watermark Law — read-once, merge-forward (universal)

Governs **every** LLM rollup (dossier, facts, Hot Buyers, any summary) **and the requests→feed sync**:
**never re-process data already consumed.** The dossier stores its text + a **`source_cursor`** (high-water
mark of everything already baked in). Each regeneration:

<Steps>
  <Step title="Read only the delta">`SELECT … WHERE synced_at > source_cursor` — only rows we haven't summarized yet.</Step>
  <Step title="Merge, don't re-summarize">Prompt = `[existing dossier] + [only the new rows]` → *"merge the new events in; leave the old as-is."*</Step>
  <Step title="Advance the watermark">Write the updated dossier + `source_cursor = max(synced_at)` consumed.</Step>
</Steps>

First pull = full history (Sonnet, cursor = now). Every pull after = only the delta, merged with Haiku —
cheap. The old views are **never re-read.** Tokens paid **once per row, ever.**

<Warning>
**The gotcha that makes it correct:** watermark on **`synced_at` (when WE ingested the row), NOT the
event's own date.** A home view from 2 years ago the scraper *just* backfilled is **new to us** —
`synced_at > cursor` catches it. Watermark on the event date and you'd silently skip every freshly-
backfilled historical row.
</Warning>
