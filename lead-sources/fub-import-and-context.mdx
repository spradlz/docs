## Overview

When a lead comes into HouseLoop from **Follow Up Boss**, this is the layer that turns it into a
fully-contextualized HouseLoop lead — its facts, its live activity, and Myli's dossier on it.

<Note>
**This is the bridge, not the plumbing.** *How FUB data lands in our `fub.*` DB* is the
[FUB Data System](/fub/scraper) (webhooks + scraper). **This page** is the layer above it. FUB is
**lead source #1** — the whole pattern (import → facts → live sync → dossier) is source-agnostic and
future sources (Zillow-direct, etc.) live under this same "Lead Sources" group.
</Note>

The model: one lead, three independent layers.

```
FUB lead (fub.* on Railway)
   │
   ├─ Layer 1  REAL-TIME SYNC ........ live counts + activity on the lead card (event-driven, no polling)
   ├─ Layer 2  FACTS (deterministic) . the 12-domain fact store — structured, no LLM
   └─ Layer 3  DOSSIER (on-demand LLM) . one Lead Context = [FUB section] + [HouseLoop section]
```

## The architecture in detail

### Layer 1 — Real-time sync (event-driven, NO polling)
```
Camp David fub.*  →  relay  →  Railway fub.*  ──NOTIFY fub_lead_changed(fub_person_id)──►
   HouseLoop pgListener  →  lead-detail SSE (/api/leads/[id]/stream)  →  card updates instantly
```
The relay fires a Postgres **`NOTIFY`** on each write. HouseLoop's existing `pgListener.ts` + the
lead-detail SSE stream push the change to any open card — **no polling** (HouseLoop law). The panel reads
the **real counts** from `fub.zillow_home_activity` / `fub.notes` / etc., never FUB's display number.

### Layer 2 — The Facts (deterministic extractor)
On first pull, the extractor structures every FUB fact into HouseLoop's **12-domain fact store** — name,
source, timeframe, price range, home views/saves/searches, Zillow Insights. **No LLM, fully structured.**

### Layer 3 — One Lead Context + the dossier (on-demand LLM)
| Section | Source | When it generates |
|---|---|---|
| **Follow Up Boss** | imported FUB history (notes, texts, calls, 1,730 views, saved searches, insights) | **once** on first pull — Sonnet, rich; then frozen until regenerated |
| **HouseLoop** | everything happening in HouseLoop going forward | live |

Myli reads the **union** (FUB KB + FUB dossier + live HouseLoop) for any action. Regen is **on-demand**,
never auto-per-fact.

## ⭐ The Watermark Law — read-once, merge-forward (universal)

Governs **every** LLM rollup (dossier, facts, Hot Buyers, any summary): **never re-run an LLM over data
it already consumed.** The dossier stores its text + a **`source_cursor`** (high-water mark of everything
already baked in). Each regeneration:

<Steps>
  <Step title="Read only the delta">`SELECT … WHERE synced_at > source_cursor` — only rows we haven't summarized yet.</Step>
  <Step title="Merge, don't re-summarize">Prompt = `[existing dossier] + [only the new rows]` → *"merge the new events in; leave the old as-is."*</Step>
  <Step title="Advance the watermark">Write the updated dossier + `source_cursor = max(synced_at)` consumed.</Step>
</Steps>

First pull = full history (Sonnet, cursor = now). Every pull after = only the delta, merged with Haiku —
cheap. The 1,730 old views are **never re-read.** Tokens paid **once per row, ever.**

<Warning>
**The gotcha that makes it correct:** watermark on **`synced_at` (when WE ingested the row), NOT the
event's own date.** A home view from 2 years ago the scraper *just* backfilled is **new to us** —
`synced_at > cursor` catches it. Watermark on the event date and you'd silently skip every freshly-
backfilled historical row.
</Warning>
