There are **three ways to search** on `houseloop.ai`, and they all run on the **same engine over the
same query envelope** — the same URL param means the same thing everywhere, and every filter is
explicit in the URL (no presets, no magic flags):

1. **The main search bar** — type a place, price, beds, home type. A deterministic classifier turns it
   into removable chips and lands you on the results gallery.
2. **The AI home search** — describe your dream home in plain English ("single-family with a private
   pool and a white kitchen in Overland Park"). Claude extracts filters constrained to our registry,
   matches our AI vision catalog, and answers superlative questions.
3. **The Details filters** — the human filter modal (status, price, beds/baths, style, and 20 detail
   facets) whose checkbox counts equal the real result counts.

All three read our own **`Listing` cache** (the Heartland MLS feed, centerbeam DB), never Repliers live.

## 1. The main search bar

The bar is the fast path: a buyer types a place / price / beds / home type and lands on the gallery. It
handles the **six deterministic facets** (location, home type, status, beds, baths, price); anything
richer (amenities, "white kitchen", "pool", floor plan) routes to the AI home search.

**Flow:** type → `POST /api/search/classify` (Haiku) → `{ intent, params }` → the bar applies the shared
deterministic defaults, renders **removable chips**, and navigates to `/listings?<params>`.

```bash
curl -X POST "https://houseloop.ai/api/search/classify" \
  -H "content-type: application/json" \
  -d '{"q":"Overland Park single family homes with a private pool"}'
# → { "intent": "normal",
#     "params": { "city": "Overland Park", "style": "Single Family Residence", "feature": "private_pool" } }
```

- **The bar NEVER asks a question.** It applies smart defaults and shows removable chips (the AI modal is
  where clarifying happens). "homes" → `style=Single Family Residence`; "single story" → `story=1`.
- **`params` carries** `city / style / story / minBeds / minBaths / minPrice / maxPrice / feature / detail.*`.
- Location typeahead comes from `GET /api/locations/autocomplete` (addresses, cities, counties,
  neighborhoods, school districts, schools). Selecting an address goes straight to the listing.

<Note>
**One brain, two surfaces.** The bar (`classify`) and the AI modal (`nlp`) share `search-defaults.mjs`, so
they emit **identical** params for the same query — the bar just never converses.
</Note>

## 2. The AI home search

The AI modal lets a visitor describe a home in plain English and **answers** instead of dumping a list.
Entry points: the **"AI Search"** nav item, the gradient **"AI Search"** button beside the hero bar, and
the **"Search smarter with AI"** row in the dropdown — all open one chat modal that calls `POST /api/search/nlp`.

**Claude Haiku 4.5** (server-side — the key never reaches the browser) runs the conversation: a friendly
reply + structured filter extraction (constrained to our registry, grounded to the buyer's words) + a
`ready` decision. When `ready`, the search resolves through a **fixed layer order** — first layer that can
handle the query wins:

<Steps>
  <Step title="Superlative answer">
    If the query carries a sort ("most expensive / cheapest / biggest / most beds"), we **answer the
    question** — singular → ONE home + a short factual narration; plural → the sorted list.
    `url = /listings?sort=<sortKey>`. (Zach's rule: "if I ask a question, answer it — not 24 matching + 455 nearby.")
  </Step>
  <Step title="Vision-tag match (PRIMARY for tagged metros)">
    Descriptive features match our **pre-computed AI vision catalog** of each listing's photos — instant
    SQL, photo-verified, exact. Skipped where a metro isn't vision-tagged yet.
  </Step>
  <Step title="Repliers NLP (untagged fallback)">
    `POST https://api.repliers.io/nlp` extracts structured filters + runs Repliers' own image search — a
    **thin fallback** only where we haven't vision-tagged.
  </Step>
  <Step title="Description re-rank (last backstop)">
    Reads listing descriptions to reject false positives ("white quartz countertops" ≠ white kitchen,
    "pool table" / "community pool" ≠ private pool).
  </Step>
</Steps>

The reply returns the match count, a `listings` preview array, and a **View N homes** URL that deep-links
into the gallery (matches pinned + badged "AI Match" via `aimatch=`).

### Vision Catalog (the primary matcher)

At ingest, **gpt-4o-mini** (vision, low detail, up to 16 photos) reads each listing's photos ONCE against a
controlled matrix of **50+ buyer-searchable features** and caches the result in `Listing.visionTags` (JSON).
Descriptive search is then **instant SQL** — no per-query LLM, no Repliers round-trip.

### Private vs HOA amenities (vision-verified)

A home's OWN pool is separated from the neighborhood's. `visionTags.pool_type = "private"` only when the
model SEES a pool in the home's own yard; community/HOA pools are tagged `"community"` and never match a
private-pool query. The `private_pool` feature **also** triggers the single-family privacy rule, so
community-pool condos never leak into a "homes with a pool" search.

## 3. The Details filters (+ results & map)

The Details modal is the human filter surface: status, price, beds/baths, style, lot size, and **20 detail
facets** (fireplace, flooring, kitchen features, exterior, etc.). Every checkbox shows a **count that equals
the real result count** (F1) and every applied pill names its field (F2).

### The query envelope (shared by every surface)

| Param | Meaning | Example |
|---|---|---|
| `city` | City name (comma-list for multi) | `city=Overland Park` |
| `state` | State abbr | `state=KS` |
| `zip` | ZIP (comma-list) | `zip=66085` |
| `status` | Listing status — CSV of group tokens. Omitted = default (For Sale + Coming Soon). | `status=forSale,sold` |
| `sort` | Sort order — a SORT, not a filter; persists across filter changes. | `sort=newest` |
| `minPrice` / `maxPrice` | Price band | `minPrice=400000` |
| `minBeds` / `minBaths` | Bed / bath floor | `minBeds=3` |
| `style` | Heartland style discriminator | `style=Single Family Residence` |
| `story` | Floor-plan stories | `story=1` |
| `feature` | Amenity/vision feature (CSV) | `feature=private_pool` |
| `detail.<field>` | A Details-facet value | `detail.fireplace=Great Room` |
| `minAcres` / `maxAcres` | Lot size band | `minAcres=2` |
| `bbox` | Viewport `W,S,E,N` (map) | `bbox=-94.8,38.8,-94.5,39.1` |
| `limit` / `page` | Page size (caps at 100) / 1-based page | `limit=24` |

**Sort values:** `newest` (**website-arrival, `firstSeenAt` desc — no status tier**), `viewsDesc`,
`priceAsc`, `priceDesc`, `bedsDesc`, `bathsDesc`, `sqftDesc`, `lotsizeDesc`, `priceDropDesc`, `recentlySold`.
Default is `newest`. Sort **persists** as you add/remove location filters.

### Status filter & counts

The `status` param is a CSV of exactly four group tokens:

| Token | Means | DB status |
|---|---|---|
| `forSale` | Active / for sale | `active` |
| `comingSoon` | Coming soon | `coming_soon` |
| `underContract` | Pending / under contract | `pending` |
| `sold` | Sold (closed) | `sold` |

**Default (no `status=` param) = `forSale,comingSoon`.** Locked because the UI status picker writes a *null*
`status` param whenever exactly that pair is selected — so "no param" and "For Sale + Coming Soon" MUST
resolve to the same set (this was the 348-vs-66 bug).

<Note>
**Status is ADDITIVE — "results include every status you pick."** Each token adds its full set; the total
is the sum. `forSale` → 58 · `comingSoon` → 8 · together → **66**. `sold` draws from the **full** sold
archive (`ListingHistory`, ~12-month window) even when combined, and the engine presents live homes first,
then sold by sale-date desc. **One count source:** every count a user sees (results header, Filters footer,
price-histogram, saved-search card) calls `/api/listings/count` with the same status draft — they always agree.
</Note>

### Facet counts come from OUR DB (F1)

`GET /api/detail-search-facets` returns each detail field's checkbox values + counts, computed from the SAME
WHERE `searchFromDb` uses (`computeDetailFacets` in `lib/listings/detail-facets-db.ts`) — so the count for a
value equals the number of homes you get when you check it, by construction. (Was Repliers aggregates, a
different dataset → every count wrong.)

### Results gallery + map

`GET /api/listings/search` returns `Listing` objects (price, address, beds/baths/sqft, photos, status,
`mlsListedAt`) + engagement stats, batched per listing. `GET /api/listings/count` is the cheap total.

`GET /api/listings/map` takes the **same envelope** plus map params so the map mirrors the list exactly:

- **Marker mode** (default, `mode:"pins"`, **no clustering, ever**): `pins` is capped at 50 per viewport
  (`MAX_MAP_PINS`, stride-sampled; AI-match pins always kept), so the UI shows "{pins.length} of {total}"
  (e.g. "46 of 312"). Zoom in → bbox shrinks → every home shows.
- **Fit mode** (`bounds=1`): returns the bounding box of the whole filtered set to auto-frame the map.
- **Search-as-you-move:** in map/split view the client pushes the live viewport `bbox` into the URL on
  every pan/zoom, so the list + count re-scope to what's on screen. Markers flip to price pills by pixel
  spacing (never stacked); AI matches always render as ✨ pills.

`GET /api/geo/place-boundary` draws the Census TIGER outline around a searched city.

## Endpoints (public website read APIs)

Served from `https://houseloop.ai/api` (prod) / `http://localhost:3001/api` (dev). Read-only `GET` unless noted.

| Endpoint | Purpose |
|---|---|
| `POST /api/search/classify` | The bar router — deterministic extraction (Haiku) → `{ intent, params }` |
| `POST /api/search/nlp` | One AI conversational turn → `reply` + (when `ready`) `count / aiMatchHlids / url / listings[] / filters` |
| `GET /api/listings/search` | Full search (envelope → results), `limit` default 24, max 100 |
| `GET /api/listings/count` | Just the total for a query (byte-identical to what `/listings` renders) |
| `GET /api/listings/map` | Map markers / fit bbox for the current viewport (same envelope) |
| `GET /api/detail-search-facets` | Details-modal checkbox values + counts (from our DB — F1) |
| `GET /api/locations/autocomplete` | Search-bar typeahead (`q` ≥ 3 chars) |
| `GET /api/geo/place-boundary` | Census TIGER named-place outline for a city |
| `GET /api/home` | The composed home page (ordered shelves + inspiration), cached 5 min |
| `GET /api/listings/{hlid}` | A single listing (full `photos.all` gallery + price `history`) |

**Home shelves:** each shelf = the **12 newest-listed homes** in its filter, new listings push in at the
front. Shelves are DB-config (`hl_home_shelf`, seeded from `prisma/seed/home-shelves.ts`) — not hardcoded.

## IDX attribution (MLS GRID / Heartland)

HouseLoop displays Heartland MLS listings via the MLS GRID IDX feed, so the
[MLS GRID IDX Rules](https://www.mlsgrid.com) govern where broker attribution + the source disclaimer must
appear. The key lever is the **200-character minimal-info exemption**.

> **§13/§23/§24/§25 (verbatim):** *"Displays of minimal information (e.g. a one-line or 'thumbnail,' text
> messages, 'tweets', etc. of two hundred (200) characters or less) are exempt from this requirement but
> only when linked directly to a display that includes all required disclosures."*

| Surface | Component | Attribution |
|---|---|---|
| Gallery / list card | `components/listing/ListingCard.tsx` | **None** — minimal-info ≤200 chars linking to detail (§13/§23). |
| Map pin preview | `components/listings/ListingsMap.tsx` | **None** — same exemption (§24/§26). |
| Listing **detail** page | `components/listing/ListingDetail.tsx` | **Required** — brokerage (§23) + `Source: Heartland MLS as distributed by MLS GRID · MLS #{n}` (§24) + Heartland logo. |
| Site **footer** (every page) | `components/site/Footer.tsx` | **Required** — §24 source + logo, §25 full disclaimer, §32 DMCA. The site-wide carrier that keeps cards/pins exempt. |

Use `components/listing/HeartlandMlsLogo.tsx` for the logo — never a raw `<img>`, and **never on a
minimal-info surface** (cards, map pins, thumbnails). Other hard rules: no seller name/phone/email (§30),
no listing-agreement type (§21), refresh ≤12h (§11), ≤500/2,500 listings per search (§27), per-address
**sold price never displayed** (Heartland §12.3(b)). Source of truth: `products/houseloop/Drop/IDX Rules.pdf`
+ `GT/GT - ACTIVE/GT_HOUSELOOP_SEO.md` §15.3.
