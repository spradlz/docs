The **HouseLoop Home Value Report** is the per-home value story for a past client. The **same underlying
report** renders in three places:

1. **The seller-report modal** (internal — what the *agent* looks at). Open it from `/settings/seller-report`.
   Third-person voice ("since **they** purchased").
2. **The seller-report dashboard** (`/settings/seller-report`) — the command center listing every past client
   with their current home + value. Third-person / agent-facing.
3. **The monthly "Home Valuation Report" email** (client-facing — what we *send the homeowner*). Second-person
   voice ("since **you** purchased").

All three read the home's value from the website `Listing` (see
[Past-Client Data Pipeline](/guides/srg-reconciliation-engine) for how that value gets there). The website is
our Zillow; the CRM pulls value FROM it by `website_hlid` and never stores or recomputes it (no copy = no drift).

<Note>
**What's built (2026-07):** The seller-report **dashboard is LIVE** (`/settings/seller-report`) and the
**modal is built + wired** — `apps/houseloop-crm/components/seller-report/SellerReportHomeModal.tsx` renders

## Where the value data lives (the ONE truth — no valuation table)

There is **no `hl_srg_home_valuations` table**. It existed only for migration `0136` and was **dropped**
afterward under [Data Doctrine](/guides/property-data-and-address-identity) law #5 — any doc or code that
still cites it is stale. Live value reads come from:

- **The web `Listing`** — `hlEstimate` / `hlEstimateLow` / `hlEstimateHigh` / `zestimate`, resolved by
  `website_hlid` (`apps/houseloop-crm/app/(crm)/settings/seller-report/page.tsx`, ~lines 210–242). This is the
  SAME value the public `/home` page renders — one source, never re-derived.
- **`hl_value_history`** (web side) — the monthly value-history series.
- **`hl_srg_past_clients`** (1,779 rows — the SSOT) + child **`hl_srg_past_client_homes`** — the person, their
  current home pin (`website_hlid`), purchase price/date, and `closings`.

## Where each section reads from

Everything is anchored on the client's **most-recent home** (`is_current_home`), keyed by `website_hlid` →
the web `Listing`.

| Section | Source | Status |
|---|---|---|
| **HouseLoop Value** distribution — Low / Estimated / High + $/ft² | web `Listing.hlEstimate` / `hlEstimateLow` / `hlEstimateHigh` (Zestimate band, or soldPrice band for a fresh sale), resolved by `website_hlid` | ✅ wired |
| **HouseLoop Value History** — line over time, % over the whole line (since 2012). **Never blank** for a home with a value: ZHVI floor → 2012 + the **real sale anchored on the line** → [Sold Data & Price History](/guides/sold-data-and-price-history) | web `Listing.valueHistory` if present; else built read-side in `get.ts` from the real sale + ZHVI; re-anchored to the oldest real price | ✅ wired |
| Purchase price · Total Gain · Avg Monthly Gain · appreciation % | `hl_srg_past_client_homes` (purchase) + the value above (current) — computed | ✅ wired |
| **Client lifetime** — # deals, lifetime volume, every closing | `hl_srg_past_clients.closings` + child homes (modal + dashboard, not the email) | ✅ wired |
| **FUB profile link(s)** — one per person (a couple = two links) | `hl_srg_past_clients.fub_id` + spouse in `fub_identity` → FUB person URL | ✅ wired |
| **Comparable sales** — ~3 addresses + sale price, each linking to a houseloop.ai saved search | a comps feed (Repliers sold comps) | ⛔ **NOT wired into the CRM yet** |
| **Neighborhood average** (last 12 months) | market/neighborhood stats (`hl_geo_market_stats` / `hl_market_index`) | ⛔ **NOT wired into the CRM yet** |

## The two value visuals (the centerpiece)

These are the two graphs from the `/home` page
(`apps/houseloop-web/components/listing/HouseLoopValueHero.tsx`):

1. **HouseLoop Value** — a bell-curve bar distribution with **Low / Estimated Value / High** columns (value +
   $/ft², circle icons), titled "Estimated Sales Range." (301 E 68th: Low `$501,244` · Est **`$527,625`** ·
   High `$554,006`.)
2. **HouseLoop Home Value History** — an interactive line/slider over time. **The line always reaches back to
   2012** (the ZHVI floor), with **every real sale anchored on it** (the line passes through the actual
   transacted price at its date), climbing to today's HouseLoop value. The **appreciation % spans the whole
   line — since the 2012 floor**, and the % **label reads the line's start year** (e.g. "+100.6% since 2012"),
   not a flat "since sold". A home with no real sale at all uses a pure ZHVI back-projection to 2012. It is
   never blank for a home that has a value.

<Note>
**Value-history graph — never blank, anchored on the real sale, floored to 2012 (Benji, 2026-06-28).**
Audit found **125 sold homes (4.3%) — 80 of them past-client (4.5%)** that had a HouseLoop value but a
**blank** value-history graph (e.g. 816 E Saddlewood). Cause: the sold/stored-value branch only read the
precomputed `Listing.valueHistory` column, which is empty for homes that sold before the disposition pipeline.
**Fixed read-side in `get.ts`:** when the column is empty, build the line as **ZHVI back-projection
(`hl_market_index`, 2012 → the sale month) + `appreciationRocket` (real sale → today)** — the real sale sits ON
the line as an anchor, and the graph always reaches **2012**. Verified: 816 E Saddlewood now runs **Jan 2012
`$228,348` → Jul 2020 `$346,000` (real sale, on the line) → Jun 2026 `$458,179`, +100.6% since 2012.**
Mechanics: [Sold Data & Price History](/guides/sold-data-and-price-history).
</Note>

<Note>
**Architecture note (do not cross-app copy).** `HouseLoopValueHero` is a **website** component built in
**Tailwind**; the CRM modal uses inline design tokens (no Tailwind). The modal **rebuilds these two visuals
CRM-native**, reading the value data off the web `Listing` (`qWeb`) — it does not import the Tailwind component
across the app boundary.
</Note>

## The seller-report modal

`apps/houseloop-crm/components/seller-report/SellerReportHomeModal.tsx` — a wide modal (the two value visuals
**side by side**), internal agent view, third person ("since **they** purchased"). Sections:

1. **Header** — client name + most-recent home address. Estimated Value + range.
2. **The two value visuals, side by side** — HouseLoop Value distribution + Value History (rebuilt CRM-native).
3. **Gain stats** — Purchase Price · Current Value · **Total Gain** (current − purchase) · **Avg Monthly Gain**
   (total ÷ months held) · appreciation %. (No mortgage/equity — deferred to Realie AVM; see the past-client
   pipeline.)
4. **Client lifetime** *(agent-only, not in the email)* — # deals, lifetime volume, every closing, formatted.
5. **FUB profile link(s)** — the client's FollowUp Boss profile; for a couple (e.g. Kyle **and** Karly
   Nations), **one link each**, resolved from `fub_id` + the spouse record in `fub_identity`.
6. **Comparable sales** *(future — needs the comps feed)* — ~3 comps as **address · sale price · link**, each
   link opening houseloop.ai with a saved search pre-loaded with those comparable properties.
7. **Neighborhood average (last 12 mo)** *(future — needs neighborhood stats wired into the CRM)*.

## The monthly email template

The **"Home Valuation Report (Monthly)"** email is sent to past clients from their assigned agent (Ana
Spradling, Zach Spradling, or Myli). It combines a short personal note with a full property valuation card.
The monthly send is the **`hvrMonthlySend`** Inngest function (`hvr-monthly-send.ts`); the subject line reads
*Your [Month Year] Home Valuation Report — [address]*.

<Warning>
**The monthly cron is NOT live yet.** `hvrMonthlySend` is **event-only** (`hvr/monthly-send.run`, dryRun default
true) — the `0 14 28 * *` schedule is gated pending Zach's sign-off (INNGEST law). The only past-client value jobs
running on a schedule today are the two WEB crons: **`pastClientZillowCache` (26th)** + **`valueHistoryPersist`
(27th)**. Until the HVR cron is wired, no report auto-sends.
</Warning>

### Email structure

**1. Personal message (above the report).** A 2–3 sentence casual note from the agent, AI-generated from merge
fields at send time but reading like a human wrote it. No branding above it — reads like a plain email.

> Hey John,
>
> Just wanted to send over your June home value update — looks like your home went up another **$13,000** this
> month! The Blackthorne Estates neighborhood is continuing to do really well, and your home is actually
> outperforming the broader KC market. Here's your full report below.
>
> **Ana Spradling** — Spradling Group | eXp Realty

**2. Report card.** A single white rounded card:

| Section | Description |
|---------|-------------|
| **Header** | Lockup A (stacked: Spradling Group eXp / Powered By / HouseLoop) + "Home Valuation Report" title + month pill |
| **Property photo** | Hero image scraped from Redfin, stored in R2 |
| **Address card** | Street, city/state/zip, beds/baths/sqft/year/subdivision |
| **Estimated value** | Big bold number on white card, 56px DM Sans |
| **Valuation range** | Low / Estimate / High bar-chart distribution (blue→green→yellow) |
| **Value history chart** | SVG line chart from purchase date to current, with gradient fill |
| **Market temperature** | Gauge bar with plain-English verdict ("Balanced & Stable") + DOM / list-to-sale / inventory |
| **Beating the market** | Three stat boxes: Your Home vs ZIP vs KC Metro appreciation |
| **Appreciation cards** | +X% / +$X card alongside purchase price/date card |
| **Comparable sales** | Horizontal scroll of comp cards with photos, address, price, sold date, bed/bath/sqft, distance, similarity % |
| **Agent + CTA** | Agent headshot + contact + "Schedule a Call" full-width button |
| **Brand band** | Dark blue footer with white Spradling Group / eXp / HouseLoop logos + disclaimer |

Design inspiration: **HomeBot** (value-over-time + market verdict), **House Canary** (Low/Med/High bar chart +
comp cards), **Zillow** (three stat boxes + history chart).

## The seller-report dashboard (LIVE)

`/settings/seller-report` (`apps/houseloop-crm/app/(crm)/settings/seller-report/page.tsx`) is the command
center for monthly home value reports. It is **live**, not "coming soon." It lists every past client with a
HouseLoop property profile and:

- Reads the past-client list from **`hl_srg_past_clients`** (1,779 — the SSOT) + child
  **`hl_srg_past_client_homes`** — NOT the raw SISU `hl_transactions` import.
- Pulls each home's **value live from the web `Listing`** by `website_hlid` (`hlEstimate` / `Low` / `High`,
  `zestimate` fallback) — never a stored copy.
- Shows current home, purchase price/date, gain, and report status per client, sortable by name / address /
  value change.
- Row → opens the seller-report modal above.

## The Seller Tools product frame (design spec v1 — NOT built yet)

<Note>
**This section is a DESIGN SPEC, not shipped work.** It's the forward product frame for everything above,
captured as one interactive artifact (v1, 2026-07-23) after a competitor teardown. The live reality is the
sections above (dashboard + modal live; monthly email, comps, neighborhood not wired). Build to this frame;
don't read it as "done."
</Note>

The teardown source is **Vox / VoChat AI** (Brandon Duncan) — a past-client seller engine. The artifact pairs
all 8 of their screens with the HouseLoop version of each, and pins these locked decisions:

- **Where it lives — a new `Seller Tools` left-nav** with two tools under it: **Seller Reports** (the
  past-client owner dashboards + the CRM report + the monthly email — everything in this guide) and **AI CMA**
  (the [CMA Tool](/listing-presentation/crm-roadmap)). The dashboard header reads only **"Seller Tools"** — no
  brokerage / "past clients" label in the chrome.
- **The value model is the unfair advantage.** Where the competitor shows one estimate, the report shows
  **Zillow + Redfin + the HouseLoop Estimate + HouseLoop Home History + Trends**, blended into one headline the
  homeowner trusts. (Zillow is cached on the `Listing`; the Redfin scraper is built + running; the HouseLoop
  Estimate is the headline — see [Houseloop Home Value](/guides/houseloop-home-value).)
- **The agent dashboard** is governed by a **HouseLoop AI · Master switch** over a **Myli Seller Engine** toggle
  ("governed by the master switch above" — master off = the engine sends nothing). Plus KPIs and a scored
  past-client table (value / equity / sell score).
- **The owner dashboard IS the full report** — the destination of the email's **"See My Full Report"** button.
  Specced in depth on **desktop and mobile**: navy hero (welcome + big value + likely range), the 3-source
  value, value-over-time chart, equity split, home facts, an ownership-history timeline, neighborhood stats,
  recent nearby sales, the agent card, and a "thinking of selling?" net sheet.
- **One value record, three surfaces (one cohesive environment):** the monthly **email** → the owner
  **dashboard** (their branded "app") → the public **`/home`** page. Same value, three renderings; the email is
  the front door into the dashboard, the public page is the same value made shareable.
- **CTA copy is locked:** "**Sell My Home**" + "**Talk to Your Agent**" (not "Request an offer" — we're not
  iBuyers). "Talk to Your Agent" launches Myli.
- **Brand:** HouseLoop tokens + Inter, single light theme; the real *Spradling Group · powered by HouseLoop*
  lockup image (never CSS text), per the lockup law.
