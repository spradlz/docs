<Note>
**Value-history graph — never blank, anchored on the real sale, floored to 2012 (Benji, 2026-06-28).**
Audit found **125 sold homes (4.3%) — 80 of them past-client (4.5%)** that had a HouseLoop value but a
**blank** value-history graph (e.g. 816 E Saddlewood). Cause: the sold/stored-value branch only read the
precomputed `Listing.valueHistory` column, and homes that sold before the disposition pipeline have an empty
column. **Fixed read-side in `get.ts`:** when the column is empty, build the line as
**ZHVI back-projection (`hl_market_index`, 2012 → the sale month) + `appreciationRocket`(real sale → today)** —
the real sale sits ON the line as an anchor, and the graph always reaches **2012**. The **appreciation % is
measured across the WHOLE line — since the 2012 floor** (the % label reads the line's start year, not a flat
"since sold"). No real sale at all → pure ZHVI 2012 → today. Verified: 816 E Saddlewood now runs **Jan 2012
$228,348 → Jul 2020 $346,000 (real sale, on the line) → Jun 2026 $458,179, +100.6% since 2012.** Mechanics:
[Sold Data & Price History](/guides/sold-data-and-price-history).
</Note>

The **HouseLoop Home Value Report** is the per-home value story for a past client. It exists in two places
that render the **same underlying report**:

1. **The seller-report modal** (internal — what the *agent* looks at). Click the eye / "View home report" on
   a client in `/settings/seller-report`. Third-person voice ("since **they** purchased").
2. **The monthly "Home Valuation Report" email** (client-facing — what we *send the homeowner*). Already
   built; see [Home Valuation Template](/guides/home-valuation-template). Second-person voice ("since **you**
   purchased").

The modal **mimics** the email report, and adds an agent-only **client lifetime** section (deals, volume) and
**FUB profile links**. Both read the home's value from the website `Listing` (see
[Past-Client Data Pipeline](/guides/past-client-data-pipeline) for how that value gets there).

## Where it reads from

Everything is anchored on the client's **most-recent home** (`is_current_home`), keyed by `website_hlid` →
the web `Listing`. The value + history are the SAME data the public `/home` page renders (one source, never
re-derived):

| Section | Source | Status |
|---|---|---|
| **HouseLoop Value** distribution — Low / Estimated / High + $/ft² | web `Listing.hlEstimate` / `hlEstimateLow` / `hlEstimateHigh` (Zestimate × 1.005 band, or soldPrice × 1.005 for a fresh sale) | ✅ data exists |
| **HouseLoop Home Value History** — line over time, % over the whole line (since 2012). **Never blank** for a home with a value: ZHVI floor → 2012 + the **real sale anchored on the line** → [Sold Data & Price History](/guides/sold-data-and-price-history) | web `Listing.valueHistory` if present; else built read-side in `get.ts` from the real sale + ZHVI; re-anchored to the oldest real price by `reanchorValueHistory` | ✅ data exists |
| Purchase price · Total Gain · Avg Monthly Gain · appreciation % | `hl_srg_past_client_homes` (purchase) + the value above (current) — computed | ✅ data exists |
| **Comparable sales** — ~3 addresses + sale price, each linking to a houseloop.ai saved search | a comps feed | ⛔ **NOT wired into the CRM** |
| **Neighborhood average** (last 12 months) | market/neighborhood stats (`hl_market_index` / market-report) | ⛔ **NOT wired into the CRM** |
| **Client lifetime** — # deals, lifetime volume, every closing | `hl_srg_past_clients` + child homes (modal only, not the email) | ✅ data exists |
| **FUB profile link(s)** — one per person (a couple = two links) | `hl_srg_past_clients.fub_id` + spouse in `fub_identity` → FUB person URL | ✅ data exists |

## The two value visuals (the centerpiece)

These are the two graphs from the `/home` page (`apps/houseloop-web/components/listing/HouseLoopValueHero.tsx`):

1. **HouseLoop Value** — a bell-curve bar distribution with **Low / Estimated Value / High** columns (value +
   $/ft², circle icons), titled "Estimated Sales Range." (301 E 68th: Low $501,244 · Est **$527,625** · High
   $554,006.)
2. **HouseLoop Home Value History** — an interactive line/slider over time. **The line always reaches back to
   2012** (the ZHVI floor), with **every real sale anchored on it** (the line passes through the actual
   transacted price at its date), climbing to today's HouseLoop value. The **appreciation % spans the whole
   line — since the 2012 floor**, and the % **label reads the line's start year** (e.g. "+100.6% since 2012"),
   not a flat "since sold". A home with no real sale at all uses a pure ZHVI back-projection to 2012. It is
   never blank for a home that has a value. Mechanics:
   [Sold Data & Price History](/guides/sold-data-and-price-history).

> **Architecture note (do not cross-app copy).** `HouseLoopValueHero` is a **website** component built in
> **Tailwind**; the CRM modal uses inline design tokens (no Tailwind). So the modal must **rebuild these two
> visuals CRM-native**, reading the value data off the web `Listing` (`qWeb`) — not import the Tailwind
> component across the app boundary.

## The seller-report modal — the build spec

A wide modal (the two value visuals **side by side**), **not super tall** (no scroll). Internal agent view —
third person ("since **they** purchased"). Sections:

1. **Header** — client name + most-recent home address. Estimated Value + range.
2. **The two value visuals, side by side** — HouseLoop Value distribution + Value History (rebuilt CRM-native).
3. **Gain stats** — Purchase Price · Current Value · **Total Gain** (current − purchase) · **Avg Monthly Gain**
   (total ÷ months held) · appreciation %. (No mortgage/equity — that's deferred to Realie; see the
   past-client pipeline.)
4. **Comparable sales** *(future — needs the comps feed)* — ~3 comps as **address · sale price · link**, each
   link opening **houseloop.ai with a saved search pre-loaded** with those comparable properties.
5. **Neighborhood average (last 12 mo)** *(future — needs neighborhood stats wired into the CRM)*.
6. **Client lifetime** *(agent-only, the part the email doesn't have)* — # deals, lifetime volume, every
   closing, formatted.
7. **FUB profile link(s)** — the client's FollowUp Boss profile; for a couple (e.g. Kyle **and** Karly
   Nations), **one link each**, resolved from `fub_id` + the spouse record in `fub_identity`.
