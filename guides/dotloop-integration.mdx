## What it does

dotloop is the transaction-management platform many brokerages use to run a deal from contract to
close — each transaction is a **loop** containing participants, documents, folders, and milestone
dates. The dotloop integration lets HouseLoop **read that transaction data through dotloop's
official Public API v2** and link each loop back to the HouseLoop lead/deal it belongs to.

We read the brokerage's *own* dotloop account. We don't move the data out of dotloop — dotloop
stays the system of record; HouseLoop reads from it.

## The two-account model

dotloop's API separates the **application** from the **data**, and so do we:

| Account | What it is | Holds |
|---|---|---|
| **Developer account** (`admin@houseloop.ai`) | the "app" — where the Application Client lives | the Client ID + Secret. No loops, no plan needed. |
| **Brokerage account** (Spradling Group) | the "data" — the real transaction file | every loop, participant, document. Untouched. |

The developer account is an empty, free dotloop account whose only job is to hold the API client
(dotloop recommends a dedicated account for this). The brokerage account is where the loops live;
it **authorizes** our client via OAuth consent, and dotloop then issues a token scoped to *that*
account's loops. The brokerage's plan, data, and workflow are never altered.

## OAuth 2.0 flow

dotloop uses three-legged OAuth 2.0 (`authorization_code` + `refresh_token` grants).

```
1. Redirect the brokerage user to the authorize URL (response_type=code, client_id, redirect_uri, state)
2. They sign in to dotloop + grant consent, dotloop redirects to our redirect_uri with ?code=...
3. Exchange the code at the token URL (grant_type=authorization_code) for access_token + refresh_token
4. Call the API with: Authorization: Bearer <access_token>
5. Access token expires (~12h), refresh with grant_type=refresh_token (the old access token is then invalid)
```

| Thing | Value |
|---|---|
| Authorize URL | `https://auth.dotloop.com/oauth/authorize` |
| Token URL | `https://auth.dotloop.com/oauth/token` |
| API base | `https://api-gateway.dotloop.com/public/v2/` |
| Redirect URI | `https://crm.houseloop.ai/api/dotloop/oauth/callback` |
| Access-token lifetime | ~12 hours (refresh with the refresh token) |
| Grants | `authorization_code`, `refresh_token` |

There is **no `scope` parameter on the authorize request** — dotloop grants whatever the client is
provisioned for at consent time, and returns the granted scopes on the token (`token.scope`). The
Client ID and Secret are stored as secrets (`DOTLOOP_CLIENT_ID` / `DOTLOOP_CLIENT_SECRET`) and are
never committed or exposed client-side. The OAuth handshake and token refresh run server-side in the
CRM. The connection (access + refresh token + granted scopes) is persisted in the CRM Postgres table
`hl_dotloop_connection`; `lib/dotloop/client.ts` → `getValidAccessToken()` auto-refreshes it.

## Access scopes

The Application Client is provisioned with these scopes (verified on the live token, 2026-06-26):

| Scope | Covers |
|---|---|
| `account:*` | account details |
| `profile:*` | the user's profiles |
| `loop:*` | the user's loops (transactions) |
| `contact:*` | the user's contacts |
| `template:*` | loop templates |
| `admin:read` | **office / all-profiles data (admin read)** |
| `document:*` | **document file content — upload AND download** |

<Warning>
**Scopes are NOT permanently locked at client creation — but widening them takes two manual steps.**
The first integration was provisioned with only the first five scopes, and document download
returned **403 "Access is denied."** It is fixed and proven. To add a scope to an existing client:

1. **Email `apisupport@dotloop.com`** and ask them to enable the capability on your client (we
   enabled **document download** + **admin read**). They flip it server-side and confirm.
2. **Re-consent — this is the step everyone misses.** A token minted *before* the scope was enabled
   keeps its old scopes forever (refresh preserves consent-time scopes), so it will keep 403-ing. You
   must mint a **fresh** token:
   - In the **brokerage** dotloop account → **My account → Apps**, click **Remove App** on our app.
     (dotloop blocks re-consent with *"… has already been connected … remove this application … and
     try again"* until you do this.)
   - Re-run `/api/dotloop/oauth/start` and **Approve** the consent screen again — it now lists the
     new permissions ("View your office data", "Download and upload loop documents"). The new token
     carries `admin:read document:*`.

Confirm with `/api/dotloop/status?probe=1` or by reading `hl_dotloop_connection.scopes`.
</Warning>

## Read endpoints

All paths are relative to the API base and require the `Authorization: Bearer` header. List
responses are wrapped in `{ "data": [ ... ], "meta": { "total": N } }`.

| Endpoint | Returns |
|---|---|
| `GET /account` | the authorized account (`id`, `email`, `firstName`, `lastName`, `defaultProfileId`) |
| `GET /profile` and `GET /profile/:id` | profiles on the account |
| `GET /profile/:id/loop` | loops (transactions) for a profile — supports `?limit=&sort=updated&direction=desc` |
| `GET /profile/:id/loop/:loopId` | one loop |
| `GET /profile/:id/loop/:loopId/detail` | loop detail — sectioned field map (address, financials, dates) |
| `GET /profile/:id/loop/:loopId/participant` | participants on a loop (name, role, email) |
| `GET /profile/:id/loop/:loopId/folder` | folders in a loop |
| `GET /profile/:id/loop/:loopId/folder/:folderId/document` | documents in a folder (metadata: `id, name, created, updated`) |
| `GET /profile/:id/loop/:loopId/folder/:folderId/document/:documentId` (`Accept: application/pdf`) | **the document's PDF content** — see [below](#downloading-document-content-pdfs) |

## Data model (what each call returns)

### Profiles
An account has many **profiles** (`{ id, name, type: TEAM | INDIVIDUAL, default }`). A loop is always
read **through a profile** (`/profile/:id/loop/...`).

<Warning>
**Inactive profiles 403 on `/loop`** with `"Profile is inactive: <id>"`. When iterating an account's
profiles, walk past the inactive ones — don't assume `profiles[0]` is readable. The `/api/dotloop/status?probe=1`
endpoint does this (it reports the first **active** profile + its loop total).
</Warning>

### Loop (list item)
`{ id, name, status, transactionType }`. `status` is an enum:
`ACTIVE_LISTING · PRIVATE_LISTING · UNDER_CONTRACT · SOLD · ARCHIVED · …`. The `name` frequently
contains the address and/or client ("12802 Walmer, Overland Park, KS 66209").

### Loop detail (`/detail`)
A map of **sections → field key/value**. Sections include `Property Address`, `Financials`,
`Contract Dates`, `Offer Dates`, `Listing Information`, `Referral`. The richest for matching:

```
Property Address:
  Street Number, Street Name, City, State/Prov, Zip/Postal Code, County, Country, MLS Number
Financials:
  Purchase/Sale Price   (string, e.g. "225,000")
Contract Dates:
  Closing Date          (e.g. "03/12/2013")
```

<Note>
Field key names vary by the loop's template, and older/archived loops are sparse (sometimes only
`Country` + `MLS Number`, with the address living in the loop `name`). **Read defensively** — treat
every field as optional and fall back to parsing the loop name.
</Note>

### Participants (`/participant`)
`{ fullName, role, email }` per person. `role` enum includes
`BUYER · SELLER · BUYING_AGENT · LISTING_AGENT · MANAGING_BROKER` (and more). This is the
name + email + side-of-deal for everyone on the transaction.

<Warning>
**A departed agent is DELETED from the participant list.** When a team agent leaves the brokerage,
dotloop removes them from the loop's people — so `/participant` shows only the agent-of-record
(often the team lead, e.g. Ana) + the clients, **not the agent who actually worked the deal**. That
agent survives ONLY inside the signed contract PDF (the "Licensee Assisting Seller" / designated-agent
field). To recover it, download the document content — now a direct API call (below).
</Warning>

## Downloading document content (PDFs)

**The document's PDF downloads directly from the OAuth Public API** — once the token carries the
`document:*` scope (see [Access scopes](#access-scopes)):

```
GET https://api-gateway.dotloop.com/public/v2/profile/:profileId/loop/:loopId/folder/:folderId/document/:documentId
Authorization: Bearer <access_token>
Accept: application/pdf
→ 200, Content-Type: application/pdf, body = the PDF bytes
```

Proven 2026-06-26: loop `223215989`, folder `161367432`, document `980358748`
("Counter Offer Addendum") → a valid **192,990-byte PDF v1.7**.

Notes, all tested:

- **The `document:*` scope is required.** Without it the same call returns **403 "Access is denied"**
  — that 403 is a *missing-scope* signal, **not** a platform limitation. Enable the scope + re-consent
  (see the warning under [Access scopes](#access-scopes)).
- **`Accept: application/pdf` is what switches metadata → file.** The same path with
  `Accept: application/json` returns the document metadata (`id, name, created, updated`).
- **Find the IDs via the read endpoints first:** `/folder` → `/folder/:id/document` (metadata) gives
  you the `folderId` + `documentId` to download.
- **Retry transient 500s.** The download endpoint occasionally returns a `500` with an empty body;
  a single retry after ~0.75s succeeds. The helper does this automatically.
- **Helper:** `lib/dotloop/client.ts` → `dotloop.downloadDocument(profileId, loopId, folderId, documentId, token)`
  returns the PDF as a `Buffer` (handles the `Accept` header + the 500 retry).

**Why this matters — the designated agent on departed-agent loops:** for a loop whose working agent
has left the brokerage, the only programmatic source of that agent's name is the signed contract
PDF. Download the loop's "Exclusive Right to Sell" (or buyer-rep) document → `pdftotext` → read the
"Licensee Assisting Seller" / designated-agent field. The agent that `/participant` no longer lists
is recoverable straight from the PDF the API now serves.

<Note>
**Historical note (superseded).** Before the `document:*` scope was enabled (22–26 Jun 2026), the
only way to get file content was dotloop's web-app REST endpoint
(`www.dotloop.com/my/rest/v1_0/document/x.pdf?documentId=<id>-1`) authenticated by a logged-in
my.dotloop.com **session cookie** (the OAuth bearer 401s there). That cookie-session workaround is
**no longer needed** — use the OAuth API download above. The web endpoint remains a manual fallback
only if the API is ever unavailable.
</Note>

## Using dotloop as a confirmation source (resolver)

dotloop is a **ground-truth confirmation layer**: per transaction it yields the **client name +
email + role**, the **property address** (street/city/state/zip/county + MLS number), the
**price + close date**, and now the **signed contract PDFs**. To confirm a past client:

1. Take the loop's `Property Address` (or parse the loop `name`) → resolve to the same
   `identityKey` / `canonicalSlug` the website uses (see
   [Property data & address identity](/guides/property-data-and-address-identity)) → match a
   `Listing`. When `MLS Number` is present, match it directly.
2. Take the `BUYER` / `SELLER` participant → that's the confirmed client + their real email + which
   side of the deal they were on.
3. For the **working agent** (when it isn't the agent-of-record in `/participant`), download the
   contract PDF (above) and read the designated-agent field.

This makes dotloop the final say on *who actually transacted what address with us* — the
confirmation step for the past-client resolver.

## Connecting an account

1. An admin opens **Settings → Integrations** and clicks **Manage** on the dotloop card (or visits
   `/api/dotloop/oauth/start`).
2. They sign in to dotloop as the **brokerage** account and grant consent.
3. dotloop redirects back to the CRM, which exchanges the code and stores the connection.
4. Verify at `/api/dotloop/status?probe=1` — it reports the connected account, scopes, token expiry,
   and a live read of the first active profile's loop total.
