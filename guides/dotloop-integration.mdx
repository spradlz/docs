## What it does

dotloop is the transaction-management platform many brokerages use to run a deal from contract to
close — each transaction is a **loop** containing participants, documents, folders, and milestone
dates. The dotloop integration lets HouseLoop **read that transaction data through dotloop's
official Public API v2** and link each loop back to the HouseLoop lead/deal it belongs to.

We read the brokerage's *own* dotloop account. We don't move the data out of dotloop — dotloop
stays the system of record; HouseLoop reads from it.

Unlike A-Frame (closed), dotloop publishes a [developer API](https://dotloop.github.io/public-api/).
Together with the [A-Frame dossier](/guides/aframe-overview) and [eSignature](/guides/esignature),
this integration is the blueprint for our own AI-driven TC platform.

## The mental model

> **A loop = one transaction.** It is a container of **folders → documents**, **participants** (people
> by role), **tasks** (in task-lists), **detail** (a sectioned field map), and an **activity** feed.
> Loops live under a **profile**; profiles live under an **account**. Templates pre-populate new loops.
> A **webhook** system pushes real-time events. E-signature exists in the product but is **not in the
> public API**.

This is the same shape as A-Frame's "xaction" — loop ≈ transaction, folder/document ≈ attachment
template, participant ≈ participant role, detail ≈ custom-field tree, activity ≈ activity types.
dotloop adds first-class **documents + e-sign** and a **webhook** system; A-Frame adds the
**task-automation/email** engine. Our platform fuses both.

## The two-account model

dotloop's API separates the **application** from the **data**, and so do we:

| Account | What it is | Holds |
|---|---|---|
| **Developer account** (`admin@houseloop.ai`) | the "app" — where the Application Client lives | the Client ID + Secret. No loops, no plan needed. |
| **Brokerage account** (Spradling Group) | the "data" — the real transaction file | every loop, participant, document. Untouched. |

The developer account is an empty, free dotloop account whose only job is to hold the API client
(dotloop recommends a dedicated account for this). The brokerage account is where the loops live;
it **authorizes** our client via OAuth consent, and dotloop then issues a token scoped to *that*
account's loops. The brokerage's plan, data, and workflow are never altered.

## OAuth 2.0 flow

dotloop uses three-legged OAuth 2.0 (`authorization_code` + `refresh_token` grants).

```
1. Redirect the brokerage user to the authorize URL (response_type=code, client_id, redirect_uri, state)
2. They sign in to dotloop + grant consent, dotloop redirects to our redirect_uri with ?code=...
3. Exchange the code at the token URL (grant_type=authorization_code) for access_token + refresh_token
4. Call the API with: Authorization: Bearer <access_token>
5. Access token expires (~12h), refresh with grant_type=refresh_token (the old access token is then invalid)
```

| Thing | Value |
|---|---|
| Authorize URL | `https://auth.dotloop.com/oauth/authorize` |
| Token URL | `https://auth.dotloop.com/oauth/token` |
| Revoke | `https://auth.dotloop.com/oauth/token/revoke` |
| API base | `https://api-gateway.dotloop.com/public/v2/` |
| Redirect URI | `https://crm.houseloop.ai/api/dotloop/oauth/callback` |
| Access-token lifetime | ~12 hours (refresh with the refresh token) |
| Grants | `authorization_code`, `refresh_token` |
| Rate limit | 100 req/min per user; watch `X-RateLimit-*` headers |
| Pagination | `batch_size` (≤100) + `batch_number` |
| Errors | RFC-style `{ "errors": [{ code, source, detail }] }` — codes 400/401/403/404/422/429 |

There is **no `scope` parameter on the authorize request** — dotloop grants whatever the client is
provisioned for at consent time, and returns the granted scopes on the token (`token.scope`). The
Client ID and Secret are stored as secrets (`DOTLOOP_CLIENT_ID` / `DOTLOOP_CLIENT_SECRET`) and are
never committed or exposed client-side. The connection (access + refresh token + granted scopes) is
persisted in the CRM Postgres table `hl_dotloop_connection`; `lib/dotloop/client.ts` →
`getValidAccessToken()` auto-refreshes it (and so does the harvest auth — see Operational hardening).

## Access scopes

The Application Client is provisioned with these scopes (verified on the live token):

| Scope | Covers |
|---|---|
| `account:*` | account details |
| `profile:*` | the user's profiles |
| `loop:*` | the user's loops (transactions) |
| `contact:*` | the user's contacts |
| `template:*` | loop templates |
| `admin:read` | **office / all-profiles data (admin read)** |
| `document:*` | **document file content — upload AND download** |

<Warning>
**Scopes are NOT permanently locked at client creation — but widening them takes two manual steps.**
To add a scope to an existing client: (1) **Email `apisupport@dotloop.com`** and ask them to enable
the capability; (2) **Re-consent** — a token minted *before* the scope was enabled keeps its old
scopes (refresh preserves consent-time scopes), so in the **brokerage** dotloop account → **My
account → Apps → Remove App**, then re-run `/api/dotloop/oauth/start` and **Approve** again. Confirm
with `/api/dotloop/status?probe=1` or by reading `hl_dotloop_connection.scopes`.
</Warning>

## Endpoint matrix (full API surface)

All paths are relative to the API base and require the `Authorization: Bearer` header. List
responses are wrapped in `{ "data": [ ... ], "meta": { "total": N } }`.

| Resource | Method · Path | Purpose |
|---|---|---|
| **Account** | `GET /account` | authenticated user (id, email, name, defaultProfileId) |
| **Profile** | `GET /profile` · `GET /profile/:id` | list / get profiles (INDIVIDUAL/TEAM/OFFICE/COMPANY/ASSOCIATION) |
| | `POST /profile` · `PATCH /profile/:id` | create / update profile |
| **Loop** | `GET /profile/:pid/loop` | list loop summaries (`?limit=&sort=updated&direction=desc&filter=&include_details=true`) |
| | `GET /profile/:pid/loop/:id` | get loop (301-redirects if merged) |
| | `POST /profile/:pid/loop` · `PATCH …/:id` | create / update loop (name, status, transactionType) |
| **Loop detail** | `GET …/loop/:id/detail` · `PATCH …/detail` | the sectioned field map (read / partial-update) |
| **Folders** | `GET …/loop/:id/folder` · `GET …/folder/:fid` | list / get folders (`include_documents`, `include_archived`) |
| | `POST …/folder` · `PATCH …/folder/:fid` | create / rename folder |
| **Documents** | `GET …/folder/:fid/document` · `GET …/document/:did` | list / get document metadata |
| | `GET …/document/:did` (`Accept: application/pdf`) | **download PDF bytes** (needs `document:*` — see below) |
| | `POST …/folder/:fid/document` | upload (multipart/form-data) |
| **Participants** | `GET …/loop/:id/participant` · `GET …/participant/:id` | list / get participants |

| **Tasks** | `GET …/loop/:id/tasklist` · `…/:tid/task` · `…/task/:id` | task-lists + tasks (**read-only via API**) |
| **Activity** | `GET …/loop/:id/activity` | loop activity feed (`meta.total` = -1, paginate; `?batch_size=200`) |
| **Loop-It** | `POST /loop-it?profile_id=:pid` | **create + populate a loop in one call** (the create facade) |
| **Contacts** | `GET /contact` · `GET /contact/:id` | list / get contacts (account rolodex) |

## Operational hardening (token self-heal + archived folders) — LOCKED 2026-06-30

Two bugs silently capped our first bulk contract harvest at **82 of 12,096 loops** (89 licensees).
Both are now fixed and are MUST-know for anyone reading documents at scale.

### 1. The token must SELF-HEAL on every 401 — never trust `expires_at` alone

The connection token (~12h) can be **rotated out from under a long-running job.** The CRM's own
`getValidAccessToken()` and any batch script share the **one** `hl_dotloop_connection` row, and a
refresh by either one **rotates the refresh_token and invalidates the other's access_token** — while
leaving a *future* `expires_at` on the now-dead token. A naive "is the timestamp still in the future?"
check then hands back a **dead** token, and every call silently `401`s for the rest of the run. That
is exactly what killed the first harvest mid-stream (it died ~partway → only the early loops got PDFs).

**The fix (brittle → resilient):** the low-level `dl()` fetch helper now **self-heals** — on ANY `401`
it FORCE-refreshes the token (`getToken(crm, {force:true})`, bypassing the `expires_at` "stillGood"
check) and **retries the call once.** Register the crm client once via `setAuthCrm(crm)` and a token
that dies mid-run heals transparently instead of poisoning the rest of the batch.
(`tools/houseloop-cards/dotloop-auth.mjs`.) **Rule: any long dotloop job must self-heal on 401; never
gate a refresh on `expires_at` alone.**

### 2. `include_archived=true` — closed-loop documents live in ARCHIVED folders

The folder-listing endpoint takes query params:
`GET /profile/:pid/loop/:lid/folder?include_documents=true&include_archived=true`. **Without
`include_archived=true` you do NOT see archived folders**, and a SOLD/closed loop keeps its signed
contracts (the "Exclusive Buyer Agency Contract", "Exclusive Right to Sell", etc.) in ARCHIVED
folders — so you find zero documents and never reach the licensee. **Always pass both params when
reading documents from closed loops.**

**Proof (2026-06-30):** with the fresh self-healing token + `include_archived=true`, loop
`248724234` returned **9 folders / 67 downloadable PDFs**, and the binding contract PDF yielded
`Licensee Assisting Seller: Bree Archer · Buyer: JaNell Simpson`. The re-harvest of all
SOLD/UNDER_CONTRACT loops then recovered real designated agents (Jodie Brethour, Alisha Hanks,
Kathy Shen, Jessica Rich, …) that `/participant` had dropped.

## Downloading document content (PDFs)

```
GET https://api-gateway.dotloop.com/public/v2/profile/:profileId/loop/:loopId/folder/:folderId/document/:documentId
Authorization: Bearer <access_token>
Accept: application/pdf
→ 200, Content-Type: application/pdf, body = the PDF bytes
```

- **`document:*` scope required** — without it = 403 (a *missing-scope* signal, not a platform limit).
- **`Accept: application/pdf` switches metadata → file** (`application/json` returns metadata only).
- **Find IDs first:** `/folder?include_archived=true` → `/folder/:id/document` → download.
- **Retry transient 500s** (~0.75s). Helper: `lib/dotloop/client.ts → dotloop.downloadDocument(...)`.

**Why it matters — the designated agent on departed-agent loops:** a fallback programmatic source of a
departed agent's name is the signed contract PDF → `pdftotext` → the "Licensee Assisting Seller/Buyer"
field. `tools/houseloop-cards/dotloop-contract-fetch.mjs` does this end-to-end (tier1 participants/detail
free, tier2 downloads the binding PDF), persisting to `hl_srg_dotloop_contracts`.

## The `.loop` fact table (`hl_srg_loop`) — deterministic dotloop facts (2026-07-01)

dotloop is **its own source of truth.** `hl_srg_loop` holds the DETERMINISTIC facts of every loop — one
row per loop — so the past-client resolver has a clean ground-truth source. **No SISU triangulation, no
LLM guessing who is "ours"** — we record who the listing agent + buyer agent were, by address, plus the
closing facts; the resolver decides "ours" later. Rolled up by `hl_srg_agent_loops` (one agent → their
loops, mirror of one home → many transactions).

**Facts per loop:** Property (`/detail`: full_address, street#, street, city, state, zip, county, legal)
· Closing (`/detail`: sale_price, earnest money, contract_date, **closing_date**, offer_date) ·
`listing_agent` · `buyer_agent` (each `{name,email,brokerage}`) · `transaction_coordinator` ·
`spradling_agents[]` (every @spradlingrealtygroup.com actor = resolver candidates) · buyers[] ·
sellers[] · participants[] · activity_actors[] · documents[] · referral_direction.

**Sourcing — deterministic-first:** (1) participant **ROLE** (`LISTING_AGENT` / `BUYING_AGENT` /
`BUYER` / `SELLER`) → (2) the **ACTIVITY LOG** → (3) documents (Haiku) only as a last-resort fallback.

### THE ACTIVITY-LOG UNLOCK (better than the contract PDF for the departed agent)

`GET /profile/:pid/loop/:lid/activity?batch_size=200` is an audit trail of every action with the
actor's name + email + brokerage: *"Walter Hawkins added document … whawkins@spradlingrealtygroup.com"*.
**dotloop scrubs a departed teammate from `/participant` but NOT from the activity log** — so a
`@spradlingrealtygroup.com` actor in the log IS the agent, recovered deterministically without reading a
single PDF. Proof — **11436 Conser**: `/participant` lists only Wendy Manning (`LISTING_AGENT`, the
other side); the buyer agent **Walter Hawkins** appears ONLY in the activity log. The contract-PDF
route is now the *fallback*, not the primary, for the departed agent.

⚠️ **Ana Spradling is the account admin on every loop** — excluded from the single-candidate agent
auto-fill unless she's the only Spradling actor (else she'd falsely become the agent on everyone's deals).

⚠️ **Rate limits:** dotloop throttles concurrent reads (100 req/min per user). `dl()` backs off
(exponential, jittered, ≤3 retries) on 429/5xx, and harvesters run ~3 parallel workers. A fetch-failure
guard means a rate-limited loop is retried, never persisted as a blank "done" row. Tool:
`tools/houseloop-cards/harvest-loop-full.mjs`.

## Webhooks (real-time events) — how the AI stays in sync

A client-app subscription pushes events to an HTTPS URL.

- **Targets:** `USER` or `PROFILE` (+ `targetId`). **Create:** `POST /subscription { targetType, targetId, eventTypes[], url, signingKey?, externalId? }`.
- **USER events:** CONTACT_CREATED/UPDATED/DELETED · PROFILE_UPDATED · USER_PROFILE_ACTIVATED/DEACTIVATED · USER_ADDED_TO_PROFILE · USER_REMOVED_FROM_PROFILE.
- **PROFILE events:** LOOP_CREATED · LOOP_UPDATED · LOOP_MERGED · LOOP_PARTICIPANT_CREATED/UPDATED/DELETED.
- **Payloads** are polymorphic: `{ id }` (loop/contact/profile), `{ id, participantId }` (participant), `{ id, fromId, toId }` (merge).
- **Verification:** `X-DOTLOOP-SIGNATURE` (HMAC-SHA1 of `timestamp.body` with `signingKey`) + `X-DOTLOOP-TIMESTAMP`.
- **Delivery:** at-least-once (dedupe), 5s timeout, backoff 30s → 1m → 15m → 30m → 1h → 2h → 4h → 8h; events retained 90 days; query the delivery log via `GET /subscription/:id/event`.

<Note>
**No saved-search / browsing webhooks** — dotloop webhooks are loop/contact/profile lifecycle only.
(This mirrors the FUB walled-garden pattern: transactional events survive, behavioral ones don't.)
</Note>

## E-signature

**Not in the public API v2.** dotloop's e-sign is a product feature (the activity feed shows
"exported … to a digitally verified, tamper-evident PDF") but there is **no documented endpoint to
request or track signatures**. For our platform, e-sign = **Documenso** (HouseLoop eSignature) — see
[eSignature](/guides/esignature). We pull the *signed result* PDF from dotloop via document download;
we never *initiate* signing through dotloop's API.

## Using dotloop as a confirmation source (resolver)

dotloop is a **ground-truth confirmation layer**: per transaction it yields the **client name + email
+ role**, the **property address** (street/city/state/zip/county + MLS number), the **price + close
date**, and the **signed contract PDFs** (→ the departed working agent). To confirm a past client:
match the loop's address → `Listing`; take the BUYER/SELLER participant → the client + side; recover
the designated agent from the activity log (or the contract PDF as fallback). dotloop is the final say
on *who actually transacted what address with us*.
