- **`/ai/prompting`** (`app/(crm)/ai/prompting/page.tsx`) — view + edit each slot; its Save **PUTs the

## The plumbing (how a slot's text actually flows)

- **Version history — `hl_ai_prompt_history`** (mig 0426): every live save appends a row (slot_key, value,
  editor, note). On a slot's **first-ever edit**, the pre-edit original is auto-seeded first
  (`seedOriginal`, `app/api/ai/prompts/[key]/route.ts:83`) so the starting point is never lost.
- **The draft mechanism**: a cockpit draft is stored BESIDE the live value at the same settings key, path
  `<jsonPath>__draft` (`lib/ai/prompt-read.ts:33`). Settings-backed slots only. A live save clears the
  draft so live and draft never diverge silently (route.ts:69-71).
- **The code-constant fallback law**: every compose path reads the live value via
  `readPromptLive(slotKey)` (`lib/ai/prompt-read.ts:14`) and falls back to its old code constant when the
  read returns null — a missing/hiccuping settings row can never break production composing.
- **Honest nuance — the gauntlet gate**: by LAW and practice (§02b #5) no prompt change goes live without
  the judged gauntlet green, but the save path does **not yet enforce it** — promote is a manual
  typed-confirm ("No automated suite is wired to this prompt yet", PromptEditorModal.tsx:197). The
  gauntlet gate replaces the confirm when the scenario suites land.

# ═══ 02 · PROMPTING DOCTRINE — how we write agent prompts ═══

Built from OpenAI's official Realtime prompting guide + cross-vendor research (ElevenLabs, Vapi, Retell)
+ our own live-call scar tissue. Full cited research: `.claude/reports/voice-agent-design-research-2026-07-20.md`.

## The skeleton (OpenAI's, adopted as law for voice agents)

`Role & Objective (as a MISSION with a checklist) → Personality & Tone (incl. Length / Pacing / Variety) →
Context (the KB, behaviorally gated) → Reference Pronunciations → Tools → Rules → Conversation Flow
(states with Goal / sample phrases / Exit-when) → Safety & Escalation.`

## Our laws (each one paid for in blood)

1. **THE PRESET IS THE LAW.** Start from the provider's official starter config/prompt structure. Never
   hand-tune provider knobs (VAD sensitivities etc.) — every deviation we tried made latency/turn-taking
   WORSE, measured.
2. **The mission is a checklist.** "You are NOT done until you have: …" — a described job produces a
   passive agent; a checklist produces drive.
3. **The follow-up-question asymmetry** (from ChatGPT's leaked voice prompt — the "always driving" feel):
   ask ONE single-sentence follow-up when natural; NO question right after answering the caller's direct
   question; never use language that signals the conversation is over.
4. **Ask, never announce.** "Let me get some details" = a stall. Sample phrases ARE the questions — the
   model closely imitates them.
5. **Answer-then-advance.** Caller asks a question → answer in one sentence → next intake question in the
   SAME breath.
6. **"Need … before" is banned.** Never frame intake as screening ("I need X before the team calls you
   back") — the agent builds the caller's file FOR them.
7. **Variety or robot.** Never the same sentence twice; re-asks must be re-phrased; after two dodges, drop
   it and move on.
8. **Entity capture:** one exact value at a time; digit-by-digit read-back for numbers/addresses;
   re-confirm after corrections; never alter a digit.
9. **Voice-specific:** 1–2 sentences per turn; prosody in words ("speak quickly and warmly") not params;
   spoken-style numbers; ~2,000-token ceiling (latency).
10. **KB without dumping:** answer ONLY from the Context block; not there → "I don't know" + offer a
    callback; never guess.

# ═══ 02b · THE CONVERSATION DIRECTOR — code owns the goal (locked 2026-07-21) ═══

The company-defining architecture verdict (2 research rounds — 6 agents over GitHub/commercial
platforms/forums — + ChatGPT's independent review, all converging; full archive:
`.claude/plans/serene-wishing-hamster.md`): **stop scaling the prompt; the runtime owns the
conversation.** The industry consensus (Retell/Bland/Vapi/OpenAI's own reference/Intercom Fin/Sierra):

1. **Per-state prompts** — the model only ever sees the CURRENT state's instructions (our
   triage→playbook `session.update` hot-swap is OpenAI's own documented pattern, extended per-stage).
2. **Checklists live in CODE, not prompts** — the prompt gets a MISSION ("leave the call so Ana can
   confidently call back"), the runtime tracks the slots. Required-field gates are runtime-enforced.
3. **Turn rules are enforced by GUARDS, not hoped from prompts** — `MyliTurnGuard` (pipecat_bot.py):
   any intake turn ending without a question → steering message + forced continuation
   (response.create); post-tool silence watchdog. Prompts LOWER the flinch rate; only code
   eliminates it (reproduced live 2026-07-20: the announce-then-silence flinch survived even the
   v2 one-breath prompt until the guard shipped).
4. **Turn-taking is config/runtime, never prompt** — SemanticTurnDetection(eagerness=low) for
   trail-offs + the `wait_for_user` no-op tool (OpenAI's documented hesitation fix).
5. **Eval-gated shipping** — no prompt change goes live without the judged gauntlet green
   (pass^k, repeat runs — single runs lie about stochastic models; Zillow ran 69%→95% this way).

**Build status (2026-07-21):** the turn-level Director is **BUILT and live** — `MyliTurnGuard`, the
post-tool watchdog, `wait_for_user`, and SemanticTurnDetection all ship in
`apps/myli-voice-agent/pipecat_bot.py` (guard class :371, `wait_for_user` :508, semantic turn :640). The
**slot-tracker checklist + scoring observer** (#2's runtime slot tracking, §07's Observer) is **FUTURE** —
today the checklist still lives in the bucket prompts.

**THE ONE-BREATH RULE** (prompt law #1, in every voice prompt): the acknowledgment and the next
question are ONE breath — a spoken turn is complete ONLY when it ends with exactly one question
(sole exception: the final goodbye, which is SPOKEN before `end_call`, never skipped). Sample
phrases are the strongest instruction-following lever (the model "strongly closely follows sample
phrases" — official guide): every state's samples end in a question.

**ONE BRAIN, MANY MOUTHS** (Zach doctrine 2026-07-20): Myli is ONE persona + ONE per-lead working
memory + ONE rulebook across voice/text/email; the MODELS are swappable per-channel engines under
the brain (phone = gpt-realtime-2.1, text = Claude Sonnet — decided; A/B parked). Deep dive:
`guides/closed-loop-follow-up`. The memory keystone: §06.

# ═══ 03 · THE VOICE LANE — how the phone persona is COMPOSED ═══

One endpoint owns it: **`GET /api/phone/myli/persona`**
(`apps/houseloop-crm/app/api/phone/myli/persona/route.ts`), called by the Pipecat voice worker at call
start. Myli's phone personality is CONFIG, not code. Runtime/hardware/deploy detail: `guides/ai-voice`.

## Cold hat vs warm hat (the company-line law, Zach 2026-07-15)

The hat is picked by who's calling **AND which line they dialed** (route.ts:44-106):

- The **company / front-door / test line is ALWAYS the cold AI Receptionist** — a total stranger. She
  NEVER name-recognizes a caller just because their number exists somewhere in the CRM.
- The **WARM hat** (greet by first name + full memory) requires BOTH: **(1)** the caller dialed the
  agent's OWN line — a number in `hl_phone_numbers` assigned to this ai_agent (route.ts:66-77) — AND
  **(2)** the matched lead is actually ASSIGNED to her (`assigned_ai_agent_id`, route.ts:103-105).
- Why: the old code flipped warm for ANY lead-matched number on ANY line, so a test lead calling the
  company line got "hey Zach" — dead wrong for a company receptionist.

## GROUND TRUTH injection (route.ts:148-161)

The model has no clock and no caller-ID access — on real calls it invented dates ("Wednesday December
18th") and digits (Zach live-test 2026-07-17). So the route injects the only two per-call facts the voice
layer actually knows: the current date/time (Central) and the caller-ID digits (or "ask and read back
digit by digit" when unavailable). The bot re-prepends the merged ground truth to a bucket when it swaps
prompts mid-call, so the clock + caller-ID survive the swap (route.ts:173-175).

## The composed base (route.ts:163-171)

```
persona = groundTruth
        + role persona            (hl_role.persona, per-agent override wins)
        + role voice doctrine     (hl_role.voice_persona, per-agent override wins)
        + memoryBlock             (warm calls only — composeMyliMemory, §06)
```

with `{{agent.*}}` / `{{ai.*}}` / `{{team.*}}` merge tokens resolved for the acting agent. **The KB block
is intentionally DROPPED from the base** — the receptionist triages first, then the bot swaps in ONE
specialized bucket; keeping the base tiny is the whole latency fix (mig 0419).

## Triage addendum + bucket hot-swap

- The **triage block** comes from the registry (`myli_voice_triage`, mig 0431 — no hidden prompt
  segments) via `readPromptLive` and is returned merged as the `triage` field (route.ts:201-206); the bot
  appends it at call start.
- The **buckets** (`myli_voice_buckets`, merge-token-resolved, route.ts:177-192) are returned as a map;
  the bot hot-swaps the session instructions to base + ONE bucket via `session.update` when the model
  calls `set_caller_type` (pipecat_bot.py:679 — OpenAI's own documented dynamic-flow pattern).
- **`?prompt_source=draft`** overlays each bucket's cockpit draft (and the triage draft) BEFORE merge, so
  the Test Cockpit chats against a draft through the exact same composition path (route.ts:182-192). The
  phone bot never sends the param — live calls unchanged.

The route also returns `voice_enabled` (the master kill switch, from the phone hat's tools) and the
per-capability toggles (capture_caller / set_callback / take_message / flag_spam).

# ═══ 04 · THE TEXT LANE — how the text system prompt is ASSEMBLED ═══

One file owns the conversational text brain: **`apps/houseloop-crm/lib/ai/lead-conversation.ts`**.

## buildSystemPrompt — the layering (lead-conversation.ts:2018-2088)

```
system = grounding      (RIGHT NOW clock/tz + WHO YOU ARE TEXTING + how-you-search-homes doctrine)
       + agentBlock     (loadDefaultAgentBlock — the acting agent identity)
       + stagesBlock    (the lead's live pipeline stages → drives appointment type)
       + memory         (composeMyliMemory — §06, the one-brain keystone)
       + persona        (role persona + per-agent override + the role's KB docs, smart-fields resolved)
```

Unlike the voice base, the text lane DOES inject the role's Knowledge Base docs into the persona stack
(:2029-2038) — latency physics differ; there is no per-turn re-read tax on text.

## The instant-ack layer (never a quiet thread)

- **`generateAck`** (:2110) — a fast Haiku micro-call reads the lead's last message + thread position and
  writes ONE fitting buy-time line ("yeah, getting that locked in for you now!") — or SKIPs when buying
  time would read weird. It is armed ONLY when the model actually invokes a slow tool and ~10s pass with
  no reply (:2197-2217) — never a blind timer on small talk.
- **Rotation ban:** the last 8 ack texts persist in `hl_settings.ai_ack_rotation` and are banned from
  reuse, so it can never read canned. **`ACK_VARIANTS`** (:2095) is the static error-fallback pool only.

## THE WHO-SPEAKS GATING CHAIN — `dispatchInboundToAi` (:225)

Every inbound text/iMessage/email walks this chain, in order:

1. **The master switch** (:229 → `aiEnabledForLead` :212): the ALL-AI failsafe
   `hl_settings ai_master.enabled` (mig 0432, `lib/ai/master-gate.ts` — off beats everything, even the
   pilot list) AND `ai_conversations.enabled` (with the `pilot_lead_ids` live-fire override).
2. **Destination-identity routing** (:237-243): WHO was contacted decides the lane. Inbound landed on an
   AGENT'S identity (their number/email) → the human owns it and their **AI clone** covers per
   config/SLA — not Myli. Landed on Myli's/company identity (or unknown) → fall through.
3. **The per-lead mic — `hl_leads.ai_responder`** (mig 0430; :245-259): `'none'` → NO AI reply, full
   stop. `'clone'` → the owner's clone lane speaks (no owner → honest silence). `'myli'` / legacy NULL →
   Myli's watch lane. The lead-card toggles write this field — they are REAL, not decorative.
4. **The clone master gate** (mig 0431, `hl_settings ai_clones.enabled`): a three-layer AND — master
   toggle × per-agent `hl_agent_clone_config.enabled` × the per-lead mic — enforced in
   `lib/agent-runtime/resolve-send-identity.ts:47` (`clonesMasterEnabled`). Master OFF = every clone
   silent; Myli does NOT take over (never a bulk write, so flipping back on restores exact prior config).
5. **Myli's watch lane requirements**: an assigned AI owner + an AI role with a non-empty persona (an
   empty persona logs a VISIBLE `ai_skipped_no_persona` activity row instead of silent death, :283-295).
   Email inbound additionally requires the email dark-gate (§05).

Myli's watch is deliberately NOT gated by the per-lead `ai_mode_enabled` escalation flag (:261-266) —
that flag belongs to the SLA/escalation layer; gating on it was exactly why "turn workflows off → Myli
went silent."

# ═══ 05 · THE FIRST-TOUCH / EMAIL LANE ═══

One function owns the first-touch brief: **`buildFirstTouchBrief`**
(`apps/houseloop-crm/lib/workflows/first-touch-brief.ts:160`). Both the live workflow node
(`execute-step.ts` → ai_message) AND the preview harness call it, so a preview is byte-for-byte what gets
sent (the three hand-copied drifting copies burned a live deal, 2026-07-18).

## Contract selection order (:176-179)

```
explicit override (cockpit re-run-with-draft)
  > tour_first_touch_contract (live slot, when the request is a tour)
  > first_touch_contract (live slot)
  > CONTEXT_FIRST_CONTRACT (the code constant — the never-break fallback)
```

## What gets stacked into the brief (:217-228)

- **The contract** (above) — the dominant, in-the-moment instruction: greeting → who you are → the REAL
  relationship (only if the dossier supports it; never invented) → the request by ADDRESS → ONE question.
  Tour first touches deliberately withhold the relationship line from message 1.
- **`intentGuidance`** (:127) per resolved intent: tour_request (acknowledge the EXACT time, never re-ask),
  property/FUB inquiry, seller_inquiry, contact_request.
- **The relationship line** (:100) pulled straight from `hl_lead_dossier` + `hl_lead_facts` so it sits
  beside the contract, not 40 facts deep in the system prompt.
- **Platform + brand lines** (:185-200): the platform the request came through ("through Zillow") and the
  company name the responder introduces with — from the lead's ROUTING RULE (mig 0436), never hardcoded.
- **`formatRuleFor(channel)`** (:54): **SMS physics** for text (aim under 160 GSM-7 chars = one billed
  segment; 320 hard ceiling; no em dashes / curly quotes — they force UCS-2 and triple carrier cost) vs
  the email body shape (2–4 short paragraphs, no subject/signature — added automatically).
- **`STYLE_RULE`** (:39 — casual tone from word choice, never dropped capitalization) and
  **`NO_REASONING`** (:46 — output ONLY the message; a leaked chain-of-thought to a real lead is a fire).

## THE EMAIL TRUTH — no separate email slot

There is **no email prompt slot and no separate email brain**. Email reuses the text brain end-to-end:
the same `dispatchInboundToAi` chain, the same `buildSystemPrompt`, the same contracts — with
`formatRuleFor('email')` swapping the output shape. Email auto-reply ships **DARK** behind
`hl_settings ai_conversations.email_enabled` (explicit `true` to activate;
lead-conversation.ts:273-277) until the e2e proves it. Texts unaffected.

# ═══ 06 · THE ONE WORKING MEMORY — the one-brain keystone ═══

**`composeMyliMemory(leadId)`** (`apps/houseloop-crm/lib/ai/myli-memory.ts:18`) is the canonical per-lead
memory composer — the working-memory layer of ONE BRAIN, MANY MOUTHS. Every channel that speaks for Myli
composes the lead's rolled-up memory through THIS one call, identically — never by assembling the blocks
itself:

- **Voice** — the persona route injects it as the warm-call memory block
  (`app/api/phone/myli/persona/route.ts:119`). The company-line cold hat simply doesn't call it for
  strangers — that's the cold-hat law, not a different memory.
- **Text/email** — `buildSystemPrompt` injects the identical composition
  (`lib/ai/lead-conversation.ts:2086`).

What it returns: the **dossier block** (structured facts) + the **context brief** (comms / browsing /
notes distillation), both built over the per-lead everything-store — `hl_lead_activity` (the canonical
run-log), `hl_messages`, `hl_calls`, and the email logs. Change the memory here and every mouth changes
together; a channel composing its own memory blocks is the drift this function exists to make impossible.

# ═══ 07 · CONVERSATION STATES & THE SCORING OBSERVER ═══

## The 3-tier state ladder (climb only on evidence)

1. **Single prompt + Conversation Flow section** — phases written in the prompt (Goal / sample phrases /
   Exit-when). Receptionist territory.
2. **Per-state instruction/tool swapping** — the runtime hot-loads instructions per phase. **BUILT — this
   is Myli voice today**: the triage → bucket hot-swap via `session.update` (§03), plus the turn-level
   guards (`MyliTurnGuard`, `wait_for_user`, semantic turn detection — §02b, all live in
   `pipecat_bot.py`). Climb here when calls consistently derail ("context rot").
3. **Hard graph with coded transitions** — compliance-heavy flows only; over-noding sounds robotic.

**The law: the LLM's job is language; the goal lives in code.** Transitions are state mutations signaled
by tool calls — never something the model is trusted to remember.

## The Observer pattern — FUTURE (how lead-qual will work)

Not yet built — the turn guards above are the shipped half of the Director; this is the unshipped half.
Fast model talks naturally. A parallel tracker (deterministic + small LLM extractor) watches the
transcript, owns the slot checklist + score, and steers by injecting context between turns ("You have X
and Y. Next goal: Z. Do not re-ask X."). The talking model never knows it exists. Scoring is ALWAYS
computed by the tracker layer, never the talking model, and persists on the CRM lead record.

## Scoring — FUTURE (the industry pattern, for the lead-qual build)

Rubric-driven (real-estate shape: timeline / motivation / price band / pre-approval / area / has-agent),
0–100 + tier, updated per response, stored as lead fields; threshold crossings fire handoff (hot transfer
/ agent alert). SMS nurture = **CRM-as-state**: each inbound rehydrates from lead fields (slots, score,
next-goal pointer) → reply → extractor updates fields → writeback → next touch scheduled (behavioral
triggers beat time drips) — maps 1:1 onto Inngest + `hl_leads`.
