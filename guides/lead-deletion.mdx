When a lead is deleted in HouseLoop CRM, **every trace is permanently removed**
from both the CRM database and the website database. There is no soft-delete, no
trash bin at the database level, and no tombstone row. The "Trash" concept lives
as a pipeline stage for leads the operator wants to hide temporarily — an actual
delete is irreversible and total.

## Why hard-delete

Soft-delete (a `deleted_at` timestamp) was removed in May 2026 because orphaned
data kept resurfacing. A lead marked as deleted would still match inbound texts
by phone number, still appear in notification queries, and still pollute pixel
analytics. The hard-delete model eliminates every class of ghost data by removing
the rows entirely.

## One source of truth: `purgeLead()`

All purge logic lives in a single function — `lib/leads/purge-lead.ts` —
exported as `purgeLead(id)`. Both delete paths call it:

- `DELETE /api/leads/[id]` — single lead
- `POST /api/leads/bulk` with `action: "delete"` — loops `purgeLead()` per id

This is deliberate. The purge used to be hand-copied into both routes, and the
two copies drifted (and both carried the same pixel bug). **There is now exactly
one nuke button**; the routes are thin wrappers around it.

<Warning>
**NEVER hand-roll a lead purge with raw SQL.** Deleting a lead means calling `purgeLead()` — via
`DELETE /api/leads/[id]`, the bulk-delete route, or the test harness (`hl_reg_purge_lead`, which calls the real
DELETE cascade). A hand-written `DELETE FROM hl_leads …` (or any subset of these steps) **skips the parts that
have no lead FK** — the anonymous pixel purge (step 2), phone-keyed inbound (step 4), and **the cross-DB website
`User` delete (step 6)** — which is exactly how a "deleted" lead leaves an orphaned website identity that
resurfaces on re-registration. If you need to purge a test identity, call the endpoint / harness. Do not
reinvent this function. *(This warning exists because a raw-SQL "purge" on 2026-07-08 skipped step 6 and
stranded a website `User` + its saved searches — the whole point of the one-button rule.)*
</Warning>

Purge is built from two layers:

1. **Structural (automatic) — `ON DELETE CASCADE` foreign keys.** Every table
   with a `lead_id` FK to `hl_leads(id)` is purged by the database itself when the
   lead row is deleted. A *new* table is covered the moment its FK is declared
   `REFERENCES hl_leads(id) ON DELETE CASCADE` — no code change to `purgeLead()`.
2. **Explicit — `purgeLead()` handles the exceptions** that have no lead FK:
   pixel events (keyed by visitor), polymorphic notes, phone-keyed inbound, and
   the cross-DB website user.

## The purge sequence

### 1. Snapshot the lead

`purgeLead()` reads the lead's `id`, `email`, and `phone` from `hl_leads`. These
drive the explicit cleanup steps below. If the lead doesn't exist it returns
`{ ok: false }` and the route responds `404`.

### 2. Purge the pixel tables (anonymous-aware)

`hl_pixel_event` and `hl_pixel_visitor` are the only two lead-referencing tables
with **no foreign key** to `hl_leads` (a visitor is anonymous-first: `lead_id` is
`NULL` until the visitor identifies). Deleting `WHERE lead_id = $id` alone is a
trap — it **misses the anonymous events** (page views recorded under the browser's
`visitor_id` before signup). Those survive and **re-link by email** the next time
the same person registers (the lead-intake backfills
`pixel_visitor/event SET lead_id WHERE email = ...`), producing "ghost views" on a
brand-new lead.

`purgeLead()` deletes every pixel row for the lead's **email-linked visitor IDs**,
anonymous rows included:

```sql
DELETE FROM hl_pixel_event
 WHERE lead_id = $1
    OR visitor_id IN (
         SELECT visitor_id FROM hl_pixel_visitor
          WHERE lead_id = $1 OR ($2 <> '' AND lower(email) = lower($2))
       );

DELETE FROM hl_pixel_visitor
 WHERE lead_id = $1 OR ($2 <> '' AND lower(email) = lower($2));
```

A future lead with the same email or browser therefore inherits **zero** history.

### 3. Purge polymorphic tables

`hl_notes` uses `entity_type` + `entity_id` instead of a direct FK:

```sql
DELETE FROM hl_notes WHERE entity_type = 'lead' AND entity_id = $1
```

### 4. Purge phone-matched messages

`hl_inbound_messages` links to leads **by phone number**, not by `lead_id`. If the
lead has a phone, `purgeLead()` normalizes it to the last 10 digits and deletes
every row where `from_number` or `to_number` matches:

```sql
DELETE FROM hl_inbound_messages
WHERE right(regexp_replace(from_number, '\D', '', 'g'), 10) = $1
   OR right(regexp_replace(to_number, '\D', '', 'g'), 10) = $1
```

This stops a future lead with the same number from inheriting the deleted lead's
message history — otherwise the [comms resolver](/guides/team-communications)
would re-attach old messages and create ghost conversations.

<Warning>
Phone-matched deletion is aggressive by design. If two leads share a phone number
(rare — see [deduplication](/guides/lead-deduplication)), deleting one removes
inbound messages that matched _both_. Intentional: nothing from a deleted lead
survives.
</Warning>

### 5. CASCADE delete from `hl_leads`

```sql
DELETE FROM hl_leads WHERE id = $1
```

Every FK that references `hl_leads.id` is `ON DELETE CASCADE`. As of the current
schema that is **42 of the 44** tables holding a `lead_id` (the other two are the
pixel tables in step 2; the newest cascade table is `hl_trigger_fire`, the trigger-fire
ledger, mig 0355). One `DELETE` wipes all dependent tables automatically —
`hl_messages`, `hl_notification`, `hl_tasks`, `hl_appointments`, `hl_calls`,
`hl_lead_tags`, `hl_lead_collaborators`, `hl_stage_transitions`,
`hl_saved_search_match`, `hl_bb_events`, `hl_lead_addresses`, and more. No
application code enumerates these — the database enforces it.

### 6. Cross-database website purge

The CRM and website run on **separate PostgreSQL databases**. The website stores a
`User` row (sessions, saved searches, `ListingReaction`/`SavedListing`, profile)
matched to the CRM lead by email. After the CRM cascade, `purgeLead()` calls
`deleteWebUserByEmail(email)` against `HL_WEB_DATABASE_URL`:

```sql
DELETE FROM "User" WHERE lower(email) = lower($1)
```

The website `User` table also cascades on all its FKs, so one delete cleans
everything (including the likes/saves that back the `/favorites` page).

<Note>
The cross-DB delete is **best-effort with error logging**. If the website DB is
unreachable, the CRM lead is still deleted and the failure is logged.
</Note>

<Warning>
**An orphaned website `User` row is NOT harmless — it silently blocks re-registration** (learned the hard way,
2026-07-08). The website's complete-profile step calls `GET /api/auth/complete-profile`, which returns
`complete = !!(name && phone)`. If a leftover `User` row still carries a name + phone, the site treats the next
sign-in with that email as a **returning, already-complete user**, **skips the complete-profile POST, and never
calls the CRM lead-intake** — so the person gets a website account but **no CRM lead and no General Registration
door fires**. The website `User` delete in step 6 is therefore **not optional cleanup; it is what makes
re-registration work.** If step 6 ever fails (web DB down), the delete is *incomplete* — re-run it, don't leave
the orphan.
</Warning>

## The guard — why you never have to remember

A schema-scan guard, `tools/lead-purge-audit.mjs`, makes a forgotten table
impossible. It reads `information_schema` and **fails (exit 1)** if any `hl_` table
has a `lead_id` column that is covered by *neither*:

- an `ON DELETE CASCADE` FK to `hl_leads`, nor
- the explicit allow-list in the guard (`hl_pixel_event`, `hl_pixel_visitor`),
  which must also be handled inside `purgeLead()`.

```bash
HL_CRM_DATABASE_URL=... node tools/lead-purge-audit.mjs
# every lead-referencing table is covered (CASCADE or explicit purge).
```

Run it in CI / drift-check. When someone adds a new lead-referencing table the
guard forces them to either give it a cascade FK or wire it into `purgeLead()` —
the table can never silently orphan on delete.

## What is NOT deleted

| Item | Why |
|---|---|
| `hl_lead_sources` | Shared config, not per-lead data. |
| `hl_stages` / `hl_pipelines` | Pipeline configuration. |
| `hl_templates` | Team-level email/text templates. |
| `hl_run_log` | Inngest run history — operational audit. |
