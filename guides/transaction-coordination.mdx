## What this section is

**Transaction Coordination (TC)** is the HouseLoop CRM section for running a deal after it goes under
contract — the coordination layer (templates, contracts, checklists, client portal) on top of the
transaction-management platform that holds the loop file (participants, documents, milestones).

Where [Website Integration](/guides/listing-lifecycle) is about the *listing* (the home) and the CRM
guides are about the *lead* (the person), **Transaction Coordination is about the deal** — the
contract-to-close record. It is modeled as a *light* version of a dedicated TC tool, AI-native
(Myli-fillable), built on HouseLoop's own rails, with **dotloop** as the integration underneath.

## The model

| Layer | Source of truth | HouseLoop role |
|---|---|---|
| **Listing** (the home) | `Listing` (web DB) | renders `/home/<slug>`, value, history |
| **Lead** (the person) | `hl_leads` (CRM DB) | CRM, AI follow-up, activity log |
| **Deal** (the transaction) | the transaction platform (dotloop) | **coordinated via this section** |

The transaction platform stays the system of record; HouseLoop reads from it (and pushes finished
documents back). We don't replace dotloop or move its data out.

## The CRM section (navigation)

Transaction Coordination is a **drill-in section** in the CRM: the main left nav has a
**Transaction Coordination ›** item; clicking it swaps the whole left menu for the TC sub-nav and a
**Back to CRM** link. The TC sub-nav (mirroring a dedicated TC tool):

- **Templates** — the Contract Engine: reusable fillable contracts (Buyer's Agency, Single Showing
  Agreement) that auto-fill from a lead's data, are verified by Myli, and push into the dotloop loop.
- **Team Info · Team Members · Lists · Client Portal · Import** — the coordination + admin surfaces.

Routes live under `/tc/*` (additive — the existing Dashboard/Leads nav is unchanged).
