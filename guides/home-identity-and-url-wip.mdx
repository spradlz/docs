## The thesis (LOCKED — Zach 2026-06-27)
**The rooftop is the identity.** Every house has ONE frozen **`HLID`** and ONE permanent `/home/<canonicalSlug>` URL.
Address / ZIP / MLS# / status are *mutable attributes*; the HLID and the URL never move — that permanence is the
local-SEO moat. (`HLP-` is the legacy prefix; grandfathered id strings stay frozen.)

A **fork** is the enemy: the *same physical home* showing up as 2+ records / `/home` pages because the address drifted
between sources — a **ZIP typo** (Zillow 64119 vs MLS 64118), **`Dr` vs `Drive`**, a **dropped directional**
(`928 NE 87th` vs `928 87th`), or a **city-line straddle** (`11900 Merritt` is one house on the KC↔Grandview line, and
even *Zillow* has it under two ZPIDs). Forks split SEO authority and confuse buyers. We kill them two ways.

## Layer 1 — Prevention at intake (the primary defense)
**Every incoming listing needs an HLID, so the only question at intake is: "is there already an HLID for this
rooftop?"** Yes → reference it. No → mint a new one. That single decision, made at *birth*, means a fork can never be
created in the first place.

- **Where:** `lib/listings/cache.ts` ingest path, before the listing's `identityKey` is stamped, via
  `lib/listings/rooftop-identity.ts` → `resolveRooftopIdentity()`. If a drifted twin already exists, the new row adopts
  the **existing** identity key, so `linkListingToProperty` gives it the same HLID and `collapseHomeCluster` folds it in.
- **Safe by construction:** the candidate set is **anchored on the house number + main street token**, so River-Hills-style
  street-level geocodes (four different houses sharing one lat/lng) can never cross-match — different houses have
  different numbers. Within that set a twin is claimed ONLY on a **conservative gate**:
  - same rooftop **coordinates (≤30 m)**, **or**
  - **identical full street + city, differing only by ZIP** (a ZIP-drift duplicate),
  - …with the **unit matching** (a condo unit is its own home) and **no directional conflict** (N vs S = distinct homes).
- **Deliberately NOT matched on structure alone** — matching sqft/year would wrongly fuse `74th Street` and `74th
  Terrace` (different parallel streets). Those ambiguous cases mint new and are caught by Layer 2.

## Layer 2 — The autonomous de-fork engine (backlog + backstop)
`apps/houseloop-web/scripts/defork-engine.ts` finds same-home forks **across keys** and resolves them with **zero human
review**. It clusters by a loose rooftop key (house# + directional/suffix-stripped core + **unit** + state — the unit is
part of the key so condo units never collapse together), then classifies every keeper↔loser pair:

| Tier | Decision | Gate |
|---|---|---|
| **A — auto-merge** | same home, merge now | coords ≤30 m · or same parcel · or identical address + ZIP · or core address + matching sqft/year |
| **B — LLM adjudicate** | Haiku rules same vs distinct | loose match, nothing confirms or conflicts (ZIP typo, doubled suffix, city-straddle). Knows `74th St` ≠ `74th Terrace`. Only **same + high-confidence** merges. |
| **C — auto-reject** | keep separate (NOT a review queue) | coords >30 m · directional conflict (N vs S) · clashing sqft/year (the "lot sale vs new-construction" trap) |

The merge is **reversible — never a destructive (Rule #0) op:** quarantine the loser (`quarantinedAt`; reads already
skip it) → write `hl_slug_alias` (old URL → keeper, **301 never 404**) + `hl_hlid_alias` → log `hl_dedup_quarantine`
(the undo trail) → **carry the loser's richer value/tax/photos forward onto the keeper**. Keeper rank: live > sold >
off_market, then most photos, then newest. Per-merge isolation: one bad row never halts the run.

## The hard part (Zach called it): UNIT + COORDINATES
A naive "collapse same-rooftop → one id" would have been a disaster. Measured: only **1.1%** of homes carry a unit;
**16,538** unit-less rows sit on rooftops shared by 2+ distinct homes (condos/duplexes). So **the unit is part of every
cluster key**, and coordinates only *confirm* a same-house-number match — they never merge on their own. Under-merge (a
leftover fork) is recoverable; over-merge (fusing two people's condos into one HLID) is not.

## Guardrails
Additive · reversible (`hl_dedup_quarantine` + `hl_slug_alias` / `hl_hlid_alias`) · **301 never 404** · house#-anchored
so street-level geocodes never cross-match · unit is part of every key (never fuse unit-less shared rooftops) ·
structure-alone never merges (74th St ≠ 74th Terrace) · no county lookups.
