## 🧭 THE DEAL LEDGER — match deals, not addresses (Zach LOCKED 2026-07-03; BUILT + at 100 percent)

**The insight that ended three weeks of resolver pain: we were answering the wrong question.** "Whose house
is this ADDRESS?" is unanswerable — addresses lie (dropped units, colliding slugs, repeating condo
floorplans, fat-fingered streets). The right question is **"which DEAL is this?"** — a deal fingerprints as
**client name + price + close date**, and independent ledgers of the same deals join deterministically.

**The five edges (in matching order):**
1. **SISU↔MLS name edge** — `sisu_transactions` ↔ `hl_srg_mls_deals` by last name + price ±$3k + close
   date ±20d.
2. **The MLS row's own client name** (`client_first`/`client_last` on 1,782 of 1,971 rows).
3. **Price+date-unique (cross-side)** — when exactly ONE listing in the whole book carries the fingerprint —
   **with the ADDRESS-CONTRADICTION VETO** (differing street numbers between the SISU address and the
   candidate reject the match; the veto un-glued 27 silent mispins the day it landed — the Garcia/Alderson
   class).
4. **The FUB person-ID link** (`hl_fub_sisu_link`) — immune to name-spelling drift.
5. **Child-row fingerprint placement** for off-market/no-MLS deals — with the **corrupt-date rule**
   (pre-2010 child dates are a known corrupt class → price must be unambiguous within the client).

Tiers: both name edges agree → **GREEN** · one edge → **YELLOW** · none → **RED** (off-market/referral =
expected, not an error). **PENDING = CLOSED** (SISU died 2026-06-19; Pending rows froze — the deal date
falls back closed → forecasted → under-contract, stamped with its source).

**The three hops (no similarity anywhere):**
> **CLIENT** →(name+price+date)→ **MLS#** →(history stamp)→ **HLID** → the card.

**The tables:** `hl_srg_deal_ledger` (1 row / client×deal, UPSERT-only, rebuilt by
`tools/houseloop-cards/build-deal-ledger.mjs`, NEVER hand-edited) + **`hl_srg_deal_overrides`** (human
verdicts — the receipts; fields WIN over SISU when present; `excluded=true` = an audited non-deal that stays
for provenance but leaves every count). **Every downstream consumer reads the ledger** — re-derivation is
how drift happens. **THE MASTER HOUSE LIST** = the `hl_srg_master_houses` VIEW (1 row per HOUSE + every
ledger transaction nested) built by `build-master-houses.mjs`, which also prints THE SCOREBOARD.

**The LLM law:** an LLM **never assigns identity** — keys assign identity. Its jobs: the dossier prose,
fact extraction from raw sources, and structured verdict PROPOSALS on residue (it proposes, code disposes,
a human confirms).

## Canonical data law — the specifics

### Structured-data source for past-client homes = OUR downloaded MLS

The structured fields on a past-client home come from **`hl_srg_mls_deals` (our downloaded Spradling Group MLS), NOT Ninja/Zillow**: `beds`, `full_baths` / `half_baths`, `sqft`, `year_built`, `lot_size`, `property_type`, `subdivision`, schools (elem/middle/high/district), `hoa_fee` / `hoa_freq`, `tax`. Match a past-client `Listing` → its deal row **by `mls_number`** (extract `\d+` from the `hl_heartlandmls_<mls>` hlid), **fallback by ADDRESS** (~95% combined), then **COALESCE-fill** the null `Listing` fields. `bathroomsTotal = full_baths + 0.5 · half_baths`. **This is FREE — never buy from Ninja what our MLS already holds.** Granular interior/exterior/pool **feature lists are NOT in `hl_srg_mls_deals`** → those come from **AI vision (gpt-4o), run on the sold set AFTER the structured fields land.** (Zillow/Ninja remains the source for Zestimate → value, value/tax/price history, GreatSchools, photos.)

### The master table + the backfill SEQUENCE (memorize the order)

`hl_srg_past_client_homes` (the child master, 1 row per past-client home) **HOLDS value columns BY DESIGN**: `current_zestimate` (+low/high), `current_hl_value` (+low/high), `value_history`, `original_zestimate`, `original_hl_value` (+low/high) — alongside the structural data. **The value lives on the CHILD master, not the parent `hl_srg_past_clients`.** The order, which never changes:

1. **Backfill the master FIRST** — each home's current value grounded to TODAY via a fresh OpenWeb Ninja Zestimate (**an old sale price is NOT today's value**); also store original value + history + structural data.
2. **Seed the website `Listing` from the master** (`hlEstimate` / Low / High / `valueHistory`) so the website value is right.
3. **Then the website is the read source** (steady state): `/home` renders from `Listing`, the CRM card pulls value by `website_hlid`.

⚠️ **Never strip the master's value columns before the website is seeded.** "CRM never copies values" is the END STATE; during the backfill the master IS the source that makes the website correct — emptying it early makes the website wrong.

### Per-field canonical sources (the remaining field rows)

- **Listing DESCRIPTION ("What's special") = Repliers `details.description` by `mls_number` — NEVER OpenWeb Ninja** (Ninja whiffs on description).
- **Photos = R2** (`Listing.photosCached` / `primaryPhotoUrl`) → Street View fallback. **NEVER the source CDN URL `cdn.repliers.io`** (unmirrored).
- **DNC (previous agent/employee) = EXACT full-name match** to SISU `agent` ∪ FUB tags `Agent:` / `Team Exit -` / `Imported Agent:`. **Never a surname-only collision**, never a literal "DNC" FUB tag.
- **Repliers agent, pick by our role:** client BOUGHT → `raw.BuyerAgentFullName`; client SOLD → `raw.ListAgentFullName`. If Repliers shows **"Spradling Group" (the team name, not a person)** → fall through to SISU for the individual.
- **Mover's NEW home = County deed by-surname name-search** (gis / AIMS / Wyandotte / Clay / Cass), then **deed-verify** the candidate — never an unverified guess.
