## ЁЯзн THE DEAL LEDGER тАФ match deals, not addresses (Zach LOCKED 2026-07-03)

**The insight that ended three weeks of resolver pain: we were answering the wrong question.** "Whose house is
this ADDRESS?" is unanswerable deterministically тАФ addresses lie (dropped unit letters, colliding slugs, condo
floorplans that repeat, miskeyed streets). Every failure of 2026-07-03 (a deleted real sale, wrong-unit pins, a
phantom HLID, "Hutchens isn't on 850th") came from reasoning by **address similarity or render quality**. The
right question is **"which DEAL is this?"** тАФ a deal fingerprints as **client name + price + close date**, and
we hold TWO independent ledgers of the same deals that join on those keys:

- **Edge 1 тАФ SISUтЖФMLS:** `sisu_transactions` (clientтЖТdeal) тЖФ `hl_srg_mls_deals` (dealтЖТlisting) by last name +
  price ┬▒$3k + close date ┬▒20d. **Measured live 2026-07-03: 85% of all SISU closings since 2015 match; 1,452 of
  1,780 past clients resolve through this edge alone.**
- **Edge 2 тАФ the MLS row's own client name:** **1,782 of 1,971 `hl_srg_mls_deals` rows carry `client_first`/
  `client_last`**; 1,492 of 1,780 past clients match it exactly.

**The three hops (no similarity anywhere):**
> **CLIENT** тЖТ(name+price+date)тЖТ **MLS#** тЖТ(history stamp)тЖТ **HLID** тЖТ child row on the card.

The MLS# is the hero: it names the **true unit** (the client's own closing listing), which kills the entire
wrong-unit/slug-collision bug class at the root.

**The ledger:** all matches materialize ONCE in **`hl_srg_deal_ledger`** (CRM DB `autorack`,
`HL_CRM_DATABASE_URL`) тАФ 1 row per client├Чdeal: `pc_id ┬╖ sisu_id ┬╖ mls_number ┬╖ hlid ┬╖ side ┬╖ price ┬╖
close_date ┬╖ agent ┬╖ match_method ┬╖ tier ┬╖ evidence jsonb`. Rules:
- Both edges agree тЖТ **GREEN** ┬╖ one edge only тЖТ **YELLOW** ┬╖ edges disagree or no key тЖТ **RED**.
- The ledger is **rebuilt idempotently by tool (`tools/houseloop-cards/build-deal-ledger.mjs`, UPSERT-only тАФ
  no DELETE/TRUNCATE ever) and NEVER hand-edited.** Human/LLM verdicts land in **`hl_srg_deal_overrides`**,
  which the rebuild consumes. A disagreement is a ROW, not an argument between agents.
- **Every downstream consumer** (child rows, GT pins, cards, fork fixes) READS the ledger instead of re-deriving
  a match. Re-derivation is how drift happens.

**The LLM law:** an LLM **never assigns identity** тАФ keys assign identity. Its three legitimate jobs: (1) the
relationship dossier prose (shipped); (2) fact extraction from unstructured sources (FUB notes тЖТ departed
agents); (3) for the RED residue (off-market, referrals, garbled source addresses), read ALL evidence and write
a structured verdict + reasoning into the review queue тАФ **it proposes, code disposes, a human confirms.**

**The fork hunt (classes AтАУF) runs as detectors over the ledger** тАФ not a manual sweep. Adam's six bug classes
(same-home multi-txn forked ┬╖ mixed units under one HLID ┬╖ client's listing not seated ┬╖ wrong-unit pin ┬╖
phantom listings ┬╖ mislabeled closings) are each ONE SQL query against the ledger + `hl_mls_property_history`
+ WEB `Listing`; the per-rooftop verdict map for the single Rule #0 fix pass is **generated**
(`tools/houseloop-cards/deal-ledger-detectors.mjs`). Regression bar: the 5 hand-verified rooftops (850th ┬╖
Granada #230 ┬╖ Fontana #306 ┬╖ 10240-F ┬╖ 10136-B) must each be re-found by their class detector.

Build contract: `.claude/state/session-scope-deal-ledger.lock`. Phases: L1 ledger тЖТ L2 detectors/verdict map тЖТ
L3 residue queue тЖТ L4 downstream reads the ledger.

## ЁЯФз THE PROCESSES / TOOLS тАФ the per-client flow

The system is **harvest-once (bulk) тЖТ resolve-per-client (deterministic) тЖТ LLM reconcile тЖТ human gate тЖТ
publish**. Boxes 1тАУ5 are bulk prep; Boxes 6тАУ7 are the per-client loop.

**Phase A тАФ Bulk harvest (run once across the book; the 5 harvest tables):**
`harvest-fub-facts.mjs` (brain, Haiku) ┬╖ `harvest-mls-history.mjs` + `resolve-mls-rooftops.mjs` (MLS
rooftop, via Camp David Matrix scraper) ┬╖ `restamp-mls-hlid.mts` + `link-hlid-on-mls.mjs` (HLID stamp) ┬╖
`harvest-loop-full.mjs` + `dotloop-auth.mjs` (dotloop `.loop`, Haiku Tier-3 only) ┬╖ `scrape-deed-homes.mjs`
(lost-buy source) ┬╖ `dedupe-past-clients.mjs` (one card / person).

**Phase B тАФ Per-client resolution (Box 6, "reconcile, don't rebuild"):**
1. `triangulate-resolver.mjs` тЖТ `createResolver(c).assemble(id)` тАФ **Stage 1, deterministic, $0, no LLM.**
   JOINs the 5 harvest tables + roster + identity/bucket + Adam's GT table, unions transactions (dedup
   rooftop+side), runs the agent waterfall, tiers GREEN/YELLOW/RED. Reads ONLY virgin sources (never its
   own prior output).
2. `triangulate-stage2.mjs` тЖТ **Stage 2 Haiku one-shot** тАФ reconciles conflicts + writes the reactivation
   dossier (2 paragraphs + facts matrix) + confirms tier. Writes STAGING (`hl_srg_resolver_stage2`).
3. `complete-closings.mjs` тЖТ materialize each SISU/lost closing as its own child row (buy/our-sell/lost-sell),
   `--write` gated, scoped to given ids.
4. `link-contacts-fub.mjs` тЖТ deterministically resolve each household member's `fub_id`/`fub_url` onto
   `dossier_json.contacts`, `--write` gated.
5. `persist-reconcile.mjs` тЖТ stage the resolver output into `hl_srg_pch_staging` (the "Reconciled тАФ &lt;date&gt;"
   audit tab).

**Phase C тАФ Human gate + publish (Box 7):**
6. `resolution-gate.mjs <pc>` тЖТ the deterministic bar (5 conditions); `--signoff` stamps
   `houseloop_resolved_at`/`_by` (**refuses to stamp a FAIL**). DNC clients bypass as a valid suppression card.
7. `fub-publish.mjs <pc> --write` тЖТ publishes the resolved card as a FUB note on every household member +
   the spouse relationship link. **Refuses any client not HouseLoop-Resolved.**

**The one card builder:** `apps/houseloop-crm/lib/pastclient/card-note.mjs` (`deriveCard` тЖТ `renderCardNote`)
is a pure function shared by BOTH the Audit route AND `fub-publish.mjs`, so the audit card and the FUB note
are **byte-for-byte identical**.

**Only 4 scripts spend LLM money** (all Haiku `claude-haiku-4-5-20251001` on `ANTHROPIC_API_KEY_HOUSELOOP`,
except s6 = Sonnet): `harvest-fub-facts` ┬╖ `harvest-loop-full` (Tier-3 doc read) ┬╖ `triangulate-stage2` ┬╖
`s6-investigator` (legacy Sonnet, **report-only**). Everything else is deterministic $0.

**Safety (every mutating tool):** `--write`/`--execute`/`--commit`/`--signoff` are **DRY-by-default** (print
the plan first) ┬╖ reversible everywhere (`_old` columns, reversal JSONs, `hl_srg_dedupe_log`,
