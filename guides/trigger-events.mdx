### The five laws (locked 2026-07-11)

1. **Integration → Source → Event.** Registration events group under the SOURCE that births the lead.
2. **Creates Lead exists ONLY on registration rows** — and it is the fail-closed **MINT gate**: OFF on a door
   means no new lead ever mints through its address (an existing lead still fires the event).
3. **ONE Existing-Lead-Activity section per integration.** Behavior is branded to the platform it happens on —
   a Zillow save can never get a HouseLoop-branded message.
4. **A birth fires only its own door.** A webinar signup runs the webinar workflow, never General Registration.
5. **Robot actions are silent.** Actions a workflow performs FOR a lead (seeded account, saved search) never
   fire trigger events; only the lead's own actions fire.

### Creating sections + custom events

Header buttons **New Section** and **Create Trigger Event** open the same four-question modal: *Name it ·
Section (a folder is born with its first event; blank = Other) · Which source feeds it · Workflow that runs.*
Create returns the event's **landing-page address** (`https://crm.houseloop.ai/api/leads/in/<source>/<event>`)
— paste it into the signup page with the source's secret key (`?key=` or the `x-houseloop-intake-secret`
header). Monthly-webinar routine: create next month's event in the folder → copy its address → paste into the
new landing page. Done.

## What a trigger event is

A thing that happened (someone did something) that can **create a lead** and/or **fire a workflow**. Every
trigger event is a row on the board with its own switches. The board groups Integration → Source → Event (v3):

| Band | Sections inside | What fires them |
|---|---|---|
| **HouseLoop.ai Website** | One **New Lead Registration folder per SOURCE** (HouseLoop.ai Organic's 9 doors · Webinars · any source you create) + your **custom sections** (e.g. Webinar Signups) + ONE **Existing Lead Activity** section (12 events, NO Creates Lead column) | Registration = a brand-new visitor through that source's pipe; existing activity = any known lead acting on the site, regardless of birth source |
| **Follow Up Boss** | **New Lead Registration · sources pulled from FUB** (Creates Lead + Sync) + **Existing Lead Activity · behavior inside FUB** (no Creates Lead) | FUB webhook signals — see the [FUB master](/fub/trigger-events) |
| **Transaction Coordination** | One section — no Creates Lead, no source (a loop always hangs on an existing lead) | e-sign documents + loop milestones |

## THE ONE-SWITCH LAW (workflow on/off lives HERE)

**A workflow is turned on or off in exactly one place: its trigger event's `Workflow Active` toggle on this
board.** Flipping it drives the workflow's real status (`hl_workflows.status`) — the Workflows page shows a
**read-only** status badge that mirrors this toggle, never an independent switch. Locked infrastructure
workflows (e.g. saved-search alert delivery) are always-on and exempt.

Each row's other switches: **Creates Lead** (does this event mint a lead?) and, for FUB rows, **Sync** +
**Notify** (see the FUB master). The event itself always fires once it exists — toggles gate the lead and the
workflow, never the firing.

## THE FIRE LEDGER — `hl_trigger_fire` (why the count never lies)

Every genuine door dispatch writes **exactly one row** to `hl_trigger_fire` (`lead_id` · `event_key` ·
`platform` · `matched` · `routed_by`), written by `dispatchDoor` in `lib/workflows/runtime.ts` — the one place
every `houseloop/door.<key>` flows through.

- The board's **TRIGGERED count** and its **click-through drill-down** ("which leads fired this") read this
  SAME table — they agree by construction.
- The native board is scoped `platform <> 'follow_up_boss'` — a FUB-origin fire can never appear on a
  HouseLoop.ai row (and vice versa).
- **Only a dispatch is a fire.** The lead activity feed is history, not a fire source — imported FUB history,
  system-seeded searches, and replayed pre-registration browsing can never masquerade as trigger fires.
  (This replaced the old two-source scoreboard that once showed "27 triggered, 0 leads.")
- `lead_id` is `ON DELETE CASCADE` — deleting a lead removes its fires.

## Routing (what happens when a door fires)

`dispatchDoor` resolves the workflow through strict tiers: the lead's **source × event assignment** first, then
platform / global fallbacks for system events. **Registration doors route ONLY via their per-source
assignment** — the fallback tiers carry a `*_registration` guard, so a first-touch outreach workflow can never
fire from a fallback match (the guard that closed the 1,421-import near-miss, 2026-07-09).
