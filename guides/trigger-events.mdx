## 2026-07-13 — SHIPPED: trigger events are CONFIGURED, not created

**The model, in one breath (Zach locked it live):** a trigger event is **ONE row per human action**, named exactly
what it is — Contact Request, Saved Home, Property Viewed. **You never hand-create a trigger event.** Every event
either comes from a **lead integration** (HouseLoop.ai, Follow Up Boss, Company Inbox, Transaction Coordination
auto-populate their events) or is **Manual**. What you do to a row is **Configure** it — the ⋮ (kebab) menu on each
row is **Configure** + **Pause**. There is no "Create Trigger Event" modal anymore.

### The Configure modal — two arms

**Configure** opens a modal with up to **two arms**, and each arm is exactly **one workflow + one notification + an
on/off**:

- **New Lead arm** — shows **only when the event can mint a lead** (the identity gate, `target.canMint`). It carries
  a workflow dropdown (+ *create* jumps to the builder), the automatic **"New Lead" notification class** (fixed),
  and its on/off. When the event can't mint, this arm is hidden.
- **Existing Lead arm** — always shows. Workflow dropdown (+ *create* → builder), a **notification dropdown** (the
  full catalog, see below), and its on/off.

**Single-arm events.** When an event can't mint (`canMint=false`), only ONE arm renders and the modal reads
**"Existing Leads Only"**:

- **HouseLoop Insights** (the computed-pattern band) is **single-arm "Existing Leads Only"** — an insight is always
  about a lead we already know (hard-locked existing).
- **Transaction Coordination** is **single-arm "Existing Leads Only"** — a loop always hangs on an existing lead.

The arm's three flags (`creates_lead` / `workflow_active` / the workflow id, and the existing-arm's
`existing_workflow_*`) are written as a unit through `PATCH /api/trigger-events`. Config lives in
`components/settings/trigger-events/ConfigureTriggerEventModal.tsx`.

### The notification dropdown is sectioned

The Existing-Lead arm's notification picker is the **full notification catalog**, grouped into the **same gray band
headers the Notifications settings page shows, 1:1** (New Lead Activity · Existing Lead Activity · Communications · …)
— driven by `lib/notifications/notification-sections.ts` and rendered with the sanctioned `HlDropdown` (a bare
`<select>` can't draw gray dividers). The New-Lead arm's notification is the fixed "New Lead" class.

### Sections = integrations · in-table bands = Activity / Communication / Insights

The board groups every trigger event under its **integration section** (a blue section header), and inside a
section, Excel-style band rows split the events by TYPE:

| Section (integration) | What it holds |
|---|---|
| **HouseLoop.ai Website** | the native website events — browsing, inquiries, registration, and the computed insights |
| **Follow Up Boss** | FUB webhook signals (see the [FUB master](/fub/trigger-events)) — **zero phone rows**: phone triggers live in the inbox sections (2026-07-16) |
| **Inbox sections** (one per inbox with doors) | **Company Inbox** (913-320-0906 · sign 913-354-8600 · email) · **Past Agents Inbox** (913-675-9877, the 37-number FUB pool) · **personal inboxes** (Zach's FUB seat 913-490-2988 · Ana legacy 913-954-7123). Rows chaptered 📞/💬/📧; a FUB-carried door's line reads "FUB …" |
| **Transaction Coordination** | e-sign documents + loop milestones (existing-lead only) |
| **Manual** | hand-run events with no integration (the catch-all — "Other/Custom" was retired) |

The in-table **bands** come from the `band` column on `hl_trigger_events`:

- **ACTIVITY** — everything the person DOES (browsing + inquiries + registration).
- **COMMUNICATION** — an actual communication received (Text / Call / Email Received — LIVE wires fired by the
  inbound writers, minting through Company Inbox).
- **INSIGHTS** — computed patterns. **Property Viewed now lives in the Insights band** (moved there in **mig 0395**).

### Drag-reorder persists (sections AND events)

Both the **integration sections** and the **individual trigger events within them** can be **drag-reordered**, and the
order **persists to `hl_lead_flow_section_order` (mig 0396)** via **`/api/lead-flow/order`**. The Trigger Events board
and the Workflows page share ONE order — reorder on either, it holds on both.

### Native insights (Zillow-parity rules)

The computed insights — **Viewed Home 10x in 7 Days** (the same home) · **Looking in a New Area** (2+ views in a ZIP
never browsed in the prior 60 days) · **Re-Engaged** (active after 14+ days of silence) — are computed hourly by the
registered Inngest fn `insight-engine` over our own pixel data (deterministic SQL, receipts in every fire, a hard
14-day per-lead-per-insight gate checked against `hl_trigger_fire`, no tags). Thresholds live in `hl_settings` key
`insight_engine`. An insight only ever aggregates patterns — it never re-fires a single event that has its own trigger.

## What a trigger event is

A thing that happened (someone did something) that can **create a lead** and/or **fire a workflow**. Every trigger
event is a row on the board with its own arms. It is **auto-populated by its integration or Manual** — you Configure
it, you don't create it.

## THE ONE-SWITCH LAW (workflow on/off lives HERE)

**A workflow is turned on or off in exactly one place: its arm's on/off inside the trigger event's Configure modal on
this board.** Flipping an arm drives that workflow's real status (`hl_workflows.status`) — the Workflows page shows a
**read-only** status badge that mirrors this toggle, never an independent switch. Locked infrastructure workflows
(e.g. saved-search alert delivery) are always-on and exempt.

The event itself always fires once it exists — the arm toggles gate the lead and the workflow, never the firing.

## THE FIRE LEDGER — `hl_trigger_fire` (why the count never lies)

Every genuine door dispatch writes **exactly one row** to `hl_trigger_fire` (`lead_id` · `event_key` · `platform` ·
`matched` · `routed_by`), written by `dispatchDoor` in `lib/workflows/runtime.ts` — the one place every
`houseloop/door.<key>` flows through.

- The board's **TRIGGERED count** and its **click-through drill-down** ("which leads fired this") read this SAME table
  — they agree by construction.
- The native board is scoped `platform <> 'follow_up_boss'` — a FUB-origin fire can never appear on a HouseLoop.ai row
  (and vice versa).
- **Only a dispatch is a fire.** The lead activity feed is history, not a fire source — imported FUB history,
  system-seeded searches, and replayed pre-registration browsing can never masquerade as trigger fires.
- `lead_id` is `ON DELETE CASCADE` — deleting a lead removes its fires.

## Routing (what happens when a door fires)

`dispatchDoor` resolves the workflow through strict tiers and **picks the arm from the dispatch's created-flag**:
`created=true` reads the New-Lead columns, `created=false` the Existing-Lead columns — fail-closed, so an unbound or
toggled-off arm is a logged skip, never a fallback guess. **Registration doors route ONLY via their per-source
assignment** — the fallback tiers carry a `*_registration` guard so a first-touch outreach workflow can never fire
from a fallback match (the guard that closed the 1,421-import near-miss, 2026-07-09).
