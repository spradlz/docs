## What we're building

TC Loop = HouseLoop's own dotloop + A-Frame + Documenso, AI-driven. A **self-contained, role-gated surface**:
a Transaction Coordinator logs in and has everything inside TC Loop (Dashboard · Loops · lead/loop detail ·
Configure) and never touches the CRM pipeline. Built human-first — a person can run a loop entirely by hand;
**Myli-TC is just another caller of the same API endpoints** (no AI backdoor). Every layer is editable +
behind a REST API so Myli drives it the same way a human does.

## How A-Frame actually works — the 3-layer engine (LOCKED 2026-06-27)

Grounded in the verbatim A-Frame recon (`.claude/reports/aframe-recon/TC_MILESTONE_TASK_MAP.md` — 139 tasks,
every one with its anchor + offset + assignee). A-Frame runs on **three layers + an offset engine**:

1. **STATUS (11)** — the headline word (Active · Under Contract · Pending · Closed…). = our **Stages**. ~5/side.
2. **DATES — only 4 KEY DATES you ENTER drive everything:** `effective` (Under Contract) · `inspection_end`
   (Inspection Period Ends) · `closing` · `on_market` (listing live). These are the **milestones'** anchors.
3. **TASKS (~139, in 8 phase folders/side)** — every task = `anchor date ± offset days`, grouped into phases
   (Intake · Contract Review · Scheduling · Communication · Compliance · Inspections · Appraisal · Closing).

**The offset engine is the trick:** enter the 4 key dates once → the system auto-schedules every milestone +
task (`anchor + offset`). e.g. Title Commitment = `effective+10`, Appraisal = `effective+25`, CDA = `closing−14`.

> **Status = headline · Milestones = the key dates that drive everything · Tasks = the auto-scheduled
> checklist hanging off those dates.** Milestones and the task list aren't rivals — milestones are the dates,
> tasks are the work.

## The status + milestone model (LOCKED 2026-06-26 — two axes)

- **STATUS** = the stepper. REUSE `hl_pipelines`/`hl_stages` + `StagesPage`, scoped to TC. No new stage table.
  - **TC Listing (sale):** `Pre-MLS → Active → Under Contract → Closed` + `Off-Market · Expired · Withdrawn · Canceled`.
  - **TC Buyer (purchase):** `Active Buyer → Under Contract → Closed` + `Lost`.
- **MILESTONES** = the dated work that drives status forward (the set above). Completing one with an
  "advances to" stage fires the guarded, forward-only advance. Status is stored (human can override).
- **The Loop is born at Appointment Set** (the container); the agreement gate (sent → signed) is milestones
  inside the already-born loop. **Deaths carry a cause** (`lost_reason`: `didnt_sign` · `canceled_listing` ·
  `fell_through`). Full spec: PRD §12.

## Architecture (locked)

1. **API-first.** Every loop action is a REST endpoint; the UI and Myli call the same one.
2. **Role-gated `/tc/**` surface.** TC role → only TC Loop. No CRM↔TC toggles.
3. **Reuse the engine, scope to TC** — Stages reuse `StagesPage`; Milestones/Tasks reuse the management-table
   + `HlFormModal` pattern on TC-scoped data; Email/Text templates reuse `hl_templates` (`scope='tc'`); Merge
   Fields extend the code catalog.
4. **Status is side-aware** (TC Buyer vs TC Listing; milestones + tasks are per-side too).
5. **E-signature = Documenso.**
6. **Tables are real CRM tables** — every `<HlDataTable>` ships the ⋮ kebab + select-all bulk-delete
   (`hl/table-requires-actions`); modals use the canonical blue-header `HlFormModal`.

## The loop engine (built + proven)

`lib/tc/loop-engine.ts`: `createLoop({ leadId, side })` (born into first stage + instantiates milestones) ·
`completeMilestone({ transactionId, milestoneKey })` (forward-only, idempotent advance) · `getLoopStatus` /
`listMilestones`. REST: `POST /api/tc/loops` · `GET /api/tc/loops/[id]` · `POST /api/tc/loops/[id]/milestone`.
Milestones: `/api/tc/milestones` (GET `?side` · POST · PATCH reorder) · `/[id]` (PATCH · DELETE, guarded).
