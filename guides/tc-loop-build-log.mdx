## What we're building

TC Loop = HouseLoop's own dotloop + A-Frame + Documenso, AI-driven. A **self-contained, role-gated surface**:
a Transaction Coordinator logs in and has everything inside TC Loop (Dashboard · Loops · lead/loop detail ·
Configure) and never touches the CRM pipeline. Built human-first — a person can run a loop entirely by hand;
**Myli-TC is just another caller of the same API endpoints** (no AI backdoor). Every layer is editable +
behind a REST API so Myli drives it the same way a human does.

## ⭐ A-Frame parity doctrine (LOCKED 2026-06-27)

**Right now the stages, milestones, and tasks are an EXACT MIRROR of A-Frame** — the TC software the team runs
on today. Names, phases, anchors, offsets, and assignees are lifted verbatim from the A-Frame recon. The **only**
thing that's ours is the UI chrome (the blue section bands, pills, tables). The content is A-Frame's.

**Why parity:** mirror A-Frame 1:1 so we can understand exactly how it operates and measure parity before we
change anything. While in parity mode, a divergence in task content is a bug, not a feature.

**Two deliberate, Zach-approved cleanups (label/UI only — not content):** (1) the listing pre-market phase
labels were renamed from meaningless "Pre-Live: Stage 1…5" to A-Frame's real time phrases, with the two
ambiguous ranges hard-set to single numbers; (2) the UI chrome is HouseLoop-native. The tasks themselves are
untouched.

**No per-task instruction manuals.** Verified from the recon: A-Frame's task `description` field is empty on all
templates, and individual steps carry no how-to field — only name · due/anchor · assignee · (optional linked
email). There was nothing to import, and we are **not** building an instructions layer now.

**Future (deferred):** this A-Frame-mirrored set eventually converts to a **HouseLoop-native TC set** (rebranded
names, possibly per-task instructions / Myli playbooks) — a later pass, only when we choose to "HouseLoop-ify"
it. Until then every task stays A-Frame-faithful.

## How A-Frame actually works — the 3-layer engine + dynamic dates (LOCKED 2026-06-27)

Grounded in the A-Frame recon (`.claude/reports/aframe-recon/TC_MILESTONE_TASK_MAP.md`) + the TC-software
research (`.claude/reports/tc-dynamic-dates-research-2026-06-27.md` — Open To Close gold reference + TRID).
Three layers + an offset engine:

1. **STATUS (11)** — the headline word. = our **Stages**. ~5/side.
2. **DATES — 4 KEY DATES you ENTER drive everything:** `effective` (Under Contract) · `inspection_end` ·
   `closing` · `on_market`. These are the **milestones'** anchors.
3. **TASKS (139, in phase folders)** — every task = a **relative rule** (`anchor + offset`), grouped into
   phases. Buyer = Intake · Contract Review · Scheduling · Communication · Compliance · Inspections · Appraisal
   · Closing. Listing adds the **pre-market run-up** (chronologically first): **Within 24h of ERTS Signed · 10
   Days Before Going Live · 5 Days Before Going Live · Listing Goes Live · Weekend After Going Live**.

**Dynamic dates = the offset engine done right (mig 0258 + `lib/tc/date-engine.ts`):** a task/milestone stores
a RELATIVE rule, and the due date **computes from the live key date — so it cascades when the key date moves.**
The rule: `anchor` (a key date) **or** another task (single-hop chain) · signed offset · **`day_unit`**
(calendar / business / calendar-skip-weekend / week / month) · **`business_day_def`** (TRID `precise` counts
Saturday, skips Sun + federal holidays; `general` skips Sat too) · **weekend/holiday roll** (lands on a
non-working day → next/prev business day) · versioned **holiday calendar** (US_FEDERAL). Plus **conditional
"If Applicable"** tasks (~25% — LBP/HOA/VA) + `task_type` (action/document). Engine proven: 13/13 unit tests
(Friday-closing Sat-counts, weekend+holiday roll chain, month clamp).

> **Status = headline · Milestones = the key dates that drive everything · Tasks = the auto-scheduled
> checklist hanging off those dates.** Enter 4 dates → the whole plan schedules; move one → it all re-shifts.

## Architecture (locked)

1. **API-first.** Every loop action is a REST endpoint; UI + Myli call the same one.
2. **Role-gated `/tc/**` surface.** No CRM↔TC toggles.
3. **Reuse the engine, scope to TC. Never hand-roll** — every Configure page is identical (Add/Refresh header
   + `PipelineSwitcher` + `HlDataTable`; modals = `HlFormModal`; filters = `HlFilterPills`; section headers =
   the shared blue `SectionBand`; no KPI cards on a template editor).
4. **Side-aware** (TC Buyer vs TC Listing).
5. **E-signature = Documenso.**
6. **Real CRM tables** — `HlDataTable` (`hl/table-requires-actions`) + canonical `HlFormModal` modals.
