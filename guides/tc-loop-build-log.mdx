## ‚≠ê Build our own dotloop ‚Äî owned end-to-end (LOCKED 2026-06-27)

**The decision:** TC Loop is **our own dotloop**, built and owned **end-to-end on our side**. We do NOT hand
dotloop our finished documents ‚Äî that would just make it "a very expensive filing cabinet." dotloop is OUT of
the critical path; at most it's a **legacy importer** (read historical SRG loops via the API we already have).

**Why own it ‚Äî the two hard reasons dotloop can't satisfy:**
1. **dotloop can't fill or sign via API.** No API-fillable smart fields, and you can't place signature fields or
   trigger a signature through its public API (UI-only). So Myli could never run a contract through dotloop.
2. **SkySlope / eXp compliance requires we own the documents end-to-end** ‚Äî we must push signed docs **directly
   to SkySlope**, only possible if the documents live on our side.

**The stack ‚Äî all ours, reusing CRM components:**
| Layer | What | Where |
|---|---|---|
| Loop | container + status/stage | `hl_transactions` + `hl_tc_stage` ‚úÖ |
| Documents | folders + document lines (**our source of truth**) | `hl_tc_loop_folder` / `hl_tc_loop_document` ‚úÖ |
| Forms + fill | our PDF templates + data-fill (Myli supplies values) | `lib/tc/{contract-forms,fill-contract,template-store}.ts` + `tools/houseloop-contracts/` ‚Äî pdf-lib **drawText overlay** (not AcroForm) ¬∑ `field-maps/` ¬∑ `vision-field-builder.mjs` auto-maps fields ¬∑ `{{signature/date/initial}}` e-sign tags ‚úÖ skeleton |

### E-signature = Documenso (self-hosted) + the compliance bar
US e-signatures are governed by the **ESIGN Act (federal) + UETA** (both adopted by **Kansas and Missouri**). An
e-signature is legally binding for agency/purchase contracts when we capture:
1. **Intent to sign** ¬∑ 2. **Consent** to do business electronically (a disclosure the signer agrees to) ¬∑
3. **Attribution** ‚Äî the signature is provably tied to the signer (email/identity verify + IP + **timestamp**) ¬∑
4. **Integrity + retention** ‚Äî the record is **tamper-evident** (document hash / digital seal) and retained,
reproducible.

The legal backbone is the **audit trail / completion certificate** (who ¬∑ when ¬∑ IP ¬∑ email-verified ¬∑ doc hash
¬∑ event log) plus the **sealed** final PDF. US real estate does **not** require an RFC-3161 trusted-timestamp
authority (that's EU/eIDAS) ‚Äî a reliable system timestamp + audit log + seal is the accepted standard (what
dotloop/DocuSign use). Broker **record retention** (KS/MO ~3 yrs) ‚Üí we store the signed doc + certificate
immutably.

**Documenso provides all of it** ‚Äî audit log, PDF digital-signature/certificate seal, completion certificate,
and an API to place fields, send, and webhook on completion ‚Äî and is **self-hostable**, so we own the data and
pay **$0 per signature**. (Chosen over SignWell for ownership + no per-sig cost + SkySlope-grade control.)

**dotloop's role now:** legacy historical-loop import only (read-only API in `lib/dotloop/client.ts`).
Documenso replaces it for signing; SkySlope is the compliance destination.

## ‚≠ê A-Frame parity doctrine (LOCKED 2026-06-27)

The stages, milestones, and tasks are an **EXACT MIRROR of A-Frame** right now (names, phases, anchors, offsets,
assignees lifted verbatim from the recon). The only thing that's ours is the UI chrome. **Why:** mirror A-Frame

## How A-Frame works ‚Äî the 3-layer engine + dynamic dates

Three layers + an offset engine:
1. **STATUS** ‚Äî the headline word = our **Stages** (~5/side).
2. **DATES ‚Äî 4 KEY DATES you ENTER drive everything:** `effective` (Under Contract) ¬∑ `inspection_end` ¬∑
   `closing` ¬∑ `on_market` ‚Äî the milestones' anchors.
3. **TASKS (139, in phase folders)** ‚Äî each = a **relative rule** (`anchor + offset`). Buyer = Intake ¬∑ Contract
   Review ¬∑ Scheduling ¬∑ Communication ¬∑ Compliance ¬∑ Inspections ¬∑ Appraisal ¬∑ Closing. Listing adds the
   pre-market run-up: Within 24h of ERTS Signed ¬∑ 10 Days Before Going Live ¬∑ 5 Days Before Going Live ¬∑ Listing
   Goes Live ¬∑ Weekend After Going Live.

**Dynamic dates (mig 0258 + `lib/tc/date-engine.ts`):** a task/milestone stores a RELATIVE rule; the due date
**computes from the live key date and cascades when it moves.** Rule = anchor (key date) or task‚Üítask chain ¬∑
signed offset ¬∑ `day_unit` ¬∑ `business_day_def` (TRID precise/general) ¬∑ weekend/holiday roll ¬∑ holiday calendar
(US_FEDERAL) ¬∑ conditional "If Applicable" ¬∑ `task_type`. Proven: 13/13 unit tests.

> **Status = headline ¬∑ Milestones = the key dates ¬∑ Tasks = the auto-scheduled checklist.** Enter 4 dates ‚Üí the
> plan schedules; move one ‚Üí it all re-shifts.

**Task counts (A-Frame parity):** 58 buyer + 81 listing = **139**. Listing's 81 = 57 contract-to-close + 24 pre-market.

## Architecture (locked)

1. **API-first.** Every loop action is a REST endpoint; UI + Myli call the same one.
2. **Role-gated `/tc` surface + an `Agent ‚áÑ TC` SURFACE-SWITCH toggle** (TC/Admin only) that routes
   `/leads/[id]` ‚áÑ `/tc/transactions/[loopId]` ‚Äî the whole shell switches, not an in-place panel.
3. **Reuse CRM components. Never hand-roll** ‚Äî `StagesPage`, `PipelineSwitcher`, `HlDataTable`, `HlFormModal`,
   `HlFilterPills`, `SectionBand`, the lead-detail lanes + comms streams + `MyliLane`.
4. **Side-aware** (TC Buyer vs TC Listing).
5. **E-signature = Documenso, self-hosted** (own data, $0/sig, cert-sealed, ESIGN/UETA). dotloop NOT in the path.
6. **We own documents end-to-end** ‚Üí push to SkySlope for compliance.
