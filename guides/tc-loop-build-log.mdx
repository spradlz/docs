## What we're building

TC Loop = HouseLoop's own dotloop + A-Frame + Documenso, AI-driven. A **self-contained, role-gated surface**:
a Transaction Coordinator logs in and has everything inside TC Loop (Dashboard · Loops · lead/loop detail ·
Configure) and never touches the CRM pipeline. Built human-first — a person can run a loop entirely by hand;
**Myli-TC is just another caller of the same API endpoints** (no AI backdoor).

## The status + milestone model (LOCKED 2026-06-26)

Built off 5 parallel research sweeps + a full recon of how appointments/outcomes/deals are wired today.
The convergent, god-tier answer — **two axes, never one**:

- **STATUS** = where the deal physically is (the stepper). A single FK pointer → a stages table. We **REUSE
  `hl_pipelines`/`hl_stages` + the existing `StagesPage` builder, scoped to TC** (this is the textbook-correct
  model — add/reorder/recolor a stage = an INSERT, no schema change). **No new stage table, no new builder.**
- **MILESTONES** = the dated work that DRIVES status forward (the centerpiece). A separate events table
  (`hl_tc_loop_event`), seeded per loop-side from a template (`hl_tc_milestone_template`, edited in the
  **Milestones builder**). **Completing a milestone fires a guarded, forward-only stage advance** (the proven
  Salesforce/HubSpot "task done → deal advances" pattern). Status stays stored so a human can override;
  milestones drive it.

**The status stages (reuse `hl_stages`; two TC pipelines):**
- **TC Listing (sale):** `Pre-MLS → Active → Under Contract → Closed` + off-ramps `Off-Market · Expired · Withdrawn · Canceled`.
- **TC Buyer (purchase):** `Active Buyer → Under Contract → Closed` + `Lost`.

**The Loop is born at Appointment Set** — it's the container; you need it before you can run a CMA or store
the agreement you're about to send. The agreement gate is **milestones inside the already-born Loop:**

```
🏠 Listing Loop (sale)                      🔑 Buyer Loop (purchase)
Appointment Set ─► LOOP BORN (Pre-MLS)      Appointment Set ─► LOOP BORN (Active Buyer)
  • Pre-Listing Packet Created                • Exclusive Buyer Agency Prepped
    (= CMA + comparables + valuation            • Buyer Agreement Sent
       + marketing packet — ONE milestone)      • Buyer Agreement Signed
  • Listing Agreement Sent                    • Under Contract  → advances status
  • Listing Agreement Signed → Active         • Closed
  • Under Contract → Closed
```

**Deaths carry a cause** — one `lost_reason` on the loop terminal, so "real losses" filter cleanly:
`didnt_sign` (lost at appointment) · `canceled_listing` (couldn't sell it) · `fell_through` (lost it under
contract). Canceling a listing and falling out of contract are **not** the same death.

**The conflation this fixes:** today an appointment outcome `listing_signed` silently advances the lead
pipeline but creates **zero** transaction record (Deals tab stays empty); the buyer side has no agreement
outcome at all. The appointment-outcome and the transaction-birth are one event modeled as two islands. The
fix: **Appointment Set creates the Loop; the agreement gate (sent → signed) is a milestone pair that advances
or kills it.** Full spec: `GT_HOUSELOOP_TRANSACTION_COORDINATION_PRD.md §12`.

## Architecture (locked)

1. **API-first.** Every loop action is a REST endpoint. The human UI calls it; Myli calls the same one.
2. **Role-gated `/tc/**` surface.** TC role → only TC Loop. Agent role → CRM. No CRM↔TC toggles.
3. **Reuse the engine, scope to TC** — dedicated TC Configure pages reuse the existing components/data:
   - **TC Stages** → reuse `components/settings/stages/StagesPage.tsx` + `PipelineSwitcher`; data = `hl_stages`
     under the TC pipelines (`hl_pipelines.scope='tc'`). **Not a fork — the same builder.**
   - **TC Milestones** → per-side editor on `hl_tc_milestone_template`, mirroring the appointment-types/outcomes
     management-table pattern (drag-reorder, kebab, bulk-delete, active toggle, add/edit modal).
   - **TC Email + Text Templates** → reuse the `hl_templates` engine; `scope='tc'`.
   - **TC Merge Fields** → extend the code catalog (`lib/merge-fields.ts`) with loop/property/agent/date tokens.
4. **Status is side-aware** (TC Buyer vs TC Listing pipelines; milestones are per-side too).
5. **The lead/loop renders in TC view by role/surface**, not a toggle widget.
6. **E-signature = Documenso** (dotloop e-sign isn't in its API).
7. **Tables are real CRM tables** — every `<HlDataTable>` ships the standard far-right ⋮ kebab (Edit / Delete)
   and a select-all → bulk-delete bar, reusing the table's real per-row delete. Enforced by the
   `hl/table-requires-actions` ESLint rule. Read-only feeds/logs opt out with a documented reason (their verb
   is Cancel/Unpublish, not delete).

## The loop engine (built + proven)

`apps/houseloop-crm/lib/tc/loop-engine.ts` — the human-first, API-first core:
- **`createLoop({ leadId, side })`** — born into the side's first TC stage; instantiates the side's milestones
  from the templates (one transaction).
- **`completeMilestone({ transactionId, milestoneKey })`** — marks it done; if it carries
  `advances_to_stage_id` and that stage is **forward** of the current one in the same pipeline, advances the
  stored stage pointer. **Forward-only, idempotent, override-able.**
- **`getLoopStatus` / `listMilestones`** — the loop-file readout.

REST: `POST /api/tc/loops` · `GET /api/tc/loops/[id]` · `POST /api/tc/loops/[id]/milestone`. The UI and Myli-TC
call the **same** endpoints. Proven against real data: a listing loop born at Pre-MLS advances to Active on
`listing_agreement_signed`, and an earlier milestone completing later does **not** regress the status.

## First target flow (the MVP we build toward)

Listing or buyer appointment set → **createLoop** (born) → run CMA / prep agreement, stored in the loop →
send the agreement via Documenso → client signs → **completeMilestone(agreement_signed)** advances status →
the contract-to-close milestones carry it to Closed. Same flow both sides, different docs.
