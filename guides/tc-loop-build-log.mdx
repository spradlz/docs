## What we're building

TC Loop = HouseLoop's own dotloop + A-Frame + Documenso, AI-driven. A **self-contained, role-gated surface**:
a Transaction Coordinator logs in and has everything inside TC Loop (Dashboard В· Loops В· lead/loop detail В·
Configure) and never touches the CRM pipeline. Built human-first вЂ” a person can run a loop entirely by hand;
**Myli-TC is just another caller of the same API endpoints** (no AI backdoor). Every layer is editable +
behind a REST API so Myli drives it the same way a human does.

## How A-Frame actually works вЂ” the 3-layer engine (LOCKED 2026-06-27)

Grounded in the verbatim A-Frame recon (`.claude/reports/aframe-recon/TC_MILESTONE_TASK_MAP.md` вЂ” 139 tasks,
every one with its anchor + offset + assignee). A-Frame runs on **three layers + an offset engine**:

1. **STATUS (11)** вЂ” the headline word (Active В· Under Contract В· Pending В· ClosedвЂ¦). = our **Stages**. ~5/side.
2. **DATES вЂ” only 4 KEY DATES you ENTER drive everything:** `effective` (Under Contract) В· `inspection_end`
   (Inspection Period Ends) В· `closing` В· `on_market` (listing live). These are the **milestones'** anchors.
3. **TASKS (~139, in 8 phase folders/side)** вЂ” every task = `anchor date В± offset days`, grouped into phases
   (Intake В· Contract Review В· Scheduling В· Communication В· Compliance В· Inspections В· Appraisal В· Closing;
   listing adds Pre-Live Stages 1вЂ“5).

**The offset engine is the trick:** enter the 4 key dates once в†’ the system auto-schedules every milestone +
task (`anchor + offset`). e.g. Title Commitment = `effective+10`, Appraisal = `effective+25`, CDA = `closingв€’14`.

> **Status = headline В· Milestones = the key dates that drive everything В· Tasks = the auto-scheduled
> checklist hanging off those dates.** Milestones and the task list aren't rivals вЂ” milestones are the dates,
> tasks are the work.

## The status + milestone model (LOCKED 2026-06-26 вЂ” two axes)

- **STATUS** = the stepper. REUSE `hl_pipelines`/`hl_stages` + `StagesPage`, scoped to TC. No new stage table.
  - **TC Listing (sale):** `Pre-MLS в†’ Active в†’ Under Contract в†’ Closed` + `Off-Market В· Expired В· Withdrawn В· Canceled`.
  - **TC Buyer (purchase):** `Active Buyer в†’ Under Contract в†’ Closed` + `Lost`.
- **MILESTONES** = the dated work that drives status forward (the set above). Completing one with an
  "advances to" stage fires the guarded, forward-only advance. Status is stored (human can override).
- **The Loop is born at Appointment Set** (the container); the agreement gate (sent в†’ signed) is milestones
  inside the already-born loop. **Deaths carry a cause** (`lost_reason`: `didnt_sign` В· `canceled_listing` В·
  `fell_through`). Full spec: PRD В§12.

## Architecture (locked)

1. **API-first.** Every loop action is a REST endpoint; the UI and Myli call the same one.
2. **Role-gated `/tc/**` surface.** TC role в†’ only TC Loop. No CRMв†”TC toggles.
3. **Reuse the engine, scope to TC** вЂ” Stages reuse `StagesPage`; Milestones/Tasks reuse the management-table
   + `HlFormModal` pattern on TC-scoped data; Email/Text templates reuse `hl_templates` (`scope='tc'`); Merge
   Fields extend the code catalog. **Never hand-roll вЂ” every Configure page is identical** (Add/Refresh header
   + `PipelineSwitcher` + `HlDataTable`; no KPI cards on a template editor).
4. **Status is side-aware** (TC Buyer vs TC Listing; milestones + tasks are per-side too).
5. **E-signature = Documenso.**
6. **Tables are real CRM tables** вЂ” every `<HlDataTable>` ships the в‹® kebab + select-all bulk-delete
   (`hl/table-requires-actions`); modals use the canonical blue-header `HlFormModal`.
