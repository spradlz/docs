## ⭐ G2 — Forms + fill INTO the loop (✅ SHIPPED 2026-06-27)

**A form is filled from the loop's own data and the filled PDF lands in a loop folder as a document — all on our
side, fill engine reused verbatim (no fork).**
- **Endpoint:** `POST /api/tc/loops/[id]/documents` `{ formKey, folderId? }` — builds a `FillContext` from the
  loop's lead/deal (same shape `ContractFormsPage` uses: buyer name, agent name, deal address from the loop row)
  → `getTemplateBytes` → `fillContract` → uploads the filled PDF to **R2** (`tc-loops/<loopId>/<form>.pdf`) →
  inserts `hl_tc_loop_document` (`file_url` + `contract_fill_id` — both already existed, **no migration**). If the
  loop has no folder it auto-creates a "Forms" folder so the doc always renders. Clean 400/404/500 (no silent fail).
- **UI:** an **"Add from Form"** button + `AddFromFormModal` in the TC **Documents** tab (reuses `HlFormModal`;
  templates from the existing `/api/tc/contract/templates`); on submit it refetches the loop bundle so the new
  doc appears under its folder. No new shell component (`hl/no-duplicate-shell-components` passes).
- **Storage = R2** (same S3 client/creds as the photo engine; `uploadBufferToR2` added to `lib/media/r2.ts`).
  Filled PDFs are **never** base64'd into Postgres.
- **Verified real:** the filled PDF fetched back HTTP 200, `application/pdf`, `%PDF-1.7`, with genuine field
  values (`f_buyerName`, `f_brokerName=eXp Realty LLC`, `f_agentName`, `f_date`). Agent view unchanged.
- **Local dev note:** `R2_PHOTOS_*` live in repo-root `.env`; mirror them into `apps/houseloop-crm/.env.local`
  (gitignored) or local "Add from Form" returns "storage not configured." Prod (Railway) already has them.

## ⭐ Build our own dotloop — owned end-to-end (LOCKED 2026-06-27)

**The decision:** TC Loop is **our own dotloop**, built and owned **end-to-end on our side**. We do NOT hand
dotloop our finished documents — that would just make it "a very expensive filing cabinet." dotloop is OUT of
the critical path; at most it's a **legacy importer** (read historical SRG loops via the API we already have).

**Why own it — the two hard reasons dotloop can't satisfy:**
1. **dotloop can't fill or sign via API.** No API-fillable smart fields, and you can't place signature fields or
   trigger a signature through its public API (UI-only). So Myli could never run a contract through dotloop.
2. **SkySlope / eXp compliance requires we own the documents end-to-end** — we must push signed docs **directly
   to SkySlope**, only possible if the documents live on our side.

**The stack — all ours, reusing CRM components:**
| Layer | What | Where |
|---|---|---|
| Loop | container + status/stage | `hl_transactions` + `hl_tc_stage` ✅ |
| Documents | folders + document lines (**our source of truth**) | `hl_tc_loop_folder` / `hl_tc_loop_document` ✅ |
| Forms + fill | our PDF templates + data-fill → filled PDF (R2) into a loop folder | `lib/tc/{contract-forms,fill-contract,template-store}.ts` + `POST /api/tc/loops/[id]/documents` + Documents-tab "Add from Form" ✅ |
| E-signature | place fields · send · seal · audit | **Documenso, SELF-HOSTED** — to build (G3) |
| Compliance archive | signed PDF + completion certificate, immutable, retained | our storage (R2) — to build |
| Broker integration | push signed docs to **SkySlope (eXp)** | to build |
| People / Tasks | participants + the A-Frame dated checklist | `hl_tc_loop_participant` ✅ · `hl_tc_loop_task` (mig 0262) ✅ |
| The UI | the existing lead-detail page in TC `view` mode (reused, not rebuilt) | spec above + `.claude/reports/aframe-recon/DOTLOOP_LOOP_LAYOUT.md` |

### E-signature = Documenso (self-hosted) + the compliance bar
US e-signatures are governed by the **ESIGN Act (federal) + UETA** (both adopted by **Kansas and Missouri**). An
e-signature is legally binding for agency/purchase contracts when we capture:
1. **Intent to sign** · 2. **Consent** to do business electronically (a disclosure the signer agrees to) ·
3. **Attribution** — the signature is provably tied to the signer (email/identity verify + IP + **timestamp**) ·
4. **Integrity + retention** — the record is **tamper-evident** (document hash / digital seal) and retained,
reproducible.

The legal backbone is the **audit trail / completion certificate** (who · when · IP · email-verified · doc hash
· event log) plus the **sealed** final PDF. US real estate does **not** require an RFC-3161 trusted-timestamp
authority (that's EU/eIDAS) — a reliable system timestamp + audit log + seal is the accepted standard (what
dotloop/DocuSign use). Broker **record retention** (KS/MO ~3 yrs) → we store the signed doc + certificate
immutably.

**Documenso provides all of it** — audit log, PDF digital-signature/certificate seal, completion certificate,
and an API to place fields, send, and webhook on completion — and is **self-hostable**, so we own the data and
pay **$0 per signature**. (Chosen over SignWell for ownership + no per-sig cost + SkySlope-grade control.)

**dotloop's role now:** legacy historical-loop import only (read-only API in `lib/dotloop/client.ts`).
Documenso replaces it for signing; SkySlope is the compliance destination.

## How A-Frame works — the 3-layer engine + dynamic dates

Three layers + an offset engine:
1. **STATUS** — the headline word = our **Stages** (~5/side).
2. **DATES — 4 KEY DATES you ENTER drive everything:** `effective` (Under Contract) · `inspection_end` ·
   `closing` · `on_market` — the milestones' anchors.
3. **TASKS (139, in phase folders)** — each = a **relative rule** (`anchor + offset`). Buyer = Intake · Contract
   Review · Scheduling · Communication · Compliance · Inspections · Appraisal · Closing. Listing adds the
   pre-market run-up: Within 24h of ERTS Signed · 10 Days Before Going Live · 5 Days Before Going Live · Listing
   Goes Live · Weekend After Going Live.

**Dynamic dates (mig 0258 + `lib/tc/date-engine.ts`):** a task/milestone stores a RELATIVE rule; the due date
**computes from the live key date and cascades when it moves.** Rule = anchor (key date) or task→task chain ·
signed offset · `day_unit` · `business_day_def` (TRID precise/general) · weekend/holiday roll · holiday calendar
(US_FEDERAL) · conditional "If Applicable" · `task_type`. Proven: 13/13 unit tests.

> **Status = headline · Milestones = the key dates · Tasks = the auto-scheduled checklist.** Enter 4 dates → the
> plan schedules; move one → it all re-shifts.

**Task counts (A-Frame parity):** 58 buyer + 81 listing = **139**. Listing's 81 = 57 contract-to-close + 24 pre-market.

## Architecture (locked)

1. **API-first.** Every loop action is a REST endpoint; UI + Myli call the same one.
2. **Role-gated `/tc` surface + an `Agent ⇄ TC` toggle** (TC/Admin only). The `/tc/transactions/[id]` loop
   route resolves the loop's lead and opens the real lead-detail page in TC `view` mode (`?view=tc`).
3. **Reuse, NEVER recreate** — `StagesPage`, `PipelineSwitcher`, `HlDataTable`, `HlFormModal`, `HlFilterPills`,
   `SectionBand`, and the lead-detail **`LeadLane` / `CenterLane` (the iPad) / `MyliLane`** + comms streams.
   Enforced by `.claude/rules/REUSE_NOT_RECREATE.md` + the `hl/no-duplicate-shell-components` ESLint rule.
4. **Side-aware** (TC Buyer vs TC Listing).
5. **E-signature = Documenso, self-hosted** (own data, $0/sig, cert-sealed, ESIGN/UETA). dotloop NOT in the path.
6. **We own documents end-to-end** → push to SkySlope for compliance.
