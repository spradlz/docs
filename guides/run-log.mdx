The **Run Log** (CRM вҶ’ **Run Log**, API `GET /api/run-log`) is HouseLoop's single
observability timeline. Every meaningful event вҖ” an inbound email, a home viewed on the
website, a Repliers listing update, an AI search, a nightly cron вҖ” lands here in real time,
and **each row carries one explicit `category` flag** that decides which tab it belongs to.

## How it works

- **One source of truth.** The category is computed **once, server-side**, from the event's
  origin (its kind + source + which Inngest function produced it) and stored on the
  row. The UI and the tabs only ever read that column вҖ” they never re-derive it.
- **Auto-categorized pills.** Every Inngest function maps to a **short type pill** (`Intake`,
  `Value`, `Sold`, `SEO`, `Photos`, `Vision`, `Alert`, `Workflow`, `System`, вҖҰ) via a
  deterministic rule on its function id вҖ” a new function self-types, and an unmatched one falls
  to `System` (never blank). Each pill deep-links to that function's run in the Inngest dashboard.
- **Real-time, no polling.** New events stream in over Server-Sent Events. A Postgres
  `LISTEN/NOTIFY` bus fires on insert, so the feed and the lead page update within ~1вҖ“2s.
- **Cross-app bridges.** `houseloop-web` (the public site, separate database) emits Inngest
  events the CRM records: `houseloop/ai-search.event` вҶ’ AI Search lane, `houseloop/idx.event`
  вҶ’ IDX Feed lane; all other functions are auto-captured via `function.finished`/`failed`.

## IDX Feed вҖ” the listing lifecycle

The IDX Feed is the **event lens** on every home: one row per Repliers webhook, with **three
facets** (the question each answers in brackets):

- **Type** *(what happened вҖ” the verb)*: `Intake` (first time we see the listing) В· `Status
  Change` (it moved in the funnel) В· `Price Change` (list price moved, status unchanged) В·
  `Disposition` (it sold) В· `Canceled` (it left the market without selling) В· `Sync` *(a no-op
  re-fire вҖ” same status + price as last time; rendered muted so the feed stays readable)*.
- **Listing Status** *(the state now вҖ” the noun)*: `Coming Soon` В· `Active` В· `Pending` В·
  `Sold` В· `Off-Market`.
- **Status** *(did our run succeed)*: `Completed` / `Failed`.

So a home reads as a story over its rows: `Intake В· Coming Soon` вҶ’ `Status Change В· Active` вҶ’
`Price Change В· Active` вҶ’ `Disposition В· Sold`. The **Type is derived server-side** from the
log itself (the row's status + price vs. the prior row for the same MLS) вҖ” no upstream stamp.

Each IDX event is *processed by* an Inngest workflow that shows in the **Inngest** tab: an
`Intake` is handled by `catalog-listing`, a `Disposition` by `listing-sold`. Same moment, two
lenses вҖ” the IDX Feed shows what happened to the home, the Inngest tab shows what the engine did.

## Inngest workflows and steps

The **Inngest** tab (and the **Home Value Update** tab) shows every Inngest function run. A
**multi-step workflow** (e.g. `catalog-listing`, which resolves вҶ’ enriches вҶ’ caches photos вҶ’ AI-tags a new listing)
renders as **one thick row with its steps stacked**, each step `вң“` (ran) or `вң—` (failed), and a
deep-link to that single run. A genuinely single-action job stays a one-line row. So you see a
workflow *and* which of its steps succeeded, never a flat atomic line per sub-task.

**How the steps get there (not hardcoded).** A function records each step's real outcome into a `steps` object as it
runs вҖ” `steps[name] = { status: 'ok' | 'failed', description }` вҖ” and **returns** it. The capture-fn
(`apps/houseloop-crm/lib/run-log/capture-fn.ts`) stores that return as `hl_run_log.payload.steps` (kept only if the
compacted output is under ~2 KB), and the feed projects `payload->'steps'` onto the row. A couple of lifecycle
functions (`catalog-listing`, `listing-sold`) additionally get a curated trail prefix in `feed.ts`; everything else вҖ”
including all four home-value engines вҖ” renders straight from its own `payload.steps`. The bullets are what the run
**actually did**, with live counts.

## What the Run Log does NOT log

A few utility functions fire **hundreds-of-thousands вҶ’ millions** of times and carry **zero
audit value** вҖ” the real artifact lives elsewhere (photos in Cloudflare R2 + `Listing.photosCached`,
data on the target row), and the per-listing intake workflow already shows them as a *step*.
Their **success** rows are deliberately skipped by `capture-fn` (failures still log вҖ” we always
want those). The skip set: `photos-cache`, `data-coverage-reconcile`, `vision-tag-listing`,
`listing-catalog-cache-one`, `sold-photo-rescue`.

<Warning>
This skip exists for a reason: `photos-cache` had logged **2.8M rows = 96% of the Run Log
table**, filling the CRM Postgres volume and 500ing the feed (2026-06-19). Per-item cache /
backfill **success** must never be logged to `hl_run_log`. When a new high-frequency backfill
job appears, add it to the skip set.
</Warning>

### Example

```bash
# Inbound + outbound communications only
curl 'https://crm.houseloop.ai/api/run-log?view=comms' -H 'Cookie: <crm session>'

# Website activity from real humans (crawlers excluded)
curl 'https://crm.houseloop.ai/api/run-log?view=website&audience=human' -H 'Cookie: <crm session>'
```

Each row returns its `category`, `status`, `label`, `preview`, `ts`, and (for leads) `lead_id`.
See the [Run Log API reference](/api-reference) for the full schema.

## Communication rows deep-link to the lead

Every **Communication** row is a link into that contact's Lead Detail page, opened on the
matching channel tab:

| Channel | Opens | Lands on |
|---|---|---|
| Email | `вҖҰ/leads/<id>?tab=Emails&focus=<sent\|recv>:<id>` | the Emails timeline, scrolled to + briefly highlighting that exact email |
| iMessage / SMS | `вҖҰ/leads/<id>?tab=Texts` | the message thread |
| Call | `вҖҰ/leads/<id>?tab=Calls` | the calls panel |
| Notification | `вҖҰ/leads/<id>?tab=All` | the full activity timeline |

Only rows tied to a CRM lead are clickable (the feed already excludes communications with no
matching lead). The `focus` token maps the Run Log's email id (`esent:`/`erecv:`) to the
thread row id (`sent:`/`recv:`) so the lead page can scroll straight to the right message.

<Note>
The Run Log is CRM-session gated (it returns `401` without a valid session). System,
website, IDX, and background-job rows are read-only вҖ” only communication rows are clickable
(they navigate to the lead). There are no bulk operations.
</Note>
