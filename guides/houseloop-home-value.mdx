## What this is

Every home on houseloop.ai has a **HouseLoop Value** and a **HouseLoop Home Value History** — our equivalent of
Zillow's Zestimate + "Zestimate history." The value is a single current number; the history is a **monthly series**
of that value going back as far as the data allows, rendered on the home page as a **graph** or a **table**.

The current value is not one algorithm — it's a **fork**. Which branch a home takes depends on what data it has,
and two different monthly jobs feed the two branches. Getting this right is the difference between a dead-on
Zillow-grade number and a rough index estimate.

## HouseLoop Value — the fork (five buckets, locked 2026-06-17, expanded 2026-07-02)

Computed at read time in `lib/listings/get.ts`. The core fork is chosen by **data presence, not a per-home flag**
(cached Zestimate → dead-on tier; none → free ZHVI tier). The 2026-07-02 past-client resolver formalized the fork
into **five value buckets**. The critical thing to understand: **every bucket is refreshed monthly on the table —
they differ only in the monthly ENGINE.** "Native" means *native value source*, not *no monthly update*.

| Bucket | When | Value | Monthly engine |
|---|---|---|---|
| **`ninja_monthly`** | sold/off-market **with** a cached Zestimate (most past clients, ~1,840) | `zestimate × 1.005` — `zestimateHouseLoopValue` | re-fetch a fresh Zestimate from OpenWeb Ninja (26th job) |
| **`zhvi_monthly`** | sold, **no** Zestimate (Zillow has the home but no Zestimate, ~90) | last **sold price × ZHVI area appreciation** — `soldValueFromAppreciation` | recompute from the fresh ZHVI index (24th job) |
| **`active`** | currently **for-sale / pending / coming-soon** | `listPrice × 1.005 ± 3%` — `activeValueFromList` | list price re-read; **flips to `ninja_monthly` on close** |
| **`native`** | an **organically-created HouseLoop card** built from our own past-client sold data, not on Zillow (e.g. `12111 W 165th Ter`) | last value/sold × ZHVI appreciation | ZHVI recompute (no Zestimate to re-fetch) |
| **`self_sourced_card`** | **new-construction** not on Zillow individually (e.g. `1809 SW Hightown Dr`, Stoney Creek) | our MLS sold data × ZHVI appreciation | ZHVI recompute |
| **`self_sourced_card_land`** | **raw land** (e.g. `00000 Old Kansas City Rd`, 11.72 ac) | the **sale price** (ZHVI is a home index, doesn't apply to land) | held at sale price; card photo = our own screenshot |

<Info>**The switch is the cached Zestimate's presence.** Past-client homes get a real Zillow Zestimate pulled onto
their `Listing` each month, so they resolve to the dead-on `ninja_monthly` tier. Everything without a cached
Zestimate falls to a ZHVI recompute. No per-home boolean — the data decides. The `native` / `self_sourced_card*`
buckets exist for homes Zillow simply doesn't carry (organic cards, new-construction, land); they still get a fresh
monthly value, driven by ZHVI instead of a Zestimate re-fetch. The bucket for every past-client home is recorded in
`.claude/reports/pastclient-canonical-FINAL-2026-07-02.csv`.</Info>

### Out-of-area homes

A home that **closed with SRG but sits outside our display MLS** (e.g. Richland, MO — a different board) still gets
a **past-client card + ZHVI value** — the relationship is the asset. But it does **not** render as a live IDX
listing, because we lack display authority from that source MLS. The empty IDX feed for such an address is the
*signal* it's out-of-area, not a data error.

### The value fields on `Listing` (centerbeam DB)

| Field | Meaning |
|---|---|
| `zestimate` (+`zestimateLow/High`) | raw Zillow Zestimate (via **OpenWeb Ninja**), cached only on homes that get the Ninja refresh |
| `hlEstimate` (+`hlEstimateLow/High`) | **the HouseLoop Value** = `zestimate × 1.005` (or list/ZHVI per the fork), persisted for seated homes |
| `hlEstimateAt` · `hlEstimateSource` | when it was computed · which branch produced it |

⚠️ **A stale `zestimate` = a stale value.** If a home's `zestimate` still holds an **old sale price** instead of a
fresh Ninja Zestimate, `× 1.005` just inflates the wrong number (e.g. a $215k old sale shows $216,075 even though
Zillow says $333k). The fix is a **Ninja refresh** for that home (the 26th job below) — not a code change.

## The monthly refresh chain — three jobs, in order

Three registered Inngest functions run on a staggered monthly cron. Each feeds a different branch of the fork:

| Day (UTC) | Function | id | Feeds |
|---|---|---|---|
| **24th** 09:00 | Zillow Research — monthly **ZHVI** ingest | `zillow-research-ingest` | the **MASS / ZHVI** branch — downloads Zillow's free ZHVI CSVs (county/city/zip/metro) → `hl_market_index`. Powers value + history for **all ~354k homes** |
| **26th** 10:00 | Past-Client Zillow Cache — dead-on Zestimate | `past-client-zillow-cache` | the **PAST-CLIENT** branch — fans out one **OpenWeb Ninja** call per past-client home → caches the real **Zestimate** (+ tax/price history) onto the `Listing`. Worklist = `hl_past_client_homes`, so new past clients auto-enroll |
| **27th** | Value-History Persist | `value-history-persist` | applies **× 1.005**, writes the current month's value + back-seeds the full monthly series into `hl_value_history` (target: `hl_past_client_homes`) |

The order matters: ZHVI (24th) lands first so the mass tier is fresh, then the Ninja Zestimates (26th) refresh the
past-client tier, then the value/history persist (27th) reads the now-current numbers and writes the monthly point.

<Info>Each is a real registered Inngest function — trigger manually via the dashboard **Invoke** button, never a
one-off script (see [Listing intake pipeline](/guides/listing-intake-pipeline)). `value-history-persist` also takes
the event `value-history/persist.run` (`{ limit?, hlids?[] }`) for targeted back-seeds/re-runs. Files:
`apps/houseloop-web/lib/inngest/functions/{zillow-research-ingest,past-client-zillow-cache,value-history-persist}.ts`.</Info>

## HouseLoop Home Value History (the monthly series)

### Where it lives — `hl_value_history`

A dedicated table in the website DB, **one row per HLID per month**, append-only:

| Column | Meaning |
|---|---|
| `hlid` | the home's HLID (permanent id) |
| `month` | first-of-month bucket (e.g. `2012-01-01`) |
| `value` (+`value_low`/`value_high`) | the HouseLoop value that month |
| `source` | `backseed` (historical) · `current` (the latest month) |
| `created_at` · `updated_at` | audit |

Primary key `(hlid, month)` — every monthly write is idempotent (re-runs never duplicate a month). A real
relational table, not a JSON blob, so the series is queryable, immutable per point, and cheap to append at scale.

### How it's built — the universal history builder

**One algorithm** (`flooredHistory` in `get.ts`, shared with the live graph and the persist job) builds the line
for **every** tier: start at the home's **oldest real sale** (which can predate 2012), otherwise floor at **2012**
using the **ZHVI** (`hl_market_index`), then climb monthly to today's HouseLoop value. The percentage shown spans
the whole line ("+X% since {year}"). The back-seed is **free** — pure index math, no Zillow API call.

### How it renders

`getListing(hlid)` reads `hl_value_history` as the source once a home is seeded (falling back to the computed line
for any home not yet seeded), and exposes it as `houseLoopValue.history`. On the home page the **HouseLoop Value**
hero shows the value range plus a **HouseLoop Home Value History** section with a **Graph ↔ Table** toggle:

- **Graph** — an interactive line + scrubber.
- **Table** — the Zillow-style monthly table: Date | HouseLoop Value, newest first, with "+X% since {year}".

## Troubleshooting a wrong value

| Symptom | Likely cause | Fix |
|---|---|---|
| Value = an **old sale price** (× 1.005) | `Listing.zestimate` holds a stale sale price, not a fresh Ninja Zestimate | re-run **`past-client-zillow-cache`** for that hlid (26th job) to pull the current Zestimate |
| Value looks **index-y / round** on a past client | home isn't in `hl_past_client_homes`, so it fell to the **ZHVI** tier | add it to the past-client set; the 26th job then enrolls it |
| `hlEstimate` **blank** | never seeded by `value-history-persist` | run `value-history/persist.run` with `{ hlids: ["…"] }` |
| **Two HLIDs for one home** | the address `identity_key` differs (e.g. a dropped `NE` directional) so the resolver treated it as two homes | identity/de-fork merge (see [Property data & address identity](/guides/property-data-and-address-identity)) |
