## What this is

Every home on houseloop.ai has a **HouseLoop Value** and a **HouseLoop Home Value History** — our equivalent of
Zillow's Zestimate + "Zestimate history." The value is a single current number; the history is a **monthly series**
of that value going back as far as the data allows, rendered on the home page as a **graph** or a **table**.

The current value is not one algorithm — it's a **fork**. Which branch a home takes depends on what data it has,
and two different monthly jobs feed the two branches. Getting this right is the difference between a dead-on
Zillow-grade number and a rough index estimate.

## HouseLoop Value — TWO monthly engines + one live state (Zach LOCKED 2026-07-03)

Computed at read time in `lib/listings/get.ts`, chosen by **data presence, not a per-home flag**. The old "5–6 bucket"
framing collapses to exactly this — **every home's value is one of these:**

| # | Engine | When | Value | Monthly engine |
|---|---|---|---|---|
| **①** | **Past-Client Home Value · Ninja Monthly** | `sold` **AND** past-client (has a cached Zestimate) | `Zestimate × 1.005` — `zestimateHouseLoopValue` | re-fetch the Zestimate from OpenWeb Ninja (26th) → persist (27th). **Zestimate is used ONLY here.** |
| **②** | **Sold Home Value · ZHVI Monthly** | `sold` **AND** (not a past client **OR** no Zestimate) | `sold price × ZHVI[now]/ZHVI[soldMonth]` — `soldValueFromAppreciation` | recompute from the fresh ZHVI index (24th) → persist (27th) |
| — | **Active** (live, no cron) | `active` / `coming_soon` / `pending` | `list price × 1.005` — `activeValueFromList` | live at render; **on close flips to ① if past-client, else ②** |

**Engine ② is the default — ALL property types fold in: single-family, condo, apartment, AND land.** There is **no
separate land / native / self-sourced bucket.**
- **Land is NOT held flat.** Land appreciates on the same zip → city → county → metro ZHVI ladder as homes. ZHVI has
  no land-specific series, so **land stays lockstep with single-family** for its area — if the zip appreciates, the
  land appreciates the same.
- **Self-sourced / new-construction not on Zillow:** Zillow carries no Zestimate for it → it runs **②** on its sold
  anchor price. We **cannot** get a Zestimate for a self-sourced home.
- **The only "held flat" case** is a home with **no anchor date at all** (an organic new-build minted from scratch,
  e.g. `12111 W 165th Ter`). The moment it has any sale/list date, Engine ② takes over. Held-flat is the *anchorless*
  exception, never a property-type rule.

<Warning>

**The fork ORDER is the contract (Zach LOCKED 2026-07-03).** The branches are evaluated in this exact
priority, and the order matters:
1. **active/pending/coming-soon** → `list × 1.005`
2. **Zestimate present** → `zestimate × 1.005` (ninja_monthly — dead-on)
3. **sold + `closePrice` + `soldDate`** → `soldValueFromAppreciation` = `closePrice × ZHVI[now]/ZHVI[soldMonth]`
   (zhvi_monthly / native / self_sourced — the appreciation branch)
4. **stored `hlEstimate`** → LAST-RESORT fallback ONLY (raw land held at sale price, or a home missing a `soldDate`).
Branch 3 (appreciation) **must** run before branch 4 (stored value). Previously the stored-`hlEstimate` branch was
evaluated first and short-circuited the appreciation — so every no-Zestimate sold home rendered **flat at its raw
close price** (e.g. 3025 NW 95th showed $596,813 forever; 10908 E 35th sold $135k in 2021 still showed $135,675).
The formula was correct but unreachable. Verified by `tools/houseloop-cards/audit-value-buckets.mjs` (independent
ZHVI recompute vs the live render; must report `FLAT_BUG = 0`).

</Warning>

<Info>**The switch is the cached Zestimate's presence.** Past-client homes get a real Zillow Zestimate pulled onto
their `Listing` each month → Engine **①**. Everything else — every non-past-client sold home, plus any home Zillow
doesn't carry a Zestimate for (new-construction, land, off-Zillow organic cards) — runs Engine **②** (ZHVI on the sold
price). No per-home flag and no `native`/`self_sourced`/`land` sub-buckets: **the data decides, and ② handles every
property type the same way.** Canonical spec: `GT/GT - ACTIVE/GT_HOUSELOOP_HOME_VALUE_ENGINE.md`.</Info>

### Out-of-area homes

A home that **closed with SRG but sits outside our display MLS** (e.g. Richland, MO — a different board) still gets
a **past-client card + ZHVI value** — the relationship is the asset. But it does **not** render as a live IDX
listing, because we lack display authority from that source MLS. The empty IDX feed for such an address is the
*signal* it's out-of-area, not a data error.

### The value fields on `Listing` (centerbeam DB)

| Field | Meaning |
|---|---|
| `zestimate` (+`zestimateLow/High`) | raw Zillow Zestimate (via **OpenWeb Ninja**), cached only on homes that get the Ninja refresh |
| `redfinEstimate` (+`redfinEstimateLow/High` · `redfinEstimateUrl` · `redfinEstimateAt`) | raw **Redfin Estimate**, scraped by ADDRESS off the Redfin property page (past-client homes only — see the Portal-value spread below) |
| `hlEstimate` (+`hlEstimateLow/High`) | **the HouseLoop Value** = `zestimate × 1.005` (or list/ZHVI per the fork), persisted for seated homes |
| `hlEstimateAt` · `hlEstimateSource` | when it was computed · which branch produced it |

⚠️ **A stale `zestimate` = a stale value.** If a home's `zestimate` still holds an **old sale price** instead of a
fresh Ninja Zestimate, `× 1.005` just inflates the wrong number (e.g. a $215k old sale shows $216,075 even though
Zillow says $333k). The fix is a **Ninja refresh** for that home (the 26th job below) — not a code change.

## The refresh jobs — three monthly + one daily backstop

Four registered Inngest functions keep the value fresh. Three run on a staggered **monthly** cron (each feeds a
different branch of the fork), and one runs **daily** to reconcile close prices and backstop any sold home that never
got dispositioned:

| Day (UTC) | Function | id | Feeds |
|---|---|---|---|
| **24th** 09:00 | Zillow Research — monthly **ZHVI** ingest | `zillow-research-ingest` | the **MASS / ZHVI** branch — downloads Zillow's free ZHVI CSVs (county/city/zip/metro) → `hl_market_index`. Powers value + history for **all ~354k homes** |
| **26th** 10:00 | Past-Client Zillow Cache — dead-on Zestimate | `past-client-zillow-cache` | the **PAST-CLIENT** branch — fans out one **OpenWeb Ninja** call per past-client home → caches the real **Zestimate** (+ tax/price history) onto the `Listing`. Worklist = `hl_past_client_homes`, so new past clients auto-enroll |
| **27th** | Value-History Persist | `value-history-persist` | calls `getListing(hlid)` (so it reuses the ONE value fork above — no second formula), writes the current month's value + back-seeds the full monthly series into `hl_value_history`, **and refreshes the stored `Listing.hlEstimate` column** so the CRM seller-report/exports that read it directly stay current (target: `hl_past_client_homes`) |
| **Daily** 08:00 | Sold data reconcile + **disposition backstop** | `sold-data-reconcile` | fills any missing `closePrice` from Repliers, then re-emits `listing.sold` for **genuinely-flat** sold homes so nothing stays flat (see [Disposition](#disposition--no-sold-home-renders-flat-hardened-2026-07-03)) |

The monthly order matters: ZHVI (24th) lands first so the mass tier is fresh, then the Ninja Zestimates (26th) refresh
the past-client tier, then the value/history persist (27th) reads the now-current numbers and writes the monthly point.

<Info>Each is a real registered Inngest function — trigger manually via the dashboard **Invoke** button, never a
one-off script (see [Listing intake pipeline](/guides/listing-intake-pipeline)). `value-history-persist` also takes
the event `value-history/persist.run` (`{ limit?, hlids?[] }`) for targeted back-seeds/re-runs. Files:
`apps/houseloop-web/lib/inngest/functions/{zillow-research-ingest,past-client-zillow-cache,value-history-persist,sold-data-reconcile}.ts`.</Info>

## Disposition — no sold home renders flat (hardened 2026-07-03)

A sold home only gets its value **minted** (and its AI overview written) when the `listing-sold` Inngest function
runs. Two gaps used to leave homes **flat** (no value / raw close price):

**1. The webhook trigger only fired on a status TRANSITION.** `apps/houseloop-web/app/api/repliers/webhook/route.ts`
required `previous.lastStatus` to be present before sending `houseloop/listing.sold`. A metadata **re-sync** of an
already-sold record carries `previous.lastStatus = null`, so if the webhook missed the *original* `Sld` transition the
sold chain never ran and the home stayed flat forever (the "1011 Fontana" bug). **Fix:** fire `listing.sold` whenever
the record is `Sld` and the row is **not already archived** — the `listing-sold` idempotency-check
(`status='sold' AND soldDate AND enrichmentSource='repliers_webhook'`) makes an already-archived re-emit a cheap
no-op, so re-firing is safe. Only genuinely-archived rows keep the `sold_resync_skipped` label.

**2. No backstop for the historical backlog.** `sold-data-reconcile` (daily) now has an `emit-missing-disposition`
step that finds the **genuinely-flat** sold homes and re-emits `listing.sold` for each. "Flat" is precise — a home
renders a value from EITHER a stored `hlEstimate` OR a `closePrice` (× ZHVI appreciation, computed read-time in
`get.ts`), so it is only flat when it has **neither**:

```sql
status = 'sold' AND "hlEstimate" IS NULL AND "closePrice" IS NULL
```

<Warning>"Flat" is **NOT** "no `hlEstimate`". ~346k sold homes carry no stored estimate on purpose and render fine
from their close price × ZHVI. Scoping the backstop to `hlEstimate IS NULL` alone matched **345,882** homes (the whole
catalog); the real genuinely-flat set is **~29**. Step 1 of the same job fills `closePrice` from Repliers first, so
only the truly-broken tail remains. Emitting is idempotent — dispositioned homes no-op via `listing-sold`.</Warning>

## The Home Value Update Run Log lane

All four engines are a first-class **Home Value Update** tab in the CRM Run Log (`/run-log?view=value`). The three
monthly engines render as ONE grouped pipeline card (`group:'home-value-update'`, 24th→26th→27th); the daily
reconcile + backstop is its own row. Registered in `apps/houseloop-crm/lib/run-log/inngest-registry.ts`
(`category:'value'`, `valueFnIds()`), surfaced via `feed.ts` (the `value_feed` classification + the `SYSTEM_SELECT`
union for the `value` view) and `RunLogTable.tsx` (the tab).

**Atomic step audit trail.** Each engine records every step's **real** status + description into its return `steps`
object — exactly the mechanism `catalog-listing` intake uses. The capture-fn
(`apps/houseloop-crm/lib/run-log/capture-fn.ts`) stores it as `hl_run_log.payload.steps`, and the Run Log renders the
green ✓ / red ✕ step bullets. Nothing is hardcoded — the bullets are what the run **actually did**, with live counts
(e.g. `value-history-persist`: ensure-table → collect-hlids → current-month → persist-value-history "3/3 homes, 524
points").

## HouseLoop Home Value History (the monthly series)

### Where it lives — `hl_value_history`

A dedicated table in the website DB, **one row per HLID per month**, append-only:

| Column | Meaning |
|---|---|
| `hlid` | the home's HLID (permanent id) |
| `month` | first-of-month bucket (e.g. `2012-01-01`) |
| `value` (+`value_low`/`value_high`) | the HouseLoop value that month |
| `source` | `backseed` (historical) · `current` (the latest month) |
| `created_at` · `updated_at` | audit |

Primary key `(hlid, month)` — every monthly write is idempotent (re-runs never duplicate a month). A real
relational table, not a JSON blob, so the series is queryable, immutable per point, and cheap to append at scale.

### How it's built — the universal history builder

**One algorithm** (`flooredHistory` in `get.ts`, shared with the live graph and the persist job) builds the line
for **every** tier: start at the home's **oldest real sale** (which can predate 2012), otherwise floor at **2012**
using the **ZHVI** (`hl_market_index`), then climb monthly to today's HouseLoop value. The percentage shown spans
the whole line ("+X% since {year}"). The back-seed is **free** — pure index math, no Zillow API call.

### How it renders

`getListing(hlid)` reads `hl_value_history` as the source once a home is seeded (falling back to the computed line
for any home not yet seeded), and exposes it as `houseLoopValue.history`. On the home page the **HouseLoop Value**
hero shows the value range plus a **HouseLoop Home Value History** section with a **Graph ↔ Table** toggle:

- **Graph** — an interactive line + scrubber.
- **Table** — the Zillow-style monthly table: Date | HouseLoop Value, newest first, with "+X% since {year}".

## Troubleshooting a wrong value

| Symptom | Likely cause | Fix |
|---|---|---|
| A **no-Zestimate** sold home shows a **flat value = its raw close price** (never appreciated) | the value fork evaluated the stored-`hlEstimate` branch before the appreciation branch (the pre-2026-07-03 bug) | fixed in `get.ts` (fork order); re-persist via `value-history/persist.run { hlids }`. Verify with `audit-value-buckets.mjs` → `FLAT_BUG = 0` |
| Value = an **old sale price** (× 1.005) | `Listing.zestimate` holds a stale sale price, not a fresh Ninja Zestimate | re-run **`past-client-zillow-cache`** for that hlid (26th job) to pull the current Zestimate |
| Value looks **index-y / round** on a past client | home isn't in `hl_past_client_homes`, so it fell to the **ZHVI** tier | add it to the past-client set; the 26th job then enrolls it |
| `hlEstimate` **blank** | never seeded by `value-history-persist` | run `value-history/persist.run` with `{ hlids: ["…"] }` |
| **Two HLIDs for one home** | the address `identity_key` differs (e.g. a dropped `NE` directional) so the resolver treated it as two homes | identity/de-fork merge (see [Property data & address identity](/guides/property-data-and-address-identity)) |

## Field provenance — the `*_source` columns + the Repliers ↔ Ninja fallback split

Every enriched field carries a `*_source` tag so we always know where a value came from:

- `value_source` = `zestimate | zhvi | algo` · `price_history_source` · `schools_source` · `tax_source`.
- Distinguish coverage with e.g. `SELECT source, count(*) FROM hl_value_history GROUP BY source` (⚠ there is NO `hl_home_value` table — the persisted series lives in `hl_value_history` + the ZHVI anchor in `hl_market_index`; fixed per the Fable 5 audit 2026-07-06).

**Repliers gives the CURRENT facts; Zillow / OpenWeb Ninja gives the RICH HISTORY + ratings.** The per-field fallback order:

- base fields (address / beds / baths / current price) + **current tax (single)** = **Repliers**.
- **tax HISTORY (multi-year)** = **OpenWeb Ninja / Zillow** → fallback Repliers current tax.
- **GreatSchools** = **OpenWeb Ninja / Zillow** ratings → fallback Repliers schools (only if the listing agent provided them).
- **price / listing history** = **default OpenWeb Ninja / Zillow** full history → fallback Repliers (partial).
- **home value** = OpenWeb Ninja Zestimate × 1.005 → fallback ZHVI → fallback the monthly algo.

The `*_source` column records which one ACTUALLY filled each field.
