## Myli's dynamic read — ONE workflow, not one per number

It is **one `(FUB · Phone Call)` workflow.** On `callsCreated` (or inbound text), Myli:

1. **Classifies human vs automated** — skips BrokerBay short-code `78340`, spam, robocalls. Only real humans get a text.
2. **Reads the dialed `To` line → its source category** (company · sign-box · Ana-cell · past-agent) → picks the copy.
3. **Sends the text-back.** Today: **from Myli's own BlueBubbles number 913-208-1633**, as herself — *"Hey, this
   is Myli — just saw a call come into our company line, wanted to reach out personally. What are you looking
   for?"* Past-agent → names the agent; Ana-cell → past-client voice. **After the port (Section 2), the reply comes
   AS the dialed line.**

## Inbound capture — every call & text lands on the lead

<Check>
**Live (shipped 2026-06-22).** Inbound calls, texts, and emails store to the lead record — **including callers
who aren't a lead yet.**
</Check>

The root cause we fixed: FUB delivers unknown callers with **`personId = 0`**, and the handler did
`if (!personId) return` — and `!0` is `true`, so **every missed call, voicemail, and first-touch text from an
unknown number was silently dropped.** Now `handleCall` / `handleTextMessage` / `handleEmail`
(`apps/dashboard/server/fub-webhooks.ts`) store `person_id = 0` as an *unmatched* record keyed by phone number.
The full inbox — missed calls and voicemails included — is captured and converts to a lead when identified.

  and it's *cleaner* than the FUB forward because the email preserves the native line.

# ═══ SECTION 2 — THE PORT-OUT PLAYBOOK: OWN EVERY NUMBER IN TWILIO ═══

**The God-tier endgame:** every business line lives in **our own Twilio**, not FUB or Google Voice. Then inbound
SMS/voice hits a **Twilio webhook → HouseLoop native → the Myli workflow**, and outbound sends **AS the exact
line** the person dialed. No middleman, no forwarding shell game, no lost source. We already own **two Twilio
accounts** (HouseLoop + SRG, `support@spradlingrealtygroup.com`, vault / Master Sheet row 78).

## Port a Google Voice number → Twilio (the 913-954-7123 recipe)

1. **Unlock it from Google.** `voice.google.com/about/port` → pay Google's **$3 port-out unlock fee** → Google
   returns the **port-out info: account number + PIN**.
2. **Submit the port-in to Twilio.** Twilio Console → Phone Numbers → **Porting & Hosting → Port a number in.**
   Provide: the number, a signed **LOA** (Letter of Authorization), the GV **account# + PIN**, a proof doc.
   Twilio runs the carrier port — **~2–4 weeks.**
3. **Land it.** Point the number's **Messaging + Voice webhooks at HouseLoop** → native inbound; send AS it.

<Warning>
**The Google Voice gotcha:** GV numbers *sometimes reject a direct port to Twilio* (Google is treated as VoIP;
the LOA/account match fails). Battle-tested fallback = a **two-step port**: GV → a cheap **prepaid mobile SIM**
(Tello / US Mobile, ~$10) first, let it settle a few days, **then** port that SIM → Twilio. Slower, near-100%.
Try direct first; fall back to two-step if Twilio rejects.
</Warning>

## Port the ~35 FUB numbers → Twilio

Same process, sourced from FUB: **request a port-out packet from Follow Up Boss support** (account# + PIN + LOA
per number) → Twilio port-in.

<Warning>
**Porting REMOVES the number from FUB** — Twilio/HouseLoop becomes the receiver. Each port is a real **cutover**,
not a copy. FUB's API does not expose number ownership; confirm each line in **FUB Admin → Phone Numbers** first.
</Warning>

## A2P 10DLC — required before outbound texting at scale

US carriers require **A2P 10DLC brand + campaign registration** to send application SMS from a 10-digit number.
Before Myli texts from ported lines at volume, register the brand + campaign in Twilio (we have the Twilio A2P
tools: `twilio_a2p_brand_status`, `twilio_a2p_campaign_register`, …). Skipping this = carrier filtering / blocks.

```

```
