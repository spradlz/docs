This is the canonical developer reference for the **Spradling Realty Group past-client data project** — one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard, every past-client home's `/home` page, **the Follow Up Boss past-client dossier**, and the **CRM Audit Report** (`Seller Report → Audit Report`). The internal strategy lives in `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` §12–§13 and the SSOT-consolidation plan in `GT_HOUSELOOP_PASTCLIENT_TRANSACTIONS_SSOT_PRD.md`.

## 🏛️ THE DATA MODEL — two tables the app reads (Zach LOCKED 2026-06-26)

> **One source of truth, two layers: the PERSON (parent) and the HOUSE (child). Nothing else.**

| Layer | Table | Grain | Holds |
|---|---|---|---|
| **PARENT — person** | `hl_srg_past_clients` | 1 / past client (~1,834) | the PERSON: name · contact · `bucket` · **`resolver_block` (the dossier)** · `resolution_status` · `website_hlid` · resolved current home. The resolver WRITES the dossier here. |
| **CHILD — house (THE SSOT)** | `hl_srg_past_client_homes` | **1 / house (~2,259)** | the ONE reconciled house record — **all facts in one flat row:** MLS facts + SISU facts + dotloop facts (loop + **contract licensee** + parties) + value/photos. The resolver READS its facts. |

**Grain (Zach 2026-06-26):** **one row per house.** A past client with multiple homes has multiple child rows. The *same* house can have multiple rows across its life (sold more than once over the years) — that's fine; each is its own transaction. **Each resolved home maps to exactly ONE HouseLoop website URL** — one `canonicalSlug` on the website `Listing`, generated/linked by the resolver. One house = one `/home/<canonicalSlug>`, always.

**Raw INPUT tables — they FEED the child; the app NEVER reads them directly:**
- `sisu_transactions` — raw SISU pipeline (agent, GCI, dates, status; 1,945 Closed)
- `hl_srg_mls_deals` — raw MLS download (our_roles, price, close date, list agent) — **retired as an SSOT**, kept only as a raw staging input
- **dotloop API** — loop → participants + the signed **contract** (the final-truth agent)

**The flow (one direction):** raw sources → **reconciled into the child** (one flat row per house) → the **S6 resolver reads the child's facts** → resolves who the house belongs to (current home / bucket / disposition / agent) → **writes the dossier + bucket onto the parent**, and the derived fields onto the child. The seller report, the website, and the Audit Report read **parent JOIN child** — no JSON scratch files, no reading a raw source. (The denormalized `home_*` columns currently on the parent move OUT to the child; see the consolidation PRD.)

### THE ARCHITECTURE LAW (Zach LOCKED 2026-06-23) — the resolver owns the output, the transport is dumb

> **The S6 LLM resolver OWNS the entire output.** Household, **labeled contacts**, **labeled transactions** (BOUGHT vs SOLD), **disposition**, **DNC screen**, **bucket**, **tags**, AND the human **NOTE**.
>
> **The transport (`dossier-to-fub.mjs`) is DUMB** — it writes the resolver's finished block to FUB + the SSOT and NEVER re-derives anything. **If the output is wrong, fix the RESOLVER, re-run, re-transport.**

## 🏷️ THE BUCKET — a resolved output (Zach LOCKED 2026-06-26)

The `bucket` is **derived from who actually repped the deal**, never a pre-baked column or a tag (tags lie — "Agent COI" might be the *agent's* COI, not Ana's). The agent is determined **per deal, by precedence:**

1. **dotloop CONTRACT licensee** (the signed `LICENSEE ASSISTING SELLER/BUYER`) — **the source of final truth**, overrides SISU
2. **SISU `agent`** (Zach Spradling → Ana Spradling) — the working agent; proved equal to the contract's designated agent
3. **MLS `our_roles`** (`list-ana`/`buy-ana` = Ana; `list-group`/`buy-group` = the team brand, NOT proof of a team agent — the Dallas bug)
4. **Zach's manual recon** (his hand-verified ~36 fact-less deals) — fallback when 1–3 are empty

| Bucket | Meaning |
|---|---|
| **Ana Only** | Ana is the agent on **every** deal |
| **Ana Handoff** | Ana touched ≥1 deal **and** a named team agent worked another (Ana's relationship, handed off) |
| **Team** | a named team agent only — **Ana never** on the deal (Earl Wilson = Jessica Rich) |
| **Internal (SG Holdings)** | a Spradling Holdings / "Ana & Zach Spradling" principal flip — **our own investor deal, excluded from reactivation** |
| **Agent DNC** | the client is an SG agent or agent family |

## ⭐ dotloop CONTRACT = the source of final truth for the agent (Zach LOCKED 2026-06-26)

The participant list **lies on departed-agent loops** — when a team agent leaves, dotloop DELETES them, leaving only Ana-of-record. The real agent survives ONLY inside the signed contract PDF (`LICENSEE ASSISTING SELLER/BUYER`). dotloop CS enabled **document download** on our OAuth client (`document:*` scope; Adam removed the app + re-consented to mint a doc-scoped token — a refresh does NOT pick up a newly-enabled scope, you must re-authorize). The chain, **all OAuth API, no Chrome**:

```
loop → GET /folder → find the Exclusive-Right-to-Sell / buyer-rep doc
     → GET /folder/:fid/document/:did  with  Accept: application/pdf   (=> 200, PDF bytes)
     → pdftotext -layout → the name on the line above "LICENSEE ASSISTING SELLER|BUYER"
```

Use `apps/houseloop-crm/lib/dotloop/client.ts → dotloop.downloadDocument()` (handles the Accept header + a transient-500 retry). Helper: `tools/houseloop-cards/dotloop-contract-agent.mjs` → `contractAgent(profileId, loopId)`. **Proven: Earl → Jessica Rich (confirms SISU), Burton → Jeremy David (recovers the departed agent SISU lost).** Full API details: [dotloop Integration](/guides/dotloop-integration). dotloop also confirms the **client + address + parties** (recon truth).

## THE LIVE-DEED DISPOSITION — county deed is primary truth

For every home the client bought, the resolver re-derives the disposition **live** (never a stored flag):

1. **County deed** (`verifyOwner()` in `county-owner-verify.mjs`; Johnson/Wyandotte KS + Jackson/Clay/Cass MO) is **primary truth.** `owner_match` = they own it; `owner_differs` = they sold/moved.
2. **Repliers** gives the clean sold date/price/office; **OpenWeb Ninja** fills the listing/buyer **agent names** Repliers lacks.
3. **DEED-LAG RULE (the Nations case):** a **recent, real** MLS sale (arm's-length: negotiated price and/or a buyer agent) **BEATS** a lagging deed → `sold=true`, `deed_lag=true`.
4. **ARBORIDGE ARTIFACT GUARD:** a "sale" priced **exactly** at list with **no buyer agent** is a non-arm's-length artifact (refi/relo/quitclaim), NOT a sale → the deed wins.
5. For a sold home, fetch the **full MLS listing** (`repliersSoldCheck(mls)`) for the real `raw.CloseDate`.

**⚠️ PARCEL LAW (the Sycamore trap):** Repliers' `addressKey` ignores ZIP, county, and directional → it MERGES different houses. A re-sale counts only if the PARCEL matches: zip + county + neighborhood + legal + coords within ~150m.

## CHAIN OF CUSTODY — the agent on every deal

Every `transactions[]` line carries `agent` = the **Spradling Group agent who repped US on THAT specific deal** (per-deal). Source precedence: dotloop contract licensee → SISU `agent` → closing `agent` → `"SRG"`. It renders as `— <agent>` on every transaction line and as the **Agent (chain of custody)** column in the Audit Report.

## 🚫 AGENT DO-NOT-CONTACT SCREEN (Step 0) — never reactivate our own agents

Before anything else the resolver loads the **agent roster** (distinct names in the SISU `agent` field) and runs `checkAgent()`: **exact full-name match** → **Agent DNC** (auto); **surname-only match** → DNC **only** with a confirming signal (an "Agent COI" note, an agent spouse/co-owner on the deed, or a deal where that agent repped the OTHER side). On DNC: a short suppression note, `tags=["HouseLoop Resolved","SRG Closed - Agent DNC"]`, bucket → `agent-dnc`. Proven: **April Atkinson** + **James Duston** (husband of agent Stephanie Duston).

## The Audit Report (`Seller Report → Audit Report`)

A full-width CRM page (`AuditReport.tsx`, fed by `/api/settings/seller-report/audit` reading `resolver_block` live). Gold layout, sticky jump bar: **All · Transaction History · Resolution Matrix · Follow-Up Boss Notes**. Each client name links to their **Follow Up Boss profile** (`https://spradlinggroup.followupboss.com/2/people/view/<fub_id>`).

## The resolver's output block — `resolver_block` jsonb on `hl_srg_past_clients` (the parent)

`contacts[]` · `transactions[]` (with `agent`) · `disposition` · `dispositions[]` (per-home deed verdict) · `dnc` · `bucket` + `bucket_basis` · `tags[]` · `note_markdown`. `resolution_status` gates the transport: `resolved` → FUB; `needs_review` → held. ⚠️ Write to **`resolver_block`**, never the pre-existing `dossier` TEXT column (Myli's case-file).

### 🔖 FUB TAG TAXONOMY (Zach LOCKED 2026-06-23)

| Tag | On which leads |
|---|---|
| `HouseLoop Resolved` | **every** resolved client |
| `SRG Closed - Ana Only` / `Ana Handoff` / `Team` / `Internal (SG Holdings)` / `Agent DNC` | the `bucket` |
| `SRG Closed - Lost Listing` | a home re-sold by another agent (seller-engagement lead) |

Tags are **ATOMIC** — never bake combinations into one tag; FUB smart lists AND them.

## The one rule that governs everything: resolve first, enrich second

A past-client home is never enriched, slugged, or surfaced until it has been resolved to a client and signed off. Per-home lifecycle: `(1) RECONCILE FACTS → (2) RESOLVE → (3) SIGN-OFF → (4) BACKFILL value/photos → (5) CANONICAL SLUG (one /home URL) → (6) SELLER REPORT`.
