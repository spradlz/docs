This is the canonical developer reference for the **Spradling Realty Group past-client data project** ‚ÄĒ one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard (past-client reactivation) and every past-client home's `/home` page. The internal strategy lives in the private GT doc `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` ¬ß12‚Äď¬ß13; this page is the **end-to-end process a developer needs to operate it.**

## The one rule that governs everything: resolve first, enrich second

**A past-client home is never enriched, slugged, or surfaced until it has been resolved to a client and signed off.** We work the book **in the order the homes sold**, one home at a time. We do **not** run a blind bulk backfill.

**Most of the book is already done.** Of ~2,259 homes, **~93% (‚Čą2,080) are already resolved and fully enriched** ‚ÄĒ they have their Zestimate, GreatSchools, tax, and history, and **we never re-run Ninja on them.** Only the **~175 still-blank homes** remain. Those are blank **by design** ‚ÄĒ they have not been resolved to an owner yet. Each one is backfilled **only after** it is resolved + signed off, one at a time, in sold order. The blanks are not a bug to fix with a bulk run; they are the queue.

### The per-home lifecycle (the gate)

```
(1) RESOLVE        reconcile engine links the home ‚Üí a client, in sold order  (S0‚ÜíS6 below)
        ‚Ēā
(2) SIGN-OFF       Zach confirms the home‚Üíclient link is correct
        ‚Ēā
(3) BACKFILL       NOW run OpenWeb-Ninja / Zillow on that ONE confirmed home:
        ‚Ēā            Zestimate (value anchor) + GreatSchools + tax + price/sale history + photos
        ‚Ēā            ‚Üí written into the MASTER table  hl_srg_past_client_homes
        ‚Ēā            (only for a still-blank home; the ~93% already enriched are skipped)
        ‚Ēā
(4) CANONICAL SLUG seat the home on the website FROM the confirmed master data
        ‚Ēā            ‚Üí one Listing row, one /home/<canonicalSlug> URL (USPS-preferred city)
        ‚Ēā
(5) SELLER REPORT  the CRM card renders off the confirmed master data
```

| Step | What happens | Output |
|---|---|---|
| **1. Resolve** | The reconcile engine (S0‚ÜíS6, below) ties each still-unresolved home to the correct client, in the order the homes sold. SISU paid-truth + MLS sold-truth + FUB evidence triangulate. | a confirmed `home ‚Üí past_client_id` link |
| **2. Sign-off** | Zach reviews the resolved home‚Üíclient and confirms it. **Nothing downstream runs without this.** | go / no-go |
| **3. Backfill** | A **manual Open Web Ninja / Zillow** call on that one newly-confirmed (still-blank) home pulls the **current Zestimate** (the value anchor), GreatSchools, tax, price/sale history, photos. Written into the **master table** `hl_srg_past_client_homes`. The ~93% already enriched are never re-run. | the home's master row, fully enriched |
| **4. Canonical slug** | The website `Listing` is **seated from the confirmed master data** ‚ÄĒ minting the canonical slug + the permanent `/home/<canonicalSlug>` URL (with the USPS-preferred city for the ZIP, never city-free). | one live `/home` page |
| **5. Seller report** | The CRM seller-report card renders the client + their confirmed homes + values. | the reactivation surface |

> **801 W Poplar St** is the worked example: it is one of the ~175 blanks, and it became eligible for backfill **only because** it had already been resolved to **Kyle & Karly Nations** and signed off. Its Zestimate ($569,600 ‚Üí HouseLoop Value ~$572,448) gets run **after** that confirmation ‚ÄĒ never before, and never as part of a bulk sweep.

## The canonical tables

There are three tables in play. The first two are the **CRM** (people + the master home snapshot); the third is the **website**, seated downstream from the master.

| Table | Database | Grain | Role |
|---|---|---|---|
| **`hl_srg_past_clients`** | CRM (`HL_CRM_DATABASE_URL`, Railway *autorack*) | 1 row / past client (~1,834) | the **PERSON**: name, `bucket`, `phone`/`email`, **`fub_id`** (‚Üí FUB), `dossier` (jsonb), `fub_identity` (jsonb), `resolved_current_home`. |
| **`hl_srg_past_client_homes`** | CRM (*autorack*) | 1 row / resolved home (~2,259) | **THE MASTER SNAPSHOT ‚ÄĒ it owns the data.** One row per home a client transacted, linked by `past_client_id`. Carries the confirmed home: `address`, `canonical_slug`, `website_hlid`, `current_zestimate` / `current_hl_value`, `schools`, `tax_history`, `sale_history`, `value_history`, structured facts, `current_home_status`. **It does NOT read from the website** ‚ÄĒ it is built first, from the resolve + Ninja backfill. |
| **`Listing`** | WEB (`HL_WEB_DATABASE_URL`, Railway *centerbeam*) = PRODUCTION | 1 row / house (~354K) | the **website house**: renders `/home/<canonicalSlug>`. For past clients it is **seated FROM the master table** after sign-off. Keyed by `identityKey`/`canonicalSlug`; carries **nothing** about clients. |

## The value rule (read this before touching value)

- **The value anchor depends on how recently the home sold.** A **years-old** sale price is stale, so a past-client home's value = the **current Open Web Ninja Zestimate** (HouseLoop Value = **Zestimate √ó 1.005**). A home that **just sold** is valued off the **real sold price √ó 1.005** by the live disposition pipeline (below) ‚ÄĒ the fresh sale is the strongest anchor. Never a stale sold price √ó a nudge.
- **GUARD:** if Ninja returns no Zestimate, the value is left **blank** ‚ÄĒ never the sold/list price.
- **There is NO separate "home value" table.** The value lives as **columns on the master home row** (`current_zestimate`, `current_hl_value`, ‚Ä¶) and is pushed to the website `Listing` when the home is seated.
- **Direction of flow:** master table first (Ninja ‚Üí master) ‚Üí website seated from the master ‚Üí **monthly, the value update appends a new `value_history` row** to the master, and the website updates from it.

### Recently-sold past clients ride the LIVE disposition pipeline (automatically)

The manual per-home backfill above is for homes sold **long ago** (no fresh sale to ride). A past-client home that **sells again now** does **not** need it ‚ÄĒ it flows through the live website **`listing-sold`** disposition pipeline on its own (see [Listing Lifecycle](/guides/listing-lifecycle)):

- The Repliers webhook fires on the real status transition ‚Üí "Sld"; the pipeline **gates on the real close date + price** (retrying until both land), persists the close price, and **mints the value from `soldPrice √ó 1.005`** ‚ÄĒ landing on the website `Listing` with zero manual work.
- **Both paths share the `ninjaEnrichedAt` stamp** (the single "was Ninja called" gate), so the live pipeline and the manual backfill **never double-enrich** the same home.
- **Worked example:** **301 E 68th Terrace** (a Nations home) sold in **2026** ‚Üí the live pipeline valued it at **$525k √ó 1.005 ‚Čą $527,625** automatically. **801 W Poplar** sold in **2021** ‚Üí it waits for the manual Ninja Zestimate backfill once resolved. Same client, two value paths, by sale recency.

## The four data sources (each authoritative for different fields)

### 1. SISU ‚ÄĒ the deal spine (who, when, how much, which agent, buy vs sell)
`sisu_transactions` (CRM, ~11,636 rows) is the source of truth that **a closing happened**, its **buy/sell role** (`transaction_type`), and the **working agent** (`agent`). **SISU is the only trusted source for agent attribution and role** (MLS rolls all agents up under `ANASPRAD`).

**The money-column law (do not conflate these):**
- **`transaction_amount` = the SALE PRICE of the home.** Never a fee, never income.
- **`gci` = GROSS commission income** (~3% of sale price).
- **`gross_paid_income` = what the team actually NETTED** after referral fees and broker splits ‚ÄĒ the real income line.

**‚ö†ÔłŹ The SISU `address` caveat:** the `address` field is **not always the property transacted** ‚ÄĒ it is sometimes the client's **residence** at the time of the deal. Resolve the true property via MLS sold-truth + triangulation, never by trusting the SISU address blindly.

**The Zillow referral fee lives in the `gci` ‚Üí `gross_paid_income` gap.** Across 1,155 closed Zillow Flex deals the team keeps ~57% of GCI vs ~74% on non-Zillow deals. ~60% of closed deals are Zillow-sourced. It is a **cost**, never a deal or income.

### 2. MLS download ‚ÄĒ structured facts + the SOLD-TRUTH (`hl_srg_mls_deals`)
`hl_srg_mls_deals` (CRM, ~1,971 rows) is the downloaded Spradling Group MLS ‚ÄĒ the **sold-truth** (real property/date/price) and the free source for structured fields: `beds`, `full_baths`/`half_baths`, `sqft`, `year_built`, `lot_size`, `property_type`, `subdivision`, schools, `hoa_fee`. (`bathroomsTotal` = `full_baths + 0.5¬∑half_baths`.)

### 3. OpenWeb Ninja / Zillow ‚ÄĒ value, value-history, schools, tax, price history, photos (the per-home backfill)
The "Real-Time Zillow Data" API (`realtime-zillow-data/search?location=` / `property-details?zpid`) provides the **Zestimate** (the value anchor for old sales), **GreatSchools**, **tax history**, and **price/sale history**. **Run per confirmed home, after sign-off, only for the still-blank ~175** ‚ÄĒ written into the master `hl_srg_past_client_homes` row. Tools: `tools/houseloop-cards/enrich-pastclient-listing.mjs`, `enrich-home.mjs`, `zillow-enrich.mjs`, `ninja-schools-tax-fill.mjs`.

### 4. Redfin photo engine ‚ÄĒ gallery fallback
Off-market homes thin to a single street-view photo on the Ninja/Zillow feed. The Redfin engine (`tools/houseloop-photos/hl-photo-engine.mjs`, real-Chrome shards on Camp David) pulls the full gallery ‚Üí uploads to **Cloudflare R2** ‚Üí writes back (improvement-only). Never hot-links Zillow/Redfin ‚ÄĒ always our R2 copy.

## How it links to the FUB scraper

`hl_srg_past_clients.fub_id` = `fub.people.id` on the Camp David `bossloop` Postgres (`fub.*` schema). That join unlocks the full communication history (`fub.text_messages` ~399K, `fub.notes` ~136K, `fub.calls` ~40K, `fub.deals` ~4.5K, `fub.assignment_history`). Two derived columns are built deterministically (no LLM):
- **`dossier`** (jsonb) ‚ÄĒ structured per-client metadata (text/email/call counts, raw notes, deals, tags, signals). The knowledge base Myli reads to write outreach.
- **`fub_identity`** (jsonb) ‚ÄĒ canonical person + FUB duplicates (same email, or phone+name) + spouse (`Secondary` tag + shared address).

## The reconcile engine (step 1 in depth ‚ÄĒ how a home gets resolved to a client)

The raw sources disagree, so resolution is a staged pipeline ‚ÄĒ **all report-only; nothing writes without human sign-off.** Tools live in `.claude/scratch/` + `tools/houseloop-cards/`:

| Stage | What it does |
|---|---|
| **S0 Identity** | collapse duplicate FUB people, link spouse. **Surname FUZZY** (catches typos like `Bohnsedt`‚Čą`Bohnstedt`), **first name TIGHT** (Tonya ‚Č† Sonja). |
| **S0.5 Forensic gather** | surface the complete household FUB cluster ‚ÄĒ *including contactless profiles* FUB buries but that hold a move or a second deal. |
| **S1 Orphan audit** | classify every closing; flag collapse-pairs (same deal twice) and orphans needing verification. |
| **S6 LLM investigator** | the capstone (claude-sonnet-4-6, report-only) ‚ÄĒ reads a six-section case file (our card ¬∑ SISU paid-truth ¬∑ MLS sold-truth ¬∑ triangulation ¬∑ house forensics ¬∑ FUB evidence) and returns a structured verdict. |

**The reconciliation canon:** MLS = sold-truth (property/date/price), SISU = paid-truth (when/how much), they **triangulate** (same price ¬Ī$3k + same date ¬Ī20 days = the same deal, even when the SISU address is a residence). Never trust a SISU address blindly; if nothing resolves, the property is **UNKNOWN** (surface candidates, never invent).

> **Worked example.** An investor's 2021 `$450k` SISU buy carried the address `10501 W 113th St Apt 5401` ‚ÄĒ their *apartment*. Triangulation matched **MLS 2355413 ‚ÄĒ `801 W Poplar St`, $450k, fourplex, 1-day gap** ‚ÄĒ the true property. That confirmed link is what makes 801 eligible for the Ninja backfill under Kyle & Karly Nations.

## How the CRM seller-report dashboard reads it

The seller-report dashboard at **`/settings/seller-report`** (`apps/houseloop-crm/app/(crm)/settings/seller-report/page.tsx`) renders **from the canonical tables, not a JSON blob**:

- **Parent `hl_srg_past_clients`** ‚Üí the person (identity, `dossier`, `resolved_current_home`).
- **Child `hl_srg_past_client_homes`** ‚Üí the homes. One query, **grouped by home** (a buy + sell of the same home collapse to one card line), with a derived **disposition** (`owns` / `sold_by_us` / `sold_lost` / `unverified`) and the **most-recent home** flagged as current-on-file.
- **Value** comes from the master row's `current_zestimate` / `current_hl_value`; each home links to its `/home/<canonical_slug>`.
- **Dossier** renders as deterministic prose built from the parent + child ‚ÄĒ never the raw JSON.

A home only appears fully populated once it has been through the resolve ‚Üí sign-off ‚Üí backfill ‚Üí slug lifecycle above (or rode the live disposition pipeline).
