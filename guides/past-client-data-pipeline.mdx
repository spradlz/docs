This is the canonical developer reference for the **Spradling Realty Group past-client data project** — one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard (past-client reactivation) and every past-client home's `/home` page. The internal strategy lives in the private GT doc `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` §12–§13; this page is the **end-to-end process a developer needs to operate it.**

## The one rule that governs everything: resolve first, enrich second

**A past-client home is never enriched, slugged, or surfaced until it has been resolved to a client and signed off.** We work the book **in the order the homes sold**, one home at a time. We do **not** run a blind bulk backfill.

**Most of the book is already done.** Of ~2,259 homes, **~93% (≈2,080) are already resolved and fully enriched** — they have their Zestimate, GreatSchools, tax, and history, and **we never re-run Ninja on them.** Only the **~175 still-blank homes** remain. Those are blank **by design** — they have not been resolved to an owner yet. Each one is backfilled **only after** it is resolved + signed off, one at a time, in sold order.

### The per-home lifecycle (the gate)

```
(1) RESOLVE        the S6 LLM resolver links the home + household → a client, on named evidence
        │
(2) SIGN-OFF       Zach confirms the home→client link is correct
        │
(3) BACKFILL       NOW run OpenWeb-Ninja / Zillow on that ONE confirmed home:
        │            Zestimate (value anchor) + GreatSchools + tax + price/sale history + photos
        │            → written into the MASTER table  hl_srg_past_client_homes
        │
(4) CANONICAL SLUG seat the home on the website FROM the confirmed master data
        │            → one Listing row, one /home/<canonicalSlug> URL (USPS-preferred city)
        │
(5) SELLER REPORT  the CRM card renders off the confirmed master data
```

| Step | What happens | Output |
|---|---|---|
| **1. Resolve** | The **S6 LLM resolver** (below) ties each still-unresolved home + household to the correct client, reading the actual evidence — never a surname match. | a confirmed `home → past_client_id` link + the household |
| **2. Sign-off** | Zach reviews the resolved home→client and confirms it. **Nothing downstream runs without this.** | go / no-go |
| **3. Backfill** | A **manual Open Web Ninja / Zillow** call on that one newly-confirmed home pulls the current Zestimate, GreatSchools, tax, price/sale history, photos → the master row. | the home's master row, fully enriched |
| **4. Canonical slug** | The website `Listing` is **seated from the confirmed master data** — minting the canonical slug + the permanent `/home/<canonicalSlug>` URL (USPS-preferred city). | one live `/home` page |
| **5. Seller report** | The CRM seller-report card renders the client + their confirmed homes + values. | the reactivation surface |

> **801 W Poplar St** is the worked example of a completed lifecycle (2026-06-22): resolved to **Kyle & Karly Nations**, signed off, then Ninja Zestimate **$569,600 → HouseLoop Value $572,448** written to the master row + the website `Listing`, `/home/801-w-poplar-street-olathe-ks-66061` live (HTTP 200).

## The canonical tables

There are three tables in play. The first two are the **CRM** (people + the master home snapshot); the third is the **website**, seated downstream from the master.

| Table | Database | Grain | Role |
|---|---|---|---|
| **`hl_srg_past_clients`** | CRM (`HL_CRM_DATABASE_URL`, Railway *autorack*) | 1 row / past client (~1,834) | the **PERSON**: name, `bucket`, `phone`/`email`, **`fub_id`** (→ FUB), `dossier` (jsonb), `fub_identity` (jsonb — incl the LLM-resolved `household_fub_ids`), `resolved_current_home`. |
| **`hl_srg_past_client_homes`** | CRM (*autorack*) | 1 row / resolved home (~2,259) | **THE MASTER SNAPSHOT — it owns the data.** One row per home a client transacted, linked by `past_client_id`. Carries the confirmed home: `address`, `canonical_slug`, `website_hlid`, `current_zestimate` / `current_hl_value`, `schools`, `tax_history`, `sale_history`, `value_history`, structured facts, `current_home_status`. **It does NOT read from the website** — built first, from the resolve + Ninja backfill. |
| **`Listing`** | WEB (`HL_WEB_DATABASE_URL`, Railway *centerbeam*) = PRODUCTION | 1 row / house (~354K) | the **website house**: renders `/home/<canonicalSlug>`. For past clients it is **seated FROM the master table** after sign-off. Keyed by `identityKey`/`canonicalSlug`; carries **nothing** about clients. |

```
hl_srg_past_clients (CRM: the person)
  ├─ fub_id              ─────────────►  fub.people (FUB scraper, Camp David)
  ├─ dossier / fub_identity ─ FUB metadata + the LLM-resolved household
  └─ id  ◄── past_client_id ── hl_srg_past_client_homes (CRM: the MASTER home snapshot — owns the data)
                                   ├─ current_zestimate / current_hl_value  (OpenWeb-Ninja, post-sign-off)
                                   ├─ schools / tax_history / sale_history / value_history
                                   └─ canonical_slug / website_hlid  ──seat──►  Listing (WEB)  ──►  /home/<slug>
```

> The old `closings` jsonb on `hl_srg_past_clients` and the retired `hl_sold_profiles` middle table are **dead**. The master home record is `hl_srg_past_client_homes`. Built by `tools/houseloop-cards/build-past-client-homes.mjs` (idempotent UPSERT on `closing_key`, never DELETE).

## How a home + household gets resolved — the S6 LLM resolver IS the authority

**Resolution is NOT deterministic string matching.** The household/spouse and the true property are decided by ONE thing: the **S6 LLM resolver** (`tools/houseloop-cards/s6-investigator.mjs`, claude-sonnet-4-6) **reading the actual evidence** — every FUB note, text, email, call, deal, the SISU closings, and the MLS truth. **Nothing links on surname or address alone.** (The earlier deterministic surname-matching pass, `_resolve_identity_v2.mjs`, is **retired** — it falsely linked different people who merely shared a last name or an apartment: Christyna→Jordyn Wyatt, Kayla→Amanda Montgomery, Shriver→Stephens.)

**The funnel — gather wide, then let the LLM decide:**

1. **Cast the net** (gathering only — decides nothing): pull every candidate that *could* be related — same/similar **last name**, same/similar **first name**, shared **email**, shared **phone**, shared **address**, any FUB **note/text** that names the client, and every **SISU closing** under the surname.
2. **Pull all the evidence** for each candidate — every note, text, email, call, and deal.
3. **The LLM triangulates** across all of it and returns the verdict — the authority for:
   - **`household`** — who is truly one household (spouse / partner / duplicate-of-the-same-person), each with a **REQUIRED evidence quote** (the exact note/text/email/deal that names the tie). The classic signal is the FUB import **"Short Summary"** note that cross-references the spouse by name + email (Field Chari's note names "Jordyn Wyatt"; hers names "Field Chari").
   - **`false_matches`** — same-surname / same-address candidates with **no named evidence**, explicitly rejected (it threw "Suzanne Tanner" out of the Chari household: *"no named connection in any note, deal, call, or text"*).
   - the true **current home**, the **deals** (with MLS/triangulation provenance), and concrete **card_errors**.
4. **Write-back** — the verdict is stamped onto the client: `resolved_current_home` and the evidence-based **`fub_identity.household_fub_ids`** (+ a full `household_llm` record). **The LLM is the only thing that sets the household.**

> **THE RULE (locked 2026-06-22):** a household link requires **named evidence** — a note/text/email/deal that names the tie, a shared email/phone, or a "Secondary"/spouse tag pointing at this person. **Surname + address alone NEVER links. When unsure, EXCLUDE.** Everything for past clients runs through THIS resolver — no other logic, no hand-rolled deterministic merge.

**The reconciliation canon:** MLS = sold-truth (property/date/price), SISU = paid-truth (when/how much), they **triangulate** (same price ±$3k + same date ±20 days = the same deal, even when the SISU address is a residence). Never trust a SISU address blindly; if nothing resolves, the property is **UNKNOWN** — surface candidates, never invent.

> **Worked example (property triangulation).** An investor's 2021 `$450k` SISU buy carried the address `10501 W 113th St Apt 5401` — their *apartment*. Triangulation matched **MLS 2355413 — `801 W Poplar St`, $450k, fourplex, 1-day gap** — the true property, under Kyle & Karly Nations.

## The value rule (read this before touching value)

- **The value anchor depends on how recently the home sold.** A **years-old** sale price is stale, so a past-client home's value = the **current Open Web Ninja Zestimate** (HouseLoop Value = **Zestimate × 1.005**). A home that **just sold** is valued off the **real sold price × 1.005** by the live disposition pipeline (below).
- **GUARD:** if Ninja returns no Zestimate, the value is left **blank** — never the sold/list price.
- **There is NO separate "home value" table.** The value lives as columns on the master home row and is pushed to the website `Listing` when the home is seated.
- **Direction of flow:** master table first (Ninja → master) → website seated from the master → **monthly, the value update appends a new `value_history` row** to the master.

### Recently-sold past clients ride the LIVE disposition pipeline (automatically)

A past-client home that **sells again now** flows through the live website **`listing-sold`** pipeline on its own (see [Listing Lifecycle](/guides/listing-lifecycle)): the Repliers webhook fires on the real "Sld" transition, the pipeline **gates on the real close date + price**, persists the close price, and **mints the value from `soldPrice × 1.005`**. Both paths share the `ninjaEnrichedAt` stamp so they never double-enrich. *Example:* **301 E 68th** (sold 2026) → valued at $525k × 1.005 ≈ **$527,625** automatically; **801 W Poplar** (sold 2021) → manual Ninja backfill once resolved.

## The four data sources (each authoritative for different fields)

### 1. SISU — the deal spine
`sisu_transactions` (CRM, ~11,636 rows) is the source of truth that a closing happened, its buy/sell role (`transaction_type`), and the working agent (`agent`). **SISU is the only trusted source for agent attribution and role.** **The money-column law:** `transaction_amount` = SALE PRICE · `gci` = GROSS commission · `gross_paid_income` = NET. **⚠️ The SISU `address` is sometimes the client's residence, not the property** — resolve via MLS + triangulation. The Zillow referral fee lives in the `gci` → `gross_paid_income` gap.

### 2. MLS download — structured facts + SOLD-TRUTH (`hl_srg_mls_deals`)
~1,971 rows — the downloaded SG MLS: real property/date/price + beds, baths, sqft, year, lot, type, subdivision, schools, HOA. (`bathroomsTotal` = `full_baths + 0.5·half_baths`.)

### 3. OpenWeb Ninja / Zillow — value, value-history, schools, tax, price history, photos (the per-home backfill)
The "Real-Time Zillow Data" API provides the **Zestimate** (value anchor for old sales), GreatSchools, tax history, and price/sale history → written into the master row. Tools: `enrich-pastclient-listing.mjs`, `enrich-home.mjs`, `zillow-enrich.mjs`, `ninja-schools-tax-fill.mjs`.

### 4. Redfin photo engine — gallery fallback
`tools/houseloop-photos/hl-photo-engine.mjs` (real-Chrome shards on Camp David) → full gallery → Cloudflare R2 → write-back (improvement-only).

## How it links to the FUB scraper

`hl_srg_past_clients.fub_id` = `fub.people.id` on the Camp David `bossloop` Postgres (`fub.*` schema). That join unlocks the full communication history (`fub.text_messages` ~399K, `fub.notes` ~136K, `fub.calls` ~40K, `fub.deals` ~4.5K, `fub.emails`, `fub.events`). Derived columns:
- **`dossier`** (jsonb) — structured per-client metadata (note/text/email/call counts + the raw notes, deals, tags, signals). The case file the S6 LLM reads.
- **`fub_identity`** (jsonb) — the canonical person + the **LLM-resolved `household_fub_ids`** (named-evidence) + `household_llm` (the verdict + evidence quotes + rejected false_matches). The spouse/household is **set by the S6 LLM resolver, never by deterministic surname/address matching.**

## How the CRM seller-report dashboard reads it

The seller-report dashboard at **`/settings/seller-report`** (`apps/houseloop-crm/app/(crm)/settings/seller-report/page.tsx`) renders **from the canonical tables, not a JSON blob**:

- **Parent `hl_srg_past_clients`** → the person (identity, `dossier`, `resolved_current_home`).
- **Child `hl_srg_past_client_homes`** → the homes, grouped by home, with a derived **disposition** (`owns` / `sold_by_us` / `sold_lost` / `unverified`) and the **most-recent home** flagged as current-on-file.
- **Per-home line:** Address · disposition · agent · purchase price · **HouseLoop value** (→ `/home/<canonical_slug>`).
- **Client header:** Purchase · Current value · **Total Appreciation** (current − purchase) · **Avg Monthly Appreciation** (total ÷ months held) · appreciation %. **No mortgage or equity is shown** (below).
- **Dossier** renders as deterministic prose from the parent + child.

### Mortgage / equity — deferred (Realie), never fabricated

True lien-based mortgage + equity is **not** on the card, and we **never fabricate it** (the old amortization guess was deleted). When enabled, the source is **Realie** (`apps/houseloop-web/lib/listings/realie.ts`): **Mortgage** = Realie `totalLienBalance` + `lenderName` (real recorded debt); **Equity** = **our HouseLoop value − the lien balance** (never Realie's own equity estimate); **owned homes only**. Deferred on cost (usage-billed, needs a monthly refresh).
