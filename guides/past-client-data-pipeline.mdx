This is the canonical developer reference for the **Spradling Realty Group past-client data project** — one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard (past-client reactivation) and every past-client home's `/home` page. The internal strategy + the in-progress reconcile algorithm live in the private GT doc `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` §13; this page documents the **shipped architecture** a developer needs to operate it.

## The two canonical tables (the whole model)

There are exactly **two** tables. Everything else is a feed or a derived view.

| Table | Database | Grain | Holds |
|---|---|---|---|
| **`hl_srg_past_clients`** | CRM (`HL_CRM_DATABASE_URL`, Railway *autorack*) | 1 row / past client (~1,834) | the **PERSON**: name, `bucket`, `closings` (jsonb deal history), `current_home`, `phone`/`email`, **`fub_id`** (→ FUB), **`website_hlid`** (→ the house), `dossier` (jsonb), `fub_identity` (jsonb) |
| **`Listing`** | WEB (`HL_WEB_DATABASE_URL`, Railway *centerbeam*) = PRODUCTION | 1 row / house (~354K) | the **HOUSE**: value, photos, schools, tax, price/sale history, beds/baths/sqft. Keyed by `identityKey`/`canonicalSlug`. Carries **nothing** about clients. |

**The link is a pointer, never a copy.** `hl_srg_past_clients.website_hlid` → `Listing.hlid`. The CRM **pulls** the home value FROM the website by that pointer; it never stores or recomputes it (no copy = no drift). One physical home = one `Listing` row = one `/home/<canonicalSlug>` URL.

```
hl_srg_past_clients (CRM: the person)
  ├─ closings[]      ─ the deals we did (from SISU)
  ├─ fub_id          ──────────────►  fub.people (FUB scraper, Camp David)
  ├─ website_hlid    ──────────────►  Listing (WEB: the house)  ──►  /home/<slug>
  ├─ dossier         ─ structured FUB metadata (texts/notes/calls/deals)
  └─ fub_identity    ─ canonical person + duplicates + spouse
```

> The retired `hl_sold_profiles` middle table (described in the original version of this guide) is **gone** — enrichment writes directly onto `Listing`.

## How the table was built (the backfill lift)

The past-client book is assembled and enriched from four data sources, each authoritative for different fields.

### 1. SISU — the deal spine (who, when, how much, which agent, buy vs sell)
`sisu_transactions` (CRM, ~11,636 rows) is the source of truth that **a closing happened**, its **buy/sell role** (`transaction_type`), and the **working agent** (`agent`). Each client's `closings[]` jsonb is built from their SISU rows. **SISU is the only trusted source for agent attribution and role** — see the source-truth hierarchy in GT §13 (MLS rolls all agents up under `ANASPRAD`, so MLS agent is never trusted).

**One SISU row = one closing card.** Its columns, grouped:

| Group | Columns | Meaning |
|---|---|---|
| Who | `client_first_name`, `client_last_name`, `address`, `transaction_type` (Buyer/Seller) | the deal |
| Our people | `agent` (cleaned, e.g. `Alisha Hanks`), `agent_raw` (tier-prefixed, e.g. `Hanks - Tier 2, Alisha`), `tc`, `team`, `appt_set_by` | `agent` is the cleaned name; `agent_raw` carries a tier/label prefix |
| 💰 Money | `transaction_amount`, `gci`, `gross_paid_income` | **SALE PRICE · GROSS commission · NET after fees/splits** |
| Lead | `lead_type`, `lead_source` | origin (e.g. `Zillow Flex Originated`, `Zillow Preferred`) |
| Timeline | `created_date` → `under_contract_date` → `closed_date` → `paid_date` (+ appt/signed/mls dates) | dates |
| Vendors | `mortgage_company`, `title_company`, `home_inspection`, `home_warranty`, `insurance_company` | service providers |
| Link / text | `fub_url` (→ the FUB person), `notes` | join to FUB + free text |

**The money-column law (do not conflate these):**
- **`transaction_amount` = the SALE PRICE of the home.** Never a fee, never income, never summed as money earned.
- **`gci` = GROSS commission income** (~3% of sale price).
- **`gross_paid_income` = what the team actually NETTED** after referral fees and broker splits — the real income line.

**⚠️ The SISU `address` caveat (critical):** the `address` field is **not always the property transacted** — it is sometimes the client's **residence/apartment** at the time of the deal. Example: an investor sold their house, moved to an apartment, then bought a fourplex — and SISU stored the *apartment* address on the fourplex purchase. Resolve the true property via the **MLS sold-truth + triangulation** below, never by trusting the SISU address blindly.

**The Zillow referral fee lives in the `gci` → `gross_paid_income` gap** (not a separate row). We pay Zillow a referral fee on Zillow-sourced (Flex/Preferred) deals. Measured across 1,155 closed Zillow Flex deals: the team keeps ~**57%** of GCI vs ~**74%** on non-Zillow deals — the ~17-point gap is the Zillow referral fee (~40% of GCI). About **60% of closed deals are Zillow-sourced** (`lead_source ILIKE '%zillow%'` OR `lead_type ILIKE '%zillow flex%'`). It is a **cost**, never a deal or income.

**"Referral" means three different things** — disambiguate by `gci` + `lead_source`:
1. **Past-agent referral as a lead source** — a past client sent us a deal we repped → full `gci` (a real repped deal, income).
2. **Outbound referral** — we referred a client out and took a cut → tiny amount + `gci = 0` (income, small).
3. **Zillow/Flex referral fee** — a percentage of our commission paid to Zillow → the `gci` → `gross_paid_income` drop (a cost).

> Rows whose `agent = "Past Agent Referrals"` use that string as a **bucket label**, not the real agent — the true agent is often embedded in the client-name field (e.g. `"David Mettman - Noelle Referral"`). The reconcile engine unmasks it, the same way it drops the MLS team-lead rollup.

### 2. MLS download — structured facts + the SOLD-TRUTH (`hl_srg_mls_deals`)
`hl_srg_mls_deals` (CRM, ~1,971 rows) is the downloaded Spradling Group MLS. It is the **sold-truth** for a deal (real property / date / price), and the **free source for the structured fields** on a past-client home: `beds`, `full_baths`/`half_baths`, `sqft`, `year_built`, `lot_size`, `property_type`, `subdivision`, schools, `hoa_fee`. The `our_roles` (`buy-*`/`list-*`) + client name tell which side we repped. The MLS backfill matches each past-client `Listing` by MLS# (extracted from `hl_heartlandmls_<mls>` hlid) → address fallback, then COALESCE-fills the null `Listing` fields. Tool: `.claude/scratch/_mls_backfill.mjs`. (`bathroomsTotal` = `full_baths + 0.5·half_baths`.)

### 3. OpenWeb Ninja / Zillow — value, value-history, schools, tax, price history, photos
The "Real-Time Zillow Data" API (`realtime-zillow-data/property-details?zpid` → `/search` fallback) provides the **Zestimate**, **GreatSchools** ratings, **tax history**, and **price/sale history**, written onto `Listing`. The **HouseLoop Value = Zestimate × 1.005** (`hlEstimate`); homes without a Zestimate fall back to the ZHVI market index, never blank. Tools: `tools/houseloop-cards/enrich-pastclient-listing.mjs`, `enrich-from-mls.mjs`, `zillow-enrich.mjs`.

### 4. Redfin photo engine — gallery fallback
Off-market homes thin to a single street-view photo on the Ninja/Zillow feed. The Redfin engine (`tools/houseloop-photos/hl-photo-engine.mjs`, 3 real-Chrome shards on Camp David) pulls the full gallery → uploads to **Cloudflare R2** (`past-clients/<streetSlug>/photo-NN.jpg`) → writes back onto `Listing.photosCached` (improvement-only). Homes with no public gallery render the Google Street View fallback (needs lat/lng). The page never hot-links Zillow/Redfin — always our R2 copy.

## How it links to the FUB scraper

`hl_srg_past_clients.fub_id` = `fub.people.id` on the Camp David `bossloop` Postgres (`fub.*` schema, the FUB scraper SSOT). That join unlocks the full communication history per client:

| FUB table | Rows | What it gives the client record |
|---|---|---|
| `fub.people` | ~49K | name, emails, phones, **addresses** (current residence), tags (`Secondary`, `Team Exit - <agent>`), stage, source |
| `fub.text_messages` | ~399K | every text (in/out, agent, date) |
| `fub.notes` | ~136K | the manual agent notes — relocation clues, spouse contacts |
| `fub.calls` | ~40K | call outcomes, duration |
| `fub.deals` | ~4.5K | deal records with agent name |
| `fub.assignment_history` | — | handoff lineage (`from_agent → to_agent`) |

Two derived columns are built deterministically from this (no LLM):
- **`dossier`** (jsonb) — structured metadata per client (text/email/call counts + first/last dates + agents, raw notes, deals, tags, lifecycle, signals). The knowledge base Myli reads to write outreach. Tool: `.claude/scratch/_build_dossier.mjs`.
- **`fub_identity`** (jsonb) — canonical person + **FUB duplicate records** (same email, or same phone+name) + **spouse** (the FUB person tagged `Secondary`, linked by shared home address). Tool: `.claude/scratch/_resolve_identity.mjs`.

## The forensic reconcile engine — how a clean record is produced

The raw sources disagree: SISU misspells names and stores residence addresses, the card price-matches deals to the wrong listing, the same deal appears twice, contactless FUB profiles (no phone/email) hide whole moves. The reconcile engine cleans this with a staged pipeline — **all report-only; nothing writes to the table without human sign-off.** Each stage is a tool in `.claude/scratch/`:

| Stage | Tool | What it does |
|---|---|---|
| **S0 Identity** | `_resolve_identity_v2.mjs` | collapse duplicate FUB people, link spouse (`Secondary` tag + shared address). **Surname FUZZY** (catches typos like `Bohnsedt`≈`Bohnstedt`), **first name TIGHT** (Tonya ≠ Sonja). |
| **S0.5 Forensic gather** | `_s05_forensic_gather.mjs` | surface the **complete** household FUB cluster — *including contactless profiles* (no phone/email) that FUB's own UI buries but that often hold a move or a second deal. Fuzzy-match SISU to flag closings missing from the card. |
| **S1 Orphan audit** | `_s1_orphan_audit.mjs` | classify every closing; flag collapse-pairs (same deal recorded twice) and orphans needing verification. |
| **S6 LLM investigator** | `_s6_investigator.mjs` | the capstone — reads a full **dossier** and returns a structured verdict (below). |

### The dossier — a human-style case file

The investigator (claude-sonnet-4-6, report-only) does **not** read raw rows — it reads a six-section case file assembled per household:

1. **Our card** — what the pipeline currently believes (may be wrong).
2. **SISU (paid-truth)** — when we got paid and how much. *Address may be the residence.*
3. **MLS sold-truth** — the real property/date/price, attributed by `our_roles` + client name.
4. **Triangulation** — for each SISU closing, MLS sold rows within **±$3k and ±20 days** are pre-computed. **Same price + same date = the same deal**, even when the SISU address is a residence and even when the MLS row has no client attribution.
5. **House forensics** — the web `Listing` per resolved MLS# (HouseLoop value, price/value/tax history, status, `/home` URL).
6. **FUB people evidence** — the full cluster (notes/texts/calls/emails/tags/events).

Each resolved deal carries a `source_of_property` provenance tag: `mls-attributed | triangulated | sisu-address | card | unknown`. **The reconciliation canon:** MLS = sold-truth (property/date/price), SISU = paid-truth (when/how much), they **triangulate** — neither overrides the other; never trust a SISU address blindly; if nothing resolves, the property is **UNKNOWN** (surface candidates, never invent).

> **Worked example.** An investor's 2021 `$450k` SISU buy carried the address `10501 W 113th St Apt 5401` — their *apartment*. Triangulation matched **MLS 2355413 — `801 W Poplar St`, $450k, fourplex, 1-day gap** — the true property, with the wrong same-price candidate (a different client's $449k single-family) correctly rejected.

## How the CRM seller-report dashboard reads it

The seller-report dashboard at **`/settings/seller-report`** (`apps/houseloop-crm/app/(crm)/settings/seller-report/page.tsx`) is the human-readable past-client surface. Per row it shows the client, their `closings`, their `bucket` (Ana Only / Ana Handoff / Spradling Team / HouseLoop), and the home value. **It reads `hl_srg_past_clients` and pulls the value LIVE from the website `Listing` by `website_hlid`** (`hlEstimate` = Zestimate × 1.005) — it does not store a separate valuation. An absent pointer renders "—".
