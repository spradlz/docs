This is the canonical developer reference for the **Spradling Realty Group past-client data project** вЂ” one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard (past-client reactivation) and every past-client home's `/home` page. The internal strategy + the in-progress reconcile algorithm live in the private GT doc `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` В§13; this page documents the **shipped architecture** a developer needs to operate it.

## The two canonical tables (the whole model)

There are exactly **two** tables. Everything else is a feed or a derived view.

| Table | Database | Grain | Holds |
|---|---|---|---|
| **`hl_srg_past_clients`** | CRM (`HL_CRM_DATABASE_URL`, Railway *autorack*) | 1 row / past client (~1,834) | the **PERSON**: name, `bucket`, `closings` (jsonb deal history), `current_home`, `phone`/`email`, **`fub_id`** (в†’ FUB), **`website_hlid`** (в†’ the house), `dossier` (jsonb), `fub_identity` (jsonb) |
| **`Listing`** | WEB (`HL_WEB_DATABASE_URL`, Railway *centerbeam*) = PRODUCTION | 1 row / house (~354K) | the **HOUSE**: value, photos, schools, tax, price/sale history, beds/baths/sqft. Keyed by `identityKey`/`canonicalSlug`. Carries **nothing** about clients. |

**The link is a pointer, never a copy.** `hl_srg_past_clients.website_hlid` в†’ `Listing.hlid`. The CRM **pulls** the home value FROM the website by that pointer; it never stores or recomputes it (no copy = no drift). One physical home = one `Listing` row = one `/home/<canonicalSlug>` URL.

  в”њв”Ђ fub_id          в”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв”Ђв–є  fub.people (FUB scraper, Camp David)

## How the table was built (the backfill lift)

The past-client book is assembled and enriched from four data sources, each authoritative for different fields.

### 1. SISU вЂ” the deal spine (who, when, how much, which agent, buy vs sell)
`sisu_transactions` (CRM, ~11,636 rows) is the source of truth that **a closing happened**, its **buy/sell role** (`transaction_type`), and the **working agent** (`agent`). Each client's `closings[]` jsonb is built from their SISU rows. **SISU is the only trusted source for agent attribution and role** вЂ” see the source-truth hierarchy in GT В§13 (MLS rolls all agents up under `ANASPRAD`, so MLS agent is never trusted).

### 2. MLS download вЂ” structured facts (`hl_srg_mls_deals`)
`hl_srg_mls_deals` (CRM, ~1,971 rows) is the downloaded Spradling Group MLS. It is the **free source for the structured fields** on a past-client home: `beds`, `full_baths`/`half_baths`, `sqft`, `year_built`, `lot_size`, `property_type`, `subdivision`, schools, `hoa_fee`. The MLS backfill matches each past-client `Listing` by MLS# (extracted from `hl_heartlandmls_<mls>` hlid) в†’ address fallback, then COALESCE-fills the null `Listing` fields. Tool: `.claude/scratch/_mls_backfill.mjs`. (`bathroomsTotal` = `full_baths + 0.5В·half_baths`.)

### 3. OpenWeb Ninja / Zillow вЂ” value, value-history, schools, tax, price history, photos
The "Real-Time Zillow Data" API (`realtime-zillow-data/property-details?zpid` в†’ `/search` fallback) provides the **Zestimate**, **GreatSchools** ratings, **tax history**, and **price/sale history**, written onto `Listing`. The **HouseLoop Value = Zestimate Г— 1.005** (`hlEstimate`); homes without a Zestimate fall back to the ZHVI market index, never blank. Tools: `tools/houseloop-cards/enrich-pastclient-listing.mjs`, `enrich-from-mls.mjs`, `zillow-enrich.mjs`.

### 4. Redfin photo engine вЂ” gallery fallback
Off-market homes thin to a single street-view photo on the Ninja/Zillow feed. The Redfin engine (`tools/houseloop-photos/hl-photo-engine.mjs`, 3 real-Chrome shards on Camp David) pulls the full gallery в†’ uploads to **Cloudflare R2** (`past-clients/<streetSlug>/photo-NN.jpg`) в†’ writes back onto `Listing.photosCached` (improvement-only). Homes with no public gallery render the Google Street View fallback (needs lat/lng). The page never hot-links Zillow/Redfin вЂ” always our R2 copy.

## How it links to the FUB scraper

`hl_srg_past_clients.fub_id` = `fub.people.id` on the Camp David `bossloop` Postgres (`fub.*` schema, the FUB scraper SSOT). That join unlocks the full communication history per client:

| FUB table | Rows | What it gives the client record |
|---|---|---|
| `fub.people` | ~49K | name, emails, phones, **addresses** (current residence), tags (`Secondary`, `Team Exit - <agent>`), stage, source |
| `fub.text_messages` | ~399K | every text (in/out, agent, date) |
| `fub.notes` | ~136K | the manual agent notes вЂ” relocation clues, spouse contacts |
| `fub.calls` | ~40K | call outcomes, duration |
| `fub.deals` | ~4.5K | deal records with agent name |
| `fub.assignment_history` | вЂ” | handoff lineage (`from_agent в†’ to_agent`) |

Two derived columns are built deterministically from this (no LLM):
- **`dossier`** (jsonb) вЂ” structured metadata per client (text/email/call counts + first/last dates + agents, raw notes, deals, tags, lifecycle, signals). The knowledge base Myli reads to write outreach. Tool: `.claude/scratch/_build_dossier.mjs`.
- **`fub_identity`** (jsonb) вЂ” canonical person + **FUB duplicate records** (same email, or same phone+name) + **spouse** (the FUB person tagged `Secondary`, linked by shared home address). Tool: `.claude/scratch/_resolve_identity.mjs`.

## How the CRM seller-report dashboard reads it

The seller-report dashboard at **`/settings/seller-report`** (`apps/houseloop-crm/app/(crm)/settings/seller-report/page.tsx`) is the human-readable past-client surface. Per row it shows the client, their `closings`, their `bucket` (Ana Only / Ana Handoff / Spradling Team / HouseLoop), and the home value. **It reads `hl_srg_past_clients` and pulls the value LIVE from the website `Listing` by `website_hlid`** (`hlEstimate` = Zestimate Г— 1.005) вЂ” it does not store a separate valuation. An absent pointer renders "вЂ”".
