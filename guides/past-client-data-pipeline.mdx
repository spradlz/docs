This is the canonical developer reference for the **Spradling Realty Group past-client data project** — one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard, every past-client home's `/home` page, **and the Follow Up Boss past-client dossier** (re-engagement + lost-listing seller leads). The internal strategy lives in the private GT doc `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` §12–§13; this page is the **end-to-end process a developer needs to operate it.**

## 🚧 CURRENT WORK IN PROGRESS — Resolver-Owned Dossier → Follow Up Boss (2026-06-23, ACTIVE)

**What we're building:** a second output of the resolver. Beyond seating the website, the resolver produces a **finished past-client DOSSIER** that lands in **Follow Up Boss** (the CRM of record) + our SSOT, so Ana & Zach can work the book — re-engagement, and especially **lost-listing seller leads** (past clients who sold a home through another agent). Powered by the **dotloop Public API** recon layer + **parcel-verified Repliers disposition.**

### THE ARCHITECTURE LAW (Zach LOCKED 2026-06-23) — the resolver owns the output, the transport is dumb

> **The S6 LLM resolver OWNS the entire output. Every byte that takes judgment is written by the resolver:** the household, the **labeled contacts** (whose phone/email is whose), the **labeled transactions** (BOUGHT vs SOLD), the **disposition** (current residence vs investment, moved, lost listing), the **tags**, AND the human **NOTE** itself.
>
> **The transport (`dossier-to-fub.mjs`) is DUMB.** It reads the resolver's finished block and writes it to FUB (note + tags + emails) + the SSOT. It **NEVER** re-derives, re-labels, re-formats, or hand-rolls anything.
>
> **If the output is wrong, the fix happens IN THE RESOLVER prompt/schema — never in the transport.** Re-run the resolver, re-transport. One brain, one place to fix.

### Why this law exists (the autopsy)

On 2026-06-22/23 the writeback **hand-rolled** the note, tags, contacts, and transactions in deterministic JS to "save LLM tokens." **Every regression came from that hand-rolled layer, never the resolver:** `roleOf()` fell back to the MLS *status* and labeled homes we **bought** as **SOLD**; `lastNameOf("Kyle & Karlie Nations")` returned **"Kyle"** and broke the SISU lookup for every couple; the contact regex **swapped** Kyle's and Karlie's phone/email; the current-home heuristic **ignored `resolved_current_home`** and called a rental their "current home." The resolver had all of it right; the transport broke it, and ~$50 of API burned. **Never again: no judgment in the transport.**

### The resolver's output block — `resolver_block` jsonb on `hl_srg_past_clients` (transported verbatim)

- **`contacts[]`** — each household member with **their own** phone + email (the LLM knows whose is whose).
- **`transactions[]`** — every deal, labeled **BOUGHT / SOLD** from SISU `transaction_type` + MLS `our_roles` (**never** the MLS status), chronological.
- **`disposition`** — current residence (or "unverified"), homes still held (residence vs **investment**), and any **lost_listings** (a home re-sold by another agent — parcel-verified).
- **`tags[]`** — the locked taxonomy below.
- **`note_markdown`** — the finished human FUB note the LLM writes.

`resolution_status` gates the transport: `resolved` (high-confidence) → written to FUB; `needs_review` (low / UNKNOWN) → **held, never written wrong**; `complete` → transported. ⚠️ Write to **`resolver_block`**, never the pre-existing `dossier` TEXT column (that's Myli's case-file).

### 🔖 FUB TAG TAXONOMY (Zach LOCKED 2026-06-23) — short, explicit, sorts every way

| Tag | On which leads | Smart list it builds |
|---|---|---|
| `HouseLoop Resolved` | **every** resolved client | everything we resolved + moved into HouseLoop |
| `SRG Closed - Ana Only` | bucket `ana-only` | Ana's closed clients |
| `SRG Closed - Ana Handoff` | bucket `ana-handoff` | handoff closed clients |
| `SRG Closed - Team` | bucket `team` | team closed clients |
| `SRG Closed - Agent DNC` | bucket `agent-dnc` | do-not-contact closed clients |
| `SRG Closed - Lost Listing` | a home re-sold by another agent | **seller-engagement leads** |

Every client carries `HouseLoop Resolved` **+** their `SRG Closed - <bucket>`; the lost ones **add** `SRG Closed - Lost Listing`. **Tags are ATOMIC — never bake combinations into one tag.** To get "Ana's seller leads," a FUB smart list ANDs `SRG Closed - Ana Only` + `SRG Closed - Lost Listing`. The resolver emits these strings; the transport writes them verbatim. Old `HouseLoop SRG …` tags were migrated 1:1 by `migrate-tags.mjs`.

### The recon + disposition tools the resolver CALLS (data only — never judgment)

- **dotloop Public API** (`apps/houseloop-crm/lib/dotloop/client.ts`, token in `hl_dotloop_connection`) = **RECON source of truth, NOT closing truth.** Per loop: client name + email + role + address + MLS# + parties + contract docs. A disorganized scratch pad (cancelled deals, listings that never sold). Use it to DIG; **SISU + MLS decide what closed. A loop ≠ a closing.**
- **Repliers parcel-verified disposition** (`repliers-sold-check.mjs` → `repliersSoldCheck(mls)`) — did a home re-sell after our sale, by another agent? **⚠️ PARCEL LAW (the Sycamore trap):** Repliers' `addressKey` ignores ZIP, county, and the N/S/E/W directional → it MERGES different houses sharing street#+name (9+ "Sycamore Ave"s in KC). A re-sale counts only if the PARCEL matches: **zip + county + neighborhood + legal description + coordinates within ~150m.** (Caught 2 false lost-listings — Field Chari's 5200 Sycamore 64129 Jackson ≠ 5200 N Sycamore 64119 Clay, 11 mi apart.)

### Branding + write order
- **SSOT first, FUB second.** Write `hl_srg_past_clients.resolver_block` + `resolution_status` FIRST; THEN write FUB; the FUB scraper mirrors it back into `fub.*`.
- **The FUB note author MUST read "HouseLoop"** — the X-System badge on FUB system id 30 (rename BossLoop→HouseLoop in the FUB dashboard; pending).
- **The FUB note = the client's lineage/history dossier.** The rich personal Sonnet dossier (anecdotes) runs HouseLoop-side LATER off the full FUB notes = Myli's knowledge base.

## The one rule that governs everything: resolve first, enrich second

**A past-client home is never enriched, slugged, or surfaced until it has been resolved to a client and signed off.** We work the book in sold order, one home at a time — never a blind bulk backfill. Of ~2,259 homes, ~93% are already resolved + enriched (we never re-run Ninja on them); only the ~175 still-blank homes remain, backfilled one at a time after resolve + sign-off.

### The per-home lifecycle (the gate)

```
(1) RESOLVE → (2) SIGN-OFF → (3) BACKFILL (Ninja → master) → (4) CANONICAL SLUG (seat website) → (5) SELLER REPORT
```

> **801 W Poplar St** worked example: resolved to **Kyle & Karlie Nations**, Ninja Zestimate $569,600 → HouseLoop Value $572,448 → master row + `Listing`, `/home/801-w-poplar-street-olathe-ks-66061` live.

## How a home + household gets resolved — the S6 LLM resolver IS the authority

The **S6 LLM resolver** (`tools/houseloop-cards/s6-investigator.mjs`, claude-sonnet-4-6) reads every FUB note/text/email/call/deal + SISU + MLS and **links a household only when a concrete artifact NAMES the tie** (required evidence quote). Surname/address alone NEVER links; when unsure, EXCLUDE (`false_matches`). It writes back `resolved_current_home`, `household_fub_ids`/`household_llm`, **and the full `resolver_block`.** The deterministic `_resolve_identity_v2.mjs` is retired.

**Reconciliation canon:** MLS = sold-truth, SISU = paid-truth, they triangulate (same price ±$3k + date ±20d = same deal). Never trust a SISU address blindly; nothing resolves → property = UNKNOWN, surface candidates, never invent.

## The four data sources

1. **SISU** (`sisu_transactions`) — deal spine: that it closed, the **buy/sell role** (`transaction_type`), the agent. ⚠️ `address` is sometimes the residence.
2. **MLS download** (`hl_srg_mls_deals`) — sold-truth + `our_roles` (buy-ana/list-ana) + structured facts.
3. **OpenWeb Ninja / Zillow** — Zestimate, schools, tax, history, photos (per-home backfill).
4. **Redfin photo engine** — gallery fallback → R2.

Plus **dotloop API** (recon) + **Repliers parcel-verified sold-check** (disposition) — see the IN-PROGRESS section.
