This is the canonical developer reference for the **Spradling Realty Group past-client data project** — one of the largest data lifts in HouseLoop. It powers the CRM seller-report dashboard, every past-client home's `/home` page, **and the Follow Up Boss past-client dossier** (re-engagement + lost-listing seller leads). The internal strategy lives in the private GT doc `GT_HOUSELOOP_SG_PAST_CLIENTS_PRD.md` §12–§13; this page is the **end-to-end process a developer needs to operate it.**

## 🚧 CURRENT WORK IN PROGRESS — Resolver-Owned Dossier → Follow Up Boss (2026-06-23, ACTIVE)

**What we're building right now:** a second output of the resolver. Beyond seating the website, the resolver also produces a **finished past-client DOSSIER** that lands in **Follow Up Boss** (the CRM of record) + our SSOT, so Ana & Zach can work the book — re-engagement, and especially **lost-listing seller leads** (past clients who sold a home through another agent). It's powered by the new **dotloop Public API** recon layer + **parcel-verified Repliers disposition.**

### THE ARCHITECTURE LAW (Zach LOCKED 2026-06-23) — the resolver owns the output, the transport is dumb

> **The S6 LLM resolver OWNS the entire output. Every byte that takes judgment is written by the resolver:** the household, the **labeled contacts** (whose phone/email is whose), the **labeled transactions** (BOUGHT vs SOLD), the **disposition** (current residence vs investment property, moved, lost listing), the **tags**, AND the human-readable **NOTE** itself. **The LLM writes the note.**
>
> **The transport layer (`dotloop-verify-writeback.mjs`) is DUMB.** It does ONE thing: take the resolver's finished block and put it where it goes — the FUB note + tags + emails, and the SSOT row. It **NEVER** re-derives, re-labels, re-formats, or hand-rolls anything. There is **no deterministic judgment in the transport.**
>
> **If the output is wrong, the fix happens IN THE RESOLVER prompt/schema — never in the transport.** Re-run the resolver, re-transport. One brain, one place to fix. That is the scalable process.

### Why this law exists (the autopsy)

On 2026-06-22/23 the writeback **hand-rolled** the note, tags, contacts, and transactions in deterministic JS to "save LLM tokens." **Every regression came from that hand-rolled layer, never the resolver:**
- `roleOf()` fell back to the MLS *status* → labeled homes we **bought** for a client as **SOLD** (Nations showed "0 buys, 3 sells" — flat wrong).
- `lastNameOf("Kyle & Karlie Nations")` returned **"Kyle"** → broke the SISU lookup for every couple.
- the contact regex **swapped** Kyle's and Karlie's phone/email.
- the current-home heuristic **ignored `resolved_current_home`** and called an investment rental their "current home."

The resolver had all of it correct. The transport re-derived it and broke it — and ~$50 of API was burned re-running batches chasing transport bugs. **Never again: no judgment in the transport.**

### The resolver's output block (emitted by S6, stored on the client, transported verbatim)

The resolver writes a complete `dossier` onto `hl_srg_past_clients` (alongside `household_llm` / `resolved_current_home`):
- **`contacts[]`** — each household member with **their own** phone + email (the LLM read the evidence and knows whose is whose).
- **`transactions[]`** — every deal, labeled **BOUGHT / SOLD** from SISU `transaction_type` + MLS `our_roles` (**never** the MLS status), chronological.
- **`disposition`** — current residence (or "unverified"), homes still held (residence vs **investment**), and any **lost listing** (a home re-sold by another agent — parcel-verified).
- **`tags[]`** — the `HouseLoop SRG` taxonomy.
- **`note_markdown`** — the finished human dossier the LLM writes.

The transport reads `note_markdown` + `tags` + `contacts` and writes them to FUB + SSOT. Nothing else.

### The recon + disposition tools the resolver CALLS (data only — never judgment)

- **dotloop Public API** (`apps/houseloop-crm/lib/dotloop/client.ts`, token in `hl_dotloop_connection`) = **RECON source of truth, NOT closing truth.** Per loop: client name + email + role + property address + MLS# + parties + the contract documents. It is a disorganized scratch pad (cancelled deals, listings that never sold). Use it to DIG; **SISU + MLS still decide what closed. A loop ≠ a closing.**
- **Repliers parcel-verified disposition** (`repliers-sold-check.mjs` → `repliersSoldCheck(mls)`) — did a home re-sell after our sale, by another agent? **⚠️ PARCEL LAW (the Sycamore trap):** Repliers' `addressKey` ignores ZIP, county, and the N/S/E/W directional, so it MERGES different houses that share street#+name (there are 9+ "Sycamore Ave"s in KC). A re-sale counts only if the PARCEL matches: **zip + county + neighborhood + legal description + coordinates within ~150m.** (Caught 2 false lost-listings — Field Chari's 5200 Sycamore 64129 Jackson ≠ the 5200 N Sycamore 64119 Clay that actually sold, 11 mi apart.)

### Write order, tags, branding (locked)
- **SSOT first, FUB second.** Write `hl_srg_past_clients` (the resolver's block + `resolution_status='complete'`) FIRST — we own it regardless of the scraper. THEN write FUB (the source of record); the FUB scraper mirrors it back into `fub.*`.
- **FUB tag taxonomy** = `HouseLoop SRG Resolution Complete` + bucket (`Ana Only` / `Ana Handoff` / `Team` / `Agent DNC`) + `HouseLoop SRG Moved - Lost Listing`. Each → a FUB smart list.
- **The FUB note author MUST read "HouseLoop"** — it's the X-System badge on FUB system id 30; rename BossLoop→HouseLoop in the FUB dashboard (pending).
- **The FUB note = the client's lineage/history dossier** (deals, parties, disposition). The rich personal Sonnet dossier (the anecdotes — "the house fire," "the watermelon") runs HouseLoop-side LATER off the full FUB notes = Myli's knowledge base.

## The one rule that governs everything: resolve first, enrich second

**A past-client home is never enriched, slugged, or surfaced until it has been resolved to a client and signed off.** We work the book **in the order the homes sold**, one home at a time. We do **not** run a blind bulk backfill.

**Most of the book is already done.** Of ~2,259 homes, **~93% (≈2,080) are already resolved and fully enriched** — they have their Zestimate, GreatSchools, tax, and history, and **we never re-run Ninja on them.** Only the **~175 still-blank homes** remain, blank **by design** — backfilled **only after** resolve + sign-off, one at a time, in sold order.

### The per-home lifecycle (the gate)

```
(1) RESOLVE        the S6 LLM resolver links the home + household → a client, on named evidence
        │
(2) SIGN-OFF       Zach confirms the home→client link is correct
        │
(3) BACKFILL       NOW run OpenWeb-Ninja / Zillow on that ONE confirmed home → MASTER table
        │
(4) CANONICAL SLUG seat the home on the website FROM the confirmed master data → one /home/<slug>
        │
(5) SELLER REPORT  the CRM card renders off the confirmed master data
```

> **801 W Poplar St** is the worked example of a completed lifecycle: resolved to **Kyle & Karlie Nations**, signed off, Ninja Zestimate **$569,600 → HouseLoop Value $572,448** to the master row + website `Listing`, `/home/801-w-poplar-street-olathe-ks-66061` live.

## How a home + household gets resolved — the S6 LLM resolver IS the authority

**Resolution is NOT deterministic string matching.** The **S6 LLM resolver** (`tools/houseloop-cards/s6-investigator.mjs`, claude-sonnet-4-6) reads every FUB note/text/email/call/deal, the SISU closings, the MLS truth — and **links a household only when a concrete artifact NAMES the tie** (with a required evidence quote). Surname/address alone NEVER links; when unsure, EXCLUDE (`false_matches`). The deterministic `_resolve_identity_v2.mjs` is **retired**.

**The funnel — gather wide, then let the LLM decide:** cast the net (similar last/first name, shared email/phone/address, notes that name the client, every SISU closing under the surname) → pull all evidence → the LLM triangulates and returns the verdict (household + evidence quotes, false_matches, the true current home, the deals with provenance) → **write-back: `resolved_current_home` + `fub_identity.household_fub_ids`/`household_llm`, and (in progress) the full `dossier` block.** The LLM is the only thing that sets any of it.

**The reconciliation canon:** MLS = sold-truth, SISU = paid-truth, they **triangulate** (same price ±$3k + date ±20 days = same deal, even when the SISU address is a residence). Never trust a SISU address blindly; if nothing resolves, the property is **UNKNOWN** — surface candidates, never invent.

> **Worked example.** An investor's 2021 `$450k` SISU buy carried the address `10501 W 113th St Apt 5401` — their apartment. Triangulation matched **MLS 2355413 — `801 W Poplar St`, $450k, 1-day gap** — the true property, under Kyle & Karlie Nations.

## The value rule

- A **years-old** sale price is stale → value = current **Open Web Ninja Zestimate** (HouseLoop Value = Zestimate × 1.005). A **just-sold** home is valued off the real sold price × 1.005 by the live `listing-sold` pipeline. **GUARD:** no Zestimate → value blank, never the list price.
- **No separate "home value" table** — value lives as columns on the master home row, pushed to the website `Listing` when seated. Monthly, a new `value_history` row appends.

## The four data sources

1. **SISU** (`sisu_transactions`, ~11,636) — the deal spine: that a closing happened, its **buy/sell role** (`transaction_type` — the trusted role source), the agent. Money law: `transaction_amount`=sale price · `gci`=gross · `gross_paid_income`=net. ⚠️ the `address` is sometimes the client's residence.
2. **MLS download** (`hl_srg_mls_deals`, ~1,971) — sold-truth + structured facts (`our_roles` = buy-ana/list-ana, beds/baths/sqft/etc.).
3. **OpenWeb Ninja / Zillow** — Zestimate, GreatSchools, tax, price history, photos (per-home backfill).
4. **Redfin photo engine** (`tools/houseloop-photos/hl-photo-engine.mjs`) — gallery fallback → R2.

Plus the recon/disposition tools in the IN-PROGRESS section: **dotloop API** (recon) + **Repliers parcel-verified sold-check** (disposition).

## How it links to the FUB scraper

`hl_srg_past_clients.fub_id` = `fub.people.id` (Camp David `bossloop` Postgres, `fub.*`). Unlocks the comms history (`fub.text_messages`, `fub.notes`, `fub.calls`, `fub.deals`, `fub.emails`). `dossier` (jsonb) = the case file the LLM reads; `fub_identity` = canonical person + LLM `household_fub_ids` + `household_llm`. **FUB is the source of record — the resolver writes the dossier note + tags into FUB, the scraper mirrors it back.**
