This is the front-door story: **what happens, in order, when a brand-new listing arrives from
Repliers and becomes a fully-resolved home on houseloop.ai.** It ties together address identity,
enrichment, photos, and rendering into one pipeline so you don't have to stitch five pages together.

<Note>
For the *identity/resolver* details see [Property Data & Address Identity](/guides/property-data-and-address-identity).
For *sold homes* see [Sold Data & Price History](/guides/sold-data-and-price-history). For *how the runs
are observed* see [Run Log](/guides/run-log).
</Note>

## The shape: one workflow per new listing

A new on-market listing fires **one** orchestrator, `catalog-listing` (an Inngest function), which runs
its steps inline as a single run ‚Äî **one run, one Run Log row** ("Cataloged ‚Äî &lt;address&gt;"). The steps
are not separate fan-out functions; they run in sequence inside the one workflow, so if a step fails the
run tells you exactly which one.

```
Repliers webhook  ‚Üí  cache upsert (SYNC, home queryable ~50ms)
                  ‚Üí  houseloop/listing.catalog  ‚Üí  catalog-listing (one run):
                        ‚Ä¢ home    ‚Äî load address
                        ‚Ä¢ photos  ‚Äî copy photos to our Cloudflare R2
                        ‚Ä¢ ai-tags ‚Äî AI photo tagging (gpt-4o vision)
                        ‚Ä¢ zillow  ‚Äî ONE OpenWeb Ninja call (tax + schools + sold history + Zestimate)
                  =  home live + fully enriched on the website
```

## Step by step (current behavior)

### 1. Webhook ‚Üí cache (synchronous)
`POST /api/repliers/webhook` fetches the full record from Repliers and calls `upsertListingToCache()`
**synchronously**. That upsert also runs the **address resolver** (`resolveHome()`) so the listing maps
to the one owned home by its city-free street+ZIP identity key (this prevents a re-list from forking a
duplicate). After the upsert returns (~50ms) the home is **queryable on the site** ‚Äî though still "thin"
(no R2 photos, vision tags, or value chart yet). The webhook then fires `houseloop/listing.catalog`
fire-and-forget and returns `200`.

### 2. `catalog-listing` ‚Äî the enrichment workflow (async)
Picked up off the Inngest queue, runs these steps in one run:

| Step | What it does | Stored on the `Listing` row |
| --- | --- | --- |
| **photos** | Downloads each Repliers photo, uploads to Cloudflare R2 | `photosCached` |
| **ai-tags** | gpt-4o vision tags buyer features (pool, garage, etc.) | `visionTags` |
| **zillow** | **One** OpenWeb Ninja call (street + ZIP, city excluded) | `valueHistory`, `taxHistory`, `schoolsGreatSchools`, `zestimate`, `annualTaxAmount` |

The **Ninja call is made exactly once per home, at intake** ‚Äî it returns the tax history, GreatSchools
ratings, prior **sold/price history** (Repliers doesn't give us sold history ‚Äî Ninja does), and the
Zestimate (which is `null` for active listings and populated for sold/off-market). Every step is
idempotent, so re-runs don't double-spend.

### 3. Value
HouseLoop Value and the value-history chart are derived from our own data (list price for active homes;
the sold anchor √ó geo appreciation for sold homes) ‚Äî anchored on the one-time Ninja base and refreshed
on render so the chart stays current as the market drifts.

## Timing

The home is **queryable in ~50ms** (the synchronous cache upsert) and **fully enriched within ~15‚Äì30s**
(the async `catalog-listing` run ‚Äî R2 photos + vision + the Ninja call). There is no sub-second
"fully resolved" guarantee, and there doesn't need to be: intake is automatic and near-real-time.

## Triggers

| Trigger | Source | Notes |
| --- | --- | --- |
| **Webhook** | Repliers `listing.created` / `listing.updated` | The primary, instant path. |
| **Cron backstop** | `listing-cache-sync` (hourly) | Catches anything the webhook missed; fires the same `houseloop/listing.catalog` event. |
| **Sold** | Repliers `lastStatus = "Sld"` transition | Fires `houseloop/listing.sold` (see [Sold Data & Price History](/guides/sold-data-and-price-history)). |

## What intake does NOT do

Intake ends when the home is live and resolved on the website. Notifying buyers is a **separate
downstream workflow** ‚Äî saved-search alerts pull listings *from* the website, they are not part of
intake. See [Saved Search Alerts](/guides/saved-search-alerts).
