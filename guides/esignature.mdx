HouseLoop eSignature is the e-signature layer of [Transaction Coordination](/guides/transaction-coordination). It takes a
real-estate contract (a flat PDF — KCRAR / Heartland MLS / eXp association forms), **auto-detects every fillable field**,
**fills it from the lead**, **sends it out for signature under the HouseLoop brand**, pulls the **executed PDF** back, and
**archives it to OUR OWN R2 on the loop document** (permanent URL — KS/MO retention). A **backup mirror** can push to
[dotloop](/guides/dotloop-integration) for eXp/SkySlope compliance — dotloop is NOT the system of record (Zach locked
Documenso-first 2026-07-06).

It is **self-hosted and open-source** — no per-document fees, no third-party "powered by" branding, no subscription.

---

## The engine: Documenso (white-labeled)

The signing substance is [Documenso](https://documenso.com) — an **open-source, self-hostable e-signature platform**
(the "open-source DocuSign"). It provides signing fields, multi-signer routing, audit trails, and a tamper-evident,
digitally-certified signed PDF — with no per-document fees and no third-party branding. We clone it, rebrand it fully to
**HouseLoop eSignature** (HouseLoop logos, brand navy `#1B3564`, product name throughout, transactional emails from
`noreply@houseloop.ai`), and self-host it on Docker.

### Why we own the signing layer

**dotloop has e-signature in its product, but NOT in its public API.** The dotloop API can read/write loops, folders,
documents, and participants, and can *download* a signed PDF — but there is **no endpoint to initiate or track a
signature request** (see the [dotloop API matrix](/guides/dotloop-api-matrix#e-signature)). A-Frame has no e-sign at all.
So to make TC AI-driven end-to-end, we run our own engine:

- **Self-hosted** → $0/document, unlimited volume, no SaaS dependency.
- **White-labeled** → signers see *HouseLoop eSignature*, not a third-party "powered by".
- **API-first** → an AI agent (Myli-TC) can create a document, place fields, add signers, and send — fully
  programmatically. That is impossible through dotloop's API and is the whole point.

## The pipeline

```
Contract PDF
  → 1. Map fields         (deterministic extraction from the PDF — see "Signature & initials placement" below)
  → 2. Fill from lead     (pdf-lib — name, price, term, address, etc.)
  → 3. Send for signature (HouseLoop eSignature — self-hosted Documenso)
  → 4. Signer signs       (browser / mobile, branded HouseLoop)
  → 5. Pull executed PDF  (signed doc + audit trail)
  → 6. Archive to our R2  (permanent, system of record) · optional backup mirror → dotloop
```

The executed PDF is archived to **HouseLoop R2** as `doc.file_url` — that is the **system of record**. Pushing it into
the dotloop loop folder is an **optional backup mirror only** for eXp/SkySlope compliance, never the destination of
record (`apps/houseloop-crm/lib/esign/send-loop-document.ts`; Documenso-first locked 2026-07-06).

Every layer is **$0/month**. The signature/initials mapping is fully deterministic ($0, no model); the optional
vision-assisted field builder costs a few cents of LLM tokens, one time per *new* form, to name/classify data blanks.

| Layer | Technology | Cost |
|---|---|---|
| Signature/initials mapping | `anchor-sig-fields.mjs` (pdftocairo + pdftotext, deterministic) | $0 |
| Data-field mapping (optional) | `vision-field-builder.mjs` (pdftotext + Claude vision) | ~cents / new form |
| Field fill | pdf-lib (embedded font) | $0 |
| E-signature | HouseLoop eSignature (Documenso, self-hosted, AGPL-3.0) | $0 |
| Archive (system of record) | HouseLoop R2 | $0 |
| Backup mirror (optional) | dotloop API | $0 (existing) |

> ⚠️ **Correction (2026-07-09):** earlier drafts named **"CommonForms / FFDNet"** as the field detector. **No such code
> ever existed in the repo** — that was aspirational. The actual tooling is
> `tools/houseloop-contracts/vision-field-builder.mjs` (built 2026-06-23, Claude vision) for data fields and
> `anchor-sig-fields.mjs` (2026-07-09, deterministic) for signature/initials boxes.

## The Documenso API / flow

Documenso exposes a REST API (+ webhooks) that the pipeline drives:

```
1. Create document           — upload the filled contract PDF
2. Add recipients            — signer(s) by name + email + signing order
3. Place fields              — signature / date / text fields at coordinates (mapped from the PDF)
4. Send for signature        — Documenso emails each signer a HouseLoop-branded signing link
5. Webhook: document.signed  — fires when all signers complete
6. Download signed PDF       — the certified, tamper-evident artifact
7. Archive to R2             — system of record; optional backup mirror → dotloop loop folder
```

Documenso webhooks (`document.created`, `document.sent`, `document.opened`, `document.signed`, `document.completed`)
let TC advance the loop's stage and run the next task the moment signing finishes — no polling, same pattern as the
[dotloop webhooks](/guides/dotloop-api-matrix#webhooks-real-time-events).

## Who signs — the loop People / Signers roster

A loop's roster (Buyer 1, Buyer 2, Buyer's Broker, Designated Agent) is resolved by `resolveLoopSigners()`
(`lib/esign/resolve-sender.ts`); the last three can be overridden per-loop with an `hl_tc_loop_participant`
row. **The email on each signer is who actually signs** — Documenso routes the signing link by email and stamps
that account's identity onto the signature line.

**The Designated Agent must be a real team member — name and email can never diverge (Zach 2026-07-22).**
Because the email decides who signs, a roster that showed *"Ana Spradling"* next to the email `zach@houseloop.ai`
sent Ana's agent line to Zach's account — so Zach's signature landed on Ana's line. The UI can't lie like that:

- **The "Change" modal** (`AddPersonModal.tsx`) renders the Designated Agent as a **picker of active team
  members** (from `/api/team/users`). Selecting an agent **binds the name + email from that member's record**
  and the email is **locked** (read-only). Co-Buyer (the client's spouse — an outside person) and Buyer's
  Broker stay free-text.
- **The participant API** (`app/api/tc/loops/[id]/participants`) enforces the same rule server-side: for
  `role='designated_agent'` it resolves the submitted email to an active human user and stamps *their* canonical
  name + email, rejecting anything that isn't a real member. So even a direct API call can't store a mismatch.

(The broker of record — Chris Cribb, `mo.sign@exprealty.net` / `ks.sign@…` — is eXp's automated managing-broker
signing service, which auto-signs the broker line; that's expected, not a stray signer.)

## Signature & initials placement — read from the PDF, never typed (locked 2026-07-09)

> **The law:** signature/initials coordinates are **extracted from the PDF itself**, never hand-typed or eyeballed.
> Full rule: `.claude/rules/SIGNATURE_ANCHORING.md`.

### The bug that kept coming back

The signature/initials boxes live in `lib/tc/contract-forms.ts` as a `signatureFields[]` per form. For months those
coordinates were **measured by hand and typed in** — re-tuned **~15 times** and still landing wrong.

**Root cause:** the initials box sits at a **different Y on every page**. Measured on the Exclusive Buyer Agency:

| Page | 1 | 2 | 3 | 4 | 5 |
|---|---|---|---|---|---|
| Initials box top (yTop) | **722** | **718** | **695** | **702** | **697** |

The code used **one hardcoded `yTop: 706`** for all five pages — so it was *mathematically impossible* to be right. The
PDF held the exact answer the whole time; we just never read it.

### The process (deterministic — run it, don't guess)

The dotloop-exported forms are **flat PDFs** (zero AcroForm fields). dotloop flattens its e-sign field boxes into the
page as **~27pt-tall vector rectangles** (the form's own data-fill underlines are ~11pt), and the printed party labels
(**BUYER / BUYER'S AGENT / BROKER'S SIGNATURE / DATE**) sit just *under* each signing line. So we read both:

```bash
node tools/houseloop-contracts/anchor-sig-fields.mjs <form.pdf>
```

1. `pdftocairo -svg` per page → parse the axis-aligned rectangles; keep the ~27pt e-sign boxes, drop the ~11pt data underlines.
2. `pdftotext -bbox` per page → the printed labels, to know **who** signs each box.
3. Classify: a narrow (`w<70`) bottom pair → **initials** (buyer left, co-buyer right); a wide box with a party label
   under it → that party's **signature**.
4. It writes the field-map (`--write lib/tc/field-maps/<key>.json`) with the **exact** coordinates, one field **per
   page**. For forms whose text layer is obfuscated (the eXp forms — `pdftotext` reads nothing), pass
   `--order buyer,co_buyer,broker,agent` to assign boxes by top-to-bottom order instead of by label.

**For a new/changed form:** run the extractor with `--write` → `contract-forms.ts` already imports
`field-maps/<key>.json`, so the fill picks it up (no coordinate ever typed) → if it warns `⚠ unassigned sign box`, add
`--order` or **one** `partyForBox()` rule → verify the render on a page whose box Y differs from page 1.

### The fill engine

`lib/tc/fill-contract.ts` (`drawSignaturePlaceholders`) draws the signature/initials as **page content** (not form
fields) in an embedded **Great Vibes** cursive — baked content renders identically in every viewer and keeps the cursive
font (a form field gets re-rendered by the viewer in its own font, which is why cursive-in-a-field showed as a block).
Initials are **vertically centered** in the ~27pt box.

**Verified (all 3 forms):** the generated field-maps drive correct placement on **Exclusive Buyer Agency** (initials
in-box on page 1 @ y=722 *and* page 3 @ y=695 — different positions, both correct; page-6 sig block), **Single
Property** and **Non-Exclusive** (via `--order`). Then a **live in-app fill** on the real Exclusive Buyer Agency (Add
Document → **Fill out** form → executed PDF on R2) confirmed "MO" in the page-1 initials box and the parties in the
page-6 boxes.

**Tools:** extractor `tools/houseloop-contracts/anchor-sig-fields.mjs` · generated field maps
`apps/houseloop-crm/lib/tc/field-maps/*.json` · imported by `apps/houseloop-crm/lib/tc/contract-forms.ts` · draw engine
`apps/houseloop-crm/lib/tc/fill-contract.ts`. Deps: `poppler` (`pdftocairo`, `pdftotext`, `pdfinfo`).

### The architecture — ONE deterministic mapper, wired into the fill (implemented 2026-07-09)

1. **Signatures / initials → vector extraction** (`anchor-sig-fields.mjs`). Reads the exact e-sign rectangle from the
   PDF. $0, pixel-exact, reproducible. ✅ Generates `field-maps/*.json`, imported by `contract-forms.ts`.
2. **Wired into the fill.** ✅ The hand-authored signature/initials arrays are gone; the fill loads the generated maps.
   Regenerate when a form revises; the fill never changes.
3. **Still authored inline** (not e-sign boxes): the eXp forms' typed print-NAME + DATE underlines merge in per form.
4. **Data blanks (name / price / term / address)** — still resolved by the vision-assisted `vision-field-builder.mjs`.
   Migrating data-field coords to deterministic text-anchor extraction is the one remaining piece.

This supersedes both the never-real "CommonForms" idea and the vision-only pixel-box approach for the signature boxes.
