HouseLoop eSignature is the e-signature layer of [Transaction Coordination](/guides/transaction-coordination). It takes a
real-estate contract (a flat PDF — KCRAR / Heartland MLS / eXp association forms), **auto-detects every fillable field**,
**fills it from the lead**, **sends it out for signature under the HouseLoop brand**, pulls the **executed PDF** back, and
**archives it to OUR OWN R2 on the loop document** (permanent URL — KS/MO retention). A **backup mirror** can push to [dotloop](/guides/dotloop-integration) for eXp/SkySlope compliance — dotloop is NOT the system of record (Zach locked Documenso-first 2026-07-06).

It is **self-hosted and open-source** — no per-document fees, no third-party "powered by" branding, no subscription.

---

## The pipeline

```
Contract PDF
  → 1. Map fields        (deterministic extraction from the PDF — see "Field & signature mapping" below)
  → 2. Fill from lead     (pdf-lib — name, price, term, address, etc.)
  → 3. Send for signature (HouseLoop eSignature — self-hosted Documenso)
  → 4. Signer signs        (browser / mobile, branded HouseLoop)
  → 5. Pull executed PDF   (signed doc + audit trail)
  → 6. Archive to our R2 (permanent) · optional backup mirror → dotloop (SkySlope / eXp compliance)
```

Every layer is **$0/month**. The signature/initials mapping is fully deterministic ($0, no model); the optional
vision-assisted field builder costs a few cents of LLM tokens, one time per *new* form, to name/classify data blanks.

| Layer | Technology | Cost |
|---|---|---|
| Signature/initials mapping | `anchor-sig-fields.mjs` (pdftocairo + pdftotext, deterministic) | $0 |
| Data-field mapping (optional) | `vision-field-builder.mjs` (pdftotext + Claude vision) | ~cents / new form |
| Field fill | pdf-lib (embedded font) | $0 |
| E-signature | HouseLoop eSignature (Documenso, self-hosted, AGPL-3.0) | $0 |
| Filing | dotloop API | $0 (existing) |

## Why these choices

- **No per-deal human.** The industry hand-tags each association form once into a maintained library; we replace that with an
  open model that auto-detects the fields, so a brand-new form is set up once (a glance) and every deal after is fully
  automatic.
- **No subscription.** Hosted e-sign vendors charge a monthly platform fee and/or per-signature fees and stamp their own
  brand on the signing experience. Self-hosting open-source gives us our own brand, unlimited volume, and $0/month.

## Signature & initials placement — read from the PDF, never typed (locked 2026-07-09)

> **The law:** signature/initials coordinates are **extracted from the PDF itself**, never hand-typed or eyeballed. Full
> rule: `.claude/rules/SIGNATURE_ANCHORING.md`.

### What we found (the bug that kept coming back)

The signature/initials boxes live in `lib/tc/contract-forms.ts` as a `signatureFields[]` per form. For months those
coordinates were **measured by hand and typed in** — and they were re-tuned **~15 times** and still landed wrong (the
cursive floating *above* the box on some pages, inside it on others).

**Root cause:** the initials box sits at a **different Y on every page**. Measured on the Exclusive Buyer Agency:

| Page | 1 | 2 | 3 | 4 | 5 |
|---|---|---|---|---|---|
| Initials box top (yTop) | **722** | **718** | **695** | **702** | **697** |

The code used **one hardcoded `yTop: 706`** for all five pages — so it was *mathematically impossible* to be right. Every
"fix" was a guess that helped one page and broke another. The PDF held the exact answer the whole time; we just never read it.

### The process (deterministic — run it, don't guess)

The dotloop-exported forms are **flat PDFs** (zero AcroForm fields / annotations). dotloop flattens its e-sign field boxes
into the page as **~27pt-tall vector rectangles** (the form's own data-fill underlines are ~11pt), and the printed party
labels (**BUYER / BUYER'S AGENT / BROKER'S SIGNATURE / DATE**) sit just *under* each signing line. So we read both:

```bash
node tools/houseloop-contracts/anchor-sig-fields.mjs <form.pdf>
```

1. `pdftocairo -svg` per page → parse the axis-aligned rectangles; keep the ~27pt e-sign boxes, drop the ~11pt data underlines.
2. `pdftotext -bbox` per page → the printed labels, to know **who** signs each box.
3. Classify: a narrow (`w<70`) bottom pair → **initials** (buyer left, co-buyer right); a wide box with a party label
   under it → that party's **signature**.
4. It prints a ready-to-paste `signatureFields` block with the **exact** coordinates, one field **per page**.

**The process for a new/changed form:** run the extractor → paste the output into that form's `signatureFields` → if it
warns `⚠ unassigned sign box`, add **one** rule to `partyForBox()` in the tool (never type a coordinate) → verify the render
on a page whose box Y differs from page 1.

### The fill engine

`lib/tc/fill-contract.ts` (`drawSignaturePlaceholders`) draws the signature/initials as **page content** (not form fields) in
an embedded **Great Vibes** cursive — baked content renders identically in every viewer and keeps the cursive font (a form
field gets re-rendered by the viewer in its own font, which is why cursive-in-a-field showed as a block). Initials are
**vertically centered** in the ~27pt box.

**Verified:** with the extracted coords, the initials land inside the box on page 1 (box at 722) *and* page 3 (box at 695) —
two different positions, both correct, from the same read-from-the-PDF data.

**Tools:** extractor `tools/houseloop-contracts/anchor-sig-fields.mjs` · field maps `apps/houseloop-crm/lib/tc/contract-forms.ts`
· draw engine `apps/houseloop-crm/lib/tc/fill-contract.ts`. Deps: `poppler` (`pdftocairo`, `pdftotext`, `pdfinfo`).

### Recommended architecture — ONE deterministic mapper, wired into the fill (Benji 2026-07-09)

The permanent fix is a single, **deterministic-first** field mapper whose output the fill engine **loads directly**, so no
coordinate is ever hand-authored:

1. **Signatures / initials → vector extraction** (the `anchor-sig-fields.mjs` method). Read the exact e-sign rectangle from
   the PDF. $0, no model, pixel-exact, reproducible. Strictly better than a vision pixel-box estimate for drawn boxes.
2. **Data blanks (name / price / term / address) → text-anchor extraction.** Anchor each value to the printed label's bbox
   (`pdftotext -bbox`). The **coordinate always comes from the PDF layer**, never from a model's guess.
3. **Vision is an OPTIONAL naming assist only.** Use it once per new form to *classify/name* an ambiguous blank — never to
   produce a coordinate. This folds the useful half of `vision-field-builder.mjs` in and drops its imprecise pixel-boxes.
4. **Wire the output into the fill.** The generator writes `field-maps/<form-key>.json`; the fill loads it (retiring the
   hand-authored arrays in `contract-forms.ts`). Regenerate when a form revises; the fill never changes.

Why this over the status quo: two detectors exist and **neither is wired in**, so coordinates get typed by hand and drift.
One generator, deterministic-first, loaded by the fill = coordinates are *generated, never typed* — the actual lock. This
supersedes both the never-real "CommonForms" idea and the vision-only `vision-field-builder.mjs` pixel-box approach.
