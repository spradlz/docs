The **Audit Report** is the human proof surface for the
[SRG Past Client Reconciliation Master](/guides/srg-reconciliation-engine). It's where Zach checks the
engine's work — every house, every deal, the evidence behind each connection, and the exact dossier card
that will land in Follow Up Boss.

It lives in the CRM under **Seller Report → Audit Report** (`/settings/seller-report/audit`).

## What it is

A full-width CRM page (`components/seller-report/AuditReport.tsx` + `MasterHouses.tsx`) backed by
`/api/settings/seller-report/audit`. It reads **tables, never blobs**: the `hl_srg_master_houses` VIEW
(derived from the deal ledger) for the house tabs, and the parent (`hl_srg_past_clients`) + child
(`hl_srg_past_client_homes`) or staging (`hl_srg_pch_staging`) for the card tabs — rendered through the ONE
shared card builder (`lib/pastclient/card-note.mjs`), so the audit card and the FUB note are byte-for-byte
identical. (The old `resolver_block` jsonb read is dead — superseded 2026-06/07.)

<Note>
**Disposition source (2026-07-16):** the disposition/value the audit cards render is now the canonical
truth from **`hl_srg_canonical`** (the MLS#-anchored resolver, 0-null), pushed onto the child
(`hl_srg_past_client_homes`) by `regenerate-notes-from-canonical.mjs` — not the child's own long-empty
`disposition` column. See the [reconciliation master's canonical map](/guides/srg-reconciliation-engine).
</Note>

Gold-table styling is deliberate (an `eslint-disable hl/no-raw-table` exception): it reads as a report,
not a data grid.

## The 5 tabs (Zach's spec, 2026-07-03)

| Tab | What it shows |
|---|---|
| **🏠 Master Houses** | RESOLVED houses only — one block per house (address = the `/home` link, HLID, value) with every deal-ledger transaction beneath it: date · BOUGHT/SOLD · client · price · agent · MLS/tier · **evidence pills**. |
| **🔧 Needs Resolution** | Opens with **Contradictions — resolutions to verify** (each card: the conflict, the resolver's proposed fix with its receipt, and a **Verify ✓** button) and **under investigation**; then field-gap houses, unplaced deals (raw SISU address as the clue), and missing-address houses. A house lives here OR on Master Houses — never both. |
| **Transaction History** | Every deal we repped, from the live child — date, side, address, price, chain-of-custody agent. |
| **FUB Notes Pending** | The staged reconcile run (`hl_srg_pch_staging`, **transactions sourced from the deal ledger** as of 2026-07-04) rendered as the full card — contacts · transactions · current/lost properties · the relationship dossier. **This is the surface Zach walks before the Rule #0 PROMOTE.** |
| **FUB Notes Final** | Signed-off cards only (`houseloop_resolved_at` set) — the green tab. |

## Trust you can see

- **Evidence pills** on every transaction row — the stored `verified_sources` from the verification engine
  (`✓ SISU addr · ✓ MLS name · ✓ Zach ✓`) or the red contradiction text. Which independent sources confirm
  each client↔house connection is visible on the row, not implied.
- **"data built at"** on the scoreboard — the page can never silently serve a stale build again.
- The **Verify ✓** button settles a contradiction: it flips the pending override to
  `verify_status='verified'` / `decided_by='zach'` and clears the row's contradiction — verification
  metadata only; identity changes go through overrides + a ledger rebuild, never a hand-edit.

## Deep links

Every client name links to their FUB profile
(`https://spradlinggroup.followupboss.com/2/people/view/<fub_id>`); every house address links to its
`/home/<slug>/<HLID>` page on houseloop.ai.

Sign-off remains **CLI-only** (`resolution-gate.mjs <pc> --signoff`) — there is no sign-off button in the
UI by design.
