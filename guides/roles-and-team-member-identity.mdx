HouseLoop separates **roles** (reusable shells) from **members** (the people and AI agents that hold
them). A member is not one role — a member holds **multiple** roles at once.

## Roles are system-level shells

A role (`hl_role`) is a reusable configuration shell, assigned to one or more members. Each role
carries a `persona`, a `tools` capability JSON, a Claude `model`, `workflow_ids`, reference `docs`,
and flags (`is_system`, `grants_tc`). The four seeded roles:

| Role | `kind` | Notes |
|------|--------|-------|
| **AI Clone** | ai | System role. A member's autonomous double that acts *as* that human. |
| **AI ISA** | ai | System role. Myli's role — the standalone inside-sales AI. |
| **Real Estate Agent** | human | The human agent shell. |
| **Transaction Coordinator** | human | The TC shell; `grants_tc` gives holders the Agent⇄TC view toggle. |

`AI Clone` and `AI ISA` are marked `is_system` (built-in, not deletable).

## A member holds MULTIPLE roles

Roles attach to members many-to-many through `hl_member_role`. One member can hold, for example,
**Real Estate Agent + AI Clone** — the human agent shell *and* their own clone shell. Myli holds
**AI ISA**, and may later *also* hold an AI Transaction Coordinator role (multi-role).

Configuration can be **overridden per member** on the join row: `persona`, `tools`, `docs`, and
`workflow_ids` set on `hl_member_role` override the shared shell's defaults for that one member. The
shell is the reusable default; the join is where a specific member diverges.

## The AI Clone role is configured at assign-time

The **AI Clone** role is a shell you assign to yourself as a member. You **configure the clone when
you assign it** — the clone's persona, tools, docs, and workflows live on that member's
`hl_member_role` override, not on the shared shell. This keeps every human's clone independently
tuned while reusing one shell.

<Note>
**Status.** The role shells, the many-to-many join, per-member overrides, and the clone-acts-as-human
dispatch are built. A dedicated UI to create and configure an AI Clone at assign-time is the intended
next step (the create-clone endpoint exists but is not yet wired to a screen). This page describes the
intended assign-time configuration model per the SSOT.
</Note>

## Clone vs Myli — two distinct AI actors

| | **AI Clone** | **Myli** |
|---|---|---|
| `type` | `ai_agent` | `ai_agent` |
| `clone_of_user_id` | → a human | NULL (standalone) |
| Comms identity | none — sends **as the human** | own email / phone / iMessage |
| Lane | **LEFT** (with the human) | **RIGHT** |
| Bound to a lead via | `ai_clone_id`, activated by `ai_mode_enabled` | `assigned_ai_agent_id` (default on every lead) |
| Team lists | hidden (one per human) | listed |

A clone acts **as** its human via `COALESCE(clone_of_user_id, …)` in `lib/ai/lead-conversation.ts` —
a human's work **never** routes to Myli. The clone is the human's own AI; Myli is a separate identity.
