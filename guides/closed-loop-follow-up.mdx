The **Closed Loop Follow-Up** system is how HouseLoop keeps every lead alive: an AI watches
behavior, follows up contextually, schedules future attention so nothing falls through, and shows
the operator *exactly* what it's doing and why. It is **signal-driven with one active workflow per
lead** — not a chatbot, drip campaign, or generic CRM automation.

The single source of truth for this system is the internal GT doc
`GT_HOUSELOOP_MYLI_CLOSED_LOOP_FOLLOWUP_v1`. This page is the developer-facing version.

## The mental model

| Object | What it is | Where it shows |
|--------|-----------|----------------|
| **Stage** | Business pipeline position (New Lead, Appointment Set, Closed…) | Left lane (lead identity) |
| **Workflow** | What the AI is *currently doing* — one goal + cadence | The cockpit "Current Workflow" |
| **Signal / Trigger** | A transient runtime event that wakes the AI | Cockpit "Active Triggers" |
| **Future Trigger** | Scheduled future attention ("the floor") | "Next Actions" |
| **Memory** | Narrative relationship intelligence | Center timeline → Memory tab |

**The active workflow *is* the state.** There is no separate state machine. "Monitoring nurture
lead" is not a mode — it is the human-readable **name of the Warm Nurture workflow** that's
currently running. "Following up after showing" = the Showing Follow-up workflow is active. One
active workflow per lead = one state. (This is why the UI never shows robotic "State: Watching ·
Mode: Listening" — the running workflow already says what the AI is doing.)

### The AI's context & memory — the standardized language (Zach locked 2026-07-17)

What the AI knows about a lead lives in exactly four places, each with ONE canonical name. This is
the vocabulary every prompt, note, and surface uses — no synonyms, no code-words:

| Canonical name | What it is | Written by | Where it shows |
|---|---|---|---|
| **Context** | The assembled facts — every text/call/email/website signal distilled into what's true right now (budget, timeline, family, homes discussed) | The context builder, on every conversation | The 🧠 Context panel on the lead |
| **Memory** | The narrative relationship story over time ("she toured in July with her sister Diane, wants a duplex") | The AI, as the relationship evolves | Center timeline → Memory tab |
| **AI Memory Update** | The AI's journal NOTE when it needs the team (or its future self) to know its working state — plain human English, always signed | Any AI writer, as an `hl_notes` row | The Notes tab + the All feed, author pill up top |
| **Instructions** | Team-authored guidance the AI must follow on this lead (showing instructions, seller constraints, do-not-disclose) | A human (or a coordinating AI), as a note | Notes; the AI checks them before acting |

The July-pilot jargon ("SHOWING PLAY", "PLAY UPDATE", "in the pocket") is retired — those notes
are preserved on their leads tagged **AI Clone (July pilot)**, but no prompt or writer uses that
language anymore. A note that needs a translator is a failed note.

## Head and brain

The system splits cleanly into two halves:

- **The HEAD (read-out)** — the lead-detail reasoning chain. It renders the AI's operating state
  as a top-down chain of causality, so an operator can *see* the AI think:
  `something happened → the AI noticed → it loaded a workflow → it did something → it scheduled
  future attention → it waits`.
- **The BRAIN (Workflow Engine)** — the native `hl_workflows` engine + Workflow Builder. It decides
  which workflow is active, advances its steps, and emits signals.

They connect through one contract so the engine drops in with **zero UI rework**.

### The reasoning chain (the HEAD)

Read top-down, in the exact order operations happen:

| Section | Answers | Fed by |
|---------|---------|--------|
| **Status line** | Who/what state | `hl_workflows.name` of the lead's one active workflow + its current step. **No fabricated default** — the old fake "Warm Nurture / Monitoring nurture lead" fallback was removed; no workflow running reads honestly as **"Myli · Standing By"**. |
| **Active Triggers** | What happened? | The signals that started/interrupted the workflow (`property_tour_requested`, `no_reply_24h`, `showing_feedback_due`…). |
| **Current Workflow** | What system is running? | The active `hl_workflows` row (name + goal). **Always present.** |
| **Current Action** | What is the AI doing right now? | The current step the workflow is executing (sending / waiting / scheduling…). |
| **Next Actions** | What happens next? | The workflow chain (`hl_scheduled_actions` — "the floor") **plus** AI-owned tasks/appointments mirrored up. Human-owned items stay in Tasks & Appointments. |
| **Recent Actions** | What already happened? | Completed steps / fired scheduled actions. |

### Where the HEAD lives

The reasoning chain renders on **both** parity lanes of the lead detail page (`apps/houseloop-crm`) —
it is **not** left-lane-only. Each lane stacks the same skeleton:

- **📋 Tasks & Appointments** — a **separate, always-on** section (dynamic tasks/appointments outside
  a workflow). It is **never gated** — it shows AI-on or AI-off, on both lanes.
- **🤖 AI Workflow** — the **gated** reasoning chain (Active Triggers → Current Workflow → Current
  Action → Next Actions → Recent Actions). The **RIGHT lane** (`MyliLane.tsx`, Myli) shows it whenever
  Myli is engaged; the **LEFT lane** (`components/leads/detail/LeadLane.tsx`, the agent's clone) shows
  it when the human's **AI Mode** is on.

Both lanes drive the same `AiReasoningChain` component via an `actor` param (`clone` | `myli`). See
[Lead Detail — Parity Lanes](/guides/lead-detail-parity-lanes) for the full skeleton and the
Next-Actions mirror rule (AI-owned tasks appear in both places; human-owned tasks stay only in Tasks
& Appointments).

<Note>
The two AIs are separate columns on `hl_leads`: **`assigned_ai_agent_id`** (Myli — the relationship
owner, right lane, swappable, on every lead) and **`ai_clone_id`** (the agent's clone — left lane,
toggled by AI Mode or SLA failover). Both can be active at once. See
[Care-Team Routing](/guides/care-team-routing).
</Note>

### The HEAD ↔ BRAIN contract

The Workflow Engine connects to the cockpit through one join:

```
hl_leads.active_workflow_id → hl_workflows.id   (the lead's one active workflow)
```

The engine sets `active_workflow_id` on a signal/trigger; the cockpit reads it. The engine emits
**signals** (lights up Active Triggers), advances **steps** (lights up Current Action), and
schedules the next step into `hl_scheduled_actions` (the floor, which already feeds Next Actions).

## THE LOCKED MODEL — the Default AI Follow-Up Workflow (Zach, 2026-07-09)

This section is the current build plan for closed-loop follow-up. It extends "the active workflow *is* the
state" to its logical end: **every lead is ALWAYS in exactly one workflow — a triggered one, or Myli's standby.**

### The design

1. **THE DEFAULT AI FOLLOW-UP WORKFLOW.** One real canvas workflow, auto-enrolled on any lead that has **no
   other active workflow**. Its steps are Myli's standing orders — visible and editable in step config, never
   buried in code:
   - *On inbound message* → read the full lead context → update context → craft and send the reply.
   - *On task due* → act on it.
   - *On silence* → the floor cadence (never the last to text; nurture touch on schedule).
   This IS "AI Follow-Up." Her operating prompt lives in the workflow's nodes (the AI Converse node from
   `GT_HOUSELOOP_AI_MODE_IS_A_WORKFLOW`, locked Option A 2026-07-04).

2. **The green dot.** The AI Follow-Up master toggle arms/disarms the default workflow globally:
   🟢 ON = every idle lead sits in Myli's standby workflow — she watches, responds, acts.
   ⚪ OFF = the default workflow is dormant; Myli takes **zero** autonomous action anywhere. It is the
   autonomy kill switch ("if the AI is going rogue") — it does **NOT** gate templated workflow sends; those are
   governed by each workflow's own Trigger Events toggle.

3. **The override / resume stack.** A trigger event fires its specific workflow → that workflow **suspends**
   the default AI Follow-Up workflow on that lead and owns it. A workflow that triggers another workflow keeps
   the override alive down the chain. When the chain **ends**, the lead **falls back into the default AI
   Follow-Up workflow** automatically. No lead ever dangles with nothing running.

4. **Auditability.** Every default-workflow action = a run-log row + a context receipt (what she saw, which
   context blocks, which tools, what she decided and why) — same audit law as every other Myli send.

### Current state (honest)

Today the default workflow **does not exist**. When a registration workflow completes, nothing is running on
the lead. The standby behaviors live as **code engines behind two toggles, both OFF**:
`ai_conversations.enabled` (inbound conversation — `lib/ai/lead-conversation.ts`) and `ai_closed_loop.enabled`
(floor cadence / tasks-due / daily home — `ensure-floor.ts`, `followup-cadence.ts`, `floor-backstop.ts`).
Those engines become the **implementation under the default workflow's nodes** when this builds — the shell
changes from invisible code to a visible canvas; the brain is already built.

### Build order (when Zach greenlights)

1. The AI Converse node type (canvas-visible, prompt-carrying, run-log + context-receipt on every action).
2. The Default AI Follow-Up workflow canvas (standing orders as steps; the engines wired underneath).
3. Auto-enroll + the override/resume stack (`hl_leads.active_workflow_id` discipline: triggered workflow
   suspends the default; chain end re-enrolls).
4. The 🟢/⚪ green-dot indicator on the AI Follow-Up nav reading the master toggle.
