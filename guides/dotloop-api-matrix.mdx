## The mental model

> **A loop = one transaction.** It is a container of **folders ‚Üí documents**, **participants** (people by role),
> **tasks** (in task-lists), **detail** (a sectioned field map), and an **activity** feed. Loops live under a
> **profile**; profiles live under an **account**. Templates pre-populate new loops. A **webhook** system pushes
> real-time events. E-signature exists in the product but is **not in the public API**.

This is the same shape as A-Frame's "xaction" ‚Äî loop ‚âà transaction, folder/document ‚âà attachment template,
participant ‚âà participant role, detail ‚âà custom-field tree, activity ‚âà activity types. dotloop adds first-class
**documents + e-sign** and a **webhook** system; A-Frame adds the **task-automation/email** engine. Our platform
fuses both.

## Authentication

- **OAuth 2.0**, 3-legged. Authorize ‚Üí code ‚Üí exchange for `access_token` + `refresh_token`.
- Base URL: `https://api-gateway.dotloop.com/public/v2/`. Authorize/token: `https://auth.dotloop.com/oauth/*`.
- Access token ~12h; refresh invalidates the prior token. Revoke via `/oauth/token/revoke`.
- **Scopes:** `account:* profile:* loop:* contact:* template:* admin:read document:*` (our live token).
- **Rate limit:** 100 req/min per user; watch `X-RateLimit-*` headers. **Pagination:** `batch_size` (‚â§100) + `batch_number`.
- Errors: RFC-style `{ "errors": [{ code, source{pointer|parameter}, detail }] }`. Codes 400/401/403/404/422/429.

## Loop lifecycle (status by transaction type)

`status` is scoped to `transactionType`:

| transactionType | Status flow |
|---|---|
| PURCHASE_OFFER | PRE_OFFER ‚Üí UNDER_CONTRACT ‚Üí SOLD ‚Üí ARCHIVED |
| LISTING_FOR_SALE | PRE_LISTING ‚Üí PRIVATE_LISTING ‚Üí ACTIVE_LISTING ‚Üí UNDER_CONTRACT ‚Üí SOLD ‚Üí ARCHIVED |
| LISTING_FOR_LEASE | PRE_LISTING ‚Üí PRIVATE_LISTING ‚Üí ACTIVE_LISTING ‚Üí UNDER_CONTRACT ‚Üí LEASED ‚Üí ARCHIVED |
| LEASE_OFFER | PRE_OFFER ‚Üí UNDER_CONTRACT ‚Üí LEASED ‚Üí ARCHIVED |
| REAL_ESTATE_OTHER / OTHER | NEW ‚Üí IN_PROGRESS ‚Üí DONE ‚Üí ARCHIVED |

(Plus `NO_STATUS`.) This is dotloop's equivalent of A-Frame's Status‚ÜíStage ‚Äî but here the *stage set itself*
switches with the transaction type, rather than statuses mapping into a fixed 5-stage enum.

## Webhooks (real-time events) ‚Äî how the AI stays in sync

A client-app subscription pushes events to an HTTPS URL.

- **Targets:** `USER` or `PROFILE` (+ `targetId`). **Create:** `POST /subscription { targetType, targetId, eventTypes[], url, signingKey?, externalId? }`.
- **USER events:** CONTACT_CREATED/UPDATED/DELETED ¬∑ PROFILE_UPDATED ¬∑ USER_PROFILE_ACTIVATED/DEACTIVATED ¬∑ USER_ADDED_TO_PROFILE ¬∑ USER_REMOVED_FROM_PROFILE.
- **PROFILE events:** LOOP_CREATED ¬∑ LOOP_UPDATED ¬∑ LOOP_MERGED ¬∑ LOOP_PARTICIPANT_CREATED/UPDATED/DELETED.
- **Payloads** are polymorphic: `{ id }` (loop/contact/profile), `{ id, participantId }` (participant), `{ id, fromId, toId }` (merge).
- **Verification:** `X-DOTLOOP-SIGNATURE` (HMAC-SHA1 of `timestamp.body` with `signingKey`) + `X-DOTLOOP-TIMESTAMP`.
- **Delivery:** at-least-once (dedupe), 5s timeout, backoff 30s‚Üí1m‚Üí15m‚Üí30m‚Üí1h‚Üí2h‚Üí4h‚Üí8h; events retained 90 days; query the delivery log via `GET /subscription/:id/event`.

<Note>
**No saved-search / browsing webhooks** here ‚Äî dotloop webhooks are loop/contact/profile lifecycle only. (This
mirrors the FUB walled-garden pattern: transactional events survive, behavioral ones don't.)
</Note>

## E-signature

**Not in the public API v2.** dotloop's e-sign is a product feature (the activity feed shows
"exported ‚Ä¶ to a digitally verified, tamper-evident PDF") but there is **no documented endpoint to request or
track signatures**. For our platform, e-sign = **Documenso** (HouseLoop eSignature) ‚Äî see
[eSignature](/guides/esignature). We pull the *signed result* PDF from dotloop via document download; we never
*initiate* signing through dotloop's API.

## Building our own (AI-driven) ‚Äî what dotloop gives us

| Capability | dotloop API | Use in our platform |
|---|---|---|
| Loop/transaction CRUD | ‚úÖ loop + Loop-It facade | AI agent opens a loop on a new deal via one Loop-It call |
| Sectioned detail (address/financials/dates) | ‚úÖ detail GET/PATCH | maps to our custom-field tree (A-Frame model) |
| Folders + documents + **download** | ‚úÖ full | the document spine; pull signed PDFs |
| Participants by role | ‚úÖ full CRUD | the people on the deal |
| Tasks / templates | ‚öÝÔ∏è read-only | we run the **automation** (A-Frame's task/email engine), not dotloop |
| Activity feed | ‚úÖ read | mirror into our run-log |
| Real-time sync | ‚úÖ webhooks (loop/contact/profile) | keep our DB live without polling |
| E-signature | ‚ùå not in API | **Documenso** |
| Contacts | ‚úÖ full CRUD (4,067 live) | optional contact sync |

**The synthesis:** dotloop supplies the *loop/document/participant/webhook* substrate; A-Frame supplies the
*task-stage automation + templated-email* engine; Documenso supplies *e-sign*; an AI agent (Myli-TC) drives all
three. That's our AI-driven TC platform.
