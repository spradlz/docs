# ═══ 01 · THE STACK (one model, audio in → audio out) ═══

Myli is built on **Pipecat** (Daily.co) — the voice framework that solves the hard audio problems (barge-in,
turn-taking, VAD, low-latency streaming, the Twilio transport) as defaults. We bring only the logic: her prompt
and the CRM wrap-up. The engine is **Gemini Live native-audio speech-to-speech** — ONE model that takes the
caller's audio and speaks back directly. No STT step, no separate TTS step, no metallic voice, no serial-hop lag.
This is the architecture UWM's "Mia" runs (Google Vertex AI / Gemini Live); we validated it against that
blueprint and the official Pipecat reference preset.

The signal flow is now two hops, not four:

| Layer | Choice | Billed by | Notes |
|---|---|---|---|
| **Transport** | Twilio Media Streams → `<Connect><Stream>` | per-minute | Live G.711 µ-law audio over WebSocket; `TwilioFrameSerializer` handles the µ-law↔PCM resample (the bug that killed the 2026-07-15 attempt). |
| **The model (S2S)** | **Gemini Live** `gemini-2.5-flash-native-audio-preview-09-2025`, voice **Leda**, `thinking_budget=0` | per audio token | Audio→audio in one model. Gemini's own VAD (LOW/LOW ~800 ms) owns turn-taking. `GEMINI_API_KEY`. |
| **Seam** | CRM `persona` (in) + `wrapup` (out) | — | Prompt + buckets in at call start; the one write out at hangup. |

**Config = the official reference preset, nothing more.** Only 4 Settings fields: `model`, `voice`,
`system_instruction`, `thinking=ThinkingConfig(thinking_budget=0)`. We tried the v1alpha "affective dialog" /
"proactive audio" knobs and **removed them** — they add latency for no clear win (`MYLI_GEMINI_EXPRESSIVE=0`
by default). The greeting is triggered with a `role:"developer"` message +
`inference_on_context_initialization=False` (so she opens on cue, not on connect).

**The runtime files** (`apps/myli-voice-agent/`): `pipecat_bot.py` (the bot — `MYLI_ENGINE=s2s` picks the
Gemini branch; `cascade` = the retired fallback) · `requirements-pcc.txt` (deps for the Pipecat Cloud image) ·
`Dockerfile.pcc` (`FROM dailyco/pipecat-base`) · `pcc-deploy.toml` (the Pipecat Cloud deploy config) ·
`pcc-build/` (the minimal 3-file build context). Every model/voice value is env-overridable
(`MYLI_ENGINE`, `MYLI_GEMINI_MODEL`, `MYLI_GEMINI_VOICE`) so we retune without a code change.

# ═══ 03 · HER PROMPT — LEAN BASE + FOUR BUCKETS ═══

Native-audio models get **worse** as the prompt gets longer and vaguer, so Myli's prompt is deliberately
small and structured. It has two parts:

**1. The lean triage base** (`hl_role.persona` on the AI Receptionist role, edited in **My Team → AI
Receptionist**). Structure, in order — the Google Live-API prompting shape:

- **Identity** — Myli, Spradling Group's front desk. Warm, upbeat, casual, quick — never a stiff IVR.
- **Voice / style** — one question per turn; she listens, she doesn't monologue; spell numbers out; never
  read a list aloud; never narrate calling a tool.
- **Job, in order** — figure out who's calling (seller / buyer / past client / other), then run that bucket.
- **Guardrails** — never fakes being human, never live-transfers; on hangup she writes the lead + a callback.

**2. The four specialized buckets** (`hl_settings.myli_voice_buckets`, seeded by migrations **0419** +
**0420**). The moment Myli identifies the caller she calls the **`set_caller_type`** function tool
(`seller` / `buyer` / `past_client` / `other`) — that RECORDS the label for the wrap-up, and the matching
bucket is the playbook she follows. Each bucket is a **collect-in-order checklist with the VERBATIM question
for each field**, so she can't paraphrase the intent away:

- **seller** → *"What's the address of the home you're thinking of selling?"* with a CAPS rule: **DO NOT accept
  just a city or an area — you need the full street address** (if they give a neighborhood, follow up "and
  what's the street address there?", then read it back). This bucket is why v2 exists — v1 said "roughly where"
  and Myli asked *"what area?"* instead of the address.
- **buyer** → *"What area are you looking to buy in?"* (or the specific listing's address if they name one),
  then budget / timeline.
- **past_client** → greet like a friend of the team, *"What can I help you with?"*, capture the reason; if it
  turns into a sale/purchase, move into the seller or buyer questions.
- **other** → vendor / general question / wrong number / asking for Zach or Ana directly — take a message + callback.

<Note>
In the S2S engine the base + all four bucket playbooks are inlined into the single `system_instruction` at call
start (native-audio can't hot-swap the system prompt mid-turn), and `set_caller_type` just records which bucket
applies for the wrap-up. Editing a bucket = a SQL update to `hl_settings.myli_voice_buckets`; editing the base
= the role persona in My Team. Neither touches the bot.
</Note>

# ═══ 04 · THE CRM SEAM (two endpoints) ═══

The bot talks to the CRM at exactly two seams:

| Route | Method | One line |
|---|---|---|
| `/api/phone/myli/persona` | GET | Returns `{ persona (lean base), ground_truth, buckets }` — everything the bot inlines into the prompt. |
| `/api/phone/myli/wrapup` | POST | **THE ONE WRITE SEAM** — every post-call CRM effect. |

**The wrap-up receipts (one POST, every effect):** spam → silent flag (no lead, no ping) · find-or-**create**
the lead (door law: source `phone_call`, assignment/stage from the source defaults, never hardcoded) · Myli
joins the care team · `hl_calls` gets summary + role + callback · a callback task on the assigned agent ·
`logActivity` `call_logged` (the All-feed + AI Follow-Up row) · `notifyLeadEvent` → the assigned agent.

<Warning>
**Transcript capture is a known follow-up.** The cascade era captured the transcript via a `TranscriptProcessor`
in the pipeline; that processor was **removed** because it isn't in the Pipecat Cloud base image's newer Pipecat.
Native-audio transcript-capture via the aggregator turn events is the re-add — tracked in §06. Until then the
wrap-up records the summary + role + callback, but not a full turn-by-turn transcript.
</Warning>

**The answer gate — 3 layers, all must pass:** (1) org master `hl_settings('ai_conversations').voice_enabled`
(fail-closed) · (2) per-line `hl_phone_numbers.answer_mode='myli'` · (3) role capability `voice_inbound`.

# ═══ 05 · THE HOST — PIPECAT CLOUD ═══

The bot no longer runs on a laptop tunnel — it's deployed to **Pipecat Cloud** (Daily's managed host for Pipecat
agents). This is the fix for the compounding latency: a free-ngrok tunnel on a laptop can't keep up with
real-time audio backpressure, so time-to-first-byte grew every turn (1→3→14 s). On a real host, latency is flat.

| Field | Value |
|---|---|
| **Org / workspace** | `spradling-group` (login = GitHub OAuth, `zachspradling@spradlingrealtygroup.com`) |
| **Agent** | `myli-voice` |
| **Profile** | `agent-2x` (1 vCPU / 2 GB dedicated — real-time audio needs non-throttled CPU) |
| **Region** | `us-west` (revisit `us-east` for Twilio latency later) |
| **Scaling** | **cold-start** (`min_agents=0`) — $0 idle, ~few-sec spin-up on the first call after idle (Zach's call: the line gets spam, so warm-idle isn't worth ~$43/mo yet). Flip to `min_agents=1` if the cold-start delay bugs us. |
| **Secret set** | `myli-voice-secrets` (`MYLI_ENGINE=s2s`, `GEMINI_API_KEY`, `HL_CRM_BASE_URL`, `HL_INTERNAL_SECRET`, `MYLI_VOICE_SECRET`, `TWILIO_*`) |
| **Status** | Deployment **ready** (2026-07-17). **Wired to Twilio 2026-07-18** — the CRM voice route streams myli-mode calls here (§06). |

**The deploy pipeline** (config in `pcc-deploy.toml`):

```bash
# from apps/myli-voice-agent/, with PIPECAT_TOKEN in env, using the venv CLI:
.venv-pipecat/bin/pipecat cloud deploy --build-dir pcc-build --dockerfile Dockerfile.pcc --yes
```

- The build context is the **minimal `pcc-build/`** dir (bot.py + Dockerfile + requirements-pcc.txt, ~32 KB).
  The full app dir is 686 MB (the venv) — over Pipecat Cloud's build-context cap, so it must NOT be the context.
- The base image imports `bot(runner_args)` from `bot.py`, so `pipecat_bot.py` is copied in as `bot.py`.
- Secrets: `.venv-pipecat/bin/pipecat cloud secrets set myli-voice-secrets --file .env.pcc --skip` (needs a
  PTY — run it under `script -q /dev/null …`).

**Cost.** Cold-start = **$0 idle**. Per active minute ≈ Gemini Live audio tokens + Twilio (~$0.0125/min) + the
Pipecat Cloud agent-2x host (~$0.02/min). Rough all-in ≈ **$0.06–0.09/min** of talk time.

<Warning>
**S2S cost metering is NOT yet wired.** The `MyliUsageObserver` + `logAiCost({category:'ai_voice'})` path was
built for the **cascade** (LLM tokens + TTS chars + Deepgram minutes). Gemini Live bills by **audio tokens**,
which the observer doesn't sum, so a live S2S call currently does NOT land a real cost row on `/ai-cost`. Re-wiring
the cost meter for the audio-token model + the Pipecat Cloud host cost is a go-live follow-up (§06).
</Warning>
