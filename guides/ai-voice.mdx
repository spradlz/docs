# ═══ 01 · THE STACK (one model, audio in → audio out) ═══

Myli is built on **Pipecat** (Daily.co) — the voice framework that solves the hard audio problems (barge-in,
turn-taking, VAD, low-latency streaming, the Twilio transport) as defaults. We bring only the logic: her
prompt and the CRM seam. The engine is **OpenAI Realtime speech-to-speech** — ONE model
(`gpt-realtime-2.1`, voice **marin**) that takes the caller's audio and speaks back directly. No STT step,
no separate TTS step, no serial-hop lag: p50 **1.3–2.2s voice-to-voice** measured on our Twilio architecture
(model TTFB 0.53s), where Gemini Live took 0.8–12.8s (HISTORY).

**The engine flag is load-bearing.** The code default is `cascade` (`pipecat_bot.py:87`) — the LIVE engine
is selected by **`MYLI_ENGINE=openai` in the deployed secret set** (`.env.pcc`). Three engines exist in the
one bot file: `openai` (**LIVE**) · `s2s` (Gemini Live — retired, HISTORY) · `cascade`
(Deepgram→Haiku→Cartesia — the outage fallback).

| Layer | Choice | Billed by | Notes |
|---|---|---|---|
| **Transport** | Twilio Media Streams → `<Connect><Stream>` | per-minute | Live G.711 µ-law audio over WebSocket; `TwilioFrameSerializer` handles the µ-law↔PCM resample (the bug that killed the 2026-07-15 attempt). |
| **The model (S2S)** | **OpenAI Realtime** `gpt-realtime-2.1`, voice **marin** (`pipecat_bot.py:92-93`) | audio tokens (blended ~$0.10/audio-min, §04) | Audio→audio in one model. `OPENAI_API_KEY` (the funded bossloop-infra account). |
| **Seam** | CRM `persona` (in) + `wrapup` (out) | — | Prompt segments in at call start; the one write out at hangup (§04). |

**The Settings shape** (`pipecat_bot.py:649-656`) — model + instruction + session properties, nothing else
(there is NO `thinking` field on this engine; that was a Gemini knob):

```python
OpenAIRealtimeLLMService.Settings(
    model=MYLI_OPENAI_MODEL,              # gpt-realtime-2.1
    system_instruction=s2s_instruction,   # base + greeting note + TRIAGE block (lean start — §03)
    session_properties=SessionProperties(
        audio=AudioConfiguration(output=AudioOutput(voice=MYLI_OPENAI_VOICE)),  # marin
        tools=s2s_tools,                  # set_caller_type · end_call · wait_for_user
        turn_detection=...,               # only when MYLI_OPENAI_TURN opts in (below)
    ),
)
```

**Turn-taking — the `MYLI_OPENAI_TURN` knob** (`pipecat_bot.py:100, 639-647`). LIVE value =
**`semantic_medium`** → `SemanticTurnDetection(eagerness="medium")` — the classifier that waits when the
caller's words sound unfinished (the documented fix for jumping in on "um..." trail-offs). The full menu:
`semantic_low` / `semantic_medium` (semantic detection at that eagerness) · `server_900` (server VAD with
`silence_duration_ms=900`, the A/B fallback arm) · unset = OpenAI's server-VAD default. The requested config
is logged at session start — verify `session.updated` in the logs before judging behavior.

**The instant greeting — canned, and NO run frame.** A pre-recorded greeting in Myli's exact live voice
(`greeting-openai.wav`, marin, generated once via `gpt-4o-mini-tts`; per-engine file pick at
`pipecat_bot.py:145-147`) plays the instant the WebSocket connects — zero model in the loop, ~400ms of
leading silence absorbing the Twilio media-leg start race (`pipecat_bot.py:770-794`). On OpenAI **no
`LLMRunFrame` is queued** (`:781`) — there is no audio gate to open, and a run frame at connect would
trigger a SECOND model greeting racing the canned one (the live-caught opening collision). The context
starts EMPTY; an instruction block tells the model its opener was already spoken, and the wrap-up restores
the greeting line into the transcript (§04). *(The Gemini audio-gate law — canned audio MUST pair with
`LLMRunFrame()` — was Gemini-ONLY; preserved in HISTORY.)*

**The runtime files** (`apps/myli-voice-agent/`): `pipecat_bot.py` (the ONE bot — all three engine
branches) · `requirements-pcc.txt` · `Dockerfile.pcc` (`FROM dailyco/pipecat-base`) · `pcc-deploy.toml`
(host config) · `pcc-build/` (the deploy context — **manually copied into; see THE CP-TRAP, §05**) ·
`greeting-openai.wav` / `greeting.wav` (per-engine canned greetings). Every model/voice value is
env-overridable (`MYLI_ENGINE`, `MYLI_OPENAI_MODEL`, `MYLI_OPENAI_VOICE`, `MYLI_OPENAI_TURN`) so we retune
without a code change.

# ═══ 03 · THE CONVERSATION DIRECTOR (the code guard layer) ═══

<Note>
This section is ARCHITECTURE — the code layer that makes the prompts hold on a live phone line. What the
prompts actually SAY (base v3.1, the triage block, the seller/buyer/past_client/other playbooks, the
ONE-BREATH RULE, versioning) is [AI Programming](/guides/ai-programming) + the registry at `/ai/prompting`.
</Note>

**Lean start → playbook HOT-SWAP.** The OpenAI session starts LEAN: `system_instruction` = the base persona
+ the greeting note + THE TRIAGE BLOCK only — **no playbooks inlined** (`pipecat_bot.py:553-580`). The
moment the model identifies the caller it fires the `set_caller_type` tool, and the bot hot-swaps the
session instructions to **base + the ONE matching playbook** via `LLMUpdateSettingsFrame` — a mid-call
`session.update`, OpenAI's own "dynamic conversation flow" pattern (`pipecat_bot.py:684-693`). The
playbooks come from `hl_settings.myli_voice_buckets` (migs 0419 + 0420) through the persona endpoint (§04),
so editing one never touches the bot.

<Warning>
**DEPRECATED/CHANGED (2026-07-19/20, the OpenAI swap).** The old rule here — *"base + all four bucket
playbooks are inlined into the single `system_instruction` at call start; native-audio can't hot-swap the
system prompt mid-turn, and `set_caller_type` just records the label"* — was the **Gemini design** and is
the **OPPOSITE of live behavior**. Gemini Live cannot `session.update`; OpenAI Realtime can, and the
hot-swap decision replaced inlining when the engine swapped. Inlining survives only on the retired
`s2s` branch (HISTORY).
</Warning>

**`MyliTurnGuard`** (`pipecat_bot.py:371-471`) — a Pipecat observer on the pipeline task; the
announce-then-silence killer, in code. Live-caught by Zach and reproduced in the cockpit even on the
one-breath prompt: prompts lower the flinch rate, **only code eliminates it**. OpenAI engine only; every
handler is guarded so the director can never crash a call.

- **The turn-completion guard.** Watches `BotStoppedSpeakingFrame`; if a spoken turn in the
  `triage`/`intake` state ends **without a "?"** in the last assistant text, it appends a steering system
  message ("continue the same breath — say ONLY the question") and forces ONE continuation via
  `LLMMessagesAppendFrame` + `LLMRunFrame` — the officially documented `response.create` correct-and-regenerate
  recipe (`:417-432`). One guard shot per bot turn; **hard cap 4 total fires per call**, and never after
  `end_call` (`:418`).
- **The post-tool watchdog** (`:434-446`). Tool results answered with `run_llm=False` (`set_caller_type`,
  `wait_for_user`) can strand a silent turn — a documented Realtime bug class. ~2.5s after such a result,
  if nothing has been spoken, the same steer fires (`arm_watchdog()` is called right after the playbook
  hot-swap, `:700`).
- **`wait_for_user`** (`:508-522`) — a NO-OP hold-the-turn tool, **OpenAI engine only**: the model calls it
  instead of speaking when the latest audio needs no spoken response (silence, background noise, hold
  music, a caller mid-thought "um..."). No "I'm here" / "take your time" filler — she just waits.
- **The state ladder + telemetry.** `triage → intake (set_caller_type) → ended (end_call)`. The wrap-up
  payload carries `director: {state, guard_fires, watchdog_fires}` (`pipecat_bot.py:833-838`) — every call
  reports how often the code had to save a turn, so guard-fire rates are a tunable metric, not folklore.

The other two tools ride alongside: **`set_caller_type`** (records the caller bucket + triggers the
hot-swap + flips the guard to `intake`) and **`end_call`** (she speaks one natural goodbye, then after a
~2.5s grace the bot completes the CALL via the Twilio REST API — ending the whole call, not just the
stream, so the TwiML can't fall through to voicemail; `pipecat_bot.py:495-534`).

# ═══ 04 · THE CRM SEAM (two endpoints) ═══

The bot talks to the CRM at exactly two seams:

| Route | Method | One line |
|---|---|---|
| `/api/phone/myli/persona` | GET | Everything the session needs: prompt segments, memory, capabilities. |
| `/api/phone/myli/wrapup` | POST | **THE ONE WRITE SEAM** — every post-call CRM effect. |

**The persona payload** (`apps/houseloop-crm/app/api/phone/myli/persona/route.ts:208-224`) returns
`{ persona, ground_truth, triage, buckets, known_lead, voice_enabled, tools_enabled, voice_config, agent }`:

- **`persona`** — the ground-truth block (live clock + caller-ID digits) + the LEAN role base + phone voice
  rules, merge tokens resolved.
- **`triage`** — THE TRIAGE BLOCK from the registry slot **`myli_voice_triage`** (mig 0431 — the
  no-hidden-prompts law: every segment the bot speaks from is visible + editable in the cockpit rail;
  `route.ts:201-206`). The bot appends it to the base at call start; a hardcoded copy in the bot is only
  the unreachable-CRM fallback.
- **`buckets`** — the 4 playbooks from `hl_settings.myli_voice_buckets`, tokens merged. **Draft-aware:**
  `?prompt_source=draft` overlays the cockpit drafts on both triage + buckets through the exact same
  composition path (`route.ts:182-192`) — the Test Cockpit chats against drafts; live calls never send the param.
- **Warm calls** — when the caller dialed the agent's OWN line AND the matched lead is assigned to her
  (both gates, `route.ts:66-106`), the persona injects the canonical memory composer `composeMyliMemory`
  (`lib/ai/myli-memory.ts`) + the `## WHO'S CALLING — YOU KNOW THEM` block (`route.ts:112-129`). The
  company/test line is ALWAYS the cold receptionist — no name-recognition for strangers.
- **`voice_enabled` / `tools_enabled` / `voice_config`** — the master capability read + per-tool gates +
  the callback-line policy, so the worker never hardcodes behavior.

**The answer gate — FOUR layers, ALL must pass** (the voice webhook,
`app/api/phone/webhooks/voice/route.ts:294-339`):

1. Org master `hl_settings('ai_conversations').voice_enabled` — **fail-closed** (`:309-314`).
2. **THE ALL-AI FAILSAFE** `aiMasterEnabled()` (mig 0432) — the one master switch off silences voice too;
   everything off, period (`:315-316`).
3. Per-line `hl_phone_numbers.answer_mode = 'myli'` — the ANSWERED BY select.
4. Role capability `voice_inbound` on the phone hat (`answers_calls` role) — AND-ed, never overwriting the
   org verdict (`:322-337`).

**The wrap-up receipts (one POST, every effect)** (`app/api/phone/myli/wrapup/route.ts`): spam → silent
flag (no lead, no ping) · find-or-**create** the lead (door law: source `phone_call`, assignment/stage from
the source defaults, never hardcoded; a caller that can't mint lands the transcript leadless instead of
500ing) · Myli joins the care team · `hl_calls` gets **the full transcript** — built by the bot from the
context messages with the canned-greeting line restored (`pipecat_bot.py:806-821`) — plus an **AI summary**
generated in the route with the same summarizer the recording path uses (`route.ts:55-69`), role label,
callback window · a callback task on the assigned agent · `logActivity` `call_logged` (the All-feed row) ·
`notifyLeadEvent` → the assigned agent · the **director telemetry** from §03 rides the same payload.

**Cost is engine-aware and WIRED** (`route.ts:80-100`): for `gpt-realtime` the route bills the
research-backed blended rate **~$0.10/audio-min** — raw token math over-bills ~25–30× because OpenAI
auto-caches the repeated context prefix at ~1/80th rate, a split our metrics can't see (live-caught: a 62s
call priced $2.85 raw); the raw token counts still ride in `data.usage` for future calibration against the
OpenAI dashboard. Add Twilio ~$0.0125/min + the Pipecat Cloud host ~$0.02/min → one
`logAiCost({category:'ai_voice'})` row per call (`:229-233`, leadless calls too) + `stampActivityTokens` on
the catalog row. The cascade fallback keeps its 3-vendor math (Deepgram + Haiku tokens + Cartesia chars).

**The deploy pipeline** (config in `pcc-deploy.toml`; from `apps/myli-voice-agent/` with `PIPECAT_TOKEN`
in env, using the venv CLI):

1. **⚠ THE CP-TRAP — copy the bot FIRST, every deploy.** `pcc-build/bot.py` is a **manual copy** of
   `pipecat_bot.py` (the base image imports `bot(runner_args)` from `bot.py`), and the deploy builds
   `pcc-build/` — skip the copy and you **silently ship the stale bot** while the CLI reports success
   (live-caught 2026-07-21):
   ```bash
   cp pipecat_bot.py pcc-build/bot.py
   ```
2. Deploy:
   ```bash
   .venv-pipecat/bin/pipecat cloud deploy --build-dir pcc-build --dockerfile Dockerfile.pcc \
     --krisp-viva-audio-filter tel --force --yes
   ```
3. Secrets (only when `.env.pcc` changed — needs a PTY, run under `script -q /dev/null …`):
   ```bash
   .venv-pipecat/bin/pipecat cloud secrets set myli-voice-secrets --file .env.pcc --skip
   ```

- **THE GEMINI LIVE ERA (2026-07-18 → 07-19) — FIRED for latency.** Myli's first S2S engine:

  - **The 4-field preset law:** exactly `model` + `voice` + `system_instruction` + `thinking` in Settings.
    Every VAD/turn override we tried (END low, START low, `silence_duration_ms=1100`) measured WORSE on
    real calls (5–15s gaps / near-mute); the v1alpha affective-dialog / proactive-audio knobs were removed
    (latency for no win). Gemini's own VAD (LOW/LOW ~800 ms) owned turn-taking.
  - **⚠ THE AUDIO-GATE LAW:** `GeminiLiveLLMService` **drops all caller audio** until
    `_ready_for_realtime_input` flips true, which happens ONLY when a context frame reaches the LLM — so
    any canned audio at call start MUST be paired with an empty-context `LLMRunFrame()`
    (`inference_on_context_initialization=False` opens the gate without a model turn; verified in library
    source `gemini_live/llm.py:813, 1523-1529, 1607-1611` after two deaf live calls). **On OpenAI this law
    INVERTS** — no gate exists, and that same `LLMRunFrame` at connect double-greets (§01).
  - **Inlined playbooks:** Gemini can't `session.update` mid-call, so base + ALL four bucket playbooks were
    inlined into the one `system_instruction` and `set_caller_type` only recorded the label — the design
    the OpenAI hot-swap replaced (§03). Also: never seed a pre-spoken assistant turn into the context
    (dangling `turn_complete=False` wedged turn-taking, 47s of silence live-caught); a module-load pre-warm
    thread exercised the Gemini path at worker boot (~1.6s turn-1 vs ~10s stone-cold).
  - **Why fired:** TTFB measured **0.8–12.8s per turn** on our real calls (0.85→3.5→7.8→9.8s swings inside
    Google, receipts in the session logs) — community-documented **unfixable** across all tiers; Vertex
    Provisioned Throughput sells throughput, not a TTFB SLA (memory: `project_myli_voice_latency_verdict`).
    **DECISION: the OpenAI Realtime swap, 2026-07-19/20** — sub-second on the identical Twilio architecture.
    The code survives behind `MYLI_ENGINE=s2s`; `GEMINI_API_KEY` idles in the secret set.
