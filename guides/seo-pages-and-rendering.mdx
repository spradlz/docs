title: "How SEO pages are built & rendered"
description: "One shared results page renders every SEO surface. The SEO block is composed dynamically from the URL, geo routes are thin shells over the taxonomy, and the sitemap enumerates the whole tree."
---

Every SEO surface — `/listings`, every county / city / zip / neighborhood page — renders the
**same** React component. There is no per-page template to drift. This page explains how that
works and how to add a new SEO page.

## One shared page

`components/listings/SearchResultsPage.tsx` is the single listings UI. Geo routes are thin
shells that resolve the URL params, then hand off:

```tsx
// app/[state]/[county]/[city]/page.tsx  (county, zip, neighborhood are identical shape)
export const dynamic = "force-dynamic";
export async function generateStaticParams() { return allCityParams(); }

export default async function CityPage({ params }) {
  const { st, co, ci } = resolveGeo(params);
  const qp = new URLSearchParams({ city: ci.name, state: st.abbr });
  return <SearchResultsPage sp={qp} geo={{ basePath: `/${st.slug}/${co.slug}/${ci.slug}` }} />;
}
```

The shell only does three things: resolve the slug to a real place, build the search params,
and pass a `basePath`. All header, results, map, and SEO rendering live in the shared page.

## The dynamic SEO block

Inside `SearchResultsPage`, the SEO section is **resolved from the URL on every surface** — so
`/listings?city=Fairway` gets the exact same breadcrumb + sections as `/kansas/johnson-county/fairway`.

```
resolveLocation(searchParams)   →  lib/geo/resolve-location.ts
   ↓  ResolvedLocation { level: state|county|city|zip|neighborhood, ... }
<LocationSeo location={resolved} />   →  components/geo/LocationSeo.tsx
   ↓ renders: breadcrumb → amenity → property type → popular searches
     → nearby cities ($) → neighborhoods ($) → zips ($) → estimate proceeds → price bands
```

- `resolveLocation` reads `city / county / zip / state / neighborhood` off the params and
  resolves them against the geo taxonomy. It returns `null` for a bare metro search, so we
  render just the directory — never a false city SEO block.
- `locationH1()` produces the server-rendered `"{Place}, {ST} Real Estate & Homes For Sale"`.
- Neighborhood dollar figures come from `getNeighborhoodValues(city, state)`
  (`lib/geo/neighborhood-stats.ts`), backed by the nightly job — see
  [SEO background jobs](/guides/seo-background-jobs).

## Rendering strategy

Geo pages are `export const dynamic = "force-dynamic"` — server-rendered on every request, so
they always reflect live inventory and live counts. `generateStaticParams` enumerates the full
taxonomy so the route set is known and the sitemap is complete. (Roadmap: flip to ISR
`revalidate` once crawl budget justifies it — see the overview's roadmap.)

## Sitemap

`app/sitemap.ts` emits a chunked sitemap via `generateSitemaps()`:

- **Chunk 0** — static pages + market-data + the whole geo tree (county → city → zip →
  neighborhood) + active listings.
- **Chunks 1..N** — remaining listing URLs (50K-URL cap per file).

The geo + neighborhood URLs are generated by walking the same taxonomy the routes use, so the
sitemap can never list a page the routes can't serve (or vice-versa).

## The geo taxonomy

The directory tree is one generated file:

- `lib/geo/heartland-geo.generated.json` — `STATES → counties → cities → { subdivisions, zips }`,
  each node carrying `avgValue` (avg **active** list price) + counts.
- Read through `lib/geo/geo.ts` (`STATES`, `findCityByName`, `geoSlug`, `findSubdivisionByName`).
- Rebuilt by `scripts/build-geo-taxonomy.ts`. `geoSlug()` is the canonical slug function — any
  code that derives a neighborhood/city slug must use it so URLs match the taxonomy.

## How to create a new SEO page

Because everything is data-driven, you rarely write a new route:

1. **New place (city / zip / neighborhood)** — it appears automatically once it's in the geo
   taxonomy. Re-run `scripts/build-geo-taxonomy.ts` to regenerate `heartland-geo.generated.json`;
   the route and its sitemap entry come for free.
2. **New SEO section** (e.g. "Homes with a Pool near {city}") — add it to
   `components/geo/LocationSeo.tsx` with its icon. It then renders on every surface at once.
3. **New page *type*** (a genuinely new route, e.g. facet pages) — add a thin shell under
   `app/.../page.tsx` that resolves params and renders `<SearchResultsPage>`, add its URLs to
   `app/sitemap.ts`, and confirm `LocationSeo` handles the new level. Never fork the results UI.

<Note>
After any `houseloop-web` deploy that changes routes or the sitemap, the change is live on the
next request — no extra step. Only **Inngest function** changes need the re-register step (see
[SEO background jobs](/guides/seo-background-jobs#operating-the-job)).
</Note>
