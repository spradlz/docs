## The doctrine (Zach 2026-06-27)

> **The canonical home is an ACCUMULATOR, not a snapshot.** The IDX feed (Repliers) only ever sends the
> *current* listing — it never carries the home's past. WE are the system of record for the home's banked
> history. One slug = one home = the full life of data, carried forward across every listing cycle. (The
> Zillow/Redfin ZPID model: the property is the permanent record; each listing is just an event on it.)

A home's banked history = **sold history · tax history · GreatSchools · HouseLoop value + value history**
(plus close price, school district). When the same address re-lists under a new MLS#, the new live row
**must inherit everything we already banked** — never render blank because the feed only sent today's snapshot.

**Forward-only.** This work does NOT merge the ~6K legacy duplicate rows (that's the reversible
reconciliation engine, Lee's lane, Rule #0). It makes the *forward* path coalesce correctly so every future
relist carries its history. The only backfill in flight is Caleb's ~2,000 past-client homes — that data is
**ground truth** and is never overwritten by a thinner IDX value.

## Root cause (confirmed in code)

The original sin: ingest keyed homes by the **listing** (`hlid = hl_heartlandmls_<MLS#>`), not the
**property** (address). So a relist forked a new row. The forward resolver (`resolveHome` / `identityKey`)
now prevents new forks — but the **carry-forward of banked data across a relist was broken**:

1. `upsertListingToCache` upserts by `hlid` (`cache.ts:413`). A relist = new MLS# = new hlid → CREATE branch
   → a **brand-new BLANK row**.
2. `collapseHomeCluster` (`cache.ts`) makes the newest cycle the winner and **soft-deletes** the older
   cycles (`deletedAt` + URL alias) — but it never copied the banked enrichment off them. The history was
   stranded on the dead row; the live relist showed blank.
3. The `/home` resolver (`app/home/[slug]/page.tsx`) + sitemap (`app/sitemap.ts`) filtered only
   `deletedAt IS NULL`, **not `quarantinedAt`** — so quarantined dupes stayed resolver candidates + sitemap
   entries, and the live-wins/recency winner could **flip** as statuses changed → an unstable 308 target →
   Google couldn't settle a canonical → SEO bleed.

What was already correct: the **same-row** IDX re-sync is improvement-only (`annualTaxAmount`/`closePrice`/
`schoolDistrict` write only when present; `taxHistory`/`schoolsGreatSchools`/`valueHistory`/`zestimate`
aren't in the update set) → a re-sync never wipes them. **The gap was only the relist → new-row → collapse path.**

## The build

### Build #1 — carry-forward on collapse (`carryForwardEnrichment`)
Before `collapseHomeCluster` retires the older cycles, it COALESCEs their banked fields onto the **winner**:
`closePrice · zestimate(+Low/High) · rentZestimate · annualTaxAmount · schoolDistrict · high/middle/
elementarySchool · taxHistory · schoolsGreatSchools · valueHistory`. Per field, the value comes from the
**newest donor cycle that has it** (`array_agg ORDER BY mlsListedAt DESC FILTER non-null`). **Winner keeps
its own non-null values; we only fill gaps.** Photos are deliberately NOT inherited — a live relist must
show its own current gallery (Repliers refreshes it within seconds), never a sold cycle's stale photos.

### Build #2 — read layer: honor `quarantinedAt` + a stable canonical
- Added `AND "quarantinedAt" IS NULL` next to every `deletedAt IS NULL` in the resolver (3 queries) and the
  sitemap (sold-archive + active queries).
- Added a stable `hlid ASC` tiebreak to the winner `ORDER` so two un-merged cycles with the same status +
  date pick the **same** row every request → the canonical 308 target can't flip.

### The coalesce precedence (the rule, baked into #1 + #2)
- **Current listing facts** (price / status / beds) → the newest row.
- **History / enrichment** → keep-existing-else-inherit.
- **Caleb's past-client backfill = ground truth** → never overwritten by a thinner IDX value.
