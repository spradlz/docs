When a buyer saves a search on `houseloop.ai`, the HouseLoop alert engine surfaces the
matching homes the moment they hit our feed — instantly per new listing, or batched daily,
weekly, or monthly. The website **computes** the matches; the **CRM workflow sends** the
email. Everything runs through Inngest — no manual triggers, no polling crons you manage.

<Note>
**Doctrine (Zach 2026-06-17): "everything goes through a workflow."** The website no longer
sends saved-search emails itself. It runs the match query (`skipSend`) and emits an event;
the CRM's **Saved Search Listing Alerts** workflow (`sys_saved_search_digest`) and the
**New Lead — Saved Search** door workflow are the only senders. This keeps every send visible
on the workflow canvas + in the Run Log (see *Inngest or it doesn't exist*).
</Note>

## How it flows

```
                         ┌─────────────────────────────────────────────┐
 Buyer saves a search    │  WEBSITE (houseloop.ai) — COMPUTES matches   │
 OR registers via a      │                                              │
 saved search            │  POST /api/saved-searches  (explicit save)   │
        │                │      → sendSavedSearchAlert(skipSend)        │
        ▼                │      → pushLeadToCRM(action:'saved_search')  │
 SavedSearch row created │                                              │
 (createdBy 'lead' or    │  seedStarterSavedSearch  (register-via-      │
  'agent')               │      saved-search, CRM lead-intake)          │
                         └───────────────────────┬─────────────────────┘
                                                 │  emits Inngest events
                                                 ▼
                         ┌─────────────────────────────────────────────┐
                         │  CRM (crm.houseloop.ai) — SENDS the email    │
                         │                                              │
                         │  New Lead — Saved Search door workflow       │
                         │      (on-create: branded email + Myli + text)│
                         │  sys_saved_search_digest workflow            │
                         │      (digests: branded matches email)        │
                         └─────────────────────────────────────────────┘
```

The website's job is to **find the homes**. The CRM's job is to **send the message**. The two
are bridged by Inngest events, never by the website calling Resend for a digest.

## The match engine (the `firstSeenAt` law)

`sendSavedSearchAlert(savedSearchId, opts)` in
`apps/houseloop-web/app/api/saved-searches/alert/logic.ts` is the ONE engine every path uses.

1. Loads the `SavedSearch` row + the owner's email (JOIN on `User`). Skips if inactive, no
   delivery email, or no user email.
2. Parses the search's own `searchUrl` through the **exact** engine the website uses
   (`parseSearchParams` + `searchFromDb`), so the alert matches what the buyer sees on
   `houseloop.ai` — including feature filters (`feature=pool`, etc.).
3. **Digest floor (`effSince`):** keeps only homes whose **`firstSeenAt >= effSince`** — i.e. the
   homes that arrived in *our feed* in this window. **Not `listDate`.** `firstSeenAt` is set once
   on create and never updated; it is the source of truth for "newest." `listDate` is the MLS
   market date (date-only) and must never gate alerts (see Dev Log 2026-06-28).
4. Resolves cover photos from `Listing.photos` (`{ primary, all, count }`).
5. **`skipSend` (the default for every automated path):** compute the matches and **return** them
   — send nothing, stamp nothing. The CRM workflow owns the email. The legacy direct-Resend send
   (`HouseLoop <updates@houseloop.ai>`) remains only for the manual `/api/saved-searches/alert`
   HTTP wrapper.
6. On a real send (non-`skipSend`): updates `lastAlertSentAt`, `lastMatchCount`, `totalMatchesSeen`.

<Note>
The alert always queries our **own `Listing` table** (centerbeam DB), never Repliers live. Results
reflect our cached MLS data, as current as the last Repliers webhook/sync.
</Note>

## The send chain (compute → emit → workflow)

`emitDigestDue()` (in `saved-search-alert.ts`) runs the match query with `skipSend`, and if there
is at least one new match, emits:

```
houseloop/saved-search.digest-due
  { email, match_count, search: { name, url, criteria, matches[] } }
```

The CRM consumes this in
`apps/houseloop-crm/lib/inngest/functions/saved-search-digest-due.ts`, resolves the lead by email,
and fires the **`sys_saved_search_digest`** ("Saved Search Listing Alerts") workflow — whose
branded email step is the only thing that actually sends. After emitting, the watermark advances
(`lastAlertSentAt = now()`, `lastMatchCount = matches`) so the next tick only picks up newer homes.

## Inngest functions

All live in `apps/houseloop-web/lib/inngest/functions/saved-search-alert.ts` and are registered in
`apps/houseloop-web/app/api/inngest/route.ts`.

| Function ID | Trigger | Audience / role |
| --- | --- | --- |
| `saved-search-on-create` | event `houseloop/saved-search.created` | **NO-OP.** The *New Lead — Saved Search* door workflow owns the on-create email. |
| `saved-search-instant-check` | **event `houseloop/idx.event`** (a new/changed listing) | `alertFrequency = 'instant'` — fires the hour a matching home hits our feed |
| `saved-search-daily-alert` | cron `0 13 * * *` (8am CT daily) | `alertFrequency = 'daily'` |
| `saved-search-weekly-alert` | cron `0 13 * * 1` (Mon 8am CT) | `alertFrequency = 'weekly'` |
| `saved-search-monthly-alert` | cron `0 13 1 * *` (1st, 8am CT) | `alertFrequency = 'monthly'` |

<Warning>
`saved-search-instant-check` is **event-driven, not a cron.** It runs on every
`houseloop/idx.event` (emitted by the Repliers webhook at
`apps/houseloop-web/app/api/repliers/webhook/route.ts` on `listing.created` / `listing.updated`).
For each instant search it checks for homes that arrived since the last send. A **never-sent**
search uses a 7-day catch-up floor for its first run; a sent search uses its `lastAlertSentAt`
watermark, so arrivals between checks are never missed and never double-sent.
</Warning>

### The digest gate

Every cron + the instant check first calls `digestGateActive()`, which reads the CRM's
`/api/workflows/digest-gate`. If the **Saved Search Listing Alerts** workflow is paused in Lead
Flow, every alert path here goes quiet. The gate **fails open** (treats unreachable as active).

## How a saved search is created

There are two creation paths, and both end with the CRM sending the email:

**1. Explicit save (`createdBy = 'lead'`).** Buyer clicks **Save Search** on `/listings`.
`POST /api/saved-searches` creates the row (defaults: `alertFrequency = 'instant'`,
`deliveryEmail = true`), runs `sendSavedSearchAlert(skipSend, isConfirmation)` to compute the
first-look matches, and calls `pushLeadToCRM({ action: 'saved_search', ... })`. The CRM's
**New Lead — Saved Search** door workflow sends the branded confirmation + matches email, the Myli
personal email, and the text.

**2. Registration via a saved search (`createdBy = 'agent'`).** When a lead registers and we have
real browsing signal, `seedStarterSavedSearch()` in
`apps/houseloop-crm/app/api/integrations/website/lead-intake/route.ts` derives a starter search
from the homes they actually viewed (`deriveSearchCriteriaFromPixel`) and raw-inserts an
`instant`, `createdBy = 'agent'` row into the web `SavedSearch` table. It is a silent insert (no
on-create email) — the `saved-search-instant-check` (with its never-sent 7-day catch-up) delivers
the recent matches on the next `idx.event`, then real-time after.

## SavedSearch data model

```prisma
model SavedSearch {
  id               String    @id @default(cuid())
  userId           String
  name             String
  searchCriteria   Json      // { city, state, zip, minPrice, maxPrice, beds, baths, ... }
  searchUrl        String    // e.g. "/listings?city=Olathe&minPrice=400000&minBeds=5"
  alertFrequency   String    @default("instant")  // instant | daily | weekly | monthly | never
  deliveryEmail    Boolean   @default(true)
  deliveryPush     Boolean   @default(false)
  isActive         Boolean   @default(true)
  createdBy        String    @default("lead")      // lead | agent (agent = seeded from browsing)
  lastAlertSentAt  DateTime?                        // watermark — advances on every emit
  lastMatchCount   Int       @default(0)            // homes the LAST alert sent (the RECENT column)
  totalMatchesSeen Int       @default(0)
  lastOpenedAt     DateTime?                         // Resend engagement (the god-tier card)
  lastClickedAt    DateTime?
  lastEmailId      String?
  createdAt        DateTime  @default(now())
  updatedAt        DateTime  @updatedAt
}
```

## The Saved Searches surface

`/saved-searches` (and the `/updates` view) render from
`apps/houseloop-web/app/api/saved-searches/enriched/route.ts`, which returns one card per active
search:

| Column | Source |
| --- | --- |
| **RESULTS** | live `totalCount` from the same query the card links to (parses `searchUrl`'s querystring so the count is byte-identical to the listing page) |
| **RECENT** | **`lastMatchCount`** — the homes the *last alert sent* (Zach 2026-06-28: "the recent has to be the homes that were sent"). Was a `daysOnMarket<=1` query that never reflected the email. |
| **STARTED** | `createdAt` |
| **FREQ** | `alertFrequency` |
| **LAST SENT** | `lastAlertSentAt` |

In the CRM, the same data drives the **Saved Searches** card on the lead (the "god-tier" card with
Resend opens/clicks/last-sent/frequency/match count). The card's count = real saved-search rows,
never website *search* (pixel) events.

## Email template

The branded matches email is rendered + sent by the **CRM workflow** (`sys_saved_search_digest` /
the New Lead — Saved Search door). The website template
`apps/houseloop-web/lib/email/saved-search-alert.ts` (search name + total match count header, up to
8 listing cards with a "fresh" badge for homes listed in the last 24h, a **View all N homes** CTA,
and an unsubscribe link) backs the legacy manual `/api/saved-searches/alert` HTTP path only.
