## The goal

An agent opens a lead in **TC view** and clicks **Add Transaction**. A native intake funnel — the Cognito-equivalent,
but in-app — **auto-fills from the lead** (name, phone, address, agent) **and the AI-created buyer-agency loop**,
captures the transaction (contract, EMD, inspections, financing, commission) + the **key dates**, and on save it
**creates/links the loop, sets the anchor dates, records the deal, and projects the dated stepper.** Buyers or
sellers. One flow. We own the data.

## The problem it fixes (current state)

Today the TC surfaces are fragmented — three unlinked pieces:

| Piece | Where it is now | Gap |
|---|---|---|
| **Create Loop** | `TcCreateLoopPrompt` — shows in TC view ONLY when the lead has *no* `hl_transactions` row | Hidden the moment any row exists (even a deal); no one can find "create new loop" |
| **Add Deal** | Deals tab → `DealMilestoneModal` (Pending / Closed / Lost) → a separate `hl_transactions` row | Creates a *deal*, never a *loop*; no intake, no dates |
| **Move to Under Contract** | opens the deal sheet, creates a deal row, commits the stage | Does **not** create/link a loop or write any milestone dates |

**Loops and deals are separate, unlinked rows** in `hl_transactions` produced by two independent flows. Nothing
joins them, nothing captures the intake, and the stepper has no dates.

<Note>
**The canonical transaction record is `hl_transactions`.** Both loops and deals live in this one table — it was
renamed from `hl_lead_deals` in migration 0241, so any older doc/code citing an `hl_lead_deals` table is stale
(the TC transactions API reads/writes `hl_transactions`:
`apps/houseloop-crm/app/api/tc/transactions/route.ts`). The whole point of this plan is to stop the two flows
producing two orphan rows and instead **link them into one transaction**.
</Note>

## The target — one "Add Transaction" flow

**One entry point** replaces the hidden Create-Loop prompt: an **Add Transaction** button on the TC Dashboard band
(and the band's empty state). It — and moving a lead to **Under Contract** — open the **same native intake funnel**:

<Steps>
<Step title="Pre-fill">
From the lead (`hl_leads`: name, phone, address, agent) + the AI-created loop if one exists (side, price, participants).
</Step>
<Step title="Capture">
The intake (Contract · EMD · HOA · Inspections · Financing · Commission — the [Cognito schema](/guides/cognito-forms-api))
+ the two **anchor key dates**: **Effective / Under Contract** and **Closing**.
</Step>
<Step title="Create / link the loop">
`createLoop` (if none for that side) seeds the 14/16 milestones + tasks from the side templates.
</Step>
<Step title="Set dates → project the stepper">
Write the anchor dates to the loop → the **date engine** projects every other milestone/task date (`anchor ± offset`)
→ the dated stepper renders.
</Step>
<Step title="Record + link the deal">
The deal (commission / close) is recorded and **linked to the loop** — deal + loop become ONE transaction, not two orphan rows.
</Step>
</Steps>

Same funnel for buyers and sellers; same funnel whether triggered by the button or the Under-Contract stage move.

## The Under Contract trigger (planned — NOT built)

The half of this plan that fires the funnel automatically is the **Under Contract stage trigger**. Today it is
spec only — the funnel exists but is only reachable manually via the band's **View / Add Transactions** hub, and
only Step 1 (Contract & Dates) is wired.

**The one idea:** moving a lead to Under Contract IS the moment a real transaction begins — so that stage move
should open the full TC intake funnel, not a one-row deal capture. This is the "section that's supposed to pop up
when you hit Under Contract."

### Where it fits — the two-chapter loop

It's the hinge between the two chapters of the TC Loop band:

1. **Chapter 1 — Shopping** (the Agency Loop): agreement prep → sent → signed. No transaction chrome.
2. **→ Under Contract** — the stage move fires **this funnel**.
3. **Chapter 2 — Contract** (the Property Loop): the funnel's anchor dates drive Under Contract → EMD → … → Possession.

The Under-Contract gate node in the shopping stepper ("→ Under Contract · starts loop") is the visual promise of
this trigger.

### The trigger (unbuilt)

| Today | Planned |
|---|---|
| Stage → Under Contract → **DealMilestoneModal** (5 fields → an `hl_transactions` deal row) | Stage → Under Contract → **the TC intake funnel** (full intake → creates/links the **loop** + records + links the deal) |
| Fires from `PipelineChipDropdown` / `CenterLane` intercept matrix | Same intercept, but routes Under Contract to the funnel instead of the deal modal |
| Captures a deal, not a transaction; no loop, no dates, no stepper | Captures the transaction, writes the 2 anchors, projects the dated stepper |

The intercept matrix lives in [Lead Detail — Stages, Deals & Audit Trail](/guides/lead-detail-stages-and-deals)
(`Anything → Under Contract`). Wiring this feature means that row opens the funnel.

### The pre-fill (two sources)

On open, the funnel auto-fills from **two sources**:

- **The lead** (`hl_leads`): name, phone, property address, assigned agent, lead source. ✅ built
  (`AddTransactionFunnel` `prefill` prop).
- **The selected loop** (`hl_transactions` + children): side, agency price, participants, and any agency-chapter
  data already captured. ⛔ **This loop-side pre-fill is NOT built.**

### The deep intake (Steps 2–6 — unbuilt)

The funnel's Steps 2–6 are the ~140-field intake, rebuilt natively from the
[Cognito Forms schema](/guides/cognito-forms-api) (TC Intake Form Buyers #103 / Sellers #108). Today they render
as **structural placeholders** — the deep field capture was deferred to "the Transaction Summary pass" and never
built. The A-Frame `Collection → Group → Field` map (Property · EMD · HOA · Inspections · Financing · Commission)
is documented in the Cognito guide.

### On save — what it must do

1. **Create / link the loop** — `createLoop` if none for that side (seeds the side's milestones + tasks), else link.
2. **Write the two anchor dates** — `EffectiveDate` (Under Contract) + `ClosingDate` → `hl_tc_loop_event.event_date`
   via `PATCH /api/tc/loops/[id]/dates` (built), so the date engine projects the rest of the stepper.
3. **Record + link the deal** — the deal milestone row (in `hl_transactions`) is created AND linked to the loop
   (one transaction, not two orphan rows). ⛔ The deal↔loop link is NOT built.
4. **Write the audit row** — `deals.under_contract` to `hl_audit_log`, same as today.

## Where the data lives (own it)

The intake is stored natively in `hl_tc_*` using the A-Frame-style **Collection → Group → Field** model
(config-not-code), seeded from the [Cognito schema](/guides/cognito-forms-api). Cognito's New Listing / Status Change
/ Pending Transaction forms map the same way as future native forms. The [key dates](/guides/cognito-forms-api)
(`ContractDetails.EffectiveDate` + `ClosingDate`) are the anchors the stepper runs on.
