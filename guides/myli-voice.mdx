## The thesis

**Myli answers THE FRONT DOOR.** One AI voice agent on ONE line (913-675-9877, which every other number
forwards into) — she greets, reads intent, filters spam, books showings and listing appointments,
live-transfers hot leads to Zach or Ana, and drops every call into HouseLoop CRM as a lead with the right
disposition. Four tools: capture caller · triage (buyer / seller / vendor / spam + urgency → disposition) ·
notify Zach · take message.

**The honest reuse note:** `apps/lee-voice-agent/agent.py` (LiveKit + Gemini 2.5 Flash native speech-to-speech,
sub-500ms, function tools) is the **framework** we reuse — but it is a *recruiting* agent today: its prompt and
tools get replaced wholesale, not re-pointed. Less than "build a voice stack," more than "swap a config."

## How a call flows

Every line lands on the same agent; the dialed number picks the personality.

```
caller dials any of the 4 lines
   │
   ▼
Twilio  (owned numbers + the legacy forward line)
   │  SIP trunk (caller-ID preserved)
   ▼
LiveKit → Myli  (Gemini native speech-to-speech)
   │  reads the DIALED number → loads that line's prompt + intro
   ▼
   greet · qualify intent · buy / sell / spam / other
   │
   ├─ HOT (buy or sell) ─▶ live_transfer to Zach/Ana  +  create_lead(hot)  +  notify
   ├─ wants an appointment ─▶ book_showing / book_listing_appt  +  create_lead
   ├─ spam ─▶ flag_spam  (no lead, logged)
   └─ other ─▶ schedule_callback  +  create_lead
   │
   ▼
HouseLoop CRM — lead + call record + recording + notification  (mirrored into FUB via API)
```

<Tip>
**This fixes the phone mess too.** Every call now flows through *your* Twilio → the `phone_call_received`
notifications finally fire, the records are yours, and Myli is the front door instead of a voicemail. The
three-source problem collapses to one pipe.
</Tip>

## Myli's call-handling tools

Function tools she calls mid-conversation — each a thin wrapper over a CRM API that already exists.

<CardGroup cols={2}>
  <Card title="live_transfer" icon="fire">Hot buyer/seller → Twilio warm-transfers live to Zach/Ana, and flags the lead HOT so you see "take this now."</Card>
  <Card title="book_showing / book_listing_appt" icon="calendar-check">Reads intent + address, offers times, books it on the calendar, confirms by text.</Card>
  <Card title="create_lead" icon="user-plus">Every call = a lead in HouseLoop with source, the line it came in on, and the recording attached.</Card>
  <Card title="flag_spam" icon="ban">Detects solicitors / robocalls, ends politely, logs it — never a junk lead or a false notification.</Card>
  <Card title="schedule_callback" icon="phone-arrow-up-right">Not-now callers → captures the ask, books a callback task, notifies the right person.</Card>
  <Card title="mirror_to_fub" icon="arrows-rotate">Writes the call back to FUB's timeline (attribution + recording) via `POST /v1/calls` so it still lives in FUB.</Card>
</CardGroup>

## Effort &amp; phasing

The agent is a **re-point, not a rebuild** — roughly a week of focused work to get Myli answering + logging +
transferring. The calendar gate is the ports (1-2 weeks, mostly waiting on carriers). So start Phase 1 on the
legacy line now, kick the port requests in parallel.

<Steps>
  <Step title="Phase 1 · days — Myli picks up + logs">
    Legacy line live (forward, no porting) · Myli answers with the per-line intro · `create_lead` + recording +
    notification. Proves the full pipe end-to-end.
  </Step>
  <Step title="Phase 2 · 1–2 weeks — intent + transfer + book">
    Buy/sell/spam/other classification · `live_transfer` on hot · `book_showing` / listing appt · the 3 ported
    owned lines join as they land.
  </Step>
  <Step title="Phase 3 · ongoing — full autonomous Myli">
    Handles the whole interaction · gathers criteria, qualifies budget/timeline · books callbacks, nurtures ·
    `mirror_to_fub` for full continuity.
  </Step>
</Steps>
