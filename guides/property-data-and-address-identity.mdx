Every HouseLoop home — coming soon, active, under contract, pending, or sold — is **one row in one table**, identified by its **address**. Not by the MLS number, not by a per-source id. This page is the source of truth for how property data is keyed, how addresses are normalized, and how an incoming address resolves to the one home we already own.

## The one table

There is exactly **one property table: `Listing`**. A home is a single row for its entire life; the lifecycle is a `status` field that flips on that same row:

```
coming_soon → active → pending → sold        (status changes; same row, same URL)
```

- **Never a new row** for a status change, **never a second slug**, **never a per-source id as identity.**
- `mlsNumber`, `zpid`, `parcelNumber` are **columns** (source references), not identity.
- Sold homes are just rows with `status='sold'`. (Historically these lived in a separate `ListingHistory` archive; that table was absorbed into `Listing` and renamed `ListingHistory_deprecated` — a reversible backup. See [Sold Data & Price History](/guides/sold-data-and-price-history).)

Reads filter the live, visible set with `deletedAt IS NULL AND quarantinedAt IS NULL` (and `status` as needed). `quarantinedAt` is a non-destructive "hide this row" flag used to retire duplicates without deleting data.

## Identity = the canonical address

A home's identity is its **canonical, normalized address** — because the address is the one thing that's true across every data source (MLS, Zillow, county), while each source has its own id (Zillow has no MLS number, it has a `zpid`; the county has a `parcelNumber`). Keying on any single source id breeds duplicate rows for the same house.

### Layer 1 — deterministic normalization (offline, free)

`lib/listings/canonical-address.ts` standardizes a raw address with zero API calls (runs at render time, in the sitemap, and as the redirect resolver):

- **Street suffix → one full word:** `Rd`/`Rd.`/`Road` → `road`; `St` → `street`; `Ave` → `avenue`; etc.
- **Directionals → one abbreviation:** `North`/`N` → `n`.
- **Unit fragments dropped** from the street line.

```
canonicalSlug({ street: "1230 W 31st St", city: "Independence", state: "MO", zip: "64055" })
  → "1230-w-31st-street-independence-mo-64055"
```

Same physical home → same slug, regardless of which spelling a source sent. This handles ~97% of addresses.

### The URL contract

- Each home renders at **`/home/{canonicalSlug}`** — one canonical URL per home.
- A request for a non-canonical variant (`…-rd-…` when canonical is `…-road-…`) **308-redirects** to the canonical URL, and the page sets `rel=canonical`.
- The sitemap emits **one URL per home** (distinct `canonicalSlug`).

## Address as a USPS-conformed unique ID

The deterministic layer can't fix everything: two sources may disagree on the **city** ("Baldwin" vs "Baldwin City" for the same ZIP) or on a suffix abbreviation, producing two slugs for one home. The U.S. Postal Service is the real ground truth for "is this a single, deliverable address," so addresses conform to USPS. This is the `lib/listings/address-identity.ts` service.

### Layer 2 — the city-independent identity key (free)

`identityKey({street, unit, city, state, zip})` = **canonical street + unit + 5-digit ZIP — city deliberately excluded.** Because the ZIP already pins the locale, "Baldwin" and "Baldwin City" produce the **same key**. This is the match/dedup key; it never depends on the city label. It is persisted on every `Listing` row as the `identityKey` column (indexed), so the resolver can match against it directly.

The **USPS-preferred city** for a ZIP comes from `zip-preferred-city.generated.json` (built by `scripts/build-zip-preferred-city.mjs` from the most-common city per ZIP among real MLS rows — Heartland MLS feeds USPS-standard cities, so this is free and self-maintaining). `preferredCity(zip)` resolves it, and `canonicalSlugPreferred()` builds the URL slug with it — so a home always slugs to one consistent, USPS-correct city.

<Note>
**Why city is never part of the key.** Kansas City-metro ZIP codes straddle municipal lines: ZIP `66215` is **Lenexa** per USPS but **Overland Park** per Zillow; `66216` shows as both **Shawnee** and **Shawnee Mission**; `64118` as both **Kansas City** and **Gladstone**. If the city were part of the match key, the same physical home would split into two records the moment two sources disagreed on the label. The ZIP pins the geography and the house number pins the home — the city is a display attribute, never an identifier.
</Note>

### Layer 3 — CASS validation (paid, reserved)

`validateAddress(parts)` standardizes an address against USPS via a pluggable provider — **Smarty** (CASS-certified, bulk-friendly) → **Google Address Validation** (we already hold the key) → **offline**. It returns the USPS-standard address, the preferred city, ZIP+4, and the **DPV delivery-point key** (`ZIP5 + ZIP4 + delivery-point` = a unique mailable address — the closest thing to a national primary key). It never throws; with no provider configured it degrades to the deterministic offline result, so identity is always resolvable.

**Cost doctrine:** never blind-validate the whole ~430K base through a paid provider. The free Layer 1 + Layer 2 pass collapses ~95%; paid CASS is reserved for **new ingest** (one cheap call per new home) and the **ambiguous residual** (the handful Layer 1+2 can't disambiguate).

## Resolving an address to one home — the resolver

Identity (above) answers *"what is this home's key?"* The **resolver** answers the operational question: *given an incoming address, which of our owned homes is this — or is it new?* It's the front door every home passes through, whether it arrives from the live Repliers feed or a one-time backfill.

`resolveHome(db, input)` (`lib/listings/resolve-home.ts`) returns a verdict — `matched` (with the owned `hlid`) or `mint` (it's new) — using three rungs, cheapest first:

1. **Identity key** — match an existing owned home on `identityKey()` (city-free street + ZIP). Handles the vast majority.
2. **Physical point** — when the key misses but coordinates are present, match on **lat/lng within ~25 m**. A house can't move, so two address strings on the same rooftop are the same home. Coordinates arrive free from the Repliers feed (`map.latitude/longitude`); a geocoder is only needed for backfill rows that lack them.
3. **Mint** — no owned home matches → this is a new home; create one row and stamp its identity.

**Where it runs.** The resolver is wired into the listing ingest path (`lib/listings/cache.ts`): every Repliers webhook/sync resolves the incoming address **before** the upsert. If we already own the home under a different `hlid` (a relist under a new MLS number), the write targets that owned row — so the home updates in place instead of forking a duplicate. New rows get their `identityKey` stamped on insert.

<Warning>
**Resolve by address, not by MLS number.** The same physical home is routinely relisted under a **new MLS number** — sometimes years apart, sometimes with a different city label (one listing says *Lenexa*, a later one says *Overland Park* for the same ZIP). A pipeline that keys homes on the MLS number forks a second row every time that happens. Resolving by **address identity** keeps one physical home as **one row** — carrying its full listing history — instead of a pile of duplicates.
</Warning>

## Data enrichment — the value/schools/tax base

Consumer data that the MLS feed doesn't carry — the home **value estimate**, **GreatSchools** ratings, and **tax history** — is sourced from Zillow data (via the OpenWeb Ninja API) **once, when the home is first resolved.** That establishes the home's **base**. The home is not re-queried against Zillow afterward: the displayed value over time, the value-history chart, and the sold close-out are all derived from **our own data** — the MLS record plus our geographic appreciation model (`lib/valuation/appreciation.ts`) — anchored on that one-time base.

Structural facts (beds/baths, square footage, year built, style, lot, features, HOA, assigned schools) come from the **MLS record via Repliers**; the **sold price and date** come from the **MLS** (and are display-restricted in non-disclosure states). Each field has exactly one authoritative source. See [Home Valuation](/guides/home-valuation-dashboard) and [Sold Data & Price History](/guides/sold-data-and-price-history).
