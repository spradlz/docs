Every HouseLoop home ‚Äî coming soon, active, under contract, pending, or sold ‚Äî is **one row in one table**, identified by its **address**. Not by the MLS number, not by a per-source id. This page is the source of truth for how property data is keyed and how addresses are normalized.

## The one table

There is exactly **one property table: `Listing`**. A home is a single row for its entire life; the lifecycle is a `status` field that flips on that same row:

```
coming_soon ‚Üí active ‚Üí pending ‚Üí sold        (status changes; same row, same URL)
```

- **Never a new row** for a status change, **never a second slug**, **never a per-source id as identity.**
- `mlsNumber`, `zpid`, `parcelNumber` are **columns** (source references), not identity.
- Sold homes are just rows with `status='sold'`. (Historically these lived in a separate `ListingHistory` archive; that table was absorbed into `Listing` and renamed `ListingHistory_deprecated` ‚Äî a reversible backup. See [Sold Data & Price History](/guides/sold-data-and-price-history).)

Reads filter the live, visible set with `deletedAt IS NULL AND quarantinedAt IS NULL` (and `status` as needed). `quarantinedAt` is a non-destructive "hide this row" flag used to retire duplicates without deleting data.

## Identity = the canonical address

A home's identity is its **canonical, normalized address** ‚Äî because the address is the one thing that's true across every data source (MLS, Zillow, county), while each source has its own id (Zillow has no MLS number, it has a `zpid`; the county has a `parcelNumber`). Keying on any single source id breeds duplicate rows for the same house.

### Layer 1 ‚Äî deterministic normalization (offline, free)

`lib/listings/canonical-address.ts` standardizes a raw address with zero API calls (runs at render time, in the sitemap, and as the redirect resolver):

- **Street suffix ‚Üí one full word:** `Rd`/`Rd.`/`Road` ‚Üí `road`; `St` ‚Üí `street`; `Ave` ‚Üí `avenue`; etc.
- **Directionals ‚Üí one abbreviation:** `North`/`N` ‚Üí `n`.
- **Unit fragments dropped** from the street line.

```
canonicalSlug({ street: "1230 W 31st St", city: "Independence", state: "MO", zip: "64055" })
  ‚Üí "1230-w-31st-street-independence-mo-64055"
```

Same physical home ‚Üí same slug, regardless of which spelling a source sent. This handles ~97% of addresses.

### The URL contract

- Each home renders at **`/home/{canonicalSlug}`** ‚Äî one canonical URL per home.
- A request for a non-canonical variant (`‚Ä¶-rd-‚Ä¶` when canonical is `‚Ä¶-road-‚Ä¶`) **308-redirects** to the canonical URL, and the page sets `rel=canonical`.
- The sitemap emits **one URL per home** (distinct `canonicalSlug`).

## Address as a USPS-conformed unique ID

The deterministic layer can't fix everything: two sources may disagree on the **city** ("Baldwin" vs "Baldwin City" for the same ZIP) or on a suffix abbreviation, producing two slugs for one home. The U.S. Postal Service is the real ground truth for "is this a single, deliverable address," so addresses conform to USPS. This is the `lib/listings/address-identity.ts` service.

### Layer 2 ‚Äî the city-independent identity key (free)

`identityKey({street, unit, city, state, zip})` = **canonical street + unit + 5-digit ZIP ‚Äî city deliberately excluded.** Because the ZIP already pins the locale, "Baldwin" and "Baldwin City" produce the **same key**. This is the match/dedup key; it never depends on the city label.

The **USPS-preferred city** for a ZIP comes from `zip-preferred-city.generated.json` (built by `scripts/build-zip-preferred-city.mjs` from the most-common city per ZIP among real MLS rows ‚Äî Heartland MLS feeds USPS-standard cities, so this is free and self-maintaining). `preferredCity(zip)` resolves it, and `canonicalSlugPreferred()` builds the URL slug with it ‚Äî so a home always slugs to one consistent, USPS-correct city.

### Layer 3 ‚Äî CASS validation (paid, reserved)

`validateAddress(parts)` standardizes an address against USPS via a pluggable provider ‚Äî **Smarty** (CASS-certified, bulk-friendly) ‚Üí **Google Address Validation** (we already hold the key) ‚Üí **offline**. It returns the USPS-standard address, the preferred city, ZIP+4, and the **DPV delivery-point key** (`ZIP5 + ZIP4 + delivery-point` = a unique mailable address ‚Äî the closest thing to a national primary key). It never throws; with no provider configured it degrades to the deterministic offline result, so identity is always resolvable.

**Cost doctrine:** never blind-validate the whole ~430K base through a paid provider. The free Layer 1 + Layer 2 pass collapses ~95%; paid CASS is reserved for **new ingest** (one cheap call per new home) and the **ambiguous residual** (the handful Layer 1+2 can't disambiguate).

## Key files

| Concern | File |
|---|---|
| Deterministic normalizer (suffix/directional) | `lib/listings/canonical-address.ts` |
