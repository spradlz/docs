## The one table (the target model)

There is one property table: `Listing`. The intended model is that a home is a single row for its entire life, and the lifecycle is a `status` field that flips on that same row:

```
coming_soon → active → pending → sold        (status changes; same row, same URL)
```

- A status change should **never** create a new row, a second slug, or use a per-source id as identity.
- `mlsNumber`, `zpid`, `parcelNumber` are **columns** (source references), not identity.
- Sold homes are rows with `status='sold'`. (Historically these lived in a separate `ListingHistory` archive; that table was absorbed into `Listing` and renamed `ListingHistory_deprecated` — a reversible backup. See [Sold Data & Price History](/guides/sold-data-and-price-history).)

Reads filter the live, visible set with `deletedAt IS NULL AND quarantinedAt IS NULL`. `quarantinedAt` is a non-destructive "hide this row" flag used to retire a duplicate without deleting data.

## Identity = the canonical address

A home's identity is its **canonical, normalized address** — the one thing true across every data source (MLS, Zillow, county), while each source has its own id. Keying on a single source id breeds duplicate rows for the same house (which is exactly how the historical duplicates were created — see below).

### Layer 1 — deterministic normalization (offline, free)

`lib/listings/canonical-address.ts` standardizes a raw address with zero API calls (runs at render time, in the sitemap, and as the redirect resolver):

- **Street suffix → one full word:** `Rd`/`Rd.`/`Road` → `road`; `St` → `street`; `Ave` → `avenue`.
- **Directionals → one abbreviation:** `North`/`N` → `n`.
- **Unit fragments dropped** from the street line.

```
canonicalSlug({ street: "1230 W 31st St", city: "Independence", state: "MO", zip: "64055" })
  → "1230-w-31st-street-independence-mo-64055"
```

Same physical home → same slug, regardless of which spelling a source sent. Handles ~97% of addresses.

### The URL contract

- Each home renders at **`/home/{canonicalSlug}/{HLID}`** — the address slug for humans + Google, the HLID for permanence (the Zillow/Redfin `…/<zpid>` pattern). One canonical URL per home.
- The **address-only** `/home/{canonicalSlug}` and any non-canonical variant (`…-rd-…` when canonical is `…-road-…`, an old slug in `hl_slug_alias`, an old `HLP-`/HLID in `hl_hlid_alias`) **301-redirects** to the canonical id-bearing URL; the page sets `rel=canonical`.
- The sitemap emits **one URL per home** — `/home/{canonicalSlug}/{HLID}`.
- The page **heading shows the Home ID** (`HLID123456`) under the address, so it's visible like a `zpid`.

## Address as a (would-be) USPS-conformed unique ID

The deterministic layer can't fix everything: two sources may disagree on the **city** ("Baldwin" vs "Baldwin City" for the same ZIP) or on a suffix abbreviation. USPS is the ground truth for "is this a single deliverable address," so the *intent* is that addresses conform to USPS. This is the `lib/listings/address-identity.ts` service.

<Warning>
**USPS/CASS conformance is currently OFF in production.** `validateAddress` has no `SMARTY_AUTH_ID/TOKEN` (and no Google fallback configured), so it runs in **offline** mode — addresses are normalized deterministically (Layer 1/2) but are **not** USPS/CASS-corrected. That means a source's **raw ZIP becomes part of the identity** as-is; nobody is correcting Zillow's `64119` to the postal-correct `64118`. This is precisely why ZIP drift (below) is unmitigated for any source that lacks coordinates. The "USPS-conformed id" is the intended model, not today's reality.
</Warning>

### Layer 2 — the city-independent identity key (free)

`identityKey({street, unit, city, state, zip})` = **canonical street + unit + 5-digit ZIP — city deliberately excluded.** The ZIP pins the locale, so "Baldwin" and "Baldwin City" produce the **same key**. It is persisted on every `Listing` row as the indexed `identityKey` column, so the resolver matches against it directly.

The **USPS-preferred city** for a ZIP comes from `zip-preferred-city.generated.json` (built from the most-common city per ZIP among real MLS rows). `preferredCity(zip)` resolves it and `canonicalSlugPreferred()` builds the URL slug with it.

<Note>
**Why city is never part of the key.** KC-metro ZIPs straddle municipal lines: `66215` is **Lenexa** per USPS but **Overland Park** per Zillow; `66216` shows as both **Shawnee** and **Shawnee Mission**. If the city were in the match key, the same home would split into two records the moment two sources disagreed on the label. The ZIP pins the geography and the house number pins the home — the city is a display attribute, never an identifier.

**But the ZIP is in the key, and ZIP *also* drifts between sources.** City drift is solved by excluding city; ZIP drift was the same problem one level down — **now solved** by the de-fork engine (coordinate-pin + the intake resolver). See [the ZIP-drift fork](#the-houseloop-property-id-and-the-zip-drift-fork) and [De-Forking](#de-forking).
</Note>

### Layer 3 — CASS validation (paid, reserved — and currently OFF)

`validateAddress(parts)` standardizes against USPS via a pluggable provider — **Smarty** → **Google Address Validation** → **offline** — returning the standardized address, preferred city, ZIP+4, and the **DPV delivery-point key**. It never throws; with no provider it degrades to the offline result. **Today there is no provider configured, so it always returns the offline result** (see the warning above). Intended for new ingest + the ambiguous residual, never a blind sweep of the whole base.

## The resolver — matching an incoming listing to one owned home

`resolveHome(db, input)` (`lib/listings/resolve-home.ts`) returns a verdict — `matched` (with the owned `hlid`) or `mint` (it's new) — using three rungs:

1. **Identity key** — match an existing owned home on `identityKey()` (city-free street + ZIP).
2. **Physical point** — when the key misses but coordinates are present, match on **lat/lng within ~25 m**.
3. **Mint** — no owned home matches → create one row and stamp its identity.

**Where it runs (live, forward).** The resolver is wired into the ingest path (`lib/listings/cache.ts`): every Repliers sync resolves the incoming address **before** the upsert. If we already own the home under a different `hlid` (a relist under a new MLS number), the write targets that owned row — so going forward the home updates in place instead of forking a new duplicate.

<Note>
**Rung 2 (lat/lng) is what actually holds single-slug together — not USPS.** Because USPS conformance is off, rung 1 (`identityKey`) keys on the raw source ZIP and *misses* across ZIP drift. What rescues the listing path is rung 2: **100% of `Listing` rows carry coordinates** (Repliers supplies them free), so two ZIP-drifted records on the same rooftop still collapse to one home via the ~25 m match. **Any source without coordinates loses this safety net** — see below.
</Note>

<Warning>
**Why duplicates existed.** The ingest historically upserted by `hlid = hl_heartlandmls_<mlsNumber>` — keyed on the MLS number, not the address — so every relist under a new MLS number forked a second row. The resolver fixes this **for new ingests**; the **pre-existing** duplicate rows are collapsed by the reconciliation engine below.
</Warning>

## The HouseLoop Property ID and the ZIP-drift fork

> **The numbers in this section are superseded** — see the 🔒 LOCKED block at the **top of this page** (HLID naming, ~2,957+ real forks, coords-now, first-pass scope, ownership) and the [De-Forking](#de-forking).

This is the part that was previously undocumented and matters most. A home's stable id lives in the **`hl_property`** table — one row per physical home, linked from every `Listing` row by `Listing.hlPropertyId`. Its primary key is the **HouseLoop Property ID (`HLP-…`)**.

<Warning>
**The HouseLoop ID is a hash OF the address — not an independent identifier.** It is generated (`lib/listings/cache.ts`) as:

```
hlPropertyId = "HLP-" + md5(identityKey).slice(0,12)      // identityKey = canonical street + unit + ZIP5
```

So `HLP-2816F8920DC0` for *411 NW 71st Terrace* is literally `md5("411 nw 71st terrace::64118")`. The id is **the address in a costume** — it is *re-derived*, every time, from `street + zip`. We do **not** have a Zillow-`zpid`-style opaque id that is minted once and survives address/ZIP changes.
</Warning>

**The consequence — ZIP drift forks a home.** Because the id (and the slug, and `identityKey`) all contain the **ZIP**, and sources disagree on a home's ZIP (Zillow: `411 NW 71st Ter, Kansas City, 64119`; MLS: `411 NW 71st Terrace, Gladstone, 64118` — the *same* house), the same physical home hashes to **two HouseLoop ids → two slugs → two pages.**

| Field | Zillow says | Our MLS (Repliers) says |
|---|---|---|
| Street | 411 NW 71st **Ter** | 411 NW 71st **Terrace** |
| City | Kansas City | **Gladstone** |
| ZIP | **64119** | **64118** |
| → `identityKey` | `411 nw 71st terrace::64119` | `411 nw 71st terrace::64118` |
| → HouseLoop ID | `HLP-<hash of 64119>` | `HLP-2816F8920DC0` |

- **What keeps it from being worse:** rung 2 (lat/lng ~25 m) + 100% coordinate coverage collapse ZIP-drifted **listings** on the Repliers ingest path. (Within a consistent ZIP it already works perfectly — e.g. all three MLS cycles of 411 share the one id `HLP-2816F8920DC0`.)
- **Where it still breaks:** any source **without coordinates** — a home a lead browsed on Zillow (FUB activity carries the Zillow address + ZIP, no lat/lng), a hand-typed address. Its `identityKey`/HLP won't match the home we own, so it fails to resolve (or, uncaught, mints a fork).
- **Blast radius (resolved 2026-06-27):** there were ~75 ZIP-drift forks (plus thousands more from dropped directionals / suffix variants). The de-fork engine merged **1,588 same-home forks** (reversibly) and intake prevention stops new ones — see [De-Forking](#de-forking).

**The durable fix (shipped 2026-06-27 — see [De-Forking](#de-forking)).** The id must be **minted once per physical home and pinned** — assigned on first sighting, after which the address/ZIP/MLS# become *attributes* that can change without changing the id. When a drifted address arrives, the resolver looks up the existing home (coords / street / **parcel**) and **reuses its HLP** instead of re-hashing the new address. `hl_property` already carries the columns for the truly-stable key — **`parcel_number` and `upi`** (the county parcel id, which survives ZIP/address/MLS changes the way `zpid` does) — they are simply not yet populated or keyed on. Two complementary moves close it: (a) **turn on Smarty** so the ZIP is USPS-corrected before it enters the key (stops new forks at the source), and (b) **mint-once on parcel/UPI** so identity stops being `md5(address)`. **As of 2026-06-27** the intake resolver (`lib/listings/rooftop-identity.ts`) reuses the existing home's id for a drifted address, and the de-fork engine has merged the existing forks (1,588, reversibly). **As of 2026-06-28 (Zach-approved, building):** HLP is replaced by a **sequence-backed `HLID123456`** — minted once from `hl_property_hlid_seq`, never re-derived from the address — and every old `HLP-…` is written to `hl_hlid_alias`. The `ON CONFLICT(identity_key)` guard on `hl_property` guarantees one HLID per rooftop forever (a relist reuses it). The HLID then goes **in the URL** (`/home/<slug>/<HLID>`). See [The HLID contract](#) at the top of this page.

## The permanent URL — the SEO moat (why identity is a minted id, not the MLS# or the address)

The single most important property of a home's URL is that it **never changes for the life of the home** — across relists, MLS# changes, ZIP corrections, status changes. This is not polish; it **is** the local-SEO strategy.

**Why a changing URL kills SEO.** Google accrues ranking authority to a *specific URL* over time. If the URL changes when a home relists (new MLS#) or its address is corrected (ZIP fix), Google must **re-index it from scratch** — the authority that URL earned is lost, and the old + new URLs split credit as **duplicate content**. Zillow and Redfin dominate local search for exactly this reason: each home has **one permanent URL** (`…/<zpid>`, `…/home/<id>`) indexed and compounding authority for years across dozens of listing cycles. The URL is pinned to a **minted, immutable id** — never the MLS#, never the raw address.

**So the identity in the URL must be a minted-once HL-ID:**

- **NOT the MLS#** — it changes every relist (a home has several over its life), so the URL would re-index every cycle.
- **NOT the raw address / canonical slug** — it gets corrected (ZIP drift), so the URL would change and re-index.
- **= a minted, opaque `HLP-…`** assigned on first sighting and **frozen forever**. Address, ZIP, MLS#, status are mutable attributes; the id (and the URL) never move.

<Warning>
**Today this is an active SEO liability, not just data hygiene.** The current URL is `/home/<canonicalSlug>` with **no permanent id** — identity *is* the slug, and the slug carries the ZIP. When the slug changes (ZIP drift, or dedup picking a different keeper), the URL changes and Google re-indexes from zero. The ~75 ZIP-drift forks are ~75 duplicate-content pairs splitting authority between two URLs for one home. Pinning identity to a minted id fixes the SEO moat and the fork in one move. **(2026-06-27: the de-fork engine now collapses these duplicate-content pairs — 1,588 merged, 301-redirected — and intake prevention stops new ones.)**
</Warning>

## The reconciliation engine — the ONE way we reconcile an address

There is exactly **one** engine that reconciles any address (a new listing, a sold disposition, a past client, the full historical backfill) to the one canonical HouseLoop home. It is a three-stage funnel — cheapest and most certain first, escalating only for what the prior stage can't settle:

1. **`identityKey` — deterministic (free).** Canonical street + unit + ZIP5 (city-free), via `lib/listings/address-identity.ts`. An exact key match is the same home. This resolves the overwhelming majority.
2. **LLM sieve — for the ambiguous tail.** A model (Claude Haiku) judges the cases the key can't equate — a dropped directional (`1270 Petzold` vs `1270 N Petzold`), an ambiguous street-type, near-identical specs — into confident-merge / confident-distinct / uncertain.
3. **Redfin ground truth — last resort only.** For the small "uncertain" residual the sieve can't settle, Redfin's address autocomplete is the tiebreaker (its property inventory is authoritative for which units exist), and Redfin's address becomes canonical. Never the first move.

<Warning>
**Google geocoding is NOT the reconciliation engine.** Geocoding (`canonicalize-addresses.mjs`) only standardizes messy postal *text*; it does not resolve an address to our canonical home. It is deprecated for reconciliation — the engine is `identityKey → LLM sieve → Redfin`.
</Warning>

**Every collapse is reversible** — a stage that's unsure leaves the rows alone rather than guessing, and each merge writes three records so it can be undone:

- `Listing.quarantinedAt = now()` — the row drops out of search but is fully restorable.
- `hl_dedup_quarantine` — an audit log of `(loser, winner, addresses, reason)`.
- `hl_slug_alias` — maps the retired URL to the keeper so the old link **301-redirects** instead of breaking.

The **keeper** of each group is chosen by completeness: most enriched (value history, photos, value) → live over sold → cleanest address → most recent.

### Keeping it collapsed (forward)

The same engine runs on the live path: when an incoming new or sold listing can't be confidently reconciled against a home we already own, the LLM sieve (and, last, Redfin) resolves it **once**, and the resulting perfect address is stored on the canonical record — so the next time that address appears, it resolves instantly against our own data and never needs an external lookup again.

## Resolving an OUTSIDE address (Zillow-browsed homes, typed addresses)

A home a lead **browsed on Zillow** (captured via FUB activity) is **not** a listing-intake event: it arrives as Zillow's address text — Zillow's ZIP, often a different city, usually **no MLS#** and **no coordinates**. It never runs through `resolveHome`, and even if it did, rung 1 misses on ZIP drift and rung 2 can't fire without coordinates. That is why a home we genuinely own (e.g. `411 NW 71st Terrace`) can fail to link from a lead's browsing history.

The correct resolution for these is the same one-resolver principle: match on the address parts that **don't** drift — **house number + directional + street name + suffix** — region-gated (same ZIP, same city, or same 3-digit ZIP prefix) so a cross-metro collision can't produce a wrong link, and resolve **once at ingest**, storing the resulting `hlPropertyId`/slug on the activity so nothing is re-matched at read time. (Today the CRM lead-activity card does a drift-tolerant read-time match as a stopgap; the durable version folds this into `resolveHome` as a street-based rung and stores the id.)

## Past-Client Address Resolution — the MLS# anchor (the canonical tier)

Past-client homes are the hardest addresses we resolve: they come from **typed source data** (old CRM/SISU
entries), so the street is often garbled — a wrong house number, a misspelled or abbreviated street, a missing
city. A missing city is the worst: CASS then geocodes the fragment to **the wrong state entirely** (`105 E
Washington` with no city → Attica, **Indiana**, not Paola, KS). Address text alone cannot be trusted for these.

**The anchor is the MLS#.** You cannot *search* Zillow by MLS number, but Zillow's **property-details** page
carries it (`attributionInfo.mlsId`). So the resolver treats the MLS# as ground truth and the typed address as a
hint:

1. **CASS-clean** the typed address (Google Address Validation, HouseLoop key `GOOGLE_GEO_SERVER_KEY`).
2. **Ninja multi-form search** — CASS address → street-type variants (`St`/`Ter`/`Dr`/`Ct`/`Ln`) → range-preserved
   (`10700-10702`) → the full `street, city, STATE zip` form (CASS mangles rural/garbled streets) → candidate `zpid`.
   Multi-unit buildings resolve by picking the unit from Zillow's autocomplete **dropdown** (a location string with
   `APT n` won't match) → the unit's own `zpid`.
3. **Validate** `property-details(zpid).attributionInfo.mlsId` against our known MLS#(s) for the home.

Each home lands in one of four states — never silently wrong:

| Status | Meaning |
|---|---|
| **confirmed** | `mlsId` matches our known MLS# → 100% certain it's the right rooftop (MLS-locked) |
| **address_match** | house # + street-name verified, Zillow exposed no `mlsId` (normal for older sales) — correct home |
| **mls_mismatch** | candidate's MLS ≠ ours → flagged for review |
| **not_found** | no Zillow record → flagged (new-construction / off-Zillow / land) |

<Note>
**The miskey lesson (LOCKED 2026-07-02): the MLS# overrides the typed address, every time.** When the typed
address and the MLS#-derived Zillow address disagree, the MLS# wins. Real fixes from the run:

## Data sources for a home

Each field has exactly one authoritative source:

- **Structural facts** (beds/baths, square footage, year built, style, lot, features, HOA, assigned schools): the **MLS record via Repliers**.
- **Sold price and date**: the **MLS** (`soldPrice` + `timestamps.closedDate`), captured always, **display-restricted** in non-disclosure states.
- **HouseLoop Value (active listing)**: derived from the **current list price** (Zillow publishes no Zestimate for on-market homes). See [Home Valuation](/guides/home-valuation-dashboard).
- **HouseLoop Value (sold / past-client)**: anchored on the sold price (or, for already-enriched past-client homes, a Zillow Zestimate captured in a prior one-time backfill).

Geographic appreciation (`lib/valuation/appreciation.ts`) drifts an anchor value over time for the value-history chart. See [Sold Data & Price History](/guides/sold-data-and-price-history).

## Carry-forward enrichment — the home as an accumulator (shipped 2026-06-27)

> **The canonical home is an ACCUMULATOR, not a snapshot.** The IDX feed (Repliers) only ever sends the
> *current* listing — never the home's past. **We are the system of record for a home's banked history.** One
> slug = one home = the full life of data, carried forward across every listing cycle. (The Zillow/Redfin ZPID
> model: the property is the permanent record; each listing is just an event on it.)

A home's **banked history** = sold history · tax history · GreatSchools · HouseLoop value + value history (+
close price, school district). When the same address re-lists under a new MLS#, the new live row **inherits
everything we already banked** — it never renders blank just because the feed sent only today's snapshot.

**Root cause that was fixed:** ingest upserts by `hlid = hl_heartlandmls_<MLS#>` (the *listing*), so a relist
created a new BLANK row; `collapseHomeCluster` then soft-deleted the older cycles **without copying their
enrichment** — the history was stranded on the dead row and the live relist showed blank. (Same-row IDX
re-syncs were already improvement-only, so they never wiped history; the gap was only the relist → new-row →
collapse path.)

**The build (forward-only — Benji):**
- **Carry-forward on collapse (`carryForwardEnrichment`, `cache.ts`):** before `collapseHomeCluster` retires the
  older cycles, it COALESCE-merges their banked fields onto the **winner** — `closePrice · zestimate(+Low/High) ·
  rentZestimate · annualTaxAmount · schoolDistrict · elementary/middle/highSchool · taxHistory ·
  schoolsGreatSchools · valueHistory`. Per field the value comes from the **newest donor cycle that has it**; the
  winner keeps its own non-null values, we only fill gaps. **Photos are NOT inherited** — a live relist shows its
  own current gallery (Repliers refreshes within seconds), never a sold cycle's stale photos.
- **Read layer honors `quarantinedAt` + a stable canonical:** every resolver/sitemap query now filters
  `quarantinedAt IS NULL` (not just `deletedAt`), and the winner `ORDER` has a stable `hlid ASC` tiebreak — so two
  un-merged cycles with the same status + date pick the **same** row every request and the canonical 308 target
  can't flip (no SEO bleed).

**The precedence rule (baked in):** current listing facts (price/status/beds) → the newest row; history /
enrichment → keep-existing-else-inherit; **Caleb's past-client backfill = ground truth, never overwritten by a
thinner IDX value.**

Governing PRD: `GT_HOUSELOOP_ADDRESS_MODEL_CARRYFORWARD_PRD.md`. Still open (tracked above, not here): the legacy
~6K-duplicate row merge (the reconciliation engine, Rule #0) and the rooftop-pinned-ID migration.

## De-Forking

The migration plan above — freeze the id, coordinate-pin to stop new forks, merge the existing forks — is now
**built, deployed, and verified**. Two layers keep one home = one HLID = one permanent URL.

### Layer 1 — prevention at intake
Every incoming listing asks **"is there already an HLID for this rooftop?"** before its `identityKey` is stamped
(`lib/listings/rooftop-identity.ts` → `resolveRooftopIdentity()`, wired into `cache.ts`). A drifted relist —
ZIP typo, `Dr`/`Drive`, dropped directional, city-line straddle — **adopts the existing identity** instead of
forking; `collapseHomeCluster` folds it in. Conservative gate, **safe by construction**: house#-anchored (so
street-level geocodes can't cross-match), match only on **coords ≤30 m** or **identical address minus ZIP**, the
**unit must match**, **no directional conflict**. Structure alone never merges (`74th St` ≠ `74th Terrace`).

### Layer 2 — the autonomous engine
Clusters same-home forks **across keys** (house# + directional/suffix-stripped core + **unit** + state), then
classifies each keeper↔loser pair:

| Tier | Decision | Gate |
|---|---|---|
| **A — auto-merge** | same home | coords ≤30 m · same parcel · identical address + ZIP · core address + matching sqft/year |
| **B — LLM adjudicate** | Haiku rules same vs distinct | loose match, nothing confirms or conflicts (knows `74th St` ≠ `74th Terrace`); only same + high-confidence merges |
| **C — auto-reject** | keep separate | coords >30 m · directional conflict · clashing sqft/year (the "lot sale vs new-construction" trap) |

It runs as a **one-time backlog sweep** (`scripts/defork-engine.ts`) and a **weekly Inngest backstop**
(`lib/inngest/functions/defork-sweep.ts` + `lib/listings/defork.ts`, Mon 15:00 UTC, scoped to recent listings).
Every merge is **reversible — never a Rule #0 op**: quarantine the loser → `hl_slug_alias` 301 (old URL →
keeper, never 404) → `hl_hlid_alias` → `hl_dedup_quarantine` undo log → carry the loser's richer value / tax /
photos onto the keeper. Keeper rank: live > sold > off_market, then most photos, then newest.

### The hard part: unit + coordinates
Only **1.1%** of homes carry a unit, and **16,538** unit-less rows sit on rooftops shared by 2+ distinct homes
(condos/duplexes). So the **unit is part of every cluster key**, and coordinates only *confirm* a same-house-number
match — they never merge on their own. Under-merge (a leftover fork) is recoverable; over-merge (fusing two
people's condos into one HLID) is not.
