Every home on HouseLoop is **one record for its entire life**. It arrives from the MLS (intake),
lives on-market, sells (disposition), and keeps the **same row and the same URL** through all of it.
Status is a field that flips — never a new row, never a second page. This is the front-door story for
how that works end to end: address identity, enrichment, photos, price history, and rendering in one
pipeline, so you don't have to stitch five pages together.

## The spine: one canonical address, one URL — forever

Before either workflow runs, one rule governs everything: **a home's identity is its address, not its
MLS number.** Every step below resolves the incoming record to the one home we already own and writes
to that single row.

- **Identity key** = canonical street + unit + 5-digit ZIP, **city excluded** (the ZIP pins the
  locale, so "Kansas City" vs "Raytown" for the same ZIP collapse to one key). Computed in
  `lib/listings/address-identity.ts`.
- **One URL** = `/home/{canonicalSlug}`, served for every status (active → pending → sold → relisted).
- **The MLS number, zpid, and parcel number are columns**, not identity. A relist under a new MLS
  number resolves to the same home and updates it in place — it never forks a duplicate.

See [Property Data & Address Identity](/guides/property-data-and-address-identity) for the resolver and
the reconciliation engine. The two key facts the lifecycle depends on:

1. **Resolution runs at write time.** Every intake/disposition write calls `resolveHome()` first, so the
   data lands on the one owned home.
2. **The URL is resilient to the city.** `/home/{slug}` resolves by the city-free identity key when the
   exact slug isn't found, then **301-redirects to the home's canonical URL**. A link with a different
   (but valid) city — `…-raytown-…` vs the canonical `…-kansas-city-…` — lands on the right home instead
   of 404ing. A real home can't dead-end on a wrong-city or wrong-suffix URL.

## Intake — a new listing becomes a live, enriched home

A brand-new on-market listing fires **one** orchestrator, `catalog-listing` (an Inngest function), which
runs its steps inline as a single run — **one run, one Run Log row** ("Cataloged — &lt;address&gt;"). The
steps are not separate fan-out functions; they run in sequence inside the one workflow, so if a step
fails the run tells you exactly which one.

```
Repliers webhook  →  cache upsert (SYNC, runs resolveHome, home queryable ~50ms)
                  →  houseloop/listing.catalog  →  catalog-listing (one run):
                        • home          — load the resolved address
                        • photos        — copy photos to Cloudflare R2 (permanent copy)
                        • ai-tags       — gpt-4o vision tags buyer features (pool, garage, …)
                        • zillow        — ONE OpenWeb Ninja call (tax + schools + sold/price history + Zestimate)
                        • price-history — write Listing.priceHistory (the home's own lifecycle ∪ Repliers history)
                  =  home live + fully enriched on the website
```

| Phase | What happens | Result |
| --- | --- | --- |
| **Webhook → cache (sync)** | `POST /api/repliers/webhook` fetches the full record and calls `upsertListingToCache()` **synchronously**, which runs `resolveHome()` so the listing maps to the one owned home. The webhook then fires `houseloop/listing.catalog` fire-and-forget and returns `200`. | Home **queryable in ~50ms** (thin — no R2 photos/value yet). |
| **photos** | Each Repliers photo is uploaded to Cloudflare R2 (our permanent copy — Repliers strips its CDN ~24h after a home leaves the feed). | `photosCached` |
| **ai-tags** | gpt-4o vision tags buyer-facing features. | `visionTags` |
| **zillow** | **One** OpenWeb Ninja call (street + ZIP, city excluded). Returns tax history, GreatSchools, prior sold/price history (Repliers doesn't supply sold history), and the Zestimate (`null` for active listings, populated for sold/off-market). | `taxHistory`, `schoolsGreatSchools`, `valueHistory`, `zestimate`, `annualTaxAmount` |
| **price-history** | Writes the **stored price-history column** (`Listing.priceHistory`) — the home's own lifecycle events (Listed / Pending / Sold / Coming soon) merged with the Repliers address history (prior cycles). This is what the page's Price History table reads. | `priceHistory` |

The Ninja call is made **exactly once per home, at intake** — every step is idempotent, so re-runs never
double-spend. **Timing:** queryable ~50ms; fully enriched within ~15–30s. There is no sub-second "fully
resolved" guarantee, and there doesn't need to be — intake is automatic and near-real-time.

### Intake triggers

| Trigger | Source | Notes |
| --- | --- | --- |
| **Webhook** | Repliers `listing.created` / `listing.updated` | The primary, instant path. |
| **Cron backstop** | `listing-cache-sync` (hourly) | Catches anything the webhook missed; fires the same `houseloop/listing.catalog` event. |
| **Sold** | Repliers `lastStatus = "Sld"` transition | Fires `houseloop/listing.sold` (see Disposition below). |

## Disposition — a home sells and becomes a permanent record

When a home sells, **one** orchestrator, `listing-sold` (an Inngest function), runs the entire sold
sequence as steps of one run. The **only** trigger is the Repliers `listing.updated` webhook firing on a
genuine status **transition** to sold (`previous.lastStatus` present, `→ "Sld"`). There is no cron, no
sweep, no polling emitter — and an idempotency check makes any stray event a cheap no-op.

```
Repliers status transition → "Sld"  →  houseloop/listing.sold  →  listing-sold (one run):
   1. idempotency-check — already archived? STOP (no-op, never re-spends R2 + LLM).
   2. load             — the live Repliers record (real close date = raw.CloseDate).
   3. mark-sold        — flip the Listing row: status='sold' + soldDate + closePrice + lastStatusChangeAt.
   4. price-history    — APPEND this sale to Listing.priceHistory from the ROW's own columns (no Repliers).
   5. value-mint       — mint the HouseLoop value from the real sold price.
   6. archive          — copy to the permanent /home page + push photos to R2.
   7. ai-overview      — generate the Sold House AI overview (house-blurb-generate).
```

Two backstop functions keep sold data complete (each a registered Inngest run, visible in the
[Run Log](/guides/run-log)):

| Function | Step | What it does |
| --- | --- | --- |
| **`sold-data-reconcile`** | `select-missing` | Fills sold homes missing value/tax/school/price-history fields. |
| **`sold-photo-rescue`** | `cache-sold-photos` | Rescues thin sold-home galleries (re-caches photos to R2). |

The sold home keeps the **same row and the same `/home/{canonicalSlug}` URL** — `status` simply flips to
`sold`. The MLS sale price is captured (`closePrice`) but **display-restricted** in non-disclosure states
(Heartland §12.3b). Value detail: [Sold Data & Price History](/guides/sold-data-and-price-history).

## Value — derived from our own data, kept current

HouseLoop Value and the value-history chart are derived from our own data, chosen at read time in
`lib/listings/get.ts` by **data presence, not a per-home flag**. This is **not a five/six-bucket fork** —
that framing was retired. It is a simple **two-engine data-presence fork** plus a live active state
(source: `docs-site/guides/houseloop-home-value.mdx`):

- **① Past-Client Home Value** — `sold` **and** past-client (has a cached Zestimate): `Zestimate × 1.005`.
  Zestimate is used ONLY here, re-fetched monthly from OpenWeb Ninja.
- **② Sold Home Value** — `sold` **and** (not a past client **or** no Zestimate): `soldPrice × ZHVI[now] / ZHVI[soldMonth]`.
  This is the default; ALL property types (single-family, condo, apartment, land) fold in, recomputed from the fresh ZHVI index.
- **Active** — `active` / `coming_soon` / `pending`: `list price × 1.005`, live at render; on close it flips to ① if past-client, else ②.

Full detail — the monthly refresh chain and the value-history series: [HouseLoop Home Value](/guides/houseloop-home-value).

## Past-client home seating (parallel to intake/disposition)

Intake and disposition above are for homes **in our IDX feed**. **SRG past-client homes are seated by a parallel,
one-time flow** — they are pre-existing homes from prior SISU/MLS transactions, so they never arrive via a Repliers
webhook and never run through `catalog-listing`.

```
SISU/MLS deal (typed address)  →  MLS# anchor resolver (offline, one-time, 2026-07-02)
   • CASS clean → Ninja multi-form search → validate mlsId vs our known MLS# → canonical address + zpid + value
   →  pastclient-canonical-FINAL-2026-07-02.csv  (1 row per HLID)
   →  seat into Listing by ADDRESS (status='sold', or 'active' if currently re-listed)  +  hl_srg_past_client_homes
   =  home lives at /home/<slug>/<HLID>, same as any listing
```

- **Resolution is by ADDRESS + MLS#, not the webhook** — the MLS# on Zillow's property-details overrides the typed
  address (the "miskey lesson"). See [Property Data & Address Identity](/guides/property-data-and-address-identity).
- **Value** follows the same two-engine data-presence fork above (past-client `Zestimate × 1.005`, else ZHVI-appreciation recompute) — see
  [HouseLoop Home Value](/guides/houseloop-home-value).
- **Seating is one-time** (the resolver ran 2026-07-02, all 1,936 homes). Going forward, a past-client home that
  **re-lists** re-enters the normal intake flow and updates in place — same row, same URL.
- Full spec: `GT_HOUSELOOP_PASTCLIENT_MLS_RESOLVER_PRD.md`. Owner of the seating/DB side: Caleb.

## Price history lives ON the row — never a live fetch

The Price History table on `/home` renders from the **stored `Listing.priceHistory` column**, not a
live Repliers call. The column is the home's running ledger of events (`{ date, event, price }`, newest
first; Sold price is `null` per the non-disclosure rule). It's populated in three ways — all of which
write the SAME column, so the page has one source of truth:

| When | Who writes it | Source |
| --- | --- | --- |
| **Intake** (new listing) | `catalog-listing` → `price-history` step | the listing's own lifecycle ∪ Repliers address history |
| **Disposition** (a sale) | `listing-sold` → `price-history` step | the **row's own columns** (the `soldDate`/`closePrice` `mark-sold` just set, plus `listPrice`/`pendingDate`/`listDate`) — **merged into** the existing column, **zero Repliers calls** |
| **Backfill** (existing homes) | `price-history-backfill` (manual Inngest fan-out, scoped) | the listing lifecycle ∪ Repliers address history, idempotent (skips a populated column) |

**The disposition path is deliberately Repliers-free.** At a sale, everything we need is already on the
row — the webhook delivered the sale and `mark-sold` wrote it — so the `price-history` step builds the new
sale's events from those columns and **appends** them to the existing history. Repliers only *announces*
the sale (it's the trigger); it is never queried to *build* the history. The engine for all three paths is
`writeListingPriceHistory()` (`lib/listings/price-history-store.ts`) with the canonical event builder in
`lib/listings/price-history-events.ts`.

**Why a stored column (not a render-time fetch):** rendering used to call Repliers live for every sold
home that lacked a stored column. That's a runtime dependency on an external API on the hot path. Storing
the history on the row removes it — the page reads one column, and the column is kept current by intake +
disposition automatically. A render-time Repliers fetch remains only as a transitional fallback for homes
not yet backfilled.

**Backfill scope (one-time):** active + under-contract + past-client homes + recently-sold are being
backfilled into the column via `price-history-backfill` (source: our Repliers IDX feed, no per-call
billing). The full historical sold set is intentionally not mass-backfilled — going forward, every sale
self-populates the column through disposition, so no historical sweep is needed.

## Why it's built this way (Inngest or it doesn't exist)

Both workflows are **registered Inngest functions**, so every intake and every sale is a real run in the
Inngest dashboard and the [Run Log](/guides/run-log) — never a one-off script or a hand-edited row. One
run = one row; if a step fails, the run names exactly which one. No backfill cron writes to the sold
column (it's a polluted cache field) — only a genuine webhook transition processes a sale.

## What the lifecycle does NOT include

- **Notifying buyers** is a separate downstream workflow — saved-search alerts pull *from* the website;
  they are not part of intake. See [Saved Search Alerts](/guides/saved-search-alerts).
- **Collapsing pre-existing duplicate rows** (homes stored more than once before the resolver existed)
  is a separate reconciliation effort — duplicates are **merged or deleted onto one canonical keeper**,
  never hidden. See [Property Data & Address Identity](/guides/property-data-and-address-identity).
