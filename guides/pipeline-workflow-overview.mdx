## 🧭 THE MENTAL MODEL — Everything is a Lead Source; `Workflow = f(source × event)` (Zach LOCKED 2026-06-30 PM)

**There is ONE platform: HouseLoop.** Like Sierra / Follow Up Boss / SureSend, you never name it — there's only
one. So there is **no "platform" and no "integration" layer.** **Everything is a LEAD SOURCE**, and a lead source
has **events**. That's the whole model.

| Concept | What it is | Lives (data) | Shows (UI) |
|---|---|---|---|
| **Lead Source** | How/where a lead enters — **HouseLoop.ai · Follow Up Boss · Manual Entry** (later: Google PPC…). Everything is one of these. | `hl_leads.source` · `hl_lead_sources` | The **Lead Sources** page (each source = a block) + the lead detail page |
| **Event** | What the lead *did*: registered, tour request, liked a home, called… | the trigger (`hl_lead_source_events`; FUB via `hl_fub_webhook_settings`) | Under its source on the Lead Sources page |
| **Workflow** | The steps that fire for a `(source × event)` — **source-specific, not agnostic** | `hl_workflows` (bound via the event row) | The **Workflows** page, **grouped by lead source** |

### The keystone — `Workflow = f(lead source × event)` (this REVERSES "agnostic")
The earlier plan said workflows were *agnostic* — "trigger on the event, filter on source, never one workflow per
source, merge fields fill the words." **That was wrong.** The **same event from a different source is a different
workflow with different copy.** Merge fields alone can't carry the difference in voice. So:

- The **Workflows page GROUPS registration workflows BY lead source** (a band per source: HouseLoop.ai · Follow Up
  Boss · Manual Entry). System Events + Follow-up workflows stay agnostic (no source — they fire for any lead).
- `/api/workflows` resolves each workflow's source from **both** binding tables (`hl_lead_source_events` **and**
  `hl_fub_webhook_settings`, since FUB binds via its own table).

### Follow Up Boss is a SPECIAL source — it pulls in sub-sources
FUB *creates* its webhook events **and pulls in the sub-source already stamped in FUB** (Zillow, realtor.com…).
So a FUB event carries a **sub-source**, and `(FUB · Zillow · Tour Request)` is its **own** workflow — distinct from
a **native** `(Zillow · Tour Request)` — because the voice differs:

```
FUB-import · Zillow · Tour Request   → "Hey Jon, this is Myli — just got your tour request for 123 Main from
                                        Zillow. You worked with Alicia Hanks on our team before — what time
                                        works best?"                          (reactivation: FUB pulled in the history)

Native · Zillow · Tour Request       → "Hey Jon, this is Myli with HouseLoop — we're the local preferred Zillow
                                        partner in Overland Park. I see you want 123 Main today at 3:00, getting
                                        that set up. Any other homes you're interested in?"   (fresh partner intro)
```

**⚠️ OPEN ARCHITECTURAL BUILD (NOT built yet — Lee to spec):** `hl_fub_webhook_settings` today has **one row per
webhook TYPE** (`property_inquiry` · `text` · `in_app` · `phone_call` · `notification`), **not per sub-source**.
To bind a distinct workflow per `(FUB × sub-source × event)`, FUB events need a **sub-source dimension** (the portal
FUB pulled from). This is the keystone the two-different-Zillow-workflows depends on.

### The surfaces (no overlap)
- **`/settings/lead-platform` — Lead Sources (the one page).** Every source is a block: split header
  (`<name>` · **⚙ Configure**) → **KPI funnel** → its **events** (ON · EVENT · WORKFLOW · ACTIONS).
  Order: HouseLoop.ai · Follow Up Boss · Manual Entry. Merges the old Registration Events + Lead Sources pages.
- **Configure** (per source) — that source's **defaults** (name · AI responder · human owner · messages-from ·
  tags · API key), one first-class page. FUB's Configure (`/settings/fub`) also holds its webhook config
  (process · creates-lead · notify · notify-via + master switch). The **only** place "Follow Up Boss" internals live.
- **`/settings/system-events` — System Events.** Existing-lead activity (tour/contact/saved…), its own page —
  it's not lead-creation.
- **`/workflows` — Workflows**, grouped by lead source (registration) + agnostic System/Follow-up bands.
- **Lead detail page** — where **source** renders for an actual lead.

**Nav under "Pipeline & Workflow": Lead Sources · System Events · Workflows.** (The standalone Registration
Events + Lead Sources + FUB Settings nav items are gone — collapsed / folded in.) **Retiring a superseded page =

### ⚠️ THE ANA 913-954-7123 GOOGLE-VOICE GOTCHA (Zach flagged — critical, do not lose this)
**913-954-7123 is Ana Spradling's actual old cell phone number.** She released it; we captured it on **Google
Voice**; a huge number of **past clients still have it saved as her cell**. **Any inbound to it = a potential past
client, full stop — a text/call from a contact reaching that line is a reactivation signal.** The problem: TODAY
it is **forwarded to the FUB inbox 913-453-5505 (and emails to the ana@spradlingrealtygroup.com inbox)**, and
**forwarding collapses the true destination** — FUB may see the forward target, not Ana's line, so we **LOSE** the
"this was Ana's cell → past client" signal. That's a shit show Zach does not want. **REQUIREMENT (Rail R9): we
must know NATIVELY that the contact came to 913-954-7123 (Ana's line), never the forward target.**

**Interim intercept — buildable NOW via the Gmail API (the email bridge).** Google Voice has **no API**, but it
already forwards **every text + voicemail (with transcription)** as an email to **`ana@spradlingrealtygroup.com`**
(today a black hole nobody reads). That forwarded email identifies **which GV line it hit** + the sender's number +
the transcript. So we point the **Gmail API** (we have Workspace admin) at that inbox, parse each Google-Voice
email → *"inbound to Ana's line from \<past client\>"* → fire the reactivation. **The black-hole inbox becomes
Myli's ears** — and it's *cleaner* than the FUB forward (the email preserves the native line). **Permanent fix:**
**port 913-954-7123 out of Google Voice into our Twilio and stop forwarding** (see the porting section below +
the deep doc). Until one of these is live, do NOT trust the sign-box tag for Ana's number.

### The workflow model — ONE workflow, Myli reads the number dynamically
It is **one `(FUB · Phone Call)` workflow, not one per number.** On `callsCreated` (or inbound text), Myli:
1. **Classifies human vs automated** — skip BrokerBay short-code `78340`, spam, robocalls. Only real humans get a text.
2. **Reads the dialed line → its source category** (company · sign-box · past-agent · Ana-cell) and picks the copy.
3. **Sends the text-back.** **Sender = Myli's own BlueBubbles number 913-208-1633** (live today, iMessage+SMS), *as
   herself* — because we CANNOT send from FUB's numbers and won't spoof. Example: *"Hey, this is Myli — just saw a
   call come into our company line, wanted to reach out personally. What are you looking for?"* Past-agent calls
   name the agent; Ana-cell calls use the past-client voice.

**After the Twilio port**, the reply instead comes **AS the exact number they dialed** (Twilio can only send from a
number provisioned to OUR account — hence the port; no spoofing/masking an unowned number). Same workflow — the
port just swaps the sender from Myli's number to the ported line.

### 📲 Porting phone numbers & communications — for everything
The God-tier endgame is to **own every number in our own Twilio** — port each line out of **Google Voice** (Ana's
913-954-7123) and **Follow Up Boss** (the company / sign / past-agent lines) — so inbound hits HouseLoop natively
and Myli sends **AS the dialed line**, no forwarding shell game. The full **Port-Out Playbook** — GV→Twilio (the
$3 unlock, LOA, the two-step SIM fallback), FUB→Twilio (port-out packet, the one-way cutover), the **Ana-first

**→ [Phone Numbers, Call Routing & Number Porting](/guides/phone-numbers-and-call-routing)** (Section 2).

## Current State — the three boxes today

HouseLoop's pipeline is "everything is a workflow" (see `GT_HOUSELOOP_EVERYTHING_IS_A_WORKFLOW_PRD.md`). A lead
enters through a **lead source**, fires an **event**, and that `(source × event)` runs its **own** workflow.

### Lead Sources *(route `/settings/lead-platform`)*
The one page. Each lead source — **HouseLoop.ai · Follow Up Boss · Manual Entry** — is a block: split header
(`<name>` · **⚙ Configure**) → **KPI funnel** (total leads · buyers/sellers UC·Sold · volume) → its
**events** (ON · EVENT · WORKFLOW · ACTIONS). **Configure** = that source's defaults (AI responder ·
human owner · messages-from · tags · API key); FUB's Configure also carries its webhook config (process ·
creates-lead · notify · notify-via + master switch). Backed by `hl_lead_sources` + `hl_lead_source_events`
(FUB via `hl_fub_webhook_settings`). Canonical: `GT_HOUSELOOP_LEAD_PLATFORM_PRD.md` *(its §2 "never one workflow
per source / merge fields fill the words" is SUPERSEDED — see the Model above)*, [Lead Sources](/guides/lead-sources).

### Workflows *(grouped by lead source)*
The canvas engine — triggers, conditions, branches, the sender chain. **Registration workflows are grouped by
lead source** (a band per source); System + Follow-up stay agnostic. Every step is visible on the canvas
(WORKFLOW_CANVAS_IS_TRUTH) and every run lands in the Run Log (INNGEST_OR_IT_DOESNT_EXIST). System workflows are
locked defaults; custom workflows are takeovers. Opening a workflow from a source event carries `?from=` so the
builder's Back returns to Lead Sources. See [Workflows](/guides/workflows), `GT_HOUSELOOP_WORKFLOW_TRIGGERS_SSOT.md`.

### System Events *(route `/settings/system-events`)*
Existing-lead activity — **viewed**/tour/contact/saved/liked/shared/saved-search fired by a lead that already
exists. Its own page (not lead-creation). Agnostic (fires for any lead regardless of source). Most attach a
workflow; **Property Viewed is a counted signal only** (see below).

**Property Viewed = a TRACKED / COUNTED event, no workflow required (Zach LOCKED 2026-07-01).** `property_viewed`
is a first-class System Event in the canonical catalog (`hl_trigger_events`, category `system`, platform
`houseloop`, migration 0276 — MousePointerClick icon). But a home view is high-frequency / low-intent, so by
default **no workflow hangs off it** — it does not fire a text/email. Instead it is **tracked as a counter**: each
view increments the lead's **Visits / Properties** stat on the lead card and shows as **"viewed N×"** on the
activity card. It CAN still *gate* a workflow with a **count condition** (e.g. "viewed the same home 3×" →
high-intent) — but a bare view just feeds the counter. That's the difference from Saved / Liked / Shared /
Contact Request, which are lower-frequency intent signals that each attach a workflow.

### Follow Up Boss — the special source *(config shipped 2026-06-29; folded into the FUB source's Configure)*
FUB is a lead source that *creates* webhook events **and pulls in its own sub-sources** (Zillow, realtor.com…) **and
is our phone system** (3 inboxes, 41 lines — see the Phone topology section above + the deep doc
[Phone Numbers, Call Routing & Number Porting](/guides/phone-numbers-and-call-routing)). Its five event groups:
**Property Inquiry · Inbound Activity · Website Activity · Zillow Insights · Phone Call Activity.** Backed by
`hl_fub_webhook_settings` (migration 0267) — one row per FUB webhook **type** with three toggles + a global master switch:

| Webhook Event | Process | Creates Lead | Notify |
|---|---|---|---|
| Property Inquiries | ✓ | ✓ | ✓ |
| Text Messages | ✓ | (opt-in) | ✓ |
| In-App Messages | ✓ | (opt-in) | — |
| Phone Calls | ✓ | (opt-in) | ✓ |
| Notifications | ✓ | — | — |

- **Process** = ingest the webhook into the lead's history. **Creates Lead** = if no lead matches, this event
  spins one up (phone calls match the caller number with company/shared-line verification; texts/in-app match
  the person). **Notify** = ping Zach when it fires. **Notify VIA** = Realloop (default) → HouseLoop (per-event cutover).
- The dashboard webhook (`apps/dashboard/server/fub-webhooks.ts`) reads this table live (60s cache) and gates
  notify + lead-creation per row. Canonical: `GT_HOUSELOOP_FUB_LEAD_IMPORT_PRD.md`, [FUB Lead Import](/fub/lead-import).
- **The open builds:** (a) rows are per webhook **type**, not per **sub-source** — the **sub-source dimension**
  (Phase 1d) is what lets a FUB-Zillow tour request fire a different workflow than FUB-realtor, both differing from a
  native Zillow lead. (b) **Email received** (`emailsCreated`) isn't a row yet (Phase 1e). (c) **Phone Call Activity**
  (Phase 4) — the ONE workflow, Myli's dynamic dialed-line read, sender = Myli's BlueBubbles now / ported number later.

## 🗄️ Quarantine — the legacy per-source FUB door model (retired, NOT deleted)

The original model made Follow Up Boss a "platform" with a Zillow.com "source" carrying per-source
registration **doors** bound to placeholder workflows. We retired it in favor of the FUB source's webhook config +
`(source × event)` workflows. These rows are **kept for audit** (Rule #0 — no destructive deletes) and hidden from
the live pages:

- `hl_lead_sources` rows with `platform='follow_up_boss'` — `fub_import` ("SRG-Follow Up Boss", displayed as
  **Follow Up Boss**, 8 inactive doors) and `zillow_com` ("Zillow.com", 2 active doors). Zillow.com survives as a
  plain **source tag** (a sub-source, not its own block).
- `hl_lead_source_events` FUB door rows (`tour_request_registration`, `contact_request_registration`, …).
- The **Coming Soon** placeholder workflows (`sys_coming_soon`, `sys_coming_soon_contact`) — zero-node no-ops
  the Zillow doors pointed at.
- The dispatch's per-source door resolution (`fub-lead-import.ts` `TRIGGER_DOORS`) → replaced by the FUB webhook
  `workflow_id` attach.

Why quarantine and not delete: the rows carry historical attribution (which leads came through which door) and
removing them is a destructive prod operation. They are flagged + hidden; the new path is the FUB source's config.
