## рҹ§ӯ THE MENTAL MODEL вҖ” Everything is a Lead Source; `Workflow = f(source Г— event)` (Zach LOCKED 2026-06-30 PM)

**There is ONE platform: HouseLoop.** Like Sierra / Follow Up Boss / SureSend, you never name it вҖ” there's only
one. So there is **no "platform" and no "integration" layer.** **Everything is a LEAD SOURCE**, and a lead source
has **events**. That's the whole model.

| Concept | What it is | Lives (data) | Shows (UI) |
|---|---|---|---|
| **Lead Source** | How/where a lead enters вҖ” **HouseLoop.ai В· Follow Up Boss В· Manual Entry** (later: Google PPCвҖҰ). Everything is one of these. | `hl_leads.source` В· `hl_lead_sources` | The **Lead Sources** page (each source = a block) + the lead detail page |
| **Event** | What the lead *did*: registered, tour request, liked a home, calledвҖҰ | the trigger (`hl_lead_source_events`; FUB via `hl_fub_webhook_settings`) | Under its source on the Lead Sources page |
| **Workflow** | The steps that fire for a `(source Г— event)` вҖ” **source-specific, not agnostic** | `hl_workflows` (bound via the event row) | The **Workflows** page, **grouped by lead source** |

### The keystone вҖ” `Workflow = f(lead source Г— event)` (this REVERSES "agnostic")
The earlier plan said workflows were *agnostic* вҖ” "trigger on the event, filter on source, never one workflow per
source, merge fields fill the words." **That was wrong.** The **same event from a different source is a different
workflow with different copy.** Merge fields alone can't carry the difference in voice. So:

- The **Workflows page GROUPS registration workflows BY lead source** (a band per source: HouseLoop.ai В· Follow Up
  Boss В· Manual Entry). System Events + Follow-up workflows stay agnostic (no source вҖ” they fire for any lead).
- `/api/workflows` resolves each workflow's source from **both** binding tables (`hl_lead_source_events` **and**
  `hl_fub_webhook_settings`, since FUB binds via its own table).

### Follow Up Boss is a SPECIAL source вҖ” it pulls in sub-sources
FUB *creates* its webhook events **and pulls in the sub-source already stamped in FUB** (Zillow, realtor.comвҖҰ).
So a FUB event carries a **sub-source**, and `(FUB В· Zillow В· Tour Request)` is its **own** workflow вҖ” distinct from
a **native** `(Zillow В· Tour Request)` вҖ” because the voice differs:

```
FUB-import В· Zillow В· Tour Request   вҶ’ "Hey Jon, this is Myli вҖ” just got your tour request for 123 Main from
                                        Zillow. You worked with Alicia Hanks on our team before вҖ” what time
                                        works best?"                          (reactivation: FUB pulled in the history)

Native В· Zillow В· Tour Request       вҶ’ "Hey Jon, this is Myli with HouseLoop вҖ” we're the local preferred Zillow
                                        partner in Overland Park. I see you want 123 Main today at 3:00, getting
                                        that set up. Any other homes you're interested in?"   (fresh partner intro)
```

**вҡ пёҸ OPEN ARCHITECTURAL BUILD (NOT built yet вҖ” Lee to spec):** `hl_fub_webhook_settings` today has **one row per
webhook TYPE** (`property_inquiry` В· `text` В· `in_app` В· `phone_call` В· `notification`), **not per sub-source**.
To bind a distinct workflow per `(FUB Г— sub-source Г— event)`, FUB events need a **sub-source dimension** (the portal
FUB pulled from). This is the keystone the two-different-Zillow-workflows depends on.

### The surfaces (no overlap)
- **`/settings/lead-platform` вҖ” Lead Sources (the one page).** Every source is a block: split header
  (`<name>` В· **вҡҷ Configure**) вҶ’ **KPI funnel** вҶ’ its **events** (ON В· EVENT В· WORKFLOW В· ACTIONS).
  Order: HouseLoop.ai В· Follow Up Boss В· Manual Entry. Merges the old Registration Events + Lead Sources pages.
- **Configure** (per source) вҖ” that source's **defaults** (name В· AI responder В· human owner В· messages-from В·
  tags В· API key), one first-class page. FUB's Configure (`/settings/fub`) also holds its webhook config
  (process В· creates-lead В· notify В· notify-via + master switch). The **only** place "Follow Up Boss" internals live.
- **`/settings/system-events` вҖ” System Events.** Existing-lead activity (tour/contact/savedвҖҰ), its own page вҖ”
  it's not lead-creation.
- **`/workflows` вҖ” Workflows**, grouped by lead source (registration) + agnostic System/Follow-up bands.
- **Lead detail page** вҖ” where **source** renders for an actual lead.

**Nav under "Pipeline & Workflow": Lead Sources В· System Events В· Workflows.** (The standalone Registration
Events + Lead Sources + FUB Settings nav items are gone вҖ” collapsed / folded in.) **Retiring a superseded page =
redirect the route + repoint inbound links.**

### вҡ пёҸ THE ANA 913-954-7123 GOOGLE-VOICE GOTCHA (Zach flagged вҖ” critical, do not lose this)
**913-954-7123 is Ana Spradling's actual old cell phone number.** She released it; we captured it on **Google
Voice**; a huge number of **past clients still have it saved as her cell**. **Any inbound to it = a potential past
client, full stop вҖ” a text/call from a contact reaching that line is a reactivation signal.** The problem: TODAY
it is **forwarded to the FUB inbox 913-453-5505 (and emails to the ana@spradlingrealtygroup.com inbox)**, and
**forwarding collapses the true destination** вҖ” FUB may see the forward target, not Ana's line, so we **LOSE** the
"this was Ana's cell вҶ’ past client" signal. That's a shit show Zach does not want. **REQUIREMENT (Rail R9): we
must know NATIVELY that the contact came to 913-954-7123 (Ana's line), never the forward target.**

**Interim intercept вҖ” buildable NOW via the Gmail API (the email bridge).** Google Voice has **no API**, but it
already forwards **every text + voicemail (with transcription)** as an email to **`ana@spradlingrealtygroup.com`**
(today a black hole nobody reads). That forwarded email identifies **which GV line it hit** + the sender's number +
the transcript. So we point the **Gmail API** (we have Workspace admin) at that inbox, parse each Google-Voice
email вҶ’ *"inbound to Ana's line from \<past client\>"* вҶ’ fire the reactivation. **The black-hole inbox becomes
Myli's ears** вҖ” and it's *cleaner* than the FUB forward (the email preserves the native line). **Permanent fix:**
**port 913-954-7123 out of Google Voice into our Twilio and stop forwarding** (see the porting section below +
the deep doc). Until one of these is live, do NOT trust the sign-box tag for Ana's number.

### The workflow model вҖ” ONE workflow, Myli reads the number dynamically
It is **one `(FUB В· Phone Call)` workflow, not one per number.** On `callsCreated` (or inbound text), Myli:
1. **Classifies human vs automated** вҖ” skip BrokerBay short-code `78340`, spam, robocalls. Only real humans get a text.
2. **Reads the dialed line вҶ’ its source category** (company В· sign-box В· past-agent В· Ana-cell) and picks the copy.
3. **Sends the text-back.** **Sender = Myli's own BlueBubbles number 913-208-1633** (live today, iMessage+SMS), *as
   herself* вҖ” because we CANNOT send from FUB's numbers and won't spoof. Example: *"Hey, this is Myli вҖ” just saw a
   call come into our company line, wanted to reach out personally. What are you looking for?"* Past-agent calls
   name the agent; Ana-cell calls use the past-client voice.

**After the Twilio port**, the reply instead comes **AS the exact number they dialed** (Twilio can only send from a
number provisioned to OUR account вҖ” hence the port; no spoofing/masking an unowned number). Same workflow вҖ” the
port just swaps the sender from Myli's number to the ported line.

### рҹ“І Porting phone numbers & communications вҖ” for everything
The God-tier endgame is to **own every number in our own Twilio** вҖ” port each line out of **Google Voice** (Ana's
913-954-7123) and **Follow Up Boss** (the company / sign / past-agent lines) вҖ” so inbound hits HouseLoop natively
and Myli sends **AS the dialed line**, no forwarding shell game. The full **Port-Out Playbook** вҖ” GVвҶ’Twilio (the

## Current State вҖ” the three boxes today

HouseLoop's pipeline is "everything is a workflow" (see `GT_HOUSELOOP_EVERYTHING_IS_A_WORKFLOW_PRD.md`). A lead
enters through a **lead source**, fires an **event**, and that `(source Г— event)` runs its **own** workflow.

### Lead Sources *(route `/settings/lead-platform`)*
The one page. Each lead source вҖ” **HouseLoop.ai В· Follow Up Boss В· Manual Entry** вҖ” is a block: split header
(`<name>` В· **вҡҷ Configure**) вҶ’ **KPI funnel** (total leads В· buyers/sellers UCВ·Sold В· volume) вҶ’ its
**events** (ON В· EVENT В· WORKFLOW В· ACTIONS). **Configure** = that source's defaults (AI responder В·
human owner В· messages-from В· tags В· API key); FUB's Configure also carries its webhook config (process В·
creates-lead В· notify В· notify-via + master switch). Backed by `hl_lead_sources` + `hl_lead_source_events`
(FUB via `hl_fub_webhook_settings`). Canonical: `GT_HOUSELOOP_LEAD_PLATFORM_PRD.md` *(its В§2 "never one workflow
per source / merge fields fill the words" is SUPERSEDED вҖ” see the Model above)*, [Lead Sources](/guides/lead-sources).

### Workflows *(grouped by lead source)*
The canvas engine вҖ” triggers, conditions, branches, the sender chain. **Registration workflows are grouped by
lead source** (a band per source); System + Follow-up stay agnostic. Every step is visible on the canvas
(WORKFLOW_CANVAS_IS_TRUTH) and every run lands in the Run Log (INNGEST_OR_IT_DOESNT_EXIST). System workflows are
locked defaults; custom workflows are takeovers. Opening a workflow from a source event carries `?from=` so the
builder's Back returns to Lead Sources. See [Workflows](/guides/workflows), `GT_HOUSELOOP_WORKFLOW_TRIGGERS_SSOT.md`.

### System Events *(route `/settings/system-events`)*
Existing-lead activity вҖ” tour/contact/saved/liked/shared/saved-search fired by a lead that already exists. Its
own page (not lead-creation). Attach a workflow to each; agnostic (fires for any lead regardless of source).

### Follow Up Boss вҖ” the special source *(config shipped 2026-06-29; folded into the FUB source's Configure)*
FUB is a lead source that *creates* webhook events **and pulls in its own sub-sources** (Zillow, realtor.comвҖҰ) **and
is our phone system** (3 inboxes, 41 lines вҖ” see the Phone topology section above + the deep doc
[Phone Numbers, Call Routing & Number Porting](/guides/phone-numbers-and-call-routing)). Its five event groups:
**Property Inquiry В· Inbound Activity В· Website Activity В· Zillow Insights В· Phone Call Activity.** Backed by
`hl_fub_webhook_settings` (migration 0267) вҖ” one row per FUB webhook **type** with three toggles + a global master switch:
