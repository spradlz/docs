## The vision: a tuning studio, not a test bench

Every AI behavior in HouseLoop - the voice answer, the first-touch text, the nurture cadence - is driven by
prompts, context blocks, and playbooks. Today those live in one place (AI Prompting + AI Context Blocks) and
the behaviors fire somewhere else (workflows, the voice line, the clone). The Test Cockpit closes the loop in
ONE surface:

**SEE the prompt → TEST it → WATCH receipts + run log → EDIT as draft → RE-RUN → PROMOTE (gauntlet-gated).**

- **Drafts never touch production.** Every compose seam accepts a prompt override, so the cockpit tests draft
  prompts without redeploying or changing what real leads experience.
- **Promote is gated on the gauntlet** - a prompt change cannot go live until its scenario suite passes.
  This is CI for prompts: every edit is proven against the trap scenarios BEFORE it talks to a real lead.
- **AI Prompting stays the single source of record.** The cockpit is where drafts get proven, then promoted
  into it. No forked prompt copies (REUSE_NOT_RECREATE).
- **The Prompt & Context rail rides every section:** whatever you are testing, the exact live prompt/persona/
  playbook/context blocks that drive it are visible right there - with Edit as draft, Diff vs live, and
  Re-run with draft.

## Where it lives (Zach decision, 2026-07-20)

**AI → Test Cockpit.** Not under Pipeline & Workflow - because the cockpit is not only testing workflows and
trigger events. Those doors inject AI context blocks and solicit conversation with the lead, so testing them
IS testing the AI. It sits next to the consolidated prompts home (AI Prompting + AI Context Blocks): one
place to author prompts, one place to prove them.

## The Prompt Editor modal - where tuning actually happens

Click **any prompt chip anywhere in the cockpit** (a context block in a receipt, a prompt-version chip in a
table, Edit-as-draft in the rail) and the big house-style modal opens (near full height, matching the CRM's
large-modal pattern):

- **Header:** the prompt's name + a **live / draft** version toggle + which surfaces use it.
- **Left - the editor:** the full prompt text, editable. Edits highlight until saved. The draft only ever
  runs inside the cockpit; live keeps answering with the current version until you promote.
- **Right - the promote path, top to bottom:** **Versions** (draft / live / history, with author) →
  **Diff** (removed lines struck red, added lines green) → **Prove it, then promote:** re-run your last test
  with the draft, run the full gauntlet vs the draft, the gate names exactly which check is failing, and the
  full-width **Promote draft to live** button unlocks only at a fully green suite.
- **Promote** writes the draft into AI Prompting as the new live version - every AI that uses the prompt
  picks it up instantly, and the record never forks.
- The **Prompt & Context rail** on the page shows the FULL live prompt at a glance (no truncation) with the
  same Edit / Diff / Re-run / Promote controls.

### How it works under the hood (BUILT, 2026-07-20)

- **One prompt system, no fork:** the cockpit rides the AI Programming registry
  (`lib/ai/prompt-registry.ts` + `hl_ai_prompt_history`). A new slot `first_touch_contract` was added,
  backed by `hl_settings ai_text_prompts.first_touch_contract` (mig 0427 seeded the exact code contract -
  1,985 chars). The same slot is editable from `/ai/prompting` AND the cockpit modal.
- **The compose reads it live:** `buildFirstTouchBrief` resolves `contractOverride` (a cockpit draft
  re-run) → the LIVE registry value → the old code constant as a never-break fallback. An edit takes
  effect on the very next compose - no deploy.
- **Drafts live BESIDE live:** `PUT /api/ai/prompts/[key] {draft:true}` stores the draft at
  `<jsonPath>__draft` - no history row, live untouched; a live save (promote) clears the draft.
  `GET /api/ai/prompts/[key]` returns `slot + live + draft + history` - the modal reads one endpoint.
- **The no-send test:** `POST /api/test/preview-firsttouch` runs the EXACT workflow compose chain
  (`buildFirstTouchBrief` → `composeAiMessage` → `cleanComposedOutput`) on the test lead with the
  editor's current text as the contract - byte-for-byte what would send, zero sends - and returns the
  message + the one-question / no-leaked-reasoning checks. Proven live: the modal composed a real first
  touch off the real dossier with both checks green and nothing sent.
- **The honest promote gate:** no scenario suite is wired to text prompts yet, so promote is a typed
  confirm (type PROMOTE). The gauntlet gate replaces it when the suites land - the UI never fakes a
  "6/7 green".

## The three sections

real world** - it says so in amber (New arm purges the test lead with the real DELETE cascade, then mints
fresh and sends real comms).

- **Controls:** What to test (trigger event OR workflow) · lead source (FUB / Direct) · email + phone · New/
  Existing arm · DISC roleplay persona · Run.
- **The receipt "white sheet"**, each card backed by its real table:
  | Step | Proof |
  |---|---|
  | Trigger fired | `hl_trigger_fire` (event_key, matched, routed_by) - rendered as the code receipt |
  | Lead created + stage | `hl_leads` + `hl_stage_transitions` (read DIRECT - workflow stage moves skip the All feed) |
  | **Context injected** | the `buildFirstTouchBrief` blocks the AI actually received - the seam between the workflow and the AI; opens in the Prompt rail |
  | Branded email | `hl_email_send_log.body_html` - the FULL sent HTML in an iframe |
  | Myli text | `hl_messages` - the words, as an iMessage bubble (delayed nurture beats show as pending) |
  | Audit scorecard | every surface verified (lead · workflow · notification · email · text · note · tag · All feed) |
- **Right rail:** the REAL bound workflow for the selected door with its ACTUAL node list (from
  `hl_workflows.workflow_config` via `GET /api/test/workflow-preview` - never a hardcoded guess) + the
  Prompt & Context rail.
- **Run history (BUILT):** `GET /api/test/history` - the real recent workflow runs + door fires for the
  current test lead, no synthetic rows (an empty lane says so). Prompt-version stamping (which contract
  version a run composed with) lands when runs record it - the column is absent until then, never faked.

### 2 · AI Voice (Lee - BUILT, live as tab 2)

Retell-style testing for Myli's voice brain: text-mode roleplay against the LIVE deployed persona + playbooks
(no calls, no sends, pennies) - shipped as a first-class page, not a wireframe.

**The UI** - `components/settings/test-cockpit/VoiceCockpit.tsx` (Zach live redesign: "top = pick the
scenario · center = a first-class texting screen, the roleplay IS the page · far right = my prompt in
plain sight"). Everything on it is real - persisted runs, the composed live persona, no mock data:

- **Scenario board:** every scenario with its latest persisted pass^k result, a per-scenario **Run** button
  (k=3 rollouts), and **Run full suite** (the client iterates scenarios one request each, so progress
  streams and no single request holds for the whole gauntlet).
- **Transcript viewer:** open any persisted run - full transcripts, the judge's verdict per check with a
  **quoted-evidence** line, plus the deterministic tool-call asserts.
- **Live roleplay chat:** the texting screen where YOU are the caller, typing against the real brain
  (stateless replay - the client holds the thread, each turn replays it into a fresh session).
- **The always-on prompt rail:** the composed brain in plain sight - base persona / triage / per-bucket
  playbook cards (seller · buyer · past client · other), each with **Edit** opening the registry
  `PromptEditorModal`, plus the **live / draft** toggle so the whole tab runs against cockpit drafts.
- **Phone smoke button:** one click dials the scripted call on the real line + a "Last smoke" line.

**The in-app engine** - `lib/test-harness/voice/` (the CLI gauntlet productized into the CRM):

- **`scenarios.ts`** - ELEVEN archetype scenarios, each trapping a specific failure: **happy-seller ·
  trail-off** (the Zach repro: the caller hesitates mid-sentence - the announce-then-silence killer) **·
  happy-buyer · buyer-specific-listing · valuation-fisher · rambler · refuser · bye-mid-intake ·
  wrong-number · injection-attacker · question-asker**. Every bad real call becomes a new scenario; the
  suite only grows. Plus **`UNIVERSAL_CHECKS`** - the binary checks every run is graded on (greeted once,
  one question per spoken turn, no announce-without-a-question, no literal screening pattern, acknowledge
  + question in the same breath, no fabricated facts) - and the exact `VOICE_TOOLS` + greeting the phone
  bot runs, kept in lockstep so text mode exercises the REAL brain contract.
- **`engine.ts`** - server-side **Realtime TEXT sessions** against the REAL composed persona (self-HTTP to
  `/api/phone/myli/persona`, the ONE composer the phone bot fetches; draft mode overlays cockpit bucket
  drafts before merging). The **caller simulator is `gpt-4o-mini`** driving persona briefs with trap
  behaviors. The **judge is cross-family `claude-sonnet-4-6`** (kills same-family self-preference bias),
  prompted with a **GROUNDING block** - the live clock, the caller-ID digits, Myli's approved-knowledge
  list, and the volunteered-info rule - because without those calibrations the judge mis-flags TRUE
  statements as fabrications. **`mustCall` tool asserts are deterministic in code**, never judged. The
  bot's **turn-completion guard is mirrored** in the harness (any intake turn that ends without a question
  gets one forced continuation - parity with `MyliTurnGuard` in `pipecat_bot.py`), and **spoken-flow turn
  merging** collapses a continuation into ONE transcript turn so the judge grades the turn a caller would
  HEAR, not the raw API events.

**The APIs + persistence** (all admin-gated):

| Route | Does |
|---|---|
| `POST /api/test/voice-gauntlet` | run ONE scenario × k against the real brain, judge, **persist to `hl_ai_voice_test_run`** (mig 0430: one row per execution, k rollouts inside `runs` jsonb), return the graded runs |
| `GET /api/test/voice-gauntlet` | the scenario board's data (latest run per scenario + which slots hold drafts) · `?runId=` opens one run's full transcripts + verdicts |
| `POST /api/test/voice-roleplay` | ONE stateless chat turn against the composed persona (text modality, fractions of a cent) |
| `GET /api/test/voice-roleplay` | the rail's brain fetch - the COMPOSED persona + triage + four playbooks, live or draft |
| `POST /api/test/voice-smoke` | place the real scripted Twilio call (~14¢) - the exact proven TwiML seller script; receipts land on the `hl_calls` row via the normal wrapup pipeline |
| `GET /api/test/voice-smoke` | the latest smoke-call row (the "Last smoke" line) |

**The CLI tier** (`tools/myli-voice/` - where the engine was born; still the fastest loop for terminal
tuning): `roleplay.mjs` (interactive text roleplay) · `gauntlet.mjs` (the full suite, `--k` / `--only`) ·
`test-call.mjs` (the scripted smoke call) · **`scripted-call.mjs`** (scripted Twilio caller, `--script
seller|buyer`, downloads the dual-channel recording) · **`measure-latency.py`** (per-turn voice-to-voice
latency straight from the WAV - caller channel vs Myli channel, pure stdlib, python3.14-safe).

**The loop, PROVEN (2026-07-21):** gauntlet round 1 scored **2/11**. Four tuning rounds later (prompt
rebuilds + judge calibration - the grounding block, volunteered-info, spoken-flow merging) - **11/11
scenarios certified pass^3 GREEN**, and on a real scripted phone call the turn-completion guard **fired
3 times live** (2 no-question continuations + 1 post-tool watchdog) with the funnel + close running end
to end. Text mode tests THE BRAIN; voice feel (turn-taking, barge-in, ASR) still needs the phone smoke +
a human ear before any voice-config release.
