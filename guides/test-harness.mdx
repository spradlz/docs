## Why this exists

Every trigger event is only trustworthy if you can fire it **for real** on a live lead and read **every receipt**.
Two production lies this session stayed hidden until a real registration + a real saved search were driven through
the actual UI: the saved-search **"80 sent / 0 delivered"** phantom (the runner skipped a locked-but-archived
workflow while the site stamped "sent"), and a **hand-rolled "100 homes"** alert (a backdated watermark faked a
result). The fix isn't more one-off manual testing — it's a **systematic, autonomous, receipt-first** harness: one
place to fire any trigger event on demand, obey the real routing, and prove it with receipts. No vibe, no
hand-rolling.

## What it is — the cockpit

Entry: on the **Trigger Events page** (`/settings/trigger-events`), every trigger-event row gets a **Test** button.
Click → a **single full-page overlay** with four zones.

1. **① Top bar — the test-lead criteria.** Source picker **Follow Up Boss** (search a real test lead, like the
   lead-integration pull-in) or **Direct Registration** (mint fresh under houseloop.ai via Playwright); an editable
   **email + phone** (default Zach's real ones); a **New-lead / Existing-lead** arm toggle; and a **role-play persona**
   (one of 4-5 DISC profiles, each with its own prompt) for the AI counterparty.
2. **② Right column — the workflow that will run.** The top-to-bottom node cards for the selected arm (reusing the
   Workflow Builder render), with live ✅/⏳/○ state and an "edit ↗" deep-link to `/workflows?wf=`.
3. **③ Center (~75%) — the live "white sheet" run log.** On **Run**, a live top-to-bottom stream: "opening
   Playwright… registering this lead…" → **trigger fired** (the `hl_trigger_fire` receipt) → **lead created / stage
   changed** (the receipt + a link to the lead) → **branded email sent** (the **full HTML render of the exact email**
   + who it sent as) → **Myli text sent** (the words) → the all-activity sheet, each row with a ✓ and its
   source-table proof.
4. **④ Bottom — the role-play / override bar.** **Autonomous run** (the DISC persona converses with Myli/the clone,
   AI-vs-AI down the whole workflow); **Override** (Zach texts the clone himself, as the lead, to his real phone, and
   watches it respond); a channel toggle **SMS · Email · Voice (soon)**.

## The architecture — the pipeline of one Run

```
Test button (row) → cockpit → Run
  1. PURGE    the test lead (clean slate)                         → purgeLead()  [reuse]
  2. FIRE     the door via its family's REAL emitter              → fireDoor(eventKey, arm, testLead)
  3. WAIT     for Inngest to settle
  4. GATE     assert the fire ledger (the primary receipt)        → hl_trigger_fire.matched / hl_tool_fire.ok
  5. RUN      the receipt reader + assemble the white sheet        → auditLead() [reuse] + receipt assembler
  6. RENDER   the cockpit: workflow (right) + receipts (center)
```

**INNGEST_OR_IT_DOESNT_EXIST:** every fire goes through the *real* production emitter — never a fake side-effect.
**The table is the truth:** every receipt reads a real table; nothing is inferred or re-rendered.

### The row source — the catalog (P0, SHIPPED)

`lib/test-harness/catalog.ts` → `buildTestCatalog()` reads `hl_trigger_events` (THE board catalog) + the two-arm seat
bindings (`hl_lead_source_events` native / `hl_fub_webhook_settings` FUB, mig 0381), so the Test page rows **can never
drift from the live Trigger Events board**. It returns **8 families / 71 events**, each with its family, fire method,
and whether each arm (new / existing) is bound + active with the real workflow name. Verified live.

| Family | Events | How the harness fires it (`fire`) |
|---|---|---|
| **Website Registration** | 30 | `ui` — Playwright over real Chrome drives houseloop.ai (direct reg, tour, contact, saved search, seller) |
| **Follow Up Boss** | 3 | `fub` — flip the FUB door on + create/scrape a real FUB person → `fub/lead.import` |
| **Behavioral (Website)** | 4 | `behavior` — emit the pixel view/like/save/share |
| **Inbox & Communication** | 5 | `webhook` — POST the inbound SMS/BB/email webhook |
| **Insights** | 5 | `cron` — invoke the `insight-engine` cron |
| **Transaction Coordination** | 3 | `tc` — e-sign send / Documenso webhook |
| **AI Tools** | 20 | `tool` — `execTool` on the bound lead |
| **Manual** | 1 | `manual` — the Add-Lead path |

Saved-search delivery has a **second half**: after the search is created via `ui`, the harness **seeds a synthetic
`Listing`** (matching the metro, `firstSeenAt=now`) → `houseloop/idx.event` → the instant alert fires → email; then
cleans up. This is the `seed` mechanism.

### The receipt "white sheet" — what proves each step

The receipt assembler (P0 next) gathers, in time order, for one `hl_workflow_runs.id`:

| Step | Proof table + column | Rendered as |
|---|---|---|
| Door fired | `hl_trigger_fire` (`event_key`, `matched`, `routed_by`) · AI: `hl_tool_fire` | the code receipt |
| Workflow ran | `hl_workflow_runs` (`status`, `outcome`, `triggered_by`) | the run spine (top) |
| Inngest ran | `hl_run_log` (`status='success'`) | the ✓ checkmark |
| Lead created / stage | `hl_leads` + `hl_stage_transitions` (⚠ NOT in the All feed — read it directly) | receipt + lead link |
| Email sent | `hl_email_send_log.body_html` (the FULL rendered HTML) + `to_email` + sender | full email in an iframe |
| Myli text | `hl_messages` (`body`, `channel`, `actor_role`) | the words, as a bubble |
| Note / tag / task / appt | `hl_notes` · `hl_lead_tags` · `hl_tasks` · `hl_appointments` | activity rows |

## Gotchas the build must honor

- **Stage change is invisible to the All feed** — workflow `update_stage` runs `logEvent:false`
  (`execute-step.ts`); the receipt MUST read `hl_stage_transitions` directly, not `hl_lead_activity`.
- **AI-tool / inbox / insight doors can't be UI-fired** — fire them through their real emitter (`execTool` / webhook
  POST / cron invoke) and gate on `hl_tool_fire` / `hl_trigger_fire`; label unbound doors honestly (`matched=0`).
- **Prod-data safety** — the web DB is the single prod Postgres; the synthetic-`Listing` seed and every test write
  hit live data, so the seeder + `purgeLead` must clean up deterministically. A real address (`zach@…`) dodges the
  `isUndeliverableAddress` guard; never `@example.com`.
- **FUB clean-slate** — the `fub_person_id` link means re-import re-pulls; scope FUB tests to a dedicated test person.
- **No new nav-label drift** — the cockpit is an overlay off the Trigger Events row, not a new left-nav leaf (avoids
  a `crm-nav-sync` slug requirement).
