A **workflow** is a trigger + a sequence of steps (send a text, send an email, wait, branch on
a condition) laid out on the **workflow canvas**. Workflows are how a lead event Рђћ a tag applied,
a saved search created, a tour requested Рђћ turns into outreach without anyone touching a keyboard.
Everything runs through **Inngest**, so every step is auditable in the [Run Log](/guides/run-log).

## How it flows

```
event (e.g. houseloop/lead.tag-added)
      Рћѓ
      Рќ╝
workflowDispatcher (Inngest)      Рєљ matches the event to workflows whose trigger fits
      Рћѓ
      Рќ╝
workflowRunner (Inngest)          Рєљ walks the canvas steps: send / wait / condition gate
      Рћѓ
      РћюРћђРќХ Send Text  Рєњ execute-step.ts Рєњ BlueBubbles / Twilio (per sender rule)
      РћюРћђРќХ Send Email Рєњ execute-step.ts Рєњ Gmail/Resend
      РћћРћђРќХ Condition  Рєњ evaluate-condition.ts (continue if true, else stop)
```

Both Inngest functions live in `apps/houseloop-crm/lib/workflows/runtime.ts`. Step execution is
`lib/workflows/execute-step.ts`.

<Note>
After deploying a new or changed workflow function, **re-sync Inngest**:
`curl -X PUT https://crm.houseloop.ai/api/inngest`. New functions don't appear in the Inngest
dashboard until you do.
</Note>

## The workflow canvas is the truth (constitutional)

<Warning>
**Every step a workflow runs must be a visible step on a visible path on the workflow canvas. No
step may be shared between two branches, and no hidden connection may exist. If you can't see the
step do the thing on the canvas, the step doesn't get to do the thing.**
</Warning>

The workflow canvas (the builder / workbench) **is** the behavior Рђћ what runs equals exactly
what's laid out on the canvas, nothing hidden. This is the workflow sibling of
[Inngest or it doesn't exist](/guides/run-log): a job doesn't exist unless it ran through Inngest
and shows in the Run Log; a workflow **step** doesn't exist unless it's a visible step on a visible
path on the canvas. Same principle Рђћ no invisible side-effects.

When a **Branch** step splits the flow, each side must be a **complete, self-contained chain**
down to its own terminal step. **Never** wire one branch into another branch's step to "reuse" it
Рђћ that makes the canvas lie about what runs.

Example Рђћ the General Registration workflow, both branches built out independently on the canvas:

```
New Registration (trigger)
  Рєњ Update Stage (New Lead)
  Рєњ Create Saved Search
  Рєњ Branch: Phone is not empty
       РћюРћђ YES Рєњ Add Note Рєњ Send Text  Рєњ Send Email Рєњ Update Stage (Attempted Contact)
       РћћРћђ NO  Рєњ Add Note Рєњ Add Tag     Рєњ Send Email Рєњ Update Stage (Attempted Contact)
                          "Registered: No Phone"
```

Both branches do their **own** Note, their **own** Email, and their **own** Update Stage. The only
difference: **YES sends a text; NO adds the "Registered: No Phone" tag and skips the text.** Neither
branch borrows a step from the other.

**Verify on the canvas, not in the database.** A green run record ("the lead advanced to Attempted
Contact") is **not** proof the workflow is correct. Open the workflow canvas and trace every branch
end-to-end Рђћ every action must be a step drawn on a drawn path.

<Note>
This rule exists because it bit us: the phone branch visually dead-ended at "Send Text" on the
canvas while secretly running the email + stage-advance through the *other* branch's steps (a
cross-link the canvas never drew). It worked in the data; the canvas lied. Fixed by giving each
branch its own complete chain. Full rule: `.claude/rules/WORKFLOW_CANVAS_IS_TRUTH.md`.
</Note>

## Data model

A workflow is one `hl_workflows` row:

| Column | Meaning |
| --- | --- |
| `trigger_type` | what starts it Рђћ e.g. `tag_added` |
| `trigger_config` | JSONB. For Tag Added: `{ tag_ids: ["Рђд","Рђд"] }` (multi-tag, "is one of") |
| `workflow_config` | JSONB `{ nodes, connections }` Рђћ the steps + connections the canvas draws and the runner walks |
| `kind` | grouping / category |

The `workflow_config.connections` array is the set of **connections** Рђћ `{ source, target }`, plus
`branch: "true" | "false"` on the two connections leaving a Branch step. Every connection in this
array is a line the canvas draws and the runner walks; there are no connections that run but aren't
drawn (see [The workflow canvas is the truth](#the-workflow-canvas-is-the-truth-constitutional)).

## Triggers

### Tag Added (FUB-style)

The Tag Added trigger fires when a tag lands on a lead **anywhere** Рђћ website intake
(`houseloop/lead.tag-added`) or a manual bulk-tag in the CRM. The editor mirrors Follow Up
Boss:

- A searchable **multi-tag picker** Рђћ type to filter, click to add. Selected tags render as
  blue, removable **tag pills** (the same blue chip used in Settings Рєњ Tags, tokenized via
  `--hl-tag-pill-border` + `components/shared/TagPill.tsx`).
- An inline **trigger-type dropdown** so you can switch the trigger without leaving the canvas.
- A canvas summary line: **"When tag is one of: A, B, C."**

Matching is **"is one of"** Рђћ the dispatcher fires if the applied tag is in `trigger_config.tag_ids`
(with back-compat for a legacy single `tag_id`, plus a no-tag fallback). The runner only fires for
a tag that was genuinely **new** on the lead (the intake INSERT Рђд RETURNING tells it which tags
were added vs already present), so re-applying a tag doesn't re-trigger.

## Send steps & the sender rule

Each Send Text / Send Email step answers one question: **"Who does this message come from?"**

| `sender_mode` | Sends as |
| --- | --- |
| `system` (default) | the HouseLoop system identity |
| `assigned` | the lead's assigned agent Рђћ and if that agent is the AI ISA, it sends **as Myli** |

When `sender_mode = 'assigned'`, the executor resolves the lead's send identity (BlueBubbles
iMessage line / inbox) via the agent runtime and sends the text **as that agent**. A message sent
as Myli logs to `hl_messages` as `imessage / ai_agent / sent` Рђћ it shows up in the lead's thread
as if Myli texted them, not as a system blast.

<Note>
**Lead-only by default.** The `houseloop.ai` lead source assigns **no human agent**. So a
saved-search workflow there is lead-only: the buyer gets a homes email + a Myli text, and no
agent gets pinged. Adding agent-notify steps to a no-agent source fails ("agent has no phone") Рђћ
that's by design.
</Note>

### Dedupe guard

Send-Text steps run through a **10-minute dedupe guard** (`alreadySent()` in `execute-step.ts`):
if an identical body already went to that lead within the window, the step is skipped and logged
as deduped. This stops a double-dispatch from double-texting a lead. (Email goes via Resend, not
`hl_messages`, so the guard is texts-only.)

## Condition & branch steps (the communication-recency matrix)

A **Condition** step is a gate (continue if true, else stop). A **Branch** step forks into a
**true** path and a **false** path Рђћ and per
[The workflow canvas is the truth](#the-workflow-canvas-is-the-truth-constitutional), each path
must be its own complete chain on the canvas. Conditions/branches are how you split "new lead vs.
returning lead," "has phone vs. no phone," or "only text if we haven't talked in 3 days."

Fields and operators live in `lib/workflows/condition-fields.ts`; evaluation in
`evaluate-condition.ts`. The communication-recency matrix:

| Field | Reads from |
| --- | --- |
| `last_communication` | any message, either direction |
| `last_contact_attempt` | any **outbound** message (text/email/call) |
| `last_contact_received` | any **inbound** message |
| `last_text` / `last_text_sent` / `last_text_received` | `hl_messages` (channel = text) by direction |
| `last_email` / `last_email_sent` / `last_email_received` | `hl_messages` (channel = email) by direction |
| `last_call` / `last_call_made` / `last_call_received` | `hl_calls` by direction |
| `last_activity` | comms Рѕф `hl_pixel_event` (texts/emails/calls *and* website activity) |

**Operators**: text/number fields support `is`, `is_not`, `contains`, **`is_not_empty`**, `is_empty`;
date-valued fields support `is_empty`, `is_not_empty`, `less_than_days`, `more_than_days`. The
`*_empty` operators take no value (the editor hides the value input for them) Рђћ e.g. the General
Registration branch gates on **"Phone is not empty."**

<Note>
**New-lead vs. returning** is the primary recency use case: gate one branch on
**"Last Contact Attempt is empty"** (brand-new lead Рєњ send the Myli intro) and the other on
**"is not empty"** (we've reached out before Рєњ send a lighter "here's your search" touch).
</Note>

Full spec: `GT/GT - ACTIVE/GT_HOUSELOOP_WORKFLOW_CONDITIONS.md`.
