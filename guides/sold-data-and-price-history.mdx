title: "Sold Data & Price History"
description: "How HouseLoop detects sold listings in real time (the Repliers webhook Sc→Sld transition), builds the permanent sold page, and renders price history. The single source of truth, and why there is no backfill cron."
---

HouseLoop has exactly **one source of truth for a sale**: the **Repliers `listing.updated` webhook**, firing on a genuine status **transition to sold**. There is no cron, no polling backfill, no "catch the diff" backup. Webhook or nothing.

## The one source of truth — the sold transition

A real close arrives as a Repliers `listing.updated` webhook whose payload diff shows the listing's `lastStatus` **changing** to `Sld`:

```
previous: { lastStatus: "Sc" }                 ← was under contract (or "New" = active)
data:     { status: "U", lastStatus: "Sld" }   ← now SOLD
```

- `lastStatus === "Sld"` is the canonical sold marker across the whole codebase (`lib/listings/providers/repliers.ts`, `market-report.ts`).
- **The discriminator** (SUPERSEDED 2026-07-03, the disposition-trigger fix; doc corrected 2026-07-06 audit): the old rule fired only on a real *transition* (`previous.lastStatus` present and `!= "Sld"`). That skipped a re-sync of an already-sold record — but if the webhook MISSED the original transition, the home's value/AI-overview chain never ran and it rendered flat forever (the 1011 Fontana bug). **New rule: whenever `data.lastStatus === "Sld"`, fire `listing.sold` UNLESS the row is already fully archived** (`Listing.status='sold'` AND `soldDate` AND `enrichmentSource='repliers_webhook'`). A genuinely-archived re-emit is skipped (`sold_resync_skipped`); a missed transition is recovered (`sold_resync_recovered`).

```
app/api/repliers/webhook/route.ts
  if (data.lastStatus === "Sld" && !alreadyArchived)   // alreadyArchived = sold + soldDate + enrichmentSource='repliers_webhook'
  → inngest.send("houseloop/listing.sold", { hlid, mls, source: "webhook" })
```

<Warning>
**Never trigger sold processing off `Listing.status='sold'`.** That column is a *cache* field written by `upsertListingToCache` (`lib/listings/cache.ts`) from the polling sync / search caching for **any** home Repliers returns as closed — it does not mean a sale just happened. A cron that swept `status='sold' AND no article` (the old `house-blurb-generate` `0 5 * * *`) backfilled that polluted pool — 100+ fake "sold processing" runs a day. **That cron was removed.** The webhook transition is the only trigger.
</Warning>

## The one workflow — `listing-sold`

The `houseloop/listing.sold` event runs a single Inngest function (`lib/inngest/functions/listing-sold.ts`) that does the whole sequence as one job, idempotently:

1. **idempotency-check** — if the `Listing` row is already fully archived (`status='sold'` AND `soldDate` AND `enrichmentSource='repliers_webhook'`), stop (ONE-TABLE law — the archive lands on the Listing row itself; a stray re-emit never re-runs R2 + LLM).
2. **load** — the live Repliers record (`fields=*,raw.CloseDate` → the real close date).
3. **mark-sold** — `Listing.status='sold'` + `soldDate = raw.CloseDate` + stamp `lastStatusChangeAt`.
4. **archive** — `archiveListingToHistory` → the archive is written ONTO the `Listing` row (photos copied to Cloudflare R2 as `photosCached`, `enrichmentSource='repliers_webhook'`) so the page survives after Repliers drops the listing. (`ListingHistory` is DEPRECATED — quarantined to `ListingHistory_deprecated`; no new rows.)
5. **ai-overview** — invokes `house-blurb-generate` per-hlid for the Sold House AI overview (Haiku).

`house-blurb-generate` has **no cron** — it runs only when `listing-sold` invokes it (or a manual per-hlid invoke). No backfill, ever.

## Observability — the inbound webhook log

Every POST to `/api/repliers/webhook` is recorded in **`hl_webhook_inbound_log`** before any decision: `received_at`, `user_agent` (e.g. `Repliers-Webhook/687`), `source_ip`, the full `raw_body`, and the `previous`/`data` status diff. This is the permanent proof of *who* hit us and *what* they sent — cross-reference it against the `listing-sold` runs in the Run Log. Real Repliers traffic shows the `Repliers-Webhook/*` user-agent and AWS IPs.

## Price history (prior sales) — live per listing

Independent of the sold pipeline: every Repliers single-listing GET returns a `history[]` array (every prior MLS record for the address). Rendered live, newest-first.

- `getPriceHistoryViaRepliers(mlsNumber)` in `lib/listings/providers/repliers.ts` (cached daily). Maps `history[]` → `PriceHistoryEntry[]`.
- `soldDate` = `timestamps.closedDate` (the actual close), **not** the entry's own `soldDate` (the contract date).
- Floored at 2016-01-01, sale records only. Renders in `components/listing/PriceHistory.tsx`.
- **Sold prices are never displayed** (Heartland IDX rule) — the UI shows the publicly-listed asking price only.

## The sold-comps archive — the `Listing` table (ListingHistory DEPRECATED)

The full sold universe (~426K Heartland sales back to 2016) lives on the ONE `Listing` table (sold rows are first-class Listing rows) — the foundation for the sold map / sold search / permanent `/home/{addressSlug}` SEO pages. The old `ListingHistory` table was quarantined to `ListingHistory_deprecated` (doc corrected 2026-07-06 audit — it is NOT the live archive).

- One-time seed: `scripts/backfill-listing-history.ts` (resumable, idempotent, INSERT-only). Source `GET /listings?status=U&lastStatus=Sld&minSoldDate=2016-01-01&boardId=157`.
- Ongoing: `listing-sold` adds a fresh, R2-backed row for each newly-sold home in real time.
- Searchable by address: `/api/locations/autocomplete` matches `ListingHistory.addressSlug` (indexed) and routes to `/home/{addressSlug}`.

<Warning>
**Repliers pagination gotcha.** Repliers paginates with **`pageNum`** + **`resultsPerPage`** — not `page`/`pageSize` (those are silently ignored → page 1 every time). Always use `pageNum` + `resultsPerPage` plus a stable `sortBy` (e.g. `createdOnDesc`).
</Warning>

## Past-client sold homes — a parallel value pipeline

The sold pipeline above is for homes **in our IDX feed** (Repliers webhook → `listing-sold`). **SRG past-client
sold homes are different** — they're pre-existing homes from prior SISU/MLS transactions, not live IDX events, so
they never arrive via the webhook. They are **batch-resolved offline** (see the MLS# anchor in
[Property data & address identity](/guides/property-data-and-address-identity)) into the canonical GT table
(`.claude/reports/pastclient-canonical-FINAL-2026-07-02.csv`), then seeded into `Listing`.

Their **value** follows the **two-engine model** (Zach LOCKED 2026-07-03), not the IDX sold path:
- **① Past-Client · Ninja Monthly** — a cached Zillow **Zestimate × 1.005**, refreshed monthly by the 26th
  `past-client-zillow-cache` job. Zestimate is used **only** for past clients.
- **② Sold · ZHVI Monthly** — every other sold home (no Zestimate): last **sold price × ZHVI area appreciation**,
  anchored on the sold month, recomputed monthly from the 24th ZHVI ingest. **All property types fold in here —
  single-family, condo, apartment, AND land. Land is NOT held flat; it rides the same zip/metro ZHVI as homes
  (single-family lockstep).** New-construction / self-sourced homes (no Zestimate) also run ②.

See [HouseLoop Home Value](/guides/houseloop-home-value) + `GT_HOUSELOOP_HOME_VALUE_ENGINE.md` for the full model.
**Price history** for these still comes from Repliers `history[]` where the MLS# resolves; for off-Zillow / off-market
homes we carry only what our own MLS record holds.
