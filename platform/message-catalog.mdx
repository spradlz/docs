## The two-beat standard (locked)

Every lead-facing workflow fires the same shape:

- **Beat 1 тАФ branded SYSTEM email, instant** (`updates@houseloop.ai`, a real named template): *"Got your
  request тАФ a HouseLoop agent will reach out shortly."*
- **Beat 2 тАФ Myli, ~2 min later** тАФ a **Myli personal email** (`{{ai.*}}` voice) **+ a Myli text**, dropping the
  lead into a Myli workflow she can converse on.
- **System Text is intentionally omitted** тАФ every text goes out as Myli (ONE NUMBER law).
- **`{{ai.*}}` / context-block copy is the accepted Myli-voice layer** тАФ real catalog rows, not vibe code.

## The Blocks Engine (Zach doctrine, LOCKED 2026-07-05)

**The law: templates own EVERY pixel of email markup; code supplies ONLY data.** Runtime *data* injection
(today's 12 matching homes) is normal; runtime *markup* composition in code is banned тАФ it made emails
invisible and uneditable.

**The honest history (why this section exists):** until 2026-07-05, the listing cards in the saved-search
digest and welcome emails тАФ photo box, price line, "View Listing" button, the hero card with the Like/Saved
pill, the "Explore the rest of the market" rows, the "See all N homes" CTA тАФ were **hardcoded HTML inside
`lib/saved-search/match-cards.ts`**. No template row, no preview, no way for a human to see or edit them.
Zach was told repeatedly this class of rot was fixed; it was not. It is now.

**How it works now (mig `0319_blocks_engine`):**
- Repeating/dynamic structures are **BLOCK rows** in `hl_templates` (`kind='block'`, folder **Blocks**),
  editable in Settings тЖТ Templates like anything else:
  `listing_card` ┬╖ `listing_hero_card` ┬╖ `search_other_results_row` ┬╖ `search_other_results_wrap` ┬╖
  `search_see_all_button`.
- The merge engine (`lib/email/template-merge.ts`) gained two primitives:
  - `{{block:listing_card list=items as=listing limit=12}}` тАФ stamps the block once per item
    (`renderBlocks` / `renderBlockOnce`).
  - `{{#ns.field}}тАж{{/ns.field}}` тАФ conditional section, kept only when the value is non-empty
    (powers the coming-soon badge, the sqft segment, the Liked-vs-Saved pill).
- `match-cards.ts` is now a **data-only module**: it fetches listings and hands item objects
  (`photo/price/beds/baths/sqft/address/url/badge`) to the block engine. Zero HTML in code.
- `{{search.matches}}` in a parent template = the `listing_card` block stamped over the lead's real
  matches. **Edit the card once in Blocks тЖТ every email that embeds homes changes.**
- Verified 2026-07-05: the block-engine output is pixel-identical to the old hardcoded render
  (all 67 email templates re-rendered through the real send path, 0 failures).

**ЁЯФС THE JOIN CONTRACT тАФ read before seating ANY card:** A card renders from `hl_trigger_events`.
`event_activity` + `event_category` are resolved in `app/api/leads/[id]/activity/route.ts` тЖТ `resolveEventCatalog()`.
The keys MUST line up or the card falls back / goes blank:
- **System event** (existing lead): resolver does `sysByType.get(canonicalEventType(kind))`. The catalog row's
  `type` must EQUAL the `hl_lead_activity.kind` verbatim (`property_inquiry`, not a past-tense form). тЪая╕П Older
  system rows (`property_viewed`, `tour_requested`) are past-tense and do NOT match their kinds (`property_view`,
  `tour_request`) тАФ their `activity` never resolves; masked only because the compact row prints the badge label.
- **Registration event** (new lead's first action): resolver keys by `normKey(action)`, so the row's `type` must
  be `<action>_registration` (e.g. `property_inquiry_registration`, action `property_inquiry`).
- **HERO vs COMPACT** is decided in `PixelFeedStream.tsx` by `isRegistration || isRequest` (catalog
  `badge.request`) тАФ NOT by `category`. Requests (tour/inquiry/contact/seller/home-value) hero; views/likes/
  saves/searches/deals stay compact. Headline for every hero = `{{first_name}} {{event.activity}}`.

## The two shapes

Every activity renders as one of two cards, decided by **category**:
- **Hero card** тАФ a **Registration** event (the lead's first action *creates* the lead). Big photo + headline +
  address + facts + message.
- **Compact card** тАФ a **System** event (an *existing* lead does the same thing again). A tight icon-tile row.

The *same* action renders both ways: a tour that creates a lead is a hero card; an existing lead's tour is compact.

## ONE template ┬╖ every value is a merge field

**There is ONE card template; every card is DATA, not code.** A card is the template filled by **merge fields**
(`lib/merge-fields.ts`) + the trigger-event catalog `hl_trigger_events`. **Add a card = add a catalog row.** Same
merge-field system Benji's New Lead notification uses (mig 0295).

| Card slot | Merge field(s) | Resolved from |
|---|---|---|
| **Card SIZE** (hero vs compact) | the event's **category** | `hl_trigger_events.category` тАФ `registration`тЖТhero, `system`тЖТcompact |
| **Headline** | `{{contact.first_name}} {{event.activity}}` | `hl_trigger_events.activity` тАФ the per-event phrase |
| **Address** | `{{listing.address}}` / `{{request.address}}` | the home on the activity |
| **Facts** | `{{listing.price}} ┬╖ {{listing.beds}} bd ┬╖ {{listing.baths}} ba ┬╖ {{listing.sqft}}` | listing |
| **Picture tile** | `{{request.photo}}` + `{{lead.source}}` | native тЖТ the photo; else the source logo/text |
| **"via" badge + links** | `{{lead.source}}` | `lib/leads/card-source.ts` тАФ HouseLoop link **left**, portal **right** |
| **Message / quote** | `{{request.message}}` | the buyer's typed text |
| **Tour date chip** | `{{request.date}} {{request.time}}` | dynamic тАФ shows only when present |
