## The two-beat standard (locked)

Every lead-facing workflow fires the same shape:

- **Beat 1 — branded SYSTEM email, instant** (`updates@houseloop.ai`, a real named template): *"Got your
  request — a HouseLoop agent will reach out shortly."*
- **Beat 2 — Myli, ~2 min later** — a **Myli personal email** (`{{ai.*}}` voice) **+ a Myli text**, dropping the
  lead into a Myli workflow she can converse on.
- **System Text is intentionally omitted** — every text goes out as Myli (ONE NUMBER law).
- **`{{ai.*}}` / context-block copy is the accepted Myli-voice layer** — real catalog rows, not vibe code.

## The Blocks Engine (Zach doctrine, LOCKED 2026-07-05)

**The law: templates own EVERY pixel of email markup; code supplies ONLY data.** Runtime *data* injection
(today's 12 matching homes) is normal; runtime *markup* composition in code is banned — it made emails
invisible and uneditable.

**The honest history (why this section exists):** until 2026-07-05, the listing cards in the saved-search
digest and welcome emails — photo box, price line, "View Listing" button, the hero card with the Like/Saved
pill, the "Explore the rest of the market" rows, the "See all N homes" CTA — were **hardcoded HTML inside
`lib/saved-search/match-cards.ts`**. No template row, no preview, no way for a human to see or edit them.
Zach was told repeatedly this class of rot was fixed; it was not. It is now.

**How it works now (mig `0319_blocks_engine`):**
- Repeating/dynamic structures are **BLOCK rows** in `hl_templates` (`kind='block'`, folder **Blocks**),
  editable in Settings → Templates like anything else:
  `listing_card` · `listing_hero_card` · `search_other_results_row` · `search_other_results_wrap` ·
  `search_see_all_button`.
- The merge engine (`lib/email/template-merge.ts`) gained two primitives:
  - `{{block:listing_card list=items as=listing limit=12}}` — stamps the block once per item
    (`renderBlocks` / `renderBlockOnce`).
  - `{{#ns.field}}…{{/ns.field}}` — conditional section, kept only when the value is non-empty
    (powers the coming-soon badge, the sqft segment, the Liked-vs-Saved pill).
- `match-cards.ts` is now a **data-only module**: it fetches listings and hands item objects
  (`photo/price/beds/baths/sqft/address/url/badge`) to the block engine. Zero HTML in code.
- `{{search.matches}}` in a parent template = the `listing_card` block stamped over the lead's real
  matches. **Edit the card once in Blocks → every email that embeds homes changes.**
- Verified 2026-07-05: the block-engine output is pixel-identical to the old hardcoded render
  (all 67 email templates re-rendered through the real send path, 0 failures).

## The Notification Contract (LOCKED 2026-07-09 — the law every notification obeys)

Every HouseLoop notification — no matter what fires it (a website pixel, a CRM action, a Myli send, or a Follow
Up Boss signal) — obeys ONE contract:

- **Owner-only.** `createNotification()` fans a lead notification to the lead's **owner** (`assigned_user_id`) —
  never collaborators, never AI agents. A user with `notif_scope='all'` (Zach) additionally receives the
  team-wide fan-out. There is no blanket team broadcast. *(This corrects the older "team fan-out" framing.)*
- **One card.** Renders through the **One-Card System** — the `notify_card` (email) / `notify_card_sms` (text)
  shells + the `hl_notify_classes` payload row. Never a hand-rolled `<div>` email, never a per-event chrome.
- **Deduped by signal-id.** A notification is keyed on its signal (recipient + event_key + entity), so if two
  paths capture the same event you are notified **exactly once**. This structurally kills the old "double
  message" fan-out.
- **Notify-on-first-capture.** Notify the first time a signal is captured, regardless of when it's processed —
  **not** a brittle "&lt; 2h old" gate. Paired with the recency gate below for connector leads.
- **Integration-aware dual-link.** A notification for an integration lead carries the integration it lives in and
  links to **both** homes where they exist:
  - **Follow Up Boss link — always** for a FUB lead → the FUB person page.
  - **HouseLoop link — only if a HouseLoop lead exists** → the CRM lead page. If **no** HouseLoop lead was created
    (Notify on, Creates-Lead off) → render **"HouseLoop — no lead created"** with no link.
- **Recency-gated (connector leads).** A `fub_created` recency gate on the FUB path guarantees **old activity
  never notifies**, even during a one-off catch-up. Details in the FUB master.

The FUB-specific half of this contract — the four toggles, the recency gate, and the 3-lane capture that feeds
it — lives in the one FUB master: [FUB Trigger Events & Notifications](/fub/trigger-events).

## The bell spine + delivery (how a notification flows)

Every notification writes one row to **`hl_notification`**, the topbar **bell** reads it, and the **dispatcher**
delivers it over the recipient's enabled channels (in-app / SMS / email) per their prefs.

<Note>
**Global delivery masters — the account-wide kill switches (Lee, 2026-07-09).** On TOP of every per-user pref sit
**two account-wide masters** — **Email** and **SMS** — stored in `hl_settings('notification_channel_master')` and
read by `dispatch.ts` on the hot send path (`getChannelMaster()`, cached ~20s, **fail-open**: an absent row or a
read error = both ON, so notifications never silently drop). **SMS master OFF ⇒ nothing texts anyone team-wide;
Email master OFF ⇒ nothing emails** — regardless of individual prefs. The **in-app bell always records** (never
gated here). Admin-only card at the top of **Settings → Notifications** ("Global delivery — master switches"),
served by `GET/PUT /api/notifications/channel-master`. This is distinct from the **FUB `_global` master** (which
gates only the Follow Up Boss pipeline — see [FUB Trigger Events](/fub/trigger-events)); these gate ALL
notification delivery by channel.
</Note>

```
event source                     spine                          surfaces
website pixel / CRM action  ─▶ createNotification() ─▶ hl_notification row  (owner-only)
FUB signal (via the router)                              │
Myli sends a text / logs a call                          ├─▶ bell (unread count + dropdown)
                                                         ├─▶ lead-detail dock badges
                                                         └─▶ dispatchNotification() → One-Card email + SMS (per prefs)
```

## The two shapes

Every activity renders as one of two cards, decided by **category**:
- **Hero card** — a **Registration** event (the lead's first action *creates* the lead). Big photo + headline +
  address + facts + message.
- **Compact card** — a **System** event (an *existing* lead does the same thing again). A tight icon-tile row.

The *same* action renders both ways: a tour that creates a lead is a hero card; an existing lead's tour is compact.

## ONE template · every value is a merge field

**There is ONE card template; every card is DATA, not code.** A card is the template filled by **merge fields**
(`lib/merge-fields.ts`) + the trigger-event catalog `hl_trigger_events`. **Add a card = add a catalog row.** Same
merge-field system Benji's New Lead notification uses (mig 0295).

| Card slot | Merge field(s) | Resolved from |

| **Card SIZE** (hero vs compact) | the event's **category** | `hl_trigger_events.category` — `registration`→hero, `system`→compact |
| **Headline** | `{{contact.first_name}} {{event.activity}}` | `hl_trigger_events.activity` — the per-event phrase |
| **Address** | `{{listing.address}}` / `{{request.address}}` | the home on the activity |
| **Facts** | `{{listing.price}} · {{listing.beds}} bd · {{listing.baths}} ba · {{listing.sqft}}` | listing |

inquiry → notifies + mints the card in `hl_lead_activity` (`native=false`; lead source paints the tile + "via"
badge + portal link). The scraper is **not** in this path — it only recovers masked text/email bodies (Lane 2).
