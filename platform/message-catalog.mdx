## The two-beat standard (locked)

Every lead-facing workflow fires the same shape:

- **Beat 1 — branded SYSTEM email, instant** (`updates@houseloop.ai`, a real named template): *"Got your
  request — a HouseLoop agent will reach out shortly."*
- **Beat 2 — Myli, ~2 min later** — a **Myli personal email** (`{{ai.*}}` voice) **+ a Myli text**, dropping the
  lead into a Myli workflow she can converse on.
- **System Text is intentionally omitted** — every text goes out as Myli (ONE NUMBER law).
- **`{{ai.*}}` / context-block copy is the accepted Myli-voice layer** — real catalog rows, not vibe code.

<Warning>

🔒 locked (verified live + committed). Cards 2–5 are seated in the DB (mig 0301) but unverified; cards 6–20 are

</Warning>

**🔑 THE JOIN CONTRACT — read before seating ANY card:** A card renders from `hl_trigger_events`.
`event_activity` + `event_category` are resolved in `app/api/leads/[id]/activity/route.ts` → `resolveEventCatalog()`.
The keys MUST line up or the card falls back / goes blank:
- **System event** (existing lead): resolver does `sysByType.get(canonicalEventType(kind))`. The catalog row's
  `type` must EQUAL the `hl_lead_activity.kind` verbatim (`property_inquiry`, not a past-tense form). ⚠️ Older
  system rows (`property_viewed`, `tour_requested`) are past-tense and do NOT match their kinds (`property_view`,
  `tour_request`) — their `activity` never resolves; masked only because the compact row prints the badge label.
- **Registration event** (new lead's first action): resolver keys by `normKey(action)`, so the row's `type` must
  be `<action>_registration` (e.g. `property_inquiry_registration`, action `property_inquiry`).
- **HERO vs COMPACT** is decided in `PixelFeedStream.tsx` by `isRegistration || isRequest` (catalog
  `badge.request`) — NOT by `category`. Requests (tour/inquiry/contact/seller/home-value) hero; views/likes/
  saves/searches/deals stay compact. Headline for every hero = `{{first_name}} {{event.activity}}`.

## The two shapes

Every activity renders as one of two cards, decided by **category**:
- **Hero card** — a **Registration** event (the lead's first action *creates* the lead). Big photo + headline +
  address + facts + message.
- **Compact card** — a **System** event (an *existing* lead does the same thing again). A tight icon-tile row.

The *same* action renders both ways: a tour that creates a lead is a hero card; an existing lead's tour is compact.

## ONE template · every value is a merge field

**There is ONE card template; every card is DATA, not code.** A card is the template filled by **merge fields**
(`lib/merge-fields.ts`) + the trigger-event catalog `hl_trigger_events`. **Add a card = add a catalog row.** Same
merge-field system Benji's New Lead notification uses (mig 0295).

| Card slot | Merge field(s) | Resolved from |
|---|---|---|
| **Card SIZE** (hero vs compact) | the event's **category** | `hl_trigger_events.category` — `registration`→hero, `system`→compact |
| **Headline** | `{{contact.first_name}} {{event.activity}}` | `hl_trigger_events.activity` — the per-event phrase |
| **Address** | `{{listing.address}}` / `{{request.address}}` | the home on the activity |
| **Facts** | `{{listing.price}} · {{listing.beds}} bd · {{listing.baths}} ba · {{listing.sqft}}` | listing |
| **Picture tile** | `{{request.photo}}` + `{{lead.source}}` | native → the photo; else the source logo/text |
| **"via" badge + links** | `{{lead.source}}` | `lib/leads/card-source.ts` — HouseLoop link **left**, portal **right** |
| **Message / quote** | `{{request.message}}` | the buyer's typed text |
| **Tour date chip** | `{{request.date}} {{request.time}}` | dynamic — shows only when present |

## The 3 source variants (every card)

A card's **picture tile + "via" badge + link row** are driven by `lib/leads/card-source.ts` from the lead's
source. **Link order is a template rule: HouseLoop link ALWAYS LEFT, source/portal link ALWAYS RIGHT** (`HomeLinks.tsx`).

| Variant | Picture tile | "via" badge | Links (HouseLoop left · source right) |
|---|---|---|---|
| **HouseLoop.ai** (native) | the real house photo | `via HouseLoop.ai` | View on HouseLoop │ View on Zillow |
| **Follow Up Boss · Zillow** | the **Zillow logo** (`/portals/zillow.png`) | `via Zillow.com` | Look up on HouseLoop │ View on Zillow |
| **Follow Up Boss · other portal** | the **source name as text** ("Realtor.com") | `via Realtor.com` | Look up on HouseLoop │ View on Realtor.com |

**The code** — the source drives exactly 3 lines (`source`, `picture`, `homeLinks`); everything else is shared:

```tsx
<ActivityHeroCard
  accentColor="#059669"                        // tour_request badge color
  actor={{ label: 'Lead', color: 'var(--hl-blue)' }}
  title="Requested a Tour"
  pill={{ label: 'Tour Request', bg: '#0596691A', fg: '#059669' }}
  calendarChip="Jul 10, 6:30 pm"               // dynamic: event_payload.requestedDate/Time
  address="14019 W 149th St, Olathe, KS 66062"
  subline="$525,000 · 4 bd · 3 ba · 2,920 sqft"
  quote="Hi — is this home still available? ..." // event_payload.message
  // ↓↓ the only 3 lines the SOURCE changes (cardSource() derives from lead.source) ↓↓
  source={{ label: sourceLabel }}              // 'HouseLoop.ai' | 'Zillow.com' | 'Realtor.com'
  picture={cardPicture({ native, source, photo })}
  homeLinks={native
    ? { houseLoop: '/home/…', portal: 'https://zillow.com/…' }
    : { portal: portalUrl, lookupUrl: '/home/…' }}
/>
```

**What fires it & where the data comes from:** `tour_request` (website) · `fub_tour_request` (FUB import).
HouseLoop.ai → Request-a-Tour form → `/api/pixel/track` → `hl_lead_activity` (`native=true`, real photo). FUB →
webhook → Camp David scraper backfills house history into `hl_lead_activity` at import (`native=false`; lead
source paints the tile + "via" badge + portal link). Renders via `components/leads/detail/ActivityHeroCard.tsx`;
picture/link logic in `lib/leads/card-source.ts`; tokens in `lib/leads/card-tokens.ts`.
