## The two-beat standard (locked)

Every lead-facing workflow fires the same shape:

- **Beat 1 ŌĆö branded SYSTEM email, instant** (`updates@houseloop.ai`, a real named template): *"Got your
  request ŌĆö a HouseLoop agent will reach out shortly."*
- **Beat 2 ŌĆö Myli, ~2 min later** ŌĆö a **Myli personal email** (`{{ai.*}}` voice) **+ a Myli text**, dropping the
  lead into a Myli workflow she can converse on.
- **System Text is intentionally omitted** ŌĆö every text goes out as Myli (ONE NUMBER law).
- **`{{ai.*}}` / context-block copy is the accepted Myli-voice layer** ŌĆö real catalog rows, not vibe code.

## The Notification Contract (LOCKED 2026-07-09 ŌĆö the law every notification obeys)

Every HouseLoop notification ŌĆö no matter what fires it (a website pixel, a CRM action, a Myli send, or a Follow
Up Boss signal) ŌĆö obeys ONE contract:

- **Owner-only.** `createNotification()` fans a lead notification to the lead's **owner** (`assigned_user_id`) ŌĆö
  never collaborators, never AI agents. A user with `notif_scope='all'` (Zach) additionally receives the
  team-wide fan-out. There is no blanket team broadcast. *(This corrects the older "team fan-out" framing.)*
- **One card.** Renders through the **One-Card System** ŌĆö the `notify_card` (email) / `notify_card_sms` (text)
  shells + the `hl_notify_classes` payload row. Never a hand-rolled `<div>` email, never a per-event chrome.
- **Deduped by signal-id.** A notification is keyed on its signal (recipient + event_key + entity), so if two
  paths capture the same event you are notified **exactly once**. This structurally kills the old "double
  message" fan-out.
- **Notify-on-first-capture.** Notify the first time a signal is captured, regardless of when it's processed ŌĆö
  **not** a brittle "&lt; 2h old" gate. Paired with the recency gate below for connector leads.
- **Integration-aware dual-link.** A notification for an integration lead carries the integration it lives in and
  links to **both** homes where they exist:
  - **Follow Up Boss link ŌĆö always** for a FUB lead ŌåÆ the FUB person page.
  - **HouseLoop link ŌĆö only if a HouseLoop lead exists** ŌåÆ the CRM lead page. If **no** HouseLoop lead was created
    (Notify on, Creates-Lead off) ŌåÆ render **"HouseLoop ŌĆö no lead created"** with no link.
- **Recency-gated (connector leads).** A `fub_created` recency gate on the FUB path guarantees **old activity
  never notifies**, even during a one-off catch-up. Details in the FUB master.

The FUB-specific half of this contract ŌĆö the four toggles, the recency gate, and the 3-lane capture that feeds
it ŌĆö lives in the one FUB master: [FUB Trigger Events & Notifications](/fub/trigger-events).

## The two shapes

Every activity renders as one of two cards, decided by **category**:
- **Hero card** ŌĆö a **Registration** event (the lead's first action *creates* the lead). Big photo + headline +
  address + facts + message.
- **Compact card** ŌĆö a **System** event (an *existing* lead does the same thing again). A tight icon-tile row.

The *same* action renders both ways: a tour that creates a lead is a hero card; an existing lead's tour is compact.
