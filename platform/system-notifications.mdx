## What it is

When something happens to a lead (new registration, stage change, inbound text, AI action, …), the CRM raises a
notification to the right agent(s) over their chosen channels (in-app bell · email · text). ~30 events across
7 groups: Lifecycle · Communications · Lead Activity · AI Follow-Up · Team · System Alerts · Website Digest.

## The law (non-negotiable)

**Every notification — email AND text — is a complete, human-editable TEMPLATE row in `hl_templates`. No code
assembles email HTML. The template PREVIEW equals exactly what SENDS.**

The old sin (killed 2026-07-02): the branded card (logo, contact card, agent, CTA, footer) was built in CODE
(`renderNotificationInner()` + `notificationShell()`) and wrapped around a bare one-line template body — so the
real email was invisible to the template editor, and previews showed a bare paragraph. That is banned. The card
is now a template.

## How it works (the path)

1. Event fires → `notifyLeadEvent()` / `createNotification()` inserts an `hl_notification` row (bell + real-time push).
2. `dispatchNotification()` (`lib/notifications/dispatch.ts`) reads the user's prefs + channels, loads the
   template, merges, and sends.
3. Template lookup: `lib/notifications/template-registry.ts` (`NOTIFICATION_TEMPLATES`) maps each event →
   `{emailKey, smsKey, group, leadScoped}`. Email via Resend, text via `sendSms` (Twilio) / BlueBubbles.
4. Every send logs to `hl_run_log` (visible in **Run Log**) and, for texts, to `hl_messages` (Communications SSOT).

## Architecture — chrome template + content template

- **`notify_base`** (`hl_templates`, kind=email) — the canonical **branded chrome**, editable HTML with ONE
  `{{notify.content}}` slot: logo → `{{notify.content}}` → contact card (`{{contact.initials/phone/email}}` +
  `{{listing.address}}`) → `{{notify.message}}` → `Assigned Agent · {{notify.agent}}` →
  `Open lead in HouseLoop CRM →` (`{{contact.crm_url}}`) → `{{company.footer}}`. **Edit the chrome once, here.**
- **`notify_base_system`** — same chrome WITHOUT the contact card (system alerts, no lead).
- **Each `notify_<event>` template = the CONTENT only** (the event headline + the enriched sub-line; the chrome adds the rest).
- **`composeNotificationEmail(baseBody, contentBody)`** (`lib/email/compose-notification.ts`) injects the content
  into the base slot. Used by **both** send and preview — they can never diverge.
- **Templates are complete HTML docs (migration 0300).** No code assembles the email — the `hl_templates` row IS
  the email (chrome baked in). See [Email Templates](/platform/email-templates-system).

### Send + preview flow
- **Send** (`lib/notifications/dispatch.ts`): resolve the merge scope → compose base+content → `applyMerge()` once
  → `sendViaResend()`.
- **Preview** (`NotificationTemplateModal` on Settings → Notifications; `EmailTemplatePreviewModal` on Settings →
  Email Templates): `resolveForPreview(compose(base, content))` → the FULL branded email, live as you edit.

## The enriched sub-line (2026-07-04, LIVE — migration 0310)

Every notification is now a **HEADING + an enriched sub-line** (what happened · when · from where · which property ·
the link), merge-field driven, and rendered **identically in email and text**. The per-event enriched-sentence
spec (heading · sub-line · exact tokens · data source) is the scoreboard in the PRD
`GT_HOUSELOOP_ENRICHED_NOTIFICATIONS_PRD.md` §2 — that table is the single source for the copy each event renders.
Each enriched block also rolls up into a reusable [AI Context Block](/platform/ai-context-blocks) Myli injects
before outreach (Layer 2). Live audit canvas: [artifact e5582966](https://claude.ai/code/artifact/e5582966-451a-4744-a717-8f80005b923c).

## Merge fields the templates use

| Field | Resolves to |
|---|---|
| `{{notify.content}}` | the per-event content (composed into the base by the composer) |
| `{{notify.message}}` | the real inbound message / AI action detail |
| `{{notify.agent}}` | the lead's ASSIGNED agent (Ana) — or Myli for AI |
| `{{notify.title}}` | the AI-composed dynamic headline (AI notifications) |
| `{{notify.link}}` | the home/search the lead acted on |
| `{{event.activity}}` · `{{event.time/date/datetime}}` | the per-event word (from `hl_trigger_events.activity`) + the timestamp |
| `{{stage.from/to/days_in_prev}}` · `{{actor.name/role}}` | stage transitions + who did it |
| `{{contact.full_name/first_name/phone/email/crm_url/initials}}` | the lead |
| `{{listing.address}}` / `{{listing.url}}` · `{{request.*}}` · `{{search.*}}` · `{{valuation.*}}` | the property / request / search / seller data the event carried |
| `{{lead.platform}}` / `{{lead.source}}` | HouseLoop.ai / Follow Up Boss · Zillow etc. |
| `{{call.*}}` · `{{email.subject/snippet}}` | call + inbound-email specifics |
| `{{company.footer}}` | the canonical CAN-SPAM brand footer |

Full vocabulary: [Merge Fields](/platform/merge-fields) (registry `lib/merge-fields.ts`).

## Where to manage it

`/settings/notifications` — every event in its group, per-channel toggles (email/text), a View/Edit modal per
template, "My Channels" (destination phone/email), and Company Senders (`notifications@houseloop.ai` team ·
`updates@houseloop.ai` client). Per-user prefs live in `hl_users.data.notification_prefs` (opt-in, gated by
`dispatch.ts`). Component: `components/settings/notifications/NotificationTemplateModal.tsx`.

## Mistakes never to repeat
1. **No code-built email HTML.** The card is a template (`notify_base`), not `renderNotificationInner`.
2. **No stubs.** Every content template is distinct + event-specific, not `body == subject` and not one shared
   `✨ {{notify.title}}` copied across all AI events.
3. **No orphaned duplicates in the folder.** `*_agent_alert` / `new_lead_saved_search_agent` are legacy, not fired
   by dispatch — they do NOT belong in User Notifications.
4. **Preview == send.** If the preview shows a bare paragraph, the composer isn't wired — that's a bug.
5. **Email == text.** The enriched sub-line renders identically in both channels.
