## Overview

HouseLoop CRM is built API-first. Every action you can take in the UI has a corresponding REST endpoint, which means Myli (and any integration partner) can do everything a human operator can.

**Base URL**
```
http://localhost:3020/api       (local dev)
https://crm.houseloop.ai/api    (production)
```

**Authentication:** Endpoints require either a valid session cookie (set by `POST /api/auth/login`) or a service API key passed as `Authorization: Bearer <key>`. Some endpoints are explicitly unauthenticated (webhooks, public health checks, OAuth callbacks).

## Big rules (read before you build)

- **Appointment disposition law** — scheduling certain types auto-advances stage; certain outcomes auto-advance further. Forward-only. See [GT_HOUSELOOP_CRM_APPOINTMENT_DISPOSITION_V1](https://github.com/spradlz/BossLoop-Gold/blob/houseloop-app/GT/GT%20-%20ACTIVE/GT_HOUSELOOP_CRM_APPOINTMENT_DISPOSITION_V1.md) for the locked spec.
- **Myli access** — Myli authenticates via the same session cookie or a service token and can call every endpoint. Mark mutations from Myli with `marked_by: 'myli'` (where supported) so the audit trail distinguishes her actions.
- **Two pipelines, one lead** — a lead can be on Buyer + Seller simultaneously. Use `POST /leads/{id}/pipeline` (not `PATCH /leads/{id}`) when you want to move the lead within one pipeline only.

## Quick Start

<CodeGroup>
```bash Create a Lead
curl -X POST https://crm.houseloop.ai/api/leads \
  -H "Authorization: Bearer $HL_SERVICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "first_name": "Jane",
    "last_name":  "Smith",
    "email":      "jane@example.com",
    "phone":      "5555550100",
    "source_id":  "<uuid>",
    "assigned_user_id": "<uuid>",
    "stage_id":   "<uuid>",
    "note":       "Met at open house"
  }'
```

```bash List Leads
curl https://crm.houseloop.ai/api/leads \
  -H "Authorization: Bearer $HL_SERVICE_TOKEN"
```

```bash Move a Lead to a Stage
curl -X PATCH https://crm.houseloop.ai/api/leads/{id} \
  -H "Authorization: Bearer $HL_SERVICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"stage_id":"<stage-uuid>"}'
```

```bash Disposition an Appointment (Myli)
curl -X PATCH http://localhost:3020/api/appointments/{id}/outcome \
  -H "Content-Type: application/json" \
  -d '{
    "outcome": "listing_signed",
    "outcome_notes": "Signed at $450k, 6-month term. Photos Wednesday.",
    "marked_by": "myli"
  }'
```

```bash Schedule a First Buyer Appointment (auto-advances stage)
curl -X POST http://localhost:3020/api/calendar/events \
  -H "Content-Type: application/json" \
  -d '{
    "lead_id": "<uuid>",
    "title":   "Buyer Consultation — Jane Smith",
    "type":    "first_buyer_appointment",
    "starts_at": "2026-05-21T15:00:00Z",
    "ends_at":   "2026-05-21T16:00:00Z"
  }'
```
</CodeGroup>

## Design Principles

- **REST + JSON** — all requests/responses are JSON
- **UUIDs** — all IDs are UUID v4
- **Timestamps** — ISO 8601 (`2026-05-18T23:45:00Z`)
- **Soft deletes** — leads are never hard-deleted; they get `deleted_at` and `status='trash'`
- **Pagination** — list endpoints support `?limit=` and `?offset=` (default limit: 500)
- **Error format** — `{ ok: false, error: "reason" }` with appropriate HTTP status (400 / 401 / 403 / 404 / 409 / 422 / 500 / 502 / 503)
- **Forward-only stage advance** — automatic stage moves driven by appointment events never drag a lead backward. Manual stage clicks bypass this rule.

## Live endpoint count

The API surface is specified in full by `openapi.yaml` — treat that file as the single source of truth for the live count. As of this writing `openapi.yaml` carries roughly **119 path entries across 27 tags** (the older "82 endpoints / 18 tags" figure is stale). Because the spec grows, don't quote a hardcoded number from this page — check `openapi.yaml` directly. If you find a UI action without a corresponding endpoint, file it in [API Gap Audit](/guides/gaps-audit).
