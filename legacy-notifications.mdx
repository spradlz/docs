## The roll-up

### [System Notifications](/platform/system-notifications)
**Covered:** THE canonical catalog of every agent/user notification (email + text) — the per-event spec table, the branded chrome, the merge fields. Absorbed Benji's `notification-templates` WIP on 2026-07-04.
**Replaced by:** [Master → part a, The One-Card System](/lee-master#a-the-one-card-system-one-shell-payload-driven-locked-2026-07-06). The 80-template catalog collapsed to ONE `notify_card` shell + the 32-row `hl_notify_classes` payload matrix; the 66 old `notify_*` rows are quarantined.
**Why kept:** the per-event spec table is the intended-state grounding for what each class should say.

### [Email Templates](/platform/email-templates-system)
**Covered:** the email template system — the three shells (`notify_base`, `notify_base_system`, `marketing_base`), full self-contained HTML since mig 0300, USED-IN classification, SENT counts.
**Replaced by:** [Master → part b, The Blocks Engine](/lee-master#b-the-blocks-engine-templates-own-every-pixel-locked-2026-07-05). Templates now own every pixel of markup; repeating structures are editable **block** rows; code supplies only data.
**Why kept:** documents the shell lineage and the "preview = send" guarantee that the Blocks Engine preserves.

### [Text Templates](/platform/text-templates-system)
**Covered:** SMS / iMessage templates — same `hl_templates` table + merge fields as email, USED-IN + SENT parity, segment-count discipline.
**Replaced by:** [Master → part b (Blocks Engine)](/lee-master#b-the-blocks-engine-templates-own-every-pixel-locked-2026-07-05) + the **ONE-NUMBER Myli text rule** in [part d](/lee-master#d-delivery-plumbing-prefs-channels-fan-out-proof) (every text goes out as Myli; system text is intentionally omitted).
**Why kept:** the parity model and segment discipline still describe how a text row is authored.

### [System Display Messages](/platform/system-display-messages)
**Covered:** the on-screen messaging layer — toasts, banners, empty-states, inline confirms/errors. Audit finding: **no unified system exists yet** — it's ad-hoc, per-component.
**Replaced by:** not yet — rolled into the master as a **flagged gap** ([part d](/lee-master#d-delivery-plumbing-prefs-channels-fan-out-proof)), so it stays on the map rather than lost.
**Why kept:** it IS the current (ad-hoc) state — the honest record that there is no display-message system yet.

### [Notifications (guide)](/guides/notifications)
**Covered:** the original CRM notification system — the `hl_notification` bell spine, the code-built centered branded email shell (FUB-style contact card + button), source-agnostic messages, the Myli format, the one matrix, and the build audit log.
**Replaced by:** [Master → part a (One-Card System)](/lee-master#a-the-one-card-system-one-shell-payload-driven-locked-2026-07-06). The **code-built email chrome is retired** — templates own the full email. The bell / `hl_notification` spine remains; connector (Zillow/FUB) leads get no system notification (Myli reaches out personally).
**Why kept:** the bell-spine flow and the original branded-shell intent are the grounding for how the delivery layer evolved.

### [Lead Activity Log](/guides/lead-activity-log) *(notification parts only)*
**Covered:** the canonical run-log (`hl_lead_activity`) — one store, one writer, one catalog, one reader; AI cost lives on the catalog. Its **notification-copy mentions** are the only part in scope here.
**Replaced by:** the notification-copy references defer to [Lee Master](/lee-master). **The rest of that doc stays LIVE and authoritative** — it is the activity-log SSOT, not a legacy notification doc, and carries no full red-line.
**Why kept:** it's a live doc; only its notification-copy overlaps this master, and that overlap now points here.
