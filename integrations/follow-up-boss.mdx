## The law: FAIL CLOSED

**No FUB signal creates a HouseLoop lead unless its trigger event's `Creates Lead` toggle is explicitly ON —
and the `_global` master switch is ON.** A signal with no config row does NOTHING (logged skip). This is
enforced in code with fail-closed defaults (`fub-lead-import.ts`) after the 2026-07-09 incident where fail-open
defaults minted 422 unauthorized leads with every toggle off — all purged, gate live-fire proven (111 backlog
imports vs the closed gate → 0 leads).

Three belts keep it that way:
1. **Import-layer door hygiene** — existing-person re-imports and behavioral/comms webhook types never fire
   native registration doors.
2. **Fail-closed config gate** — no row / toggle off / `_global` off → no lead, no workflow, receipt logged.
3. **Dispatch registration guard** — a `*_registration` door can never route via a fallback tier
   (`runtime.ts`), so first-touch outreach to imports is structurally impossible.

## How a FUB signal flows

```
Zillow / Realtor / PPC → Follow Up Boss
  → native webhook → fub.* mirror (raw truth, always captured)
  → classification (classifyFubEvent → the atomic trigger event)
  → hl_fub_webhook_settings row = THE decision:
       Sync (ingest to history) · Creates Lead · Workflow · Notify — four independent toggles
  → creates_lead ON → hl_leads (platform='follow_up_boss') → the FUB door fires → hl_trigger_fire ledger
```

FUB-platform fires show on the **Follow Up Boss** section of [Trigger Events](/guides/trigger-events) — never
on the native HouseLoop.ai board (ledger scoped by platform).

## What a FUB signal becomes in HouseLoop — the 2026-07-10/11 hardening

Once captured, a lead's activity is mirrored into HouseLoop the right way — governed by **one master**, **forward-only**,
with **real timestamps**. What shipped this pass (deep detail: [FUB Lead Import](/fub/lead-import)):

- **ONE FUB master switch.** There were secretly **two** — the UI toggle wrote `hl_settings.fub_sync_master` while
  the fail-closed notify/process gate read `hl_fub_webhook_settings._global.enabled`. They diverged, `_global` was
  orphaned OFF, and it **silently killed every FUB notification for hours**. Collapsed: `getFubSyncMaster` /
  `setFubSyncMaster` now read/write `_global.enabled` — the flag the UI shows and the code obeys (notify + process +
  forward-sync). It can't orphan again.
- **The lead-detail feed is FORWARD-ONLY.** A pulled-in lead starts CLEAN — pre-import texts/calls/emails stay
  context-only under the FUB tab; only new comms flow into the feed + Texts/Calls/Emails sub-tabs going forward
  (`backfillFubComms` watermark floored at the lead's `created_at`).
- **Real timestamps, not "22 hours ago."** Property view/save activity is stamped from `scraped_at` (the real
  trigger time, matches FUB to the minute) instead of a date-only midnight, and the feed renders the actual date +
  time (Central), not a relative "Xh ago."
- **ONE home-activity writer.** The dashboard's `parseZillowViewNote` ghost writer — which hardcoded every Zillow
  home note as `activity_type='viewed'` (so **saved** homes recorded as **viewed**) — was removed;
  `lib/fub/sync-lead-record.ts` is the sole, correct writer (derives saved-vs-viewed, stamps `fub_event_id`). 1,318
  legacy mislabeled rows were cleaned (565 dup deletes + 753 corrected → `saved`).
- **Inquiries/tours flow like texts.** `fub.events` is now stored **unconditionally** in the webhook router, so a
  new inquiry/tour always lands and renders forward (the July-7 ingest ceiling is lifted).
- **Myli's facts are distilled, not dumped.** The extractor locks the slot vocabulary (`canonicalSlot`) so facts
  can't bloat, and raw house-view lists collapse into ONE synthesized signal ("viewed 32 homes in 2 days → high
  urgency"). A bloated lead went **55 junk facts → 13** meaningful ones, stable on re-run.
- **The import wizard is a filter + assign + tag builder.** Select who comes in by FUB **stage** and **tag** (a
  searchable modal over all ~1,276 live FUB tags, AND'd with stages, with a live match count); **assign** every
  imported lead to a person (or Myli-AI-only, never ownerless); **apply HouseLoop tags** (pick existing or create
  new) on top of the origin tag + the lead's own imported FUB tags. One-way, stage-mapped, Rule #0-gated on the full run.

## Lead-source canonicalization — the six buckets

Follow Up Boss carries **hundreds of messy source strings** ("Zillow Preferred", "Premier Agent", "Trulia.com",
"Agent COI", "&lt;unspecified&gt;", "FiveStreet", "Opcity"…). Importing those verbatim would give HouseLoop an
un-filterable soup. So every FUB lead's source is **folded into exactly SIX canonical buckets** — the normalized
list you import into and report on. Locked by Zach 2026-07-04: *"clean lead sources — six buckets, everything
else is noise."*

| Canonical bucket | `hl_lead_sources` key | What folds in |
|---|---|---|
| **Zillow** | `fub_zillow` | Zillow, Zillow Preferred, Premier Agent, Zillow Flex, **Trulia** (Zillow-owned) |
| **Google PPC** | `fub_google_ppc` | anything carrying the `Google Adwords PPC` tag (tag wins, even over a Zillow source string) |
| **Google Organic** | `fub_google_organic` | `google` / `google organic` source text |
| **Realtor.com** | `fub_realtor` | `realtor.com` |
| **PC SOI** | `fub_pc_soi` | Agent COI, SOI, **Referral** (a referral is relationship-based), SRG past client |
| **Other** | `fub_other` | everything else — FiveStreet, Opcity, Homes.com, HomeLight, unknown/new strings |

**The fold happens in ONE place** — `lib/fub-import/source-canonicalize.ts` (`canonicalizeFubSource`), the single
map. Three rules, in order: ① the **`Google Adwords PPC` tag wins** → Google PPC (it beats the source field);
② the **fold map** matches the source text → its bucket; ③ **default = Other** — an unknown or brand-new string
NEVER invents a bucket, it lands in Other. Nothing is lost: the fine original origin is preserved on the lead as
a **`Lead Source - <original>` tag** (e.g. `Lead Source - Opcity`). The buckets are seeded by **migration 0315**;
the old hardcoded `Zillow.com` import label was killed so imports stamp the *real* resolved bucket.

### We do NOT rewrite FUB's Source field — we tag (it's immutable)

**Follow Up Boss's `source` field cannot be changed via the API** (probed 2026-07-04 → "No valid fields").
So inside FUB, a lead's Source stays exactly what it always was — "Zillow Preferred", "Premier Agent",
"zillow.com" — **forever**. We never touch it. The normalized bucket lives in FUB as a **tag** instead.

The one-time backfill — a registered Inngest function (`fub-source-sweep`, event `fub/source-sweep.run`,
**manual, no cron**; Rule #0-gated, reversal file saved first) — stamped the entire book:
- **`HL Source: <bucket>`** — the canonical bucket (e.g. `HL Source: Zillow`), for filtering inside FUB.
- **`Lead Source - <origin>`** — the exact original string preserved (e.g. `Lead Source - Zillow Preferred`).

Tag writes are GET → merge → PUT off the **live** FUB record (never the mirror), so a stale array can't delete a
tag a lead just earned. **Verified live (mirror, this session):** **48,920 of 49,234** people carry an
`HL Source:` tag (**28,072 = Zillow**); only **64** Zillow-family stragglers still need it.

<Warning>
**The `HL Source:` tag is the normalized signal in FUB — the Source field is NOT.** Inside FUB, "Zillow
Preferred" (and the other raw strings) will always show as the Source; that's expected and unavoidable (FUB owns
that field). The `HL Source: Zillow` tag is where the bucket lives. Likewise, **do not read
`fub.people.source` in our mirror as the canonical bucket** — the mirror re-syncs FUB's raw source over any fold,
so it also shows the raw strings. The canonical bucket is derived by the resolver / carried by the tag, never
stored durably in a source column.
</Warning>

### How HouseLoop gets the bucket (it derives, not reads the tag)

When a FUB lead **imports into HouseLoop**, the import runs the raw Source string through the fold map
(`canonicalizeFubSource`) → resolves the bucket → stores it on the HouseLoop lead (`hl_leads.source_id`) and
adds the `Lead Source - <origin>` tag. So the HouseLoop lead's Source = **Zillow**, even though FUB's field says
"Zillow Preferred". HouseLoop derives the bucket from the raw string — it does **not** depend on the
`HL Source:` tag (the only tag it reads as an override is `Google Adwords PPC` → Google PPC).

**Where you see it:** the **Platform / Source** page — a platform card + the six-bucket KPI rows (click a bucket
to filter its leads); each imported lead shows its bucket as its Source, plus the `Lead Source - <origin>` chip
on lead detail. New inbound is canonicalized the same way at import time. Live bucket spread across 49,234
`fub.people` (by `HL Source:` tag): **Zillow ≈ 28,072**, the rest split across Google PPC · PC SOI · Realtor.com ·
Google Organic · Other.

## Boundaries that never move

- **Deleting a HouseLoop lead NEVER touches Follow Up Boss** — the delete path has zero FUB references
  ([Lead Deletion](/guides/lead-deletion)); FUB people stay intact no matter what's purged here.
- **The `fub.*` mirror is read-only truth** — capture everything, act on nothing without a toggle.
- Notifications ship from the System line/address — never a person's, never Myli's.
