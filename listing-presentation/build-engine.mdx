## The scripts

| Script | Input | Output |
| --- | --- | --- |
| `build-lp-v3.py` | brand assets, `curated-reviews.json` | `sg-sellers-page.html` — the 17-page presentation deck |
| `rebrand-cma.py` | the Claude-Cowork CMA HTML (in Downloads) | `cma-branded.html` — the CMA in our brand |
| `compress-cma.py` | `cma-branded.html` | `cma-branded-web.html` — a &lt;16MB copy for the on-screen artifact only |

The print PDF is produced by rendering each HTML with headless Chrome and merging with `pdfunite`.

## What `rebrand-cma.py` does (in order)

It reads Cowork's raw CMA (navy + **gold**, Segoe UI) and applies deterministic transforms:

1. **Swap the entire stylesheet** for our token system — navy `#0F2153`, brand blue `#2D6DFF`, no gold,
   serif display face, our card/table/pill styles.
2. **Drop our white lockup** on the cover (transparent PNG, never a white box).
3. **Fix the fake galleries.** Cowork padded each data sheet's "19-photo" gallery by repeating the one
   real photo 19+ times. The engine hoists that single real photo to the **top of the sheet as a hero**
   and deletes the duplicate grid.
4. **Add the difference column** to every comp table — the your-home-vs-comp delta, computed from the
   row values: green when we're ahead, amber when we're behind (**never red**), a dash when there's no
   clean numeric diff.
5. **Retune the numbers + commentary** to the reconciled recommendation and remove contradictions. These
   are the per-listing content edits (the pricing story) — the one part that is judgment, expressed as an
   explicit, reviewable list of find-and-replace edits so it's still reproducible from the script.
6. **Merge the price graphic into the Executive Summary** (Min / Recommended / Max) and even up the
   comp columns.

## Reproduce it

```bash
cd .claude/reports/listing-presentation-work

# 1. build both halves
python3 build-lp-v3.py          # -> sg-sellers-page.html  (presentation)
python3 rebrand-cma.py          # -> cma-branded.html      (CMA)

# 2. serve locally so Chrome can render with images
python3 -m http.server 8811 &

# 3. render each to a full-resolution PDF
CHROME="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
"$CHROME" --headless --disable-gpu --no-pdf-header-footer \
  --print-to-pdf=pres.pdf "http://localhost:8811/sg-sellers-page.html"
"$CHROME" --headless --disable-gpu --no-pdf-header-footer \
  --print-to-pdf=cma.pdf  "http://localhost:8811/cma-branded.html"

# 4. merge into one print PDF
pdfunite pres.pdf cma.pdf "$HOME/Desktop/Spradling-Group-Listing-Presentation-<name>.pdf"
```

## Print discipline (non-negotiable)

- **Every `.sheet` / `.page` is one US-Letter page** — `@page{size:letter;margin:0}` + `min-height:11in`
  + `page-break-after:always`, with `page-break-after:auto` on the last sheet so there's no trailing blank.
- **Measure, don't assume.** After any change, each page's rendered height must stay under the ~1056px
  (11in @ 96dpi) print boundary. The build checks this with Playwright before rendering the PDF; a page
  that spills to a second physical sheet is a bug, fixed by tightening spacing — not shipped.
- **Verify the actual PDF**, not just the browser: `pdfinfo` must report the expected page count with
  every page `612 x 792 pts (letter)`, and individual pages get rendered to PNG to confirm nothing
  bleeds across a break.

## The brand law it enforces

Ported straight from our UI doctrine so the CMA matches the CRM and the website:

- Navy `#0F2153` is the only navy; brand blue `#2D6DFF` is the accent. **No gold, no green** used
  decoratively (green only ever means genuine positive/advantage — e.g. a "+" difference).
- The HouseLoop lockup is **always transparent**, never on a white box.
- Real photography only. No duplicated or fabricated imagery — if we only have one real photo, we show
  one, not a faked gallery.
- Tokens, not raw hex, for anything that can be a token.

## Cloud CMA → HouseLoop (the current, correct process)

The CMA half is now generated in **Cloud CMA** and rebuilt in our brand. Cloud CMA exports a **teal,
non-branded** PDF (its own template, all pages selected). **You do not staple that in** — it clashes with our
navy/blue, repeats itself, and pads galleries. You **extract its data + photos and rebuild it** with
`build-cma.py`. (`rebrand-cma.py` above is for the older hand-authored CMA HTML — a Cloud CMA PDF uses
`build-cma.py` instead.)

<Steps>
<Step title="Get a non-branded Cloud CMA (read-only)">
The agent generates the CMA in Cloud CMA (all pages selected) and exports the PDF — it lands in Downloads with a
hash name. **Never edit or delete it.** All extraction is read-only.
</Step>
<Step title="Extract data + ALL photos">
`pdfimages -j -p <cma.pdf> img/p` (every photo, `p-<page>-<n>.jpg`) and `pdftotext -layout <cma.pdf>` (the comp
text). Parse into `data.json` (subject + comps + pricing) and `photos.json` (a front-exterior hero per comp;
galleries use **all** of each comp's photos, deduped).
</Step>
<Step title="Rebuild in our brand">
`python3 build-cma.py` → `cma-13210-branded.html`: a CMA cover, then **per comp a 2-page flow** — page 1 =
hero + subject-vs-comp comparison table with the difference column + price + bottom line; page 2+ = **all** the
photos in a 4-col wall. Navy/blue tokens (`cma-style.css`). No teal, no gold, no 6-photo caps, no repeated
hero/stats.
</Step>
</Steps>

### The final merged order (proven on the Viktor Okoth build)

One PDF, this exact order: **Cover → By the Numbers → Listing presentation body → CMA cover → CMA comps →
Suggested Price Range → Thank You.** The Suggested Price Range page's recommended field is a **white write-in
box** (blue `$` on the left, blank) for the agent to fill in pen; Min/Max are filled. Range + Thank You are
**dead last**. Assemble with `pdfseparate` + `pdfunite` (build the input list as a zsh array). Full recipe,
pain-points, and the locked source/data/photos: `.claude/reports/listing-presentation-work/viktor-okoth/BUILD-HANDOFF.md`.

### Reproduce for ANY property — env-driven (proven on the Merriam build)

All three builders are property-agnostic; nothing is hardcoded to a listing. Drive them with env vars:

- **`build-cma.py`** — `CMA_EX=<extract dir>` (contains `data.json` + `photos.json` + `img/`). Everything else
  (seller, subject, comps, per-comp `photoPages`, the "closest match" anchor) comes from `data.json`.
- **`build-lp-v3.py`** (deck) — `LP_SELLER`, `LP_ADDR` (cover), `LP_SUBDIV`, `LP_ADDR2`, `LP_MIN`, `LP_MAX`
  (Suggested Price Range page). Defaults are the last listing.
- **`build-track-record.py`** (By the Numbers) — `TR_AREA` (header label) + `TR_SALES_JSON` (a JSON array of
  `[address, city, listStr, soldStr, pctStr]`, pulled from `hl_srg_mls_deals` for the subject's city + price band).

**Two comp sections.** A comp in `data.json` may set `"tier":"additional"` (default `"closest"`) and
`"imgDir":"img2"` when its photos came from a *second* Cloud CMA export. Multiple tiers render a navy
`.page.divider` section header — **Closest Comparables** (same-complex anchors) then **Additional Area
Comparables** (context). The "closest match" anchor is only ever chosen from the `closest` tier.

**Watch-outs the Merriam run surfaced:** the seller name is never in the Cloud CMA (comes from the call sheet /
you); a thin market may only export 2 comps (widen the Cloud CMA + re-export to add more, then confirm they're
in the PDF); the anchor must exclude `additional` comps; the section divider needs the high-specificity
`.page.divider` rule to render navy (not white).
