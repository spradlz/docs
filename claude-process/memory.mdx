Claude's context is wiped between sessions and compacted mid-session. So "memory" here is not the model
remembering — it's a set of **durable files that get injected back into context** at session start and after every
compaction. The discipline is: **a fact worth keeping goes into exactly one durable artifact, and that artifact is
auto-loaded** — never left only in the conversation, where it evaporates.

## The layers

| Layer | Where | What it holds | Updated by |
|---|---|---|---|
| **Working memory** | `.claude/state/progress/<subproject>.md` | The live scoreboard for the active subproject — current step, step log, decisions, next action. | Auto-written after every edit; auto-synthesized; injected on session start + post-compaction. |
| **Shipped/closures** | `.claude/state/progress/_shipped/<subproject>.md` | What's been replaced or killed, so dead work isn't redone. | Auto-detected from git diffs. |
| **Lessons** | `.claude/lessons/` | Topic-scoped "never do this again" write-ups. | Written when a lesson is learned; scanned by topic each session. |
| **Rules (constitutional)** | `.claude/rules/*.md` | Hard laws that override defaults (e.g. `UI_ITERATION_HTML_FIRST`, `WORKFLOW_CANVAS_IS_TRUTH`). | Written once, injected every session via `CLAUDE.md`. |
| **Long-term memory** | the agent memory store (`MEMORY.md` index + one file per fact) | Durable facts about the user, feedback, projects, and references. | Written as facts are learned; the index is loaded every session; individual files recalled on relevance. |
| **This process doc** | `docs-site/claude-process/` | The operating methods + the lessons that shaped them. | Updated section-by-section, on demand (see below). |

## How memory gets updated

- **Automatically** — working-memory progress files and closure detection run on hooks (after edits, before
  compaction, on a cron). No manual journaling; the scoreboard maintains itself.
- **On a lesson** — when we learn a better way, it's captured in the right layer: a **rule** if it's a hard law, a
  **lesson file** if it's topic guidance, a **memory file** if it's a durable fact, and a **section of this doc** if
  it's a working method. The same lesson can land in more than one layer (a rule *and* this doc) so it's both
  enforced and explained.
- **Before saving** — check for an existing file that already covers it and update that, rather than creating a
  duplicate. One fact, one home.

## This doc as a memory surface

The **Claude Process** tab is itself a memory layer — the human-readable home for *how we work*. It updates
**section-by-section, on demand**: Zach names the section (UI Design, Memory, Deploys…) and the lesson we just
learned, and that section is rewritten to fold it in. It is not auto-generated prose; it is deliberately curated so
each section stays the current best method. New domains become new sections as they're learned.

<Note>
The test for whether a fact is "remembered": will it be injected back into context next session **without anyone
re-typing it?** If yes, it's in memory. If it only lives in the chat, it's already gone.
</Note>
