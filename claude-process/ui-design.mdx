title: "Claude Process — UI Design"
description: "How we iterate UI on HouseLoop: design fast in HTML, lock the wireframe, derive an exact spec from that HTML node-by-node, then port it to React verbatim. The wireframe is the source of truth; the build is a translation, never a reconstruction."
---

UI moves fastest when the thing we're approving **is** the thing we build from. Claude can iterate an HTML
wireframe in seconds — change a layout, restyle a card, move a control — and Zach can eyeball and approve it
instantly. That speed is the point. The failure mode is letting the approved *pixels* get re-described in prose and
then rebuilding from the prose. This section is the process that keeps the pixels authoritative end to end.

## The method — HTML first, port second

### Phase A — iterate in HTML (fast, WYSIWYG)

1. Claude builds an **HTML wireframe** using the real HouseLoop tokens (navy `#1B3A8A`, blue `#2D6DFF`, hairline
   `#E5E7EB`, radii, Inter) and publishes it as an **artifact** Zach can open.
2. Zach directs edits; Claude edits the **same HTML file** live and re-publishes. Repeat until Zach says
   **"the HTML is locked."**
3. **Every edit lands in the HTML first.** A verbal change ("stack these two", "abbreviate that") is not a
   decision until it's *in the wireframe* and re-confirmed. The approval surface and the build source never diverge.

### Phase B — port the locked HTML (deterministic, no eyeballing)

Once locked, Claude does **not** rebuild from memory. It runs a fixed translation:

1. **Freeze + pin** the locked HTML into the repo (`docs/wireframes/…`) so it's version-pinned, not an ephemeral file.
2. **Derive the spec FROM the HTML, node-by-node** — a mapping table, one row per element:

   | HTML node | Exact CSS (height, grid, gaps, border, font-px, color→token) | Verbatim text | React component (reuse) |
   |---|---|---|---|

   Every spec row traces to a line in the HTML. **No English design-prose** — it's a literal translation table.
3. **Hard values become named constants** — every dimension the HTML fixes (`height:70px`, grid ratios, gaps)
   is a constant in the spec and the code. **No `flex` auto-expand or wrapping where the HTML fixes a size.**
4. **Strings are copied verbatim** — "Inspect Ends", "Inspect Res", "Pending", "Clear", "Walkthru" — lifted from
   the nodes, never paraphrased or "improved."
5. **Reuse existing components** — the table maps each node to the real component (`StatCard`, the pills, `IpadModal`),
   never a hand-rolled lookalike (see the `REUSE_NOT_RECREATE` law).
6. **Verify by overlay** — screenshot the built React at the wireframe's viewport, place it beside the HTML render,
   and diff **every** element: outline present? label text exact? height matched? position right? "Matches the
   wireframe," not "matches my memory." Only then is it done.

## The lesson — the mode-bands divergence (2026-07-03)

Zach signed off on a pixel-exact HTML wireframe for the lead-detail **TC Dashboard** and **Referral Dashboard**
bands. The shipped React looked nothing like it: no card outlines, full words where abbreviations were locked
("Inspection Ends" instead of "Inspect Ends"), an invented Buyer/Seller toggle in the wrong corner, a band that
auto-expanded and wrapped instead of holding a fixed height.

**The mechanism** (this is the part that matters):

1. After sign-off, the design was re-described in a **PRD + plan in English prose** ("hero + 5 TC fields as inline
   blocks", "6 KPIs in navy"). That prose was a **lossy compression** of the HTML — it silently dropped the
   outlined cards, the exact grid, the switch structure, the abbreviation strings, and the pixel heights.
2. The React was then built **from the prose and memory — the HTML file was never re-opened.** Everything that
   lived only in the pixels got **reconstructed from a vague mental model. That reconstruction is the hand-rolling.**
3. Two verbal edits Zach made were never written back into the HTML, so there was no spec to port them from, and
   they were implemented wrong from memory.

In HTML, Claude was editing the actual pixels Zach was looking at — one source, WYSIWYG. In React, it was working
from a text abstraction one step removed, so drift entered at every gap between the prose and the pixels.

<Warning>
**A signed-off wireframe is a spec, not a suggestion.** Building UI from a prose description of an approved
wireframe — instead of from the wireframe itself — is the root cause. The prose is a *pointer* to the HTML, never
a replacement for it.
</Warning>

## The rule

> **The wireframe is the truth. Edits land in the HTML first and are re-confirmed; the spec is derived from the
> locked HTML node-by-node; the build is a verbatim port of that spec; done means an overlay diff against the
> wireframe, not against memory.**

Enforced by `.claude/rules/UI_ITERATION_HTML_FIRST.md` (constitutional). This is the design-artifact counterpart to
`WORKFLOW_CANVAS_IS_TRUTH` — same principle, applied to UI: what runs equals what was drawn, nothing reconstructed.
